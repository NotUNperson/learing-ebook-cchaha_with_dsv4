# 04-01 SSL/TLS 基础

## 本节你会学到什么

- 理解 HTTP 明文传输面临的三大安全威胁
- 区分对称加密和非对称加密各自的作用
- 了解证书、CA 和信任链的运作机制
- 理解 TLS 握手过程为什么被称为"握手"

## 为什么 HTTP 不够安全

你寄明信片给朋友，邮递员、分拣员、任何一个经手的人都可以翻过来看上面写了什么。HTTP 协议也是如此——数据在网络上以明文传输，任何在传输路径上的人都能看到内容。

具体来说，HTTP 面临三大威胁：

**窃听**——就像有人在电话线上搭了一条分机，偷听你和朋友的对话。你在咖啡厅连公共 WiFi 时，同一个网络里的其他人可以抓包看到你访问的所有 HTTP 内容，包括密码、银行卡号。

**篡改**——运营商在网页里插广告，就是典型的篡改。你在浏览器里看到的内容，在传输途中被人修改过。好比有人截获你寄的信，修改了里面的金额数字再寄出去。

**冒充**——你以为是银行官网，其实是个山寨网站。没有验证机制的情况下，你无法确认跟你通信的是不是本人。这就像接到一个电话，对方自称是快递员，但你没办法确认他到底是不是。

HTTPS 就是来解决这三个问题的：加密防止窃听，完整性校验防止篡改，证书验证防止冒充。

## 对称加密：你和朋友共用一把钥匙

想象你和朋友约定用一个密码本来通信——写的时候用密码本把每个字转换成暗号，读的时候用同一本密码本还原。这本密码本就是"密钥"。加密和解密用同一把密钥，这就是**对称加密**。

对称加密的特点是快、效率高。但有一个致命问题：你怎么把这本密码本安全地交给朋友？如果你们本来就不能安全地通信，那怎么传递密钥？这叫"密钥分发问题"。

## 非对称加密：一个信箱的两个用途

非对称加密解决了这个问题。想象你有一个特殊信箱：信箱上有一条缝，任何人都可以把信塞进去（公钥加密），但只有你有开信箱的钥匙（私钥解密）。

这个信箱有两个关键特性：
- 用公钥加密的数据，只有对应的私钥能解开
- 用私钥加密的数据，公钥能解开（这就是"数字签名"的原理）

非对称加密安全性高，但速度慢，比对称加密慢几百倍。所以实际使用中，HTTPS 把两者结合起来：用非对称加密来安全地传递一把"临时对称密钥"，然后后续的数据传输全部改用对称加密。这样既解决了密钥分发问题，又保证了传输效率。

## 证书：你是谁？证明给我看

有了非对称加密，你拿到了服务器的公钥。但你怎么知道这个公钥真的是 `taobao.com` 的，而不是某个攻击者伪造的？

这就需要一个大家都信任的第三方来作证。这个机制就是**证书和 CA（Certificate Authority，证书颁发机构）**。

想象你要办护照：你自己声称"我是张三，这是我的照片和签名"，这没用。你必须去公安局，公安局核实你的身份后，在你的申请表上盖章。公安局就是 CA，盖章后的申请表就是证书。

换个更形象的类比：古代写信时用蜡封（火漆印章）。你收到一封信，上面有蜡封印记。如果你认得这个印记属于某人，就说明这封信确实是他寄的，而且中途没有被拆过——因为一旦拆开，蜡封就会碎掉。

数字证书的工作原理类似：
1. CA 用自己的私钥在证书上"盖章"（数字签名）
2. 浏览器用 CA 的公钥验证这个签名——就像检查蜡封是否完整、印记是否匹配
3. 如果签名验证通过，说明这个证书确实是 CA 签发的，网站身份可信

浏览器和操作系统内置了一份"受信任的 CA 名单"——就像你只认可公安局、民政局这些官方机构盖章的证明。

## 信任链：层层盖章的证明

有时候 CA 不会直接用根证书给网站签名，而是通过中间 CA。这就形成了**信任链**：

```
根 CA（浏览器信任列表里自带的）
  └── 中间 CA（根 CA 给中间 CA 签发的证书）
        └── 网站证书（中间 CA 给网站签发的证书）
```

想象你去开一个工作证明：你拿着部门主管的签字去找 HR，HR 盖了章。银行之所以相信你的证明，是因为银行的"受信任名单"里有这家公司公章的样子。它一层层验证：公司的公章是真的 -> HR 的签字有权用这个章 -> 所以你的证明是真的。

## TLS 握手：见面先确认身份

TLS 握手过程就像一个正式的商务会面：

1. **打招呼（Client Hello）**：浏览器说"你好，我支持这些加密算法，这是我的随机数 A"
2. **回应（Server Hello）**：服务器说"你好，我选这个算法，这是我的证书，这是我的随机数 B"
3. **验证证书**：浏览器检查证书——就像你检查对方名片，确认他真的来自他声称的公司
4. **生成会话密钥**：浏览器生成一个随机数 C，用服务器公钥加密后发送过去。两边用 A + B + C 共同计算出同一把"对称密钥"
5. **握手完成**：双方说"好，我们后面就用这把密钥来通信了"

至此，后续的所有数据都改用对称加密传输，既安全又高效。

整个过程只有几百毫秒，但确保了通信双方的身份和后续数据的安全。

## 动手试试

打开浏览器，随便访问一个 HTTPS 网站（比如 `https://www.baidu.com`），点击地址栏左边的小锁图标，查看证书详情。找到以下信息：

- 证书是哪个 CA 签发的？
- 证书的有效期是多久？
- 点击"证书路径"（或"信任链"），能看到几层证书？

然后打开终端，用 curl 的 verbose 模式观察 TLS 握手过程：

```bash
curl -v https://www.baidu.com 2>&1 | head -30
```

注意看输出中的 `SSL connection using` 和 `Server certificate` 相关行。

## 本节小结

HTTPS 用非对称加密传递对称密钥、用对称加密传输数据、用证书体系验证身份，三者共同作用实现了安全的网络通信。
