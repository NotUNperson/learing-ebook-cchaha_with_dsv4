## 知识脉络回顾

本章我们从"为什么 HTTP 不安全"出发，逐步搭建起了 HTTPS 的完整知识框架。下面这个链条串起了所有要点：

```
威胁认知 --> 加密方案 --> 身份验证 --> 证书生成 --> Nginx 配置 --> 强制跳转
   |            |            |            |              |             |
窃听/篡改/    对称+非对称   证书+CA+     openssl     ssl_certificate   301 + HSTS
冒充          组合使用      信任链        三步走      + ssl_protocols
```

### 对称加密 vs 非对称加密

| 维度 | 对称加密 | 非对称加密 |
|------|---------|-----------|
| 类比 | 两个人共用一把钥匙开同一把锁 | 信箱投递口（公钥）+ 开箱钥匙（私钥） |
| 速度 | 快（硬件加速） | 慢（数学运算密集） |
| 密钥管理 | 分发困难（怎么安全地交换钥匙？） | 只需保护私钥，公钥可以公开 |
| 在 HTTPS 中 | 加密大量传输数据 | 握手阶段传递对称密钥 |

HTTPS 的精髓在于**取长补短**：用非对称加密的安全性和对称加密的高效率，组合出既安全又实用的方案。这就像你要给朋友送一份机密文件——你用密码箱（对称加密）装着文件，但密码箱的密码写在便条上，用你朋友的专属信箱（非对称加密）寄给他。这样文件传得快，密码又不会被偷看。

### 证书验证链

```
浏览器内置信任（根 CA 公钥）
        |
        | 验证签名
        v
中间 CA 证书（根 CA 签发的）
        |
        | 验证签名
        v
网站证书（中间 CA 签发的）
```

每一层的逻辑都一样：上一级用自己的私钥给下一级的证书签名，验证者用上一级的公钥验证签名。这个链条的根部——根 CA——是浏览器/操作系统出厂时就内置的，所以整个信任体系得以成立。

### Nginx HTTPS 配置关键指令速查

| 指令 | 作用 | 关键点 |
|------|------|--------|
| `listen 443 ssl` | 在 443 端口启用 SSL | 不加 ssl 参数就是普通 HTTP |
| `ssl_certificate` | 指定证书文件路径 | 这个是公开的 |
| `ssl_certificate_key` | 指定私钥文件路径 | 这个要保密，权限 600 |
| `ssl_protocols` | 限制 TLS 协议版本 | 推荐 TLSv1.2 TLSv1.3 |
| `ssl_ciphers` | 指定加密套件 | 选带 ECDHE 的支持前向保密 |
| `ssl_session_cache` | 缓存会话参数 | 减少重复握手 |
| `return 301 https://...` | HTTP 跳转 HTTPS | 放在 80 端口的 server 块 |
| `add_header Strict-Transport-Security` | HSTS 头 | 让浏览器以后直接走 HTTPS |

## 常见问题排查

**问题一：浏览器显示"SSL 证书无效"**

这个错误说明证书没有被浏览器信任。如果是你自己的自签名证书，这是正常的。如果是买的证书还报这个错，检查：证书是否过期？域名是否匹配？是否需要中间证书（有的 CA 签发的证书需要把中间证书也配上）？

**问题二：curl 返回 "SSL routines:ssl3_read_bytes:tlsv1 alert protocol version"**

表示客户端和服务器之间没有共同的 TLS 协议版本。可能的原因：服务器只开了 TLSv1.3，但客户端（旧版 curl）不支持。临时解决办法：在 `ssl_protocols` 中加上 TLSv1.2。

**问题三：curl 返回 "unable to get local issuer certificate"**

curl 在尝试验证服务器证书的签发者，但找不到签发者的证书。对于自签名证书，用 `-k` 参数跳过验证。

**问题四：HTTPS 站点被反向代理后，后端收到的是 HTTP 请求**

Nginx 作为反向代理时，默认用 HTTP 连接后端。后端应用需要知道原始请求是 HTTPS 的。解决方法：设置 `proxy_set_header X-Forwarded-Proto https;`。

## 动手试试

完成下面的综合练习，把本章所有知识点串起来：

1. **生成证书**：进入 `examples/04-02/`，运行 `bash generate-cert.sh`，生成自签名证书
2. **配置 Nginx**：使用 `examples/04-03/nginx.conf`，或者手动写一个 HTTPS server 块，配好证书路径
3. **验证 HTTPS**：启动 Nginx，用 `curl -k https://localhost` 确认 HTTPS 正常工作；用 `openssl s_client -connect localhost:443` 查看 TLS 握手细节
4. **配置跳转**：用 `examples/04-04/nginx.conf`，添加 80 端口的 301 跳转 block 和 HSTS 头
5. **测试跳转**：
   ```bash
   # 测试 301 跳转
   curl -I http://localhost
   # 跟随跳转并查看最终结果
   curl -L -k http://localhost
   # 查看 HSTS 头
   curl -I -k https://localhost 2>/dev/null | grep -i strict
   ```
6. 在浏览器中访问，观察地址栏的锁标志（虽然自签名证书会显示"不安全"，但你能看到 HTTPS 确实在工作）

## 本节小结

HTTPS = 非对称加密交换密钥 + 对称加密传输数据 + 证书体系验证身份 + HSTS/301 强制加密；整个体系解决的是嗅探、篡改和冒充三大安全威胁。
