# 05-01 Gzip 压缩

## 本节你会学到什么

- 理解 Gzip 压缩的原理和为什么文本文件能压缩 70% 以上
- 掌握 Nginx 中 Gzip 的配置指令及其含义
- 学会在压缩率和 CPU 开销之间做权衡
- 能够验证 Gzip 是否真正生效

## 真空压缩袋的启示

每次换季收拾衣服，你一定会用到真空压缩袋。蓬松的羽绒服塞进去，抽掉空气后立刻变成薄薄一片，体积缩小了 70%。但毛衣本身没有变化——你需要穿的时候，打开袋子，它还会蓬起来。

Gzip 压缩就是这个道理。网页文件（HTML、CSS、JS）里充满了重复的标签、属性和空格——就像羽绒服里的空气。Gzip 做的就是"把空气抽走"的工作：找出重复模式，用更短的编码替代。浏览器收到后再"打开袋子"，把文件解压回原样。整个过程对用户完全透明，页面看起来没有任何区别，但传输体积小了 70% 以上。

但要注意，有些东西不能压缩。已经烤干的压缩饼干（JPG、PNG、MP4 等已经高度压缩的文件），你没法再抽出什么来，硬要放进真空袋只会浪费时间。

## Gzip 压缩是怎么做到的

Gzip 背后有两个核心算法，用生活场景来理解：

**LZ77 算法——用"同上"来节省时间。** 想象你在课堂上传纸条，上面写着"nginx nginx nginx 很好用"。你可以写成"nginx(同上)(同上) 很好用"——用三个字的引用代替了两次重复的五个字母。LZ77 做的就是这件事：在数据流里找重复出现的字节串，用"往回看 N 个字节，复制 M 个字节"这样的指令来替代。

**哈夫曼编码——给高频词分配短代码。** 想象你在超市收银台，最常用的"散装称重"键最大最显眼，不常用的"特殊商品"键在角落里。同样，在 HTML 中，`<`、`>`、空格这些字符出现频率极高，哈夫曼编码给它们分配最短的二进制编码；不常用的字符分配长编码。高频字符用短码，整体长度自然就缩短了。

## Nginx Gzip 配置指令详解

Gzip 在 Nginx 中的配置项不少，但核心的就几个。下面逐一说明：

### gzip on —— 总开关

```nginx
gzip on;
```

这就像压缩袋的抽气口——你不打开它，后面再怎么操作都没用。默认是 `off`，所以第一步必须是 `gzip on`。

### gzip_types —— 该压缩哪些文件

```nginx
gzip_types text/css application/javascript text/xml application/json;
```

这就像你只往压缩袋里装衣服（文本文件），不会把铁块（图片、视频）也塞进去。Nginx 默认只压缩 `text/html`，其他的 MIME 类型需要你手动指定。

常见的需要压缩的类型：
- `text/css` —— 样式表
- `application/javascript` / `text/javascript` —— JS 文件
- `application/json` —— API 返回的 JSON 数据
- `text/xml` / `application/xml` —— XML 数据
- `image/svg+xml` —— SVG 矢量图（本质是 XML，可以被压缩）

不需要压缩的：`image/jpeg`、`image/png`、`image/gif`、`video/mp4`、`application/pdf` 等已经压缩过的格式。

### gzip_min_length —— 太小的文件不值得压缩

```nginx
gzip_min_length 1000;  # 单位：字节，只压缩大于 1KB 的文件
```

压缩小文件可能出现"越压越大"的情况，因为压缩算法本身也要附带一个很小的头部信息。就像你用真空压缩袋装一双袜子——袋子的封口条都比袜子重，不值得。一般设置为 1000（1KB）到 2048（2KB）就足够了。

### gzip_comp_level —— 压缩有多猛

```nginx
gzip_comp_level 6;  # 级别 1-9，默认 1
```

级别 1 = 压缩最快但压缩率最低，级别 9 = 压缩最狠但最慢。对于大多数网站，级别 6 是黄金平衡点——压缩率接近最高（约为级别 9 的 95%），但 CPU 消耗只有级别 9 的约一半。

用快递打包来类比：级别 1 是随手揉一揉塞袋子里，快但占空间；级别 9 是精细折叠、真空压缩、防震泡沫全上，慢但节省空间；级别 6 是认真折叠但不追求极致，大部分人都会选这个。

### gzip_vary —— 告诉缓存该怎么做

```nginx
gzip_vary on;
```

这个指令让 Nginx 在响应头里自动加上 `Vary: Accept-Encoding`。意思是告诉中间的缓存服务器（CDN、代理）:"同一个 URL 可能有压缩版和非压缩版，缓存的时候按客户端的 `Accept-Encoding` 请求头来区分。"

用餐厅来类比：同一个菜（同一个 URL），有人要辣的有人要清淡的（压缩 vs 不压缩）。`Vary` 头就像给服务员说"这个客人要辣的，在菜单上标注一下"，下次同一个客人再来，不用重新问他口味。

### gzip_proxied —— 代理请求也压缩

```nginx
gzip_proxied any;
```

当 Nginx 作为反向代理时，后端服务器返回的内容是否也要压缩？`any` 表示不管什么情况都压缩。如果你用的是 Nginx 直接处理静态文件，这条可以不配。但如果你是反向代理场景（Nginx 前面接客户端，后面接后端），建议加上。

## 完整配置

```nginx
gzip on;
gzip_types text/plain text/css application/json application/javascript text/xml application/xml text/javascript;
gzip_min_length 1000;
gzip_comp_level 6;
gzip_vary on;
gzip_http_version 1.1;
gzip_proxied any;
```

把这个配置放在 `http` 块中会全局生效，放在 `server` 块中只对该虚拟主机生效，放在 `location` 块中只对该路径生效。建议放在 `http` 块中做全局配置，个别需要特殊处理的 location 再覆盖。

## 如何验证压缩是否生效

**用浏览器的开发者工具**（最直观）：
1. 按 F12 打开开发者工具
2. 切换到 Network（网络）标签
3. 刷新页面
4. 点击任意一个 HTML/CSS/JS 请求
5. 查看 Response Headers（响应头），找 `Content-Encoding: gzip`
6. 同时对比 Size 列的两个数字——一个是传输大小，一个是实际大小，传输大小应该明显更小

**用 curl**：

```bash
# 发送带 Accept-Encoding: gzip 头的请求，观察响应头
curl -H "Accept-Encoding: gzip" -I http://localhost/index.html | grep -i content-encoding

# 看实际压缩效果（-s 静默模式，-w 显示传输大小）
curl -s -H "Accept-Encoding: gzip" -o /dev/null -w "传输大小: %{size_download} bytes\n" http://localhost/index.html
```

## 动手试试

1. 用 `examples/05-01/nginx.conf` 配置启动 Nginx（或者在你的配置中添加上面的 Gzip 配置）
2. 访问 `examples/05-01/site/index.html`（这是一个大约 15KB 的测试页面，包含大量重复文本）
3. 打开浏览器开发者工具（F12），在 Network 标签中查看 `index.html` 请求
4. 对比传输大小和实际大小，计算压缩率：(实际大小 - 传输大小) / 实际大小
5. 临时把 `gzip_comp_level` 改为 1 和 9，观察不同压缩级别下的传输大小和页面响应时间
6. 试试把 `gzip_min_length` 改成一个很大的值（比如 100000），刷新页面，看看 Gzip 是否还会生效——是不是因为文件没达到最小长度？

## 本节小结

Gzip 压缩像真空压缩袋一样把网页文件中的"空气"（重复模式）抽走，Nginx 用寥寥几行配置就能让传输体积减少 70% 以上，是性价比最高的性能优化手段。
