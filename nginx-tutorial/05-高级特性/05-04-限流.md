# 05-04 限流

## 本节你会学到什么

- 理解为什么需要限流（防止滥用和攻击）
- 用 limit_req_zone/limit_req 控制请求频率（漏桶算法）
- 用 limit_conn_zone/limit_conn 控制并发连接数
- 理解 burst 和 nodelay 参数的含义和配合

## 景区限流的智慧

每年国庆节，热门景区都会限流。景区不是无限制地往里放人——5 万人进园之后，大门关闭，后来的人排队等候。里面出来一批，外面放进去一批。这么做的原因很简单：景区容量有限，放太多人进去，所有人都别想好好玩，而且有踩踏风险。

你的服务器就是那个景区。带宽、CPU、内存都是有限的。如果有人写了个脚本疯狂请求你的网站，或者某个 API 突然被高频调用，服务器就像国庆的故宫一样——直接瘫掉。

限流的目的不是为难正常用户，而是保护服务器，确保大多数正常用户能正常使用。这就像景区限流不是为了不让游客玩，恰恰是为了让进去的游客有良好的体验。

## 漏桶算法：水龙头和桶

Nginx 的请求频率限制基于**漏桶算法（Leaky Bucket）**。想象一个场景：

水龙头（请求）往桶里放水（进来请求），桶底有个小孔（处理能力），水以固定的速度流出去（处理请求）。如果水龙头开太大，水（请求）就会溢出（被拒绝）。

这个算法的核心特征是：不管水龙头开得多猛（请求多密集），桶底流出去的水永远是匀速的。这就是限流的本质——把突发的高速请求"平滑"成匀速处理。

关键参数说明：

- **rate**：桶底小孔的大小，即每秒能处理多少个请求
- **burst**：桶的容量（在桶底小孔之上的额外缓冲区），即允许的突发请求数
- **nodelay**：有 burst 时，突增的请求是立即处理还是排队等待

## limit_req_zone 和 limit_req：请求频率限制

配置分为两步：先定义限流区域（zone），再在具体 location 中应用。

**第一步：定义限流区域（放在 http 块中）**

```nginx
# 语法: limit_req_zone key zone=名称:内存大小 rate=速率;
limit_req_zone $binary_remote_addr zone=req_limit:10m rate=10r/s;
```

拆解这条配置：
- `$binary_remote_addr` —— 限流的"key"，这里是按客户端 IP 来限流。写成 `$binary_remote_addr` 而不是 `$remote_addr` 是因为前者是二进制格式，只占 4 字节（IPv4），而后者是字符串占 7-15 字节，省内存。
- `zone=req_limit:10m` —— 分配 10MB 的共享内存来存储每个 IP 的请求计数。10MB 大约可以存储 16 万个 IP 的记录。
- `rate=10r/s` —— 每秒允许 10 个请求。`r/m` 表示每分钟。

**第二步：在 location 中应用限流**

```nginx
location /api/ {
    limit_req zone=req_limit burst=20 nodelay;
    # ...
}
```

- `zone=req_limit` —— 使用前面定义好的限流规则
- `burst=20` —— 允许在速率限制之外额外排队 20 个请求
- `nodelay` —— 突增请求立即处理（而不是按 rate 的速率排队等待）

## burst 和 nodelay 的故事

理解 burst 和 nodelay 最好的方式是用排队吃饭类比：

假设一家网红餐厅，店里只有 10 个座位（rate=10），每桌客人吃饭时间不定，但平均每 6 分钟翻台。

**场景一：burst=0（无排队区）**
门口写着"满座请稍后"。人满了就硬拦在门外。第十一个人只能在门口干等，等有人出来了才能进门。如果有个人每隔 3 秒就来推门问"有空位没"——他会被直接赶走。用户体验极差，就像限速 10r/s 且没有 burst。

**场景二：burst=20（有排队区，无 nodelay）**
门口有 20 个取号排队位。店里满 10 个人，后来的人取号等位。服务生按固定的节奏（rate=10r/s）叫号。你的号可能要等好几轮，但至少你知道排队是在进行的。

**场景三：burst=20 + nodelay（有排队区，立即可进）**
门口的 20 个位子直接可以坐——先到先得。第 11 到第 30 个人直接进去，不需要等待。但第 31 个人就会被挡在外面，直到前面有空位。这就是 `burst=20 nodelay` 的效果——允许一定量的突发流量立即通过，超出的才被限流。

**选择建议**：
- 对普通用户访问的页面：`burst=20 nodelay`（允许突发，体验好）
- 对登录接口：`burst=3 nodelay`（小 burst，防止暴力破解但允许正常的错误重试）
- 对 API 调用：看情况，对机器间调用可以不加 nodelay（让它们排队更公平）

## limit_conn_zone 和 limit_conn：并发连接数限制

请求频率限制控制的是"单位时间内来多少次请求"，而并发连接数限制控制的是"同一时间占着多少个连接"。

```nginx
# 定义（http 块）
limit_conn_zone $binary_remote_addr zone=conn_per_ip:10m;

# 应用（location 块）
limit_conn conn_per_ip 10;    # 每个 IP 最多 10 个并发连接
```

两者的区别用一个电话客服中心的例子来说明。假设你是客服中心的经理：
- **请求频率限制**：限制每个人一小时最多打几次电话。防止有人反复骚扰。
- **并发连接数限制**：限制同时有多少条电话线被占用。一个人只能占用一条线，不能同时用 20 部电话打进来。

在网站场景中，并发连接数限制特别适用于下载站——防止下载工具开几百个连接同时下载，把带宽占满。

## 带宽限制：给每个连接限速

除了限制连接数，还可以限制每个连接的下载速度：

```nginx
location /download/ {
    limit_rate 500k;             # 每个连接限速 500KB/s
    limit_rate_after 5m;         # 前 5MB 不限速，超过后才限速
}
```

这就像商场的扶梯限速——不是不让你坐，但不能跑太快。`limit_rate_after` 则像"前 10 层免费通行的扶梯，之后才开始限速"，给小文件的传输留了快速通道。

## 完整配置示例

```nginx
http {
    # 定义限流区域
    limit_req_zone  $binary_remote_addr zone=req_limit:10m rate=10r/s;
    limit_req_zone  $binary_remote_addr zone=api_limit:10m rate=5r/s;
    limit_conn_zone $binary_remote_addr zone=conn_per_ip:10m;

    server {
        listen 80;

        location / {
            limit_req zone=req_limit burst=20 nodelay;
            limit_conn conn_per_ip 10;
        }

        location /api/ {
            limit_req zone=api_limit burst=10 nodelay;
        }

        location /login/ {
            # 登录接口：严格限流，防止暴力破解
            limit_req zone=req_limit burst=3 nodelay;
        }
    }
}
```

## 动手试试

1. 使用 `examples/05-04/nginx.conf` 配置启动 Nginx
2. 用 Apache Bench（ab）或类似工具模拟并发请求：

```bash
# 安装 ab（如果没有的话）
apt-get install apache2-utils

# 模拟 50 个请求，并发 10 个
ab -n 50 -c 10 http://localhost/

# 观察输出中的 "Non-2xx responses"——失败的请求数量
# 你会看到有些请求返回了 503（被限流了）
```

3. 把 `burst` 改大一点（比如 50），再用 ab 测试，观察 503 是否减少
4. 把 `rate` 改成 `1r/s`，在浏览器里快速刷新页面，观察什么时候开始出现 503
5. 临时注释掉 `limit_req` 行，在 `/download/` 位置放一个大文件，用两三个浏览器标签页同时下载，看看把 `limit_conn` 设置成 1 或 2 后对并发下载的影响

## 本节小结

请求频率限流是景区限流（漏桶匀速放行 + 排队区缓冲），连接数限流是电话线占线管理，两者搭配保护服务器不被突发流量冲垮，同时保证正常用户体验不受影响。
