# 05-02 浏览器缓存控制

## 本节你会学到什么

- 理解 expires 指令如何设置缓存的"保质期"
- 区分强制缓存（Cache-Control）和协商缓存（ETag/Last-Modified）
- 掌握不同文件类型的缓存策略
- 学会在浏览器开发者工具中查看缓存行为

## 图书馆借书的缓存策略

想象你去图书馆借书。管理员告诉你："这本书你可以借 7 天，7 天内不需要再来问我能不能看，直接看就行。"第 8 天你再去借同一本书时，管理员说："我得查一下库里还有没有这本书，最近有没有人借过。"

浏览器缓存和这个流程一模一样：

- **借期 7 天** = 缓存的过期时间（Expires/max-age）。这 7 天内，浏览器直接从本地拿，根本不问服务器。
- **第 8 天去确认** = 协商缓存。浏览器拿着"上次看到的版本信息"去问服务器："这本书有没有出新版本？"服务器说"没有"（304 Not Modified），浏览器继续看本地缓存；服务器说"有"（200 OK），浏览器下载新版本。

这套机制让用户第二次访问你的网站时，大部分静态资源（CSS、JS、图片）根本不需要重新下载，页面秒开。

## 咖啡馆菜单：强制缓存 vs 协商缓存

你经常去一家咖啡馆。你对菜单已经很熟了，但你不会每次都问服务员"菜单有没有更新"——你对每个菜品有不同的处理方式：

- **固定菜品（摩卡、拿铁、美式）**——你知道这些永不会变，从不问，直接点（强制缓存，长期有效）
- **季节限定**——你会留意门口海报，问一句"这个季节有没有新品"（协商缓存，每次都验证一下）
- **今日特价**——你每次都会问服务员"今天有什么特价"（不缓存，每次都重新获取）

网站的资源也是同样的逻辑：
- **CSS/JS/图片/字体** = 固定菜品。不常变，适合强制缓存，设一个很长的过期时间
- **HTML 首页** = 季节限定。可能经常更新，用协商缓存，每次问一下服务器有没有变化
- **API 返回的用户数据** = 今日特价。每次访问都可能不一样，不缓存

## expires 指令：设置保质期

```nginx
# 静态图片缓存 30 天
location ~* \.(jpg|jpeg|png|gif|ico)$ {
    expires 30d;
}

# JS 和 CSS 缓存 7 天
location ~* \.(css|js)$ {
    expires 7d;
}
```

`expires` 指令会自动在响应头中生成 `Expires`（HTTP/1.0）和 `Cache-Control: max-age`（HTTP/1.1）两个头。它的单位很灵活：`30d`（天）、`24h`（小时）、`60m`（分钟）、`expires max`（永不过期，相当于 2037 年）。

## Cache-Control：更精细的控制

`expires` 很简单，但配置不够灵活。对于需要更精细控制的场景，直接用 `add_header` 设置 `Cache-Control` 头：

```nginx
# 强缓存：告诉浏览器"这个文件 30 天内不用来问我"
location ~* \.(jpg|png|gif)$ {
    expires 30d;
    add_header Cache-Control "public";
}

# 协商缓存：告诉浏览器"用之前先来问我有没有更新"
location ~* \.html$ {
    add_header Cache-Control "no-cache";
}

# 不缓存：每次都要重新获取
location /api/ {
    add_header Cache-Control "no-store";
}
```

Cache-Control 的几个关键值：

| 值 | 含义 | 类比 |
|-----|------|------|
| `public` | 任何缓存服务器都可以缓存 | 这本书任何人都可以借 |
| `private` | 只有浏览器可以缓存，中间代理不能 | 这本书只能本人借阅 |
| `max-age=3600` | 3600 秒内直接使用缓存 | 这本书可以借 1 小时，1 小时内不用还 |
| `no-cache` | 可以缓存，但每次使用前必须验证 | 每次看书前先问管理员有没有新版本 |
| `no-store` | 完全不缓存 | 这本书不借出，只能在馆内查阅 |
| `immutable` | 内容永不改变，连验证都不需要 | 这本字典，内容永远不会变 |

这里有一个常见的认知错位：`no-cache` 不是"不缓存"，而是"可以缓存，但每次用前必须验证"。真正的不缓存是 `no-store`。这个命名确实容易让人误解，就像"无菌"不是说没有细菌，而是细菌被灭活了。

## ETag 和 Last-Modified：协商缓存的凭据

当浏览器拿着"上一个版本"的信息来问服务器时，服务器需要一种方式来判断"你有没有最新版本"。两种方式：

**Last-Modified** —— 文件最后修改时间。

服务器说："这个 HTML 文件的最后修改时间是 2026-05-20 10:00，你手里的版本是什么时间的？"浏览器把自己的时间发给服务器，服务器比较一下，如果一致就返回 304。

但这个方案有时间精度问题——如果文件在 1 秒内修改两次，Last-Modified 分辨不出来。而且某些文件虽然内容没变但修改时间变了（比如用 rsync 同步时），也会导致不必要的重新下载。

**ETag** —— 文件内容的唯一指纹。

服务器说："这个文件的指纹是 `abc123`，你手里的指纹是什么？"浏览器把指纹发过去，服务器对比，一致就 304。

ETag 就像文件内容的 DNA——内容不变，指纹永远不变。它完全基于内容，不受修改时间影响，是最可靠的验证方式。Nginx 默认自动生成 ETag（`etag on;`），你不需要额外配置。

完整的一次协商缓存流程：

```
浏览器请求 /style.css
  --> 发送请求头: If-None-Match: "abc123"
      服务器检查: 当前 ETag 还是 "abc123"
  <-- 返回 304 Not Modified（响应 body 为空）
      浏览器使用本地缓存中的 style.css
```

对比两种缓存策略的流量消耗——304 响应通常只有几百字节（仅响应头），而重新下载可能几 KB 到几百 KB。一次协商缓存省下的流量也许不多，但成千上万的用户每天访问你的网站，累积下来的带宽节省非常可观。

## 版本化文件名的终极策略

聪明的做法是给文件加上版本指纹——文件名包含内容哈希值：

```
style.a3f2b1c.css   # 内容变了，哈希变了，文件名变了
```

对于这种文件，可以设永久缓存：

```nginx
# 匹配文件名中包含 8 位以上十六进制哈希的文件
location ~* \.[a-f0-9]{8,}\.(js|css|png|jpg)$ {
    expires max;
    add_header Cache-Control "public, immutable";
}
```

`immutable` 是较新的指令，它告诉浏览器"这个文件的内容绝对永远不会变，验证都不用验证"。当用户刷新页面时，浏览器对于 `immutable` 的资源不会发送验证请求，直接使用本地缓存。这个指令对于带哈希的 Webpack/打包产出来说是最完美的。

这就像图书馆给每本书贴了唯一的流水号——书的内容改了，流水号就变了。你借的书流水号是 A001，只要这个流水号还在，内容就永远是那个版本。

## 不同文件类型的推荐缓存策略

| 文件类型 | 策略 | 配置 | 原因 |
|---------|------|------|------|
| HTML 页面 | 协商缓存 | `no-cache` | 更新频繁，但希望利用缓存验证 |
| CSS/JS（无版本号） | 短期强缓存 | `expires 7d` | 偶尔更新，设一周缓存 |
| CSS/JS（带 hash） | 永久缓存 | `expires max; immutable` | 内容变了文件名就变 |
| 图片（无版本号） | 长期强缓存 | `expires 30d` | 很少变化 |
| 图片（带 hash） | 永久缓存 | `expires max; immutable` | 跟 CSS/JS 同理 |
| 字体文件 | 永久缓存 | `expires 1y` | 几乎从不变 |
| API 响应 | 不缓存 | `no-store` | 每次都可能变化 |

## 动手试试

1. 使用 `examples/05-02/nginx.conf` 配置启动 Nginx
2. 在网站目录中放一个 HTML 页面和一张图片
3. 打开浏览器开发者工具（F12），切换到 Network 标签，**不要勾选"Disable cache"**
4. 第一次访问页面，观察所有请求的状态都是 200，传输大小等于实际大小
5. 刷新页面（F5），观察：
   - 对于图片请求，状态变成了 304 Not Modified（或 "from disk cache"），传输大小极小或为 0
   - 对于 HTML，因为配置了 `no-cache`，可以看到浏览器每次都会发送验证请求
6. 修改图片文件但不改文件名，刷新页面——如果缓存还在有效期内，你可能看到的仍然是旧图片（这就是为什么带 hash 的文件名很重要）
7. 打开 chrome://version 或浏览器的缓存目录，感受一下浏览器缓存的原理

## 本节小结

缓存策略就是给不同文件类型设定不同的"借阅期限"，静态资源长缓存、HTML 协商缓存、API 不缓存，配合文件名版本化实现缓存和更新的完美平衡。
