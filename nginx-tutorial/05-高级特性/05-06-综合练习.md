# 05-06 综合练习

## 本节你会学到什么

- 把 Gzip、缓存、访问控制、限流、自定义日志整合到一个配置中
- 理解各项配置之间的配合关系
- 能够排查多项配置同时生效时可能的冲突

## 本章知识串联

回想一下本章我们学到了什么：

```
你的 Nginx 服务器现在像一个功能齐全的智能大楼：

1. Gzip 压缩        = 快递站自动把包裹压缩打包，节省运输空间
2. 浏览器缓存        = 便利店在每栋楼里设分仓，常用品不用每次从总库调
3. 访问控制          = 大门门禁 + 电梯刷卡，未经授权进不来
4. 限流             = 扶梯限速 + 客流量管制，防止挤爆
5. 自定义日志        = 监控摄像头 + 进出门登记，随时回溯
```

每一块都可以单独使用，但真正的威力在于把它们组合起来。一个生产级别的 Nginx 配置通常同时使用上述所有特性。

## 设计思路：分类处理不同请求

整合配置的关键思路是——**按请求类型分而治之**。不同类型的请求有不同的需求：

| 请求类型 | Gzip | 缓存 | 访问控制 | 限流 | 日志 |
|---------|------|------|---------|------|------|
| 首页 HTML | 开启 | 协商缓存 | 无 | 适度 | JSON 格式 |
| CSS/JS | 开启 | 7 天强缓存 | 无 | 无 | 关闭 |
| 图片 | 略过（已压缩） | 30 天强缓存 | 无 | 无 | 关闭 |
| 管理后台 | 开启 | 不缓存 | 密码+IP | 适度 | JSON |
| API 接口 | 开启 | 不缓存 | 视情况 | 严格 | 详细格式 |
| 登录接口 | 开启 | 不缓存 | 视情况 | 最严格 | JSON |

## 完整配置解析

下面逐块分析 `examples/05-06/nginx.conf` 中的配置：

**http 块：全局配置。** Gzip 和限流 zone 定义在 http 块中，这样所有 server 和 location 都能共享这些配置，不需要每个 location 重复定义。限流 zone 使用 `shared` 共享内存意味着即使 Nginx 有多个 worker 进程，它们看到的限流计数器也是一致的。

```nginx
http {
    # Gzip 全局开启
    gzip on;
    gzip_types text/plain text/css application/json application/javascript ...;
    gzip_min_length 1000;
    gzip_comp_level 6;

    # 限流 zone 定义
    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=5r/s;
    limit_req_zone $binary_remote_addr zone=login_limit:10m rate=1r/s;
    limit_conn_zone $binary_remote_addr zone=conn_per_ip:10m;

    # 日志格式
    log_format json ...;
    log_format detailed ...;
}
```

**静态资源 location：压缩 + 缓存。** CSS、JS、图片、字体这四个 location 各司其职，设置不同的缓存期限。为什么要分开？因为实际使用中它们的更新频率不同——字体几乎万年不变（1 年缓存），图片偶尔更新（30 天），JS/CSS 经常更新（7 天）。分开管理让缓存策略更贴合实际。

**管理后台 location：双重认证 + satisfy any。** 内网 IP 用户直接访问（无需密码），外网用户需要输密码。这个"内外有别"的逻辑在实际工作场景中很实用——大多数公司的管理后台都是部署在内网的。

**API location：限流 + 独立日志。** API 是机器间调用，响应时间敏感。限流防止某个客户端失控刷爆后端，独立日志方便单独分析 API 性能。接口类请求通常不需要缓存（因为每次返回的数据都不同），所以不设缓存策略。

**登录 location：最严格限流。** `rate=1r/s burst=3` 意味着每秒最多 1 次登录尝试，可以突发 3 次。这个策略专门对抗暴力破解：攻击者用字典尝试几千个密码时，他们会在前 3 次尝试后立即被限速到 1 次/秒，而正常用户输错两三次密码是完全可以接受的。

## 一些需要注意的坑

**配置继承关系。** Nginx 中某些指令在 http -> server -> location 之间是继承的（如 `gzip_types`），某些是被替换的（如 `access_log`）。比如在 location 中写了 `access_log off;`，它会覆盖上层的 `access_log` 设置，而不是追加。最好在测试环境中先验证，不要想当然。

**限流粒度。** `limit_req_zone` 的 key 如果设成 `$binary_remote_addr`，按 IP 限流。但如果你前面有 CDN 或反向代理，所有请求的 IP 都变成了代理的 IP，限流就失效了——相当于景区检票口看到的都是同一个旅行团的领队，没法区分每个团员。这时候要用 `$http_x_forwarded_for` 或者 `$http_x_real_ip` 来获取真实的客户端 IP。

**Gzip + 缓存。** 浏览器缓存的是压缩后的内容。如果同一个 URL 被浏览器缓存了 Gzip 压缩版本，但服务器上后来改了 Gzip 配置（比如关了 Gzip 或改了压缩级别），已缓存的用户逻辑上不应该受影响——但如果缓存到期后用户重新请求，可能观测到行为的变化。这不是 bug，但理解这一点有助于排查问题。

**日志性能。** 日志写到磁盘会消耗 I/O。如果日志量很大，考虑：
- 静态资源关闭日志（`access_log off;`）
- 健康检查关闭日志
- 写日志到 `/dev/shm`（内存文件系统）再定期归档

## 动手试试

1. 把 `examples/05-06/nginx.conf` 作为 Nginx 的主配置文件
2. 启动 Nginx，用浏览器或 curl 访问不同的路径：

```bash
# 访问首页（Gzip + JSON 日志）
curl -H "Accept-Encoding: gzip" -I http://localhost/

# 访问管理后台（触发认证）
curl -I http://localhost/admin/

# 访问 API（触发限流，连续发送 20 个请求）
for i in $(seq 1 20); do
  curl -s -o /dev/null -w "请求 #$i: HTTP %{http_code}\n" http://localhost/api/test
done
```

3. 观察日志输出：

```bash
# 查看 JSON 日志
tail -3 /var/log/nginx/access.log

# 查看 API 的详细日志
tail -3 /var/log/nginx/api_access.log
```

4. 验证 Gzip：在 Network 标签中确认 `Content-Encoding: gzip` 响应头存在
5. 验证缓存：刷新页面两次，观察静态资源的传输大小是否变成 0（from cache）
6. 验证限流：连续快速请求 API，确认有请求返回 503
7. 验证访问控制：从未授权的 IP 访问 `/admin/`，确认返回 401 或 403

## 本节小结

生产级 Nginx 配置 = 静态资源（Gzip+长缓存+关闭日志）+ 管理后台（密码+IP白名单）+ API 接口（限流+独立日志），按请求类型分类处理，每类请求享受不同的安全和性能策略组合。
