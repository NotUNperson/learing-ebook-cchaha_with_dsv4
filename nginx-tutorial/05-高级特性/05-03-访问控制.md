# 05-03 访问控制

## 本节你会学到什么

- 用 allow 和 deny 指令实现 IP 黑白名单
- 用 auth_basic 实现 HTTP 基本认证（用户名密码登录）
- 理解 satisfy 指令如何组合 IP 控制和密码认证
- 掌握 htpasswd 的用法和密码安全注意事项

## 小区门禁卡的三种档次

想象你住在一个小区，进入小区的方式有好几种：

**第一种——不要门禁，大门敞开。** 任何人从大街上直接走进来。这相当于没有任何访问控制的网站，公网上谁都能访问。

**第二种——刷卡进门。** 小区门口有一道闸机，住户刷卡才能进。没卡的外人必须找保安登记。这相当于 IP 白名单——只有特定 IP 能访问，其他 IP 拒绝。

**第三种——刷脸进门。** 不仅要刷卡（IP 白名单），还要密码或人脸识别（HTTP 认证）。双重验证，安全性高但用起来麻烦。

Nginx 的访问控制正好对应这三种模式：你可以不做任何控制（全开放），可以只限制 IP（门禁卡），也可以 IP 加密码一起上（刷卡加刷脸），还可以灵活选择是"满足一个就行"还是"两个都必须满足"。

## IP 访问控制：allow 和 deny

这是最简单的访问控制方式——根据访问者的 IP 地址决定是否放行。

```nginx
location /admin/ {
    allow  192.168.1.0/24;   # 允许内网 192.168.1.x 网段
    allow  10.0.0.0/8;       # 允许 VPN 网段
    deny   all;              # 拒绝所有其他 IP
}
```

Nginx 按照**从上到下的顺序**逐条匹配，一旦匹配到就停止。所以规则的顺序很重要。

用航空安检通道来类比：VIP 通道和普通通道。进来一个人，先看是不是 VIP（allow 192.168.1.100），是，直接放行；不是，继续看是不是公司员工（allow 192.168.1.0/24），是，放行；都不是，到了最后一条（deny all），抱歉，请走普通通道——被拦截。

一个常见的错误是把顺序写反了：

```nginx
# 错误示例——永远拒绝所有人
deny  all;              # 这个先匹配到了
allow 192.168.1.0/24;   # 永远不会被执行
```

或者反过来：

```nginx
# 错误示例——所有人都能进
allow all;              # 这个先匹配到了
deny  192.168.1.100;    # 永远不会被执行
```

前缀 `/24` 的含义：IP 地址是一个 32 位二进制数，`/24` 表示"前 24 位是网络地址，后 8 位是主机地址"。所以 `192.168.1.0/24` 匹配的是 `192.168.1.1` 到 `192.168.1.254`。就像小区的"号"表示几号楼，`/24` 是"192.168.1. 这栋楼的住户编号 1 到 254"。

常用网段建议：
- 本地：`127.0.0.1`
- 家庭/办公局域网：`192.168.0.0/16`
- 企业内网：`10.0.0.0/8`
- Docker 默认网桥：`172.17.0.0/16`

## HTTP Basic 认证：用户名 + 密码

HTTP Basic 认证是最简单的用户认证方式——浏览器弹出一个小对话框，让你输入用户名和密码。

```nginx
location /admin/ {
    auth_basic            "Admin Area - Please Login";   # 提示文字（会显示在对话框中）
    auth_basic_user_file  /etc/nginx/htpasswd;            # 密码文件路径
}
```

启用了 `auth_basic` 后，Nginx 在收到请求时首先检查请求头中是否包含有效的 `Authorization` 头（Base64 编码的用户名和密码）。没有或无效就返回 401，浏览器收到 401 后自动弹出登录对话框。你可以把它想象成餐厅卫生间门口的锁——按对了密码才能进去，这个锁虽然不高级，但能拦住一般人。

**如何生成密码文件：**

```bash
# 安装 htpasswd 工具
# Debian/Ubuntu:
apt-get install apache2-utils

# CentOS/RHEL:
yum install httpd-tools

# 创建密码文件并添加第一个用户（-c 表示创建新文件）
htpasswd -c /etc/nginx/htpasswd admin

# 添加更多用户（不要加 -c，否则会覆盖已有文件）
htpasswd /etc/nginx/htpasswd editor

# 查看密码文件内容
cat /etc/nginx/htpasswd
# 输出类似: admin:$apr1$iRZfP3Kx$Qq8X5J2nY7rVtLmW3kHpB.
```

密码文件的格式是 `用户名:加密后的密码`。`$apr1$` 前缀表示使用 Apache 的 MD5 加密。

HTTP Basic 认证有一个重要的安全缺陷：用户名和密码在网络上传输时只是做了 Base64 编码，不是真正的加密。Base64 编码就像把"admin:123456"翻译成一种人人都能翻译回来的语言，任何能抓包的人都能解码。所以**HTTP Basic 认证必须配合 HTTPS 使用**，否则等于把你的密码写在了明信片上寄出去。

## satisfy：组合两种验证

你有 IP 白名单和密码认证两种方式，想让它们怎么配合？

```nginx
location /admin/ {
    allow  192.168.1.0/24;
    deny   all;
    auth_basic            "Admin";
    auth_basic_user_file  /etc/nginx/htpasswd;

    # satisfy any  -> IP白名单 或 密码，满足其一即可
    # satisfy all  -> IP白名单 且 密码，两个都要满足（默认）
    satisfy any;
}
```

用小区进出场景来理解：
- **satisfy any** = 小区有三个门：业主刷卡（IP 白名单）可以进，访客登记后输密码（auth_basic）也可以进。两个条件满足任意一个就行。就像你到了前台，要么刷门禁卡，要么登记身份证，二者选其一即可。
- **satisfy all** = 你不仅要刷卡（IP 白名单），还得输密码（auth_basic），两个都对才放行。适用于银行金库级别的安全需求。

## 实际场景推荐

| 场景 | 配置方案 | 理由 |
|------|---------|------|
| 公司内部管理系统 | IP 白名单 | 员工都在办公室，限制内网 IP 即可 |
| 博客后台/WordPress | HTTP Basic + HTTPS | 加了 HTTPS 后基本安全，且配置简单 |
| API 接口 | IP 白名单 | 一般是服务器间通信，不需要用户交互 |
| 金融后台 | IP 白名单 + auth_basic (satisfy all) | 双重保险，安全性最高 |
| 开发环境 | allow/deny（可选） | 选择性地限制，取决于团队需求 |

## 动手试试

1. 安装 htpasswd 工具，生成一个密码文件：

```bash
# 安装工具
apt-get install apache2-utils    # Debian/Ubuntu
# 或
yum install httpd-tools          # CentOS

# 创建用户
htpasswd -c /tmp/htpasswd admin
# 输入密码两次
```

2. 使用 `examples/05-03/nginx.conf` 配置（修改 `auth_basic_user_file` 为你的实际密码文件路径）
3. 创建一个 `/admin/` 目录，放一个测试页面
4. 启动 Nginx 后用浏览器访问 `/admin/`，观察是否弹出登录对话框
5. 输入用户名密码，确认能正常访问
6. 修改配置文件，把 `satisfy` 改成 `any`，再测试——如果你是本地访问（127.0.0.1 在 allow 列表中），会不会跳过密码？
7. 尝试用 curl 测试：

```bash
# 不带认证信息——应该返回 401
curl -I http://localhost/admin/

# 带认证信息——应该返回 200
curl -u admin:yourpassword http://localhost/admin/
```

## 本节小结

Nginx 的 IP 白名单是门禁卡、HTTP Basic 认证是门锁密码，satisfy 指令决定是"刷卡或输密码"还是"刷卡且输密码"；密码认证必须配 HTTPS 才安全。
