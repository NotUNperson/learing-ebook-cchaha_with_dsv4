# 05-05 日志自定义

## 本节你会学到什么

- 用 log_format 自定义 Nginx 的日志输出格式
- 掌握常用的日志变量及其含义
- 学会给不同的 location 配置不同的日志文件
- 了解 JSON 格式日志的优势和适用场景

## 行车记录仪的启示

开车的人大多装了行车记录仪。它的价值在哪里？不是用来欣赏沿途风景的，而是**出事之后回放定位问题**。你平时从来不看——但一旦刮蹭了，你会一帧一帧地检查："对方是从哪个方向过来的？当时车速多少？我有没有打转向灯？"

Nginx 的访问日志就是服务器的"行车记录仪"。正常运行时你可能半年不看一次日志。但一旦出了问题——有人报告网站打不开、某个 API 突然变慢、怀疑被人扫描攻击——日志就是你定位问题的唯一线索。

行车记录仪你可以选不同型号：有的只记录前方影像、有的还记录 GPS 坐标和车速、有的甚至能记录刹车和转向信号。Nginx 的日志也一样——你可以定义记录哪些信息、用什么格式、存到哪个文件。

## log_format：定义你自己的"行车记录仪参数"

Nginx 默认的日志格式（combined format）长这样：

```
192.168.1.1 - - [22/May/2026:10:15:30 +0800] "GET /index.html HTTP/1.1"
200 2048 "http://google.com" "Mozilla/5.0..."
```

对于大多数场景，默认格式够用了。但如果你需要记录更多信息（比如请求处理时间、上游服务器响应时间等），就需要自定义格式。

```nginx
# 定义在 http 块中
log_format detailed '$remote_addr - $remote_user [$time_local] '
                    '"$request" $status $body_bytes_sent '
                    '"$http_referer" "$http_user_agent" '
                    'rt=$request_time uct="$upstream_connect_time" '
                    'uht="$upstream_header_time" urt="$upstream_response_time" '
                    'host="$host"';
```

然后在 server 或 location 中使用：

```nginx
access_log /var/log/nginx/access.log detailed;
```

## 常用日志变量速查表

把日志变量想象成你在找房子时关心的各种信息——每个变量对应一个你关心的维度：

| 变量 | 含义 | 生活类比 |
|------|------|---------|
| `$remote_addr` | 客户端 IP 地址 | 拜访者的来源地址 |
| `$time_local` | 本地时间 | 拜访的时间 |
| `$request` | 完整的请求行（方法 + URI + 协议） | "我想进这间房，方式是推门" |
| `$status` | 响应状态码（200/404/500...） | 门开了没？200=开了，404=没找到门 |
| `$body_bytes_sent` | 响应体大小（字节） | 递出去的东西有多重 |
| `$request_time` | 处理时间（从收到第一个字节到发送完最后一个字节，单位秒） | 从你敲门到拿到东西花了多久 |
| `$http_referer` | 来源页面（从哪个页面跳转过来的） | 是谁介绍你来的 |
| `$http_user_agent` | 浏览器标识 | 你用的什么交通工具来的（公交/开车/步行） |
| `$host` | 请求的域名 | 你到了哪个小区 |
| `$upstream_addr` | 上游服务器地址 | 实际是谁服务你的（物业找了哪个维修工） |
| `$upstream_response_time` | 上游服务器处理时间 | 维修工花了多久修好 |
| `$upstream_connect_time` | 连接上游服务器的耗时 | 维修工接电话花了多久 |
| `$scheme` | 协议（http 或 https） | 你是走路还是坐装甲车来的 |
| `$ssl_protocol` | SSL 协议版本 | 装甲车的防护等级 |
| `$http_x_forwarded_for` | 原始客户端 IP（经过代理后） | 托人办事时，你最初的身份 |

## JSON 格式日志：让机器能读懂

如果你的日志量很大，或者需要送到日志分析平台（ELK、Splunk、Grafana Loki），JSON 格式是更好的选择：

```nginx
log_format json escape=json '{'
    '"timestamp":"$time_iso8601",'
    '"remote_addr":"$remote_addr",'
    '"request":"$request",'
    '"status":$status,'
    '"body_bytes_sent":$body_bytes_sent,'
    '"request_time":$request_time,'
    '"upstream_response_time":"$upstream_response_time",'
    '"http_referer":"$http_referer",'
    '"http_user_agent":"$http_user_agent",'
    '"host":"$host"'
'}';

access_log /var/log/nginx/access.log json;
```

JSON 格式的好处：每条日志是一个完整的 JSON 对象，日志分析工具可以直接解析。`escape=json` 会让 Nginx 自动转义 JSON 中的特殊字符。就像你把行车记录换成了黑匣子——虽然平时看不出区别，但一旦需要分析，标准化的数据格式让分析效率大幅提升。

用"给货物贴标签"来类比：传统日志是手写标签（"收件人张三、物品衣服"），人看得懂但机器难解析；JSON 日志是标准快递面单二维码，扫码枪"滴"一下，所有字段整整齐齐地出现了。

## 不同 location 使用不同日志

不是所有的请求都值得记录。就像你不会把行车记录仪整天开着录停车场里的静止画面，有些请求可以完全不记日志：

```nginx
# API 接口：详细格式 + 独立文件，方便事后排查慢请求
location /api/ {
    access_log /var/log/nginx/api_access.log detailed;
    error_log  /var/log/nginx/api_error.log  info;
}

# 静态资源：不记录日志（请求量太大，且没什么分析价值）
location /static/ {
    access_log off;
}

# 健康检查端点：不记录日志（负载均衡器几秒就检查一次）
location /health {
    access_log off;
    return 200 "OK";
}

# favicon 和 robots.txt：请求频率高，不记录
location = /favicon.ico {
    access_log off;
    log_not_found off;
}
```

`log_not_found off` 的意思是即使请求的文件不存在（返回 404），也不在错误日志中记录。因为 favicon.ico 和 robots.txt 经常不存在，这些 404 是正常的，没必要记录下来干扰视线——就像你知道停车场里会有行人路过，不需要每次都录下来标记为"异常"。

## 错误日志的级别

错误日志（error_log）有多个级别，从详细到简略依次是：

```
debug > info > notice > warn > error > crit > alert > emerg
```

生产环境通常设置为 `warn` 或 `error`，开发调试时可以临时调成 `info` 或 `debug`（debug 需要在编译时开启）。

```nginx
error_log /var/log/nginx/error.log warn;
```

## 动手试试

1. 使用 `examples/05-05/nginx.conf` 启动 Nginx
2. 访问网站几次，然后查看日志文件：

```bash
# 查看 JSON 格式的访问日志（漂亮的格式化）
cat /var/log/nginx/access.log | python3 -m json.tool | head -50

# 如果没有 Python，直接看也行
tail -5 /var/log/nginx/access.log
```

3. 对比默认 combined 格式和 JSON 格式的日志，想想哪种更适合你目前的场景
4. 设置对 `/static/` 目录关闭日志，访问几个不存在的文件，观察访问日志中确实没有这些记录
5. 尝试在日志格式中加入一个新的变量（比如 `$gzip_ratio` 获取压缩比），观察输出变化

## 本节小结

自定义日志就像配置行车记录仪的录像参数，用 log_format 决定记录哪些字段，用 JSON 格式方便机器分析，对不同 location 可以灵活开关日志避免 I/O 浪费。
