1. 什么是 Gzip 压缩

Gzip 是一种通用的数据压缩算法，由 Jean-loup Gailly 和 Mark Adler 在 1992 年开发。它的核心思想是找出数据中的重复模式，用更短的编码来表示它们。在 Web 领域，Gzip 已经成为了最广泛使用的 HTTP 压缩方式，几乎所有现代浏览器和服务器都支持它。

对于 HTML、CSS 和 JavaScript 这些文本文件，Gzip 的压缩率通常能达到 70% 到 80%。也就是说，一个 100KB 的 HTML 文件，经过 Gzip 压缩后可能只剩下 20KB 到 30KB。

2. 压缩的原理（简述）

Gzip 使用了两种核心算法。第一种是 LZ77 算法，它会寻找数据中重复出现的字符串。比如在一段文本中，"nginx" 这个词出现了 20 次，LZ77 就会用类似"往回看 5 个字符，复制 5 个字符"这样的短指令来替代重复的部分。

第二种是哈夫曼编码。统计每个字符出现的频率，出现频率越高的字符，给它分配的编码越短。比如在英文中字母 'e' 最常见，在 HTML 中 '<' 和 '>' 最常见，这些高频字符都会得到短编码。两种算法结合，达到了惊人的压缩效果。

3. 哪些文件适合压缩

文件类型压缩比说明
HTML70-80%充满重复的标签和属性，压缩效果极好
CSS70-80%选择器和属性名重复度高
JavaScript60-75%代码中有大量重复的关键字和模式
JSON70-80%键名重复，结构规整
SVG60-70%本质是 XML，压缩效果好
图片 (JPG/PNG)几乎无已经是压缩格式，再压缩没用还浪费 CPU
视频/音频几乎无已经是高度压缩的格式
PDF极少内部已经压缩过

4. 压缩的代价

收益：节省带宽 70%+，网页加载更快，用户体验更好，用户流量消耗更少（移动端尤其重要）。

代价：服务器需要 CPU 时间来压缩，客户端需要 CPU 时间来解压。但好在现代 CPU 处理 Gzip 非常快，这点开销相对于节省下来的传输时间来说，完全值得。

权衡：压缩级别越高，文件越小，但 CPU 消耗越大。级别 6 通常是一个很好的平衡点——压缩率接近最高，但 CPU 开销仍然可控。

5. Nginx Gzip 配置示例

以下是推荐的 Nginx Gzip 配置，适合大多数场景：

gzip on;
gzip_types text/plain text/css application/json application/javascript text/xml application/xml text/javascript;
gzip_min_length 1000;
gzip_comp_level 6;
gzip_vary on;

这个配置的含义是：开启 Gzip，对指定的 MIME 类型进行压缩，只压缩大于 1000 字节的文件（太小的文件压缩反而可能变大），压缩级别设为 6。

6. 如何验证 Gzip 是否生效

在浏览器开发者工具的 Network 标签中，查看响应头是否包含 Content-Encoding: gzip。也可以对比文件的实际大小和传输大小，传输大小应该明显更小。

或者用 curl 命令：curl -H "Accept-Encoding: gzip" -I http://your-server/，查看响应头中的 Content-Encoding 字段。
