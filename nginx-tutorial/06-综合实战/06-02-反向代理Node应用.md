# 06-02 反向代理 Node 应用

## 本节你会学到什么

- 理解 Nginx 和 Node.js 的职责分工模式
- 配置 Nginx 分离处理静态资源和动态请求
- 学会设置正确的代理请求头
- 能运行一个完整的 Nginx + Node.js 组合服务

## 餐厅的前厅和后厨

你去一家餐厅吃饭，流程是这样的：你推门进去，迎宾员接待你，安排座位，递上菜单。你点完菜后，服务员把订单送到后厨。后厨的厨师做好菜，服务员再把菜端给你。

你有没有见过哪个餐厅是让客人直接冲进厨房对着厨师喊"给我来份宫保鸡丁"的？没有。因为这会导致：
- 厨师一边炒菜一边应付客人，效率极低
- 厨房的布局不适合接待客人（油烟、高温、危险）
- 你想喝的酒水饮料在吧台，但你人在厨房，得绕路

Nginx + Node.js 的组合就是这个模式：
- **Nginx = 前厅（服务员 + 吧台 + 迎宾）**。它处理静态资源（CSS、JS、图片、字体）就像服务员直接给你倒水——简单、快速、不需要厨师。它对外的接口（HTTP 端口）友好、安全、稳定。
- **Node.js = 后厨（厨师）**。它处理动态请求（API、页面渲染、业务逻辑）——需要"加工"的东西，就像厨师做菜需要时间、火候和专业技巧。

分工之后的好处：
1. **静态资源不消耗 Node.js 资源**。一张图片让 Node.js 来服务就像让主厨去端盘子——浪费高级技能
2. **Nginx 能做的都给 Nginx 做**。Gzip、缓存、限流、HTTPS，这些 Node.js 也能做，但 Nginx 用 C 语言实现，效率高出很多
3. **前后端解耦**。你可以单独更新 Node.js 代码，Nginx 不需要改；你也可以调 Nginx 的 Gzip 配置，Node.js 不受影响

## 实战配置分析

我们来看一个真实的场景：Node.js 应用跑在 `127.0.0.1:3000`，只监听本地回环地址，外部无法直接访问。Nginx 监听 80 端口，对外提供服务。

```nginx
server {
    listen 80;
    server_name myapp.test;

    # 策略 1：静态资源，Nginx 直接处理
    location /static/ {
        alias /var/www/myapp/static/;
        expires 7d;
        add_header Cache-Control "public";
        access_log off;
    }

    # 策略 2：API 接口，代理到 Node.js
    location /api/ {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # 策略 3：其他所有请求，代理到 Node.js
    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host              $host;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
    }
}
```

### alias 和 root 的区别

注意静态资源用了 `alias` 而不是 `root`。用一个简单的比喻来解释：

你要找一份文件。**root** 的方式是："去 `/var/www/myapp/` 这个总目录里找，路径是 `static/style.css`"，最终位置是 `/var/www/myapp/static/style.css`。**alias** 的方式是："`/static/` 这个路径就等于 `/var/www/myapp/static/` 这个目录"，所以在 `/static/` 下找 `style.css` 时，Nginx 直接到 `/var/www/myapp/static/style.css`。

两种写法都能找到同一个文件，区别在于路径拼接的逻辑。很多初学者在这里踩坑——`alias` 后面的路径末尾要加 `/`，否则路径可能对不上。

### proxy_set_header：递上一张完整的快递单

`proxy_set_header` 的作用是修改或添加发给后端服务器的请求头。为什么需要它？

想象你是快递员，接到一个包裹要送到"3 号楼 502"。但物业说："快递不能直接送到住户，统一放在物业前台"。于是你把包裹给了物业前台（Nginx），但你在包裹上贴了张便条："发件人是张三（X-Forwarded-For），原本要送到 3 号楼 502（Host），发件人要求在下午 3 点前送达（其他信息）"。

后端 Node.js 看到的是 Nginx 的请求（因为 Nginx 把请求转发过来），如果不设置这些头，Node.js 就不知道原始请求来自哪里了。几个关键的头：

| 头名称 | 含义 | 为什么需要 |
|--------|------|-----------|
| `Host` | 用户访问的原始域名 | 后端可能根据域名做不同处理 |
| `X-Real-IP` | 用户的真实 IP | 后端记录日志需要知道真实来源 |
| `X-Forwarded-For` | 经过的代理链 | 跟 X-Real-IP 类似，但能记录完整的代理链路 |
| `X-Forwarded-Proto` | 原始协议（http/https） | 后端需要知道原始请求是否安全 |

### proxy_buffering：一次性上菜还是分批上菜

```nginx
location /api/ {
    proxy_buffering off;    # API 响应实时传输
}
```

静态页面和 API 对缓冲的需求不同。代理缓冲就像餐厅上菜——大份菜（大 HTML 页面）可以先在备餐台缓冲一下，聚齐后一次性端上桌。但 API 数据通常很小，每个请求的数据量也不大，而且用户期待实时返回，缓冲反而会增加不必要的延迟。所以对 API 关掉缓冲（`proxy_buffering off`），对页面保持缓冲。

## 运行步骤

**第一步：启动 Node.js 后端**

```bash
node examples/06-02/app/server.js
# 输出: [Node.js] 后端服务已启动: http://127.0.0.1:3000
```

**第二步：配置 Nginx**

把 `examples/06-02/nginx.conf` 放到 Nginx 的配置目录中。修改 `root` 和 `alias` 路径为实际路径。然后重载：`nginx -s reload`

**第三步：验证**

```bash
# 访问首页（由 Node.js 渲染，通过 Nginx 代理）
curl http://myapp.test/

# 访问静态资源（Nginx 直接处理）
curl -I http://myapp.test/static/style.css
# 观察响应头有 X-Served-By: Nginx-Direct

# 调用 API（代理到 Node.js）
curl http://myapp.test/api/info
# 观察 JSON 响应中的 backend 字段显示端口 3000
```

## 动手试试

1. 启动 Node.js 后端：`node examples/06-02/app/server.js`
2. 配置 Nginx 使用 `examples/06-02/nginx.conf`（调整路径后）
3. 浏览器访问 `http://myapp.test/`，观察页面是否能正常加载
4. 在开发者工具的 Network 标签中，查看静态文件（style.css）的响应头，确认 `X-Served-By: Nginx-Direct`
5. 点击页面上的 API 测试按钮，观察 API 响应和静态资源的不同响应头
6. 停止 Node.js 进程后刷新页面——Nginx 会返回 502 Bad Gateway 错误，这说明动态请求确实经过了代理
7. 注意观察：停止 Node.js 后，`/static/` 路径的静态资源仍然能正常访问——证明 Nginx 在独立处理这些请求

## 本节小结

Nginx + Node.js 的分工就像餐厅的前厅和后厨，Nginx 处理静态资源（快速、零等待），Node.js 处理动态内容（API、业务逻辑），各司其职，效率最优。
