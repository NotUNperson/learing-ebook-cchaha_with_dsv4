# 06-03 负载均衡集群实战

## 本节你会学到什么

- 在多实例场景中配置 Nginx 负载均衡
- 理解 least_conn 算法和健康检查的工作原理
- 学会用 Docker Compose 搭建完整的负载均衡环境
- 能够观察和验证故障转移行为

## 银行多个柜台

去银行办业务，你不会冲向某个特定的柜台，而是先取号，然后等着叫号。哪个柜台空出来了就叫下一个号。这就是负载均衡。

银行有 3 个柜台，假设每个柜台的服务速度差不多。如果 1 号和 2 号柜台正在给客户办复杂的业务（转账、开户，耗时较长），而 3 号柜台刚办完一个简单的查询业务空闲着，聪明的叫号系统会把下一个客户分配给 3 号柜台——这就是 `least_conn` 策略的目标，把请求发给当前"最闲"的后端。

如果某个柜台的柜员突然临时离开（后端挂了），叫号系统应该自动跳过这个柜台，等柜员回来后再重新启用——这就是健康检查。

                      |

   +----+----+   +----+----+   +----+----+

   |  :3001  |   |  :3002  |   |  :3003  |
   +---------+   +---------+   +---------+

## upstream 块详解

```nginx
upstream node_cluster {
    least_conn;

    server 127.0.0.1:3001 weight=1 max_fails=2 fail_timeout=30s;
    server 127.0.0.1:3002 weight=1 max_fails=2 fail_timeout=30s;
    server 127.0.0.1:3003 weight=1 max_fails=2 fail_timeout=30s;
}
```

每个参数的详细说明：

**least_conn**：分配算法。把请求发给当前活跃连接数最少的后端。活跃连接包括正在处理和等待处理的连接。适合后端处理时间不均衡的场景——如果每个请求处理时间不同（有的 API 10ms，有的 2s），`least_conn` 能避免把请求全堆积到处理慢的实例上。

**weight=1**：权重。所有实例权重相同（1:1:1），请求均匀分配。如果你想给某个配置更强的服务器多一些请求，可以设 `weight=2`，表示这个实例接收两倍的请求量。就像银行里经验丰富的柜员（高性能服务器）可以多分一些复杂的业务给 ta 办。

**max_fails=2**：健康检查。在 `fail_timeout` 时间段内，如果这个后端连续失败 2 次，Nginx 会把它标记为"不健康"，暂时不再转发请求给它。这就像柜台的柜员突然请假——叫号系统暂时跳过这个柜台，直到这个柜员回来（`fail_timeout` 到期后 Nginx 会重新尝试）。

**fail_timeout=30s**：失败超时时间。被标记为不健康后，30 秒内都不再转发请求；30 秒后 Nginx 发一个请求试试它好了没有，好了就恢复，没好继续等。

## proxy_next_upstream：请求失败后的重试策略

当 Nginx 把请求发给某个后端，但后端返回了错误，Nginx 是否应该自动尝试下一个后端？

```nginx
proxy_next_upstream error timeout http_500 http_502 http_503;
proxy_next_upstream_tries 2;        # 最多重试 2 次
proxy_next_upstream_timeout 10s;    # 重试总时长上限
```

这就好比你去银行柜台办业务，窗口的柜员说"系统卡了我办不了"。你不会就此放弃，而会去另一个窗口试试。但如果连试了两个窗口都办不了，你就该意识到可能不是柜员的问题，而是你的材料本身有问题——再试也是徒劳。

重要提示：`proxy_next_upstream` 只对**读请求（GET/HEAD）安全**。对于写请求（POST/PUT/DELETE），重试可能导致操作被执行多次。比如转账请求，第一个后台实际已经转成功了但返回了超时，Nginx 重试到第二个后台又转了一次——这就重复扣款了。所以生产环境中要小心配置，对于写操作通常不启用自动重试。

## 负载均衡算法选择指南

Nginx 支持多种负载均衡算法，选择哪一种取决于你的业务特点：

| 算法 | 指令 | 适合场景 | 类比 |
|------|------|---------|------|
| 轮询（默认） | 不写就是轮询 | 后端性能一致、请求耗时均匀 | 雨露均沾式分发传单 |
| 加权轮询 | weight=2 | 后端性能不一致 | 大卡车多装、小推车少装 |
| 最少连接 | least_conn | 请求耗时差异大 | 排最短的队 |
| IP 哈希 | ip_hash | 需要会话保持 | 固定去同一个窗口 |
| 通用哈希 | hash $var | 自定义分发规则 | 按姓氏首字母分窗口 |

IP 哈希（`ip_hash`）是一个特殊的方法，它保证同一个客户端 IP 的请求始终路由到同一个后端。这解决了"会话保持"的问题——如果你用内存存储 session 而不是共享 session 存储时，`ip_hash` 确保用户始终与同一个后端通信，不至于登录完后下一个请求跑到另一个后端又需要重新登录。

## 运行步骤

### 方法一：Docker Compose（推荐）

使用配套的 `docker-compose.yml`：

```bash
# 进入 examples/06-03/ 目录
cd examples/06-03

# 启动所有服务（Nginx + 3 个 Node.js 后端）
docker-compose up -d

# 查看所有容器状态
docker-compose ps

# 查看日志
docker-compose logs -f

# 停止并删除所有容器
docker-compose down
```

### 方法二：手动启动

```bash
# 终端 1：启动后端 1
PORT=3001 INSTANCE_ID=node-3001 node examples/06-03/backend/server.js

# 终端 2：启动后端 2
PORT=3002 INSTANCE_ID=node-3002 node examples/06-03/backend/server.js

# 终端 3：启动后端 3
PORT=3003 INSTANCE_ID=node-3003 node examples/06-03/backend/server.js

# 终端 4：配置并启动 Nginx
# 把 nginx.conf 放到 Nginx 配置目录
nginx -s reload
```

## 动手试试

1. 启动集群（推荐用 Docker Compose 一键启动）
2. 浏览器访问 `http://localhost/`，**连续快速刷新 10 次**，观察每次页面显示的 Instance ID 和端口号：是不是在 node-3001、node-3002、node-3003 之间轮转？
3. 点击页面上的"测试慢响应"按钮（`/slow` 路径模拟 2 秒延迟）。观察 Nginx 是否把这个请求分配到了一个空闲的实例上
4. 手动停掉一个后端实例：
   ```bash
   # 用 Docker 的话
   docker stop backend-3002
   ```
   然后刷新页面几次，观察 Nginx 是否自动跳过已停止的实例，请求只在剩下的两个实例之间分配
5. 查看 Nginx 的错误日志，确认被停止的实例被标记为"不健康"
   ```bash
   docker logs lb-nginx 2>&1 | grep -i "upstream"
   ```
6. 重新启动被停止的实例：
   ```bash
   docker start backend-3002
   ```
   等 30 秒后刷新页面，观察该实例是否自动恢复（健康检查到期后 Nginx 会重新尝试连接）

## 本节小结

负载均衡集群 = Nginx 做调度中心 + 多实例并行处理 + 健康检查自动摘除故障节点，`least_conn` 算法确保请求优先发到最空闲的后端。
