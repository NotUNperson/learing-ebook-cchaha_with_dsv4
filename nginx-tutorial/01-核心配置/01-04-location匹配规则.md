# 01-04 location 匹配规则

## 本节你会学到什么

- 掌握四种 location 匹配方式及其语法
- 理解各匹配方式的优先级规则
- 学会用快递分拣的类比记住匹配顺序
- 能够写出精确的 location 配置来控制请求路由

## 正文

### location 是干什么的

`location` 指令是 Nginx 配置中最精细的路由控制器。如果说 `server` 块决定了"请求进哪个门"，那 `location` 就决定了"进门之后往哪个房间走"。它根据 URL 路径把请求分配到不同的处理逻辑。

每一个 `location` 就像一条规则，告诉 Nginx：当 URL 路径长这样时，按如下方式处理。

### 四种匹配方式

Nginx 的 location 支持四种匹配语法，每一种前面有不同的修饰符：

| 修饰符 | 含义 | 示例 |
|--------|------|------|
| `=` | 精确匹配 | `location = /login` |
| `^~` | 优先前缀匹配 | `location ^~ /images/` |
| `~` | 区分大小写的正则匹配 | `location ~ \.php$` |
| `~*` | 不区分大小写的正则匹配 | `location ~* \.jpg$` |
| （无） | 普通前缀匹配 | `location /api/` |

没有修饰符的 `location /api/` 是最基础的前缀匹配——只要 URL 路径以 `/api/` 开头就行。

### 快递分拣的类比

理解 location 的匹配优先级，最好的类比是**快递自动分拣中心**。传送带上的包裹（HTTP 请求）依次经过不同的分拣口，每个口有一条规则。但规则之间不是平等的——有些口有"优先权"。

假设分拣中心有以下规则：

1. **精确规则（`=`）**：写着"收件人=张三"的包裹，直接走这个口。不经过后面的任何规则。——最高优先级，一次命中，立即处理。
2. **优先前缀规则（`^~`）**：写着"地址以'上海市'开头"的包裹。匹配到这个之后，不再检查后面的正则规则。——高优先级，命中后跳过正则。
3. **正则规则（`~` 和 `~*`）**：写正则表达式的规则，比如"地址包含'科技'二字"。——中等优先级，按出现顺序匹配，第一个命中的生效。
4. **普通前缀规则（无修饰符）**：最基础的规则，比如"地址以'北京'开头"。——最低优先级，但如果前面所有类型都没匹配上，会选匹配得最长的那条。

### 详细的匹配流程

Nginx 处理每个请求时，location 匹配的完整流程如下：

**第一步：精确匹配检查**

Nginx 首先检查所有 `=` 精确匹配的 location。如果找到一个完全匹配的，立刻用它，后续全部跳过。

```nginx
location = / {
    # 只有访问 / 且仅 / 时才用这个
    # /index.html 不会匹配这里！
}

location = /favicon.ico {
    # 专门处理网站图标的请求
    # 很多浏览器会自动请求这个文件
}
```

精确匹配通常用于高频但路径固定的请求。比如网站首页 `/` 和网站图标 `/favicon.ico`，加一个精确匹配可以减少不必要的正则计算。

**第二步：前缀匹配收集**

Nginx 扫描所有普通前缀 location（包括带 `^~` 的），记录下匹配得**最长**的那个。暂时不处理，先记下来。

```nginx
location / {
    # 匹配所有请求（最短前缀匹配）
}

location /images/ {
    # 匹配 /images/ 开头的请求（比 / 长）
}

location ^~ /images/thumbnails/ {
    # 匹配 /images/thumbnails/ 开头的请求（最长）
    # ^~ 标记表示：如果这条最长，就不做正则检查了
}
```

对于请求 `/images/thumbnails/photo.jpg`：
- `location /` 匹配（匹配长度 1）
- `location /images/` 匹配（匹配长度 8）
- `location ^~ /images/thumbnails/` 匹配（匹配长度 22）

**最长的是第三条**，而且它带了 `^~`，所以直接用它，**跳过第三步的正则匹配**。

如果最长匹配没带 `^~`（是普通前缀匹配），则进入第三步。

**第三步：正则匹配检查**

按配置文件中的出现顺序，依次检查所有正则 location（`~` 和 `~*`）。**第一个匹配成功的正则 location 胜出，立即使用。

```nginx
location ~ \.php$ {
    # 匹配 .php 结尾的请求（区分大小写）
    # /index.php 匹配，/index.PHP 不匹配
}

location ~* \.jpg|\.png|\.gif$ {
    # 匹配图片文件（不区分大小写）
    # /photo.JPG 和 /photo.jpg 都匹配
}
```

**第四步：兜底——使用第二步中记录的最长前缀匹配**

如果正则在第三步中没有任何匹配，Nginx 就回到第二步的结果，使用那个记录下来的"最长前缀匹配"。

### 踩坑实例

理解了这个流程，来看几个容易搞混的实际场景。

**场景一：前缀 vs 正则，到底谁赢？**

```nginx
location /api/ {
    return 200 "前缀匹配 /api/";
}

location ~ /api/ {
    return 200 "正则匹配 /api/";
}
```

访问 `/api/users` 会返回什么？答案是**"正则匹配 /api/"**。因为普通前缀匹配虽然命中了，但正则匹配的优先级更高（前提是没有 `^~` 阻挡）。

**场景二：加了 `^~` 之后**

```nginx
location ^~ /api/ {
    return 200 "优先前缀 /api/";
}

location ~ /api/ {
    return 200 "正则匹配 /api/";
}
```

这次访问 `/api/users` 返回**"优先前缀 /api/"**。因为 `^~` 阻止了正则匹配的检查。

**场景三：多个正则，谁先谁赢**

```nginx
location ~ \.php$ {
    return 200 "第一个正则：.php";
}

location ~ \.php|\.html$ {
    return 200 "第二个正则：.php 或 .html";
}
```

访问 `/index.php` 返回**"第一个正则：.php"**。因为正则匹配是按出现顺序，第一个命中就赢了。第二个也匹配，但没机会执行。

**场景四：精确匹配的"霸道"**

```nginx
location = /api {
    return 200 "精确匹配 /api";
}

location /api/ {
    return 200 "前缀匹配 /api/";
}
```

访问 `/api` 返回**"精确匹配 /api/"**。访问 `/api/users` 返回**"前缀匹配 /api/"**。注意：`/api/` 不匹配 `= /api`，但匹配 `/api/`。

### 优先级记忆口诀

总结成一个容易记的规则：

```
精确匹配（=）最优先，一击命中就完事。
优先前缀（^~）排第二，命中挡住正则路。
正则匹配（~, ~*）排第三，按序检查先到先得。
普通前缀排最后，没人要时才上场——但上场的是匹配最长的那一个。
```

## 动手试试

1. 在你的 Nginx 配置中创建 5 个 location，分别使用 `=`、`^~`、`~`、`~*`、和普通前缀匹配，每个返回不同的文字内容（用 `return 200 "xxx"` 来快速验证）。
2. 分别访问不同的 URL 路径，验证你的理解是否和实际行为一致。
3. 故意写两个冲突的 location（比如一个普通前缀和一个正则匹配同一个路径），看看实际谁生效。
4. 用 `nginx -t && nginx -s reload` 重载后，用 `curl -v http://localhost/xxx` 来观察返回结果（curl 会在终端打印出完整的 HTTP 响应）。

## 本节小结

location 有四种匹配方式，优先级从高到低为：精确匹配（`=`）、优先前缀（`^~`）、正则匹配（`~` / `~*`）、普通前缀。正则按出现顺序第一个命中生效，普通前缀选最长匹配。
