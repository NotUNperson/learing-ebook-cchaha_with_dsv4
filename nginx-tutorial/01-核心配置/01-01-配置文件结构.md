# 01-01 配置文件结构

## 本节你会学到什么

- 理解 nginx.conf 的整体层级结构
- 掌握上下文（context）和指令（directive）的概念
- 用"公司组织架构"类比理解 http/server/location 的包含关系
- 学会使用 include 指令拆分管理配置

## 正文

### 配置文件在哪里

Nginx 的一切行为都由配置文件决定。找到配置文件是第一步：

- Linux（apt）：`/etc/nginx/nginx.conf`
- Linux（yum）：`/etc/nginx/nginx.conf`
- Windows：`C:\nginx\conf\nginx.conf`
- macOS（brew）：`/usr/local/etc/nginx/nginx.conf`

用文本编辑器打开这个文件，你会看到一堆花括号、分号和关键字。第一次看到可能会觉得像天书，但别担心，它其实非常有规律。十分钟后你就能看懂它的骨架。

### 整体结构一览

简化来看，一个 nginx.conf 长这样：

```nginx
# 全局上下文（main）
worker_processes  auto;

events {
    # events 上下文
    worker_connections  1024;
}

http {
    # http 上下文
    include       mime.types;
    default_type  application/octet-stream;

    server {
        # server 上下文
        listen       80;
        server_name  localhost;

        location / {
            # location 上下文
            root   html;
            index  index.html;
        }

        location /images/ {
            root   /var/www;
        }
    }
}
```

### 公司组织架构的类比

把 nginx.conf 的层级结构想象成一家公司的组织架构：

- **全局上下文（main）** = 公司层面的规定。比如"全公司统一上班时间是 9 点"（`worker_processes`）、"文件保存路径是某某目录"（`error_log` 位置）。这个级别的规定对全公司有效。

- **http 上下文** = 公司成立了"网络服务事业部"。这个部门负责所有和 HTTP/HTTPS 相关的事。所有 Web 服务的规则都在这里定义。一家公司可以有多个事业部，但通常网络服务只有一个。

- **server 上下文** = 事业部下面的"项目组"。每个项目组负责一个网站。比如 A 组管 `www.site-a.com`，B 组管 `www.site-b.com`。每个 server 块可以有不同的域名、端口和规则。

- **location 上下文** = 项目组内部的"具体分工"。比如你负责首页的响应逻辑，他负责图片文件的路由，另一个人负责 API 接口的转发。不同的 URL 路径对应不同的 location。

这个类比对应关系如下：

```
公司（main）
  └── 网络服务事业部（http）
        ├── 网站A项目组（server）
        │     ├── 首页负责人（location /）
        │     ├── 图片负责人（location /images/）
        │     └── API负责人（location /api/）
        └── 网站B项目组（server）
              ├── 首页负责人（location /）
              └── 后台负责人（location /admin/）
```

理解了这层嵌套关系，再看配置文件就不觉得混乱了——每一层有它自己的职责范围。

### 指令和块的语法

Nginx 配置文件有两种基本元素：

**1. 指令（Directive）—— 一行一个，分号结尾**

```nginx
worker_processes  8;       # 指定 worker 进程数为 8
listen            80;      # 监听 80 端口
server_name       example.com;  # 域名
root              /var/www/html;  # 网站根目录
```

指令就像下达命令："你把 worker 数量设成 8"，"你监听 80 端口"。每条指令以分号结尾，很像 C 语言或 JavaScript 的写法。忘了分号会导致配置文件语法错误。

**2. 块（Block）—— 花括号包裹，内部可以有更多指令**

```nginx
server {
    listen 80;
    server_name example.com;
}
```

块把相关的指令组织在一起，表示"下面这些设定都属于这个 server 的"。块可以嵌套——server 块在 http 块里面，location 块在 server 块里面。

**3. 注释 —— # 开头**

```nginx
# 这是一行注释，Nginx 会忽略它
listen 80;  # 行尾注释也可以，从 # 开始到行末都被忽略
```

### 关键指令速览

在深入各个上下文之前，先认识几个最常见的指令：

| 指令 | 所属上下文 | 作用 |
|------|-----------|------|
| `worker_processes` | main | worker 进程数量 |
| `worker_connections` | events | 每个 worker 的最大并发连接数 |
| `listen` | server | 监听的 IP 和端口 |
| `server_name` | server | 匹配的域名（虚拟主机） |
| `root` | http/server/location | 网站文件的根目录 |
| `index` | http/server/location | 默认首页文件名 |
| `error_page` | http/server/location | 自定义错误页面 |
| `include` | 任何位置 | 引入其他配置文件 |

### include 指令：把配置拆成多个文件

当你的 Nginx 只有一个网站时，所有配置写在一个文件里没毛病。但当你有十个、二十个网站时，一个文件会变得非常臃肿。这时候 `include` 就上场了。

`include` 的作用是把其他文件的内容"插入"到当前位置。就像写论文时引用其他文献——内容不在正文里，但引用之后就像写在那里一样。

一个典型的拆分方式：

```
/etc/nginx/
├── nginx.conf              # 主配置（全局设定 + include）
├── conf.d/
│   ├── site-a.conf         # 网站 A 的 server 块
│   └── site-b.conf         # 网站 B 的 server 块
└── snippets/
    ├── ssl.conf            # SSL 通用配置片段
    └── security.conf       # 安全头通用配置片段
```

主配置文件中的相关行：

```nginx
http {
    include /etc/nginx/conf.d/*.conf;   # 引入所有网站配置
    include /etc/nginx/snippets/*.conf; # 引入通用配置片段
}
```

这种组织方式带来的好处是：
- 每个网站的配置独立一个文件，修改 A 站不会误伤 B 站
- 启用/禁用某个网站只需要删除/添加对应的配置文件，然后 reload
- 团队成员各管各的文件，不会同时编辑同一个文件导致冲突

用**乐高积木**来类比：`nginx.conf` 是底板，`include` 进来的文件是各种积木块。你可以随时增减积木块而不影响底板和其他积木。

## 动手试试

1. 找到你的 `nginx.conf` 文件，用文本编辑器打开，完整读一遍（不用看懂每一行，先感受整体结构）。
2. 在文件中数一数有几对花括号 `{}`，每一对代表一个块。确认你能认出 main、events、http、server、location 这些层级。
3. 在 `http` 块中，找到或添加一行 `include` 指令。观察你的 Nginx 安装目录中，哪些文件是通过 include 引入的。
4. 执行 `nginx -t` 确认你当前的配置文件语法正确。如果有错误，根据提示修正。

## 本节小结

nginx.conf 采用 main > events > http > server > location 的层级结构，用花括号分隔上下文块，用分号结尾每条指令。把配置拆成多个文件用 include 引入是管理复杂配置的最佳实践。
