# 01-05 错误页面与访问日志

## 本节你会学到什么

- 使用 `error_page` 自定义各种 HTTP 错误页面
- 理解访问日志的默认格式和自定义格式
- 掌握错误日志的级别分类和用途
- 学会通过日志定位问题

## 正文

### 为什么要自定义错误页面

浏览器自带的错误页面大家都见过——白底黑字写着 "404 Not Found" 或 "500 Internal Server Error"。对于访问者来说，这种页面又冷又硬，传递的信息是"出错了，但我不告诉你为什么，也不告诉你接下来该怎么办"。

自定义错误页面就像一个贴心的大楼前台。当访客问"XX 公司在几楼"时，如果这家公司已经搬走了，前台不是说"没这人"就沉默，而是说"那家公司已经搬到隔壁 B 座 15 楼了，需要我帮你查一下具体地址吗？"

一个好的自定义错误页面至少应该：
- 用友好的语言告诉用户发生了什么
- 提供一个返回首页的链接
- 保持网站的视觉风格一致
- 最好再加一个搜索框

### error_page 指令

`error_page` 的语法非常直接：

```nginx
error_page 状态码 处理方法;
```

具体来说：

```nginx
# 为单个错误码指定页面
error_page 404 /404.html;

# 为多个错误码指定同一个页面
error_page 500 502 503 504 /50x.html;

# 指定绝对 URL（重定向到外部页面）
error_page 502 = https://status.example.com/down.html;

# 更改返回的状态码
error_page 404 =200 /missing.html;
```

最常用的配置：

```nginx
server {
    listen 80;
    server_name example.com;
    root /var/www/example;

    # 自定义 404 页面
    error_page 404 /404.html;
    location = /404.html {
        # 404 页面通常放在网站根目录
        # 不需要 try_files，直接用 root
    }

    # 服务器错误页面（500/502/503/504）
    error_page 500 502 503 504 /50x.html;
    location = /50x.html {
        # 50x 页面可以放一个通用位置
    }
}
```

注意那个 `location = /404.html` 使用了精确匹配。这是必要的，否则当 Nginx 内部重定向到 `/404.html` 时，可能再次触发找不到文件（因为根目录中真的没有这个文件），形成无限循环。

### 访问日志：网站的"监控摄像头"

访问日志（access_log）记录了每一个到达 Nginx 的请求。它就像大楼门口的监控摄像头——记录着谁（IP）、什么时候（时间）、从哪里来（Referer）、做了什么（请求方法+路径）、结果如何（状态码）。

**默认日志格式**

Nginx 默认的日志格式叫 `combined`，在 `nginx.conf` 中通常由 `log_format` 定义。我们来逐字段拆解：

```nginx
log_format combined '$remote_addr - $remote_user [$time_local] '
                    '"$request" $status $body_bytes_sent '
                    '"$http_referer" "$http_user_agent"';
```

一条实际的日志长这样：

```
192.168.1.100 - - [22/May/2026:14:30:45 +0800] "GET /images/logo.png HTTP/1.1" 200 20480 "http://example.com/" "Mozilla/5.0 (Windows NT 10.0) AppleWebKit/537.36"
```

| 字段 | 变量 | 含义 | 这条记录的值 |
|------|------|------|------------|
| IP 地址 | $remote_addr | 访问者的 IP | 192.168.1.100 |
| 用户名 | $remote_user | HTTP 基本认证用户名 | -（没有） |
| 时间 | $time_local | 服务器本地时间 | 2026年5月22日 14:30 |
| 请求行 | $request | 完整的 HTTP 请求行 | GET /images/logo.png |
| 状态码 | $status | HTTP 响应状态码 | 200（成功） |
| 发送字节 | $body_bytes_sent | 响应体大小 | 20480 字节（20KB） |
| 来源页 | $http_referer | 从哪个页面点过来的 | http://example.com/ |
| 浏览器 | $http_user_agent | 用户浏览器信息 | Chrome on Windows |

### 自定义日志格式

默认格式适合大多数场景，但你也可以定义自己的格式来记录特定的信息：

```nginx
# 定义一个更详细的日志格式
log_format detailed '$remote_addr - $remote_user [$time_local] '
                    '"$request" $status $body_bytes_sent '
                    '"$http_referer" "$http_user_agent" '
                    'rt=$request_time uct="$upstream_connect_time" '
                    'uht="$upstream_header_time" urt="$upstream_response_time"';

# 对特定 location 使用不同的日志格式
location /api/ {
    proxy_pass http://backend;
    access_log /var/log/nginx/api_access.log detailed;
}
```

这个 `detailed` 格式额外记录了：
- `$request_time`：Nginx 处理请求的总时间（秒）
- `$upstream_connect_time`：连接后端服务器花了多久
- `$upstream_response_time`：后端服务器响应花了多久

你可以关闭某个 location 的访问日志（比如健康检查请求，天天写日志没意义）：

```nginx
location /health-check {
    access_log off;
    return 200 "OK";
}
```

### 错误日志：网站的"维修记录本"

access.log 记录的是"正常营业中发生了什么"，error.log 记录的是"出了什么故障"。用餐厅来类比：access.log 是前台登记簿（来了几桌客人、点了什么菜），error.log 是维修记录本（什么时候下水道堵了、哪个灯泡坏了）。

```nginx
# 全局错误日志（main 上下文）
error_log logs/error.log warn;

# 某个 server 或 location 覆盖错误日志级别
location /critical-api/ {
    error_log logs/critical_error.log debug;
    proxy_pass http://backend;
}
```

错误日志有 7 个级别，从详细到简略：

| 级别 | 含义 | 什么时候用 |
|------|------|-----------|
| `debug` | 调试信息，极其详细 | 开发调试时 |
| `info` | 一般信息 | 排查奇怪问题时 |
| `notice` | 值得注意但不严重 | 默认生产的合理选择 |
| `warn` | 警告，建议看一下 | 推荐的生产环境级别 |
| `error` | 出了错误 | 发生了需要关注的问题 |
| `crit` | 严重错误 | 影响服务但不致命 |
| `alert` | 需要立即处理 | 服务即将不可用 |
| `emerg` | 系统级紧急 | 服务已经不可用 |

生产环境通常使用 `warn` 级别——太详细会刷爆硬盘，太简略则出了事看不到线索。调试时临时改到 `debug`，调完改回来。

**注意：** `debug` 级别需要 Nginx 编译时带上 `--with-debug` 参数才能使用。如果你用的是包管理器安装的版本，可能不支持 debug 日志。

### 日志切割

日志文件会一直增长。如果不加管理，一个高流量的网站几天就能把 access.log 撑到好几个 GB。因此需要定期"切日志"：

```bash
# 方法一：手动切割（向 Nginx 发送 USR1 信号）
mv /var/log/nginx/access.log /var/log/nginx/access.log.20260522
kill -USR1 $(cat /var/run/nginx.pid)
# Nginx 收到 USR1 信号后会重新打开日志文件，旧文件被 mv 走了不影响写入

# 方法二：使用 logrotate（Linux 自动工具）
# 大多数 Linux 发行版安装 Nginx 后会自动配置好 logrotate
```

logrotate 会在 `/etc/logrotate.d/nginx` 中自动配置，通常按天切割，保留 14 天，超过的自动删除。就像写字楼物业定期清理旧的访客登记本——保留最近一段时间的就够了，太旧的没必要存着。

## 动手试试

1. 创建一个 `404.html`，要求有友好的提示文字和一个返回首页的链接。放在 Nginx 的 html 目录下。
2. 修改 Nginx 配置，添加 `error_page 404 /404.html;` 和对应的 location 块。
3. 用 `nginx -t && nginx -s reload` 重载，然后访问一个不存在的路径（比如 `http://localhost/不存在的页面`），确认看到的是你的自定义页面而不是浏览器的默认 404。
4. 查看 `access.log`，找到这条 404 请求的日志条目。注意状态码、请求路径和来源页字段。
5. 尝试在配置中定义一个自定义日志格式（比如加一个 `$request_time`），然后对某个 location 使用这个新格式。重载后访问，看看日志有什么变化。

## 本节小结

`error_page` 让你自定义错误页面来提升用户体验；`access_log` 记录请求细节是排错和监控的基础；`error_log` 记录运行故障，生产环境建议使用 `warn` 级别。
