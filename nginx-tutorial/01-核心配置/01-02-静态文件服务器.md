# 01-02 静态文件服务器

## 正文

### 静态文件服务器是什么

静态文件服务器，从名字就能猜到它的工作：把硬盘上的文件原封不动地发送给浏览器。HTML 页面、CSS 样式、JS 脚本、图片、字体、视频——这些不需要后端程序处理的内容，都由静态文件服务器直接提供。

Nginx 的看家本领之一就是静态文件服务。它在这方面的性能远超大多数同类产品，因为它的设计目标之一就是"用最少的内存和 CPU，以最快的速度把文件发给用户"。

### 核心指令一：root —— 指定网站根目录

`root` 指令告诉 Nginx："网站文件放在哪个文件夹里"。这是最基础也最重要的指令。

```nginx
server {
    listen 80;
    server_name www.example.com;

    location / {
        root /var/www/example;
        index index.html;
    }
}
```

这个配置的意思是：当有人访问 `http://www.example.com/` 时，Nginx 去 `/var/www/example/` 目录下找文件。具体来说：

- 请求 `/` -> 找 `/var/www/example/index.html`
- 请求 `/about.html` -> 找 `/var/www/example/about.html`
- 请求 `/images/logo.png` -> 找 `/var/www/example/images/logo.png`

`root` 的拼接规则很简单：**root 路径 + URL 路径 = 文件系统路径**。

### 核心指令二：alias —— 路径别名

`alias` 和 `root` 功能类似，但有一个关键区别。这让很多初学者栽过跟头。

用 `root` 时，Nginx 把 URL 路径完整拼接到 root 路径后面。
用 `alias` 时，Nginx 用 alias 路径**替换** URL 中 location 匹配到的部分。

看例子就明白了：

```nginx
# 方式一：用 root
location /static/ {
    root /var/www;
}
# 请求 /static/logo.png -> 找 /var/www/static/logo.png

# 方式二：用 alias
location /static/ {
    alias /var/www/assets/;
}
# 请求 /static/logo.png -> 找 /var/www/assets/logo.png
```

注意区别：`root` 保留了 `/static/` 这一段，而 `alias` 把 `/static/` 替换成了 `/var/www/assets/`。

用**商场里的指示牌**来类比：

- `root` 像是地标导航："去服装区，从商场大门进来后，沿着走廊走到服装区"。你仍然走在"商场路径"里。
- `alias` 像是快捷通道："去服装区，从这里直接穿过去就是"。你跳过了一段路径，直接从 A 点到了 B 点，中间的过程被替换掉了。

**重要提醒：** `alias` 后面的路径必须以 `/` 结尾，否则可能不工作。这是一个常见的坑：

```nginx
location /images/ {
    alias /var/www/pictures/;   # 正确：以 / 结尾
}
```

### 核心指令三：index —— 默认首页

当你访问一个目录而不是具体文件时（比如直接访问 `http://www.example.com/`），Nginx 怎么知道返回哪个文件？这就是 `index` 指令的作用。

```nginx
location / {
    root /var/www/example;
    index index.html index.htm default.html;
}
```

`index` 指令定义了一个**按顺序查找**的列表。当请求 `/` 时，Nginx 会依次查找：

1. 先找 `/var/www/example/index.html`
2. 找不到就找 `/var/www/example/index.htm`
3. 还找不到就找 `/var/www/example/default.html`
4. 全都没有就返回 404

这有点像你去餐厅看菜单——优先推荐招牌菜，招牌卖完了推荐特色菜，特色也卖完了推荐今日特价。

### 核心指令四：try_files —— 按顺序尝试

`try_files` 是 Nginx 中最强大也最常用的指令之一。它告诉 Nginx："按我这个清单依次查找文件，找到了就返回，全都找不到就执行最后一个动作。"

```nginx
location / {
    root /var/www/example;
    try_files $uri $uri/ /index.html;
}
```

这段配置的执行逻辑：

1. 先找 `$uri`（请求的原始路径对应的文件），比如请求 `/about` 就找文件 `/var/www/example/about`
2. 找不到就找 `$uri/`（把请求路径当目录，找目录下的 index），比如找 `/var/www/example/about/index.html`
3. 还找不到就**内部转发**到 `/index.html`，相当于在 Nginx 内部重新发了一个到 `/index.html` 的请求

### try_files 的内部转发机制

"内部转发"是理解 try_files 的关键。用**图书馆找书**来类比：

你去图书馆找一本书《Nginx 入门》（请求路径 `/nginx-intro`）：

1. 自助查询机告诉你应该在 3 楼 A 区第 5 排（$uri——直接找对应文件）
2. 结果去了发现那个位置是空的。——没事，查查看整排书架，也许被放错了（$uri/——当目录找 index）
3. 还是没找到。——图书馆有个规则："找不到的书统一申请馆际互借"（内部转发到 `/index.html`）

内部转发的妙处在于：浏览器地址栏显示的仍然是 `http://example.com/nginx-intro`，但实际上 Nginx 已经把请求在内部重新路由到了 `/index.html`，由这个页面（通常是一个前端单页应用的主入口）来接管路由。

这正是 React、Vue.js 等前端框架配合 Nginx 部署时的关键配置：

```nginx
location / {
    try_files $uri $uri/ /index.html;
}
```

前端应用只加载了一个 `index.html`，路由由 JavaScript 控制。但用户可能直接访问 `/user/profile` 这个前端路由——这个路径在硬盘上根本没有对应文件。`try_files` 发现找不到文件，就把请求内部转发到 `/index.html`，让前端应用自己处理路由。

### 完整示例：搭建一个静态网站

假设你有一个简单的博客网站，目录结构如下：

```
/var/www/myblog/
├── index.html          (首页)
├── about.html          (关于页)
├── css/
│   └── style.css       (样式)
├── js/
│   └── main.js         (脚本)
└── images/
    └── logo.png        (Logo)
```

对应的 Nginx 配置：

```nginx
server {
    listen 80;
    server_name myblog.local;

    # 网站根目录
    root /var/www/myblog;

    # 默认首页
    index index.html;

    # 主站路由
    location / {
        try_files $uri $uri/ =404;
    }

    # 图片设置缓存（浏览器可以缓存 30 天）
    location /images/ {
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # CSS 和 JS 设置短期缓存
    location ~* \.(css|js)$ {
        expires 7d;
        add_header Cache-Control "public";
    }
}
```

这里用到了 `=404` 作为 try_files 的最后一个参数——意思是"全都找不到就返回 404"。`~*` 是正则匹配（下一节详细讲），表示匹配以 `.css` 或 `.js` 结尾的请求。

## 动手试试

1. 在你的 Nginx html 目录下创建一个新文件夹，比如 `mysite`。
2. 在 `mysite` 里创建三个文件：`index.html`（写一句"这是首页"）、`about.html`（写"这是关于页"）、`404.html`（写"页面未找到"）。
3. 修改 `nginx.conf`（或创建一个新的 server 配置文件），配置 `root` 指向你刚创建的 `mysite` 目录。
4. 添加 `try_files` 配置，让找不到的文件返回自定义的 404 页面。
5. 用 `nginx -t` 测试，然后用 `nginx -s reload` 重载。分别访问 `http://localhost`、`http://localhost/about.html`、`http://localhost/不存在的页面`，观察结果。

## 本节小结

`root` 指定文件根目录（路径拼接），`alias` 做路径替换，`index` 定义默认首页的查找顺序，`try_files` 按顺序尝试多个文件并在找不到时做内部转发——这四个指令是配置静态文件服务器的核心。
