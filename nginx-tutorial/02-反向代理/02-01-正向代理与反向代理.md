# 02-01 正向代理与反向代理

## 本节你会学到什么

- 理解"代理"这个概念的本质含义
- 区分正向代理（帮客户端）和反向代理（帮服务端）
- 理解为什么反向代理是现代 Web 架构的标配
- 认识 Nginx 作为反向代理在实际架构中的核心地位

## 正文

### 什么是代理

"代理"这个词在法律和商业中很常见——你委托律师代理你打官司，你委托房产中介代理你买房。在计算机网络中，代理的含义完全一样：**一个中间人，代表某一方去和另一方打交道**。

代理的核心价值在于"间接性"——本来 A 和 B 直接对话，现在 A 通过 C 和 B 对话。这看似多此一举，但 C 作为中间人可以提供很多额外价值：缓存、过滤、负载分流、安全防护。

用**外卖平台**来理解最直观：

- 你直接去餐厅吃饭 = 客户端直连服务器
- 你通过美团点外卖 = 通过代理访问。美团帮你找到餐厅（路由）、帮你下单（转发）、告诉餐厅你的地址（传递信息），但美团不是餐厅本身。

### 正向代理：帮客户端的

正向代理（Forward Proxy）站在客户端这边，代表客户端去访问外部资源。

**典型场景：**

公司的网络管理。公司的所有电脑通过一台代理服务器上网。你想访问 `google.com`，请求先到公司的代理服务器，代理服务器帮你访问 Google，然后把结果返回给你。

```
你（客户端） --> 正向代理服务器 --> Google.com（公网）
```

正向代理的作用：

- **访问控制**：公司可以屏蔽某些网站（比如社交媒体），代理服务器直接拒绝转发
- **隐藏客户端身份**：Google 看到的是代理服务器的 IP，不知道是公司里哪台电脑在访问
- **内容缓存**：如果 100 个同事今天都访问了同一篇新闻，代理服务器只去公网拉一次
- **突破网络限制**：你挂了 VPN 才能访问某些网站，VPN 就是典型的正向代理

用**代购**来类比：你在国内买不到海外的商品，你找了一个海外代购。代购去帮你买，然后用他的名义寄给你。卖东西的商家只知道买家是这个代购，并不知道背后真正的买家是你。这个代购就是正向代理。

### 反向代理：帮服务端的

反向代理（Reverse Proxy）站在服务端这边，代表服务器去接收客户端的请求。

**典型场景：**

你的网站后端有 3 台 Node.js 应用服务器跑在内网。你不可能把它们全部直接暴露在互联网上。于是你在前面放了一台 Nginx，它接收所有来自互联网的请求，再转发给后面的应用服务器处理。

```
互联网用户 --> 反向代理（Nginx） --> 应用服务器 1（内网 192.168.1.10:3000）
                                  --> 应用服务器 2（内网 192.168.1.11:3000）
                                  --> 应用服务器 3（内网 192.168.1.12:3000）
```

反向代理的作用：

- **隐藏内部架构**：外界只知道 Nginx 的地址，不知道后面有几台服务器、各自在哪
- **负载均衡**：把流量均匀分给多台后端服务器
- **安全防护**：过滤恶意请求，起到防火墙的作用
- **SSL 卸载**：Nginx 处理 HTTPS 加密解密，后端用 HTTP 通信，减轻后端压力
- **缓存加速**：缓存后端响应，下次相同请求直接返回缓存内容

用**公司电话总机**来类比：你作为客户，想联系某公司的技术支持。你拨打的是公司对外公布的统一热线（400 电话）。总机接到电话后，根据你的问题转接给合适的技术人员——可能是小张，可能是小李，也可能是外包的第三方团队。但你从头到尾只知道那个 400 号码，内部怎么分配的对你来说完全透明。这个总机就是反向代理。

### 正向代理 vs 反向代理：一张表看懂

| 维度 | 正向代理 | 反向代理 |
|------|---------|---------|
| 为谁服务 | 客户端 | 服务端 |
| 谁部署 | 客户端一侧（公司、个人） | 服务端一侧（网站运营方） |
| 隐藏谁的身份 | 客户端 | 服务器 |
| 典型用途 | VPN、上网行为管理 | 负载均衡、安全网关、SSL 终端 |
| 客户端知情吗 | 知道（需要在浏览器设置代理） | 不知道（以为在和服务器直接对话） |
| 生活类比 | 代购帮你买东西 | 公司前台帮你转接电话 |

用一句话总结：**正向代理是客户端的代表，反向代理是服务端的代表。**

### 反向代理是现代架构的标配

在今天，几乎没有正经的 Web 应用会把后端服务器直接暴露在公网上。Nginx 作为反向代理几乎是所有架构的"标准第一层"。

一个典型的请求流程：

```
1. 用户输入 https://www.example.com
2. DNS 解析到 Nginx 服务器的公网 IP
3. Nginx 收到 HTTPS 请求，解密（SSL 卸载）
4. Nginx 查看 URL 路径：
   - /static/logo.png  -> 直接返回静态文件
   - /api/users        -> 转发到后端 API 服务器
   - /admin/           -> 转发到后台管理服务器
5. 后端服务器返回响应
6. Nginx 把响应加密（如果需要）后返回给用户
```

整个流程中，用户完全不知道 Nginx 后面还站着多少台服务器。对用户来说，`www.example.com` 就是一个完整的网站。Nginx 像一层透明的玻璃，用户透过它看到内容，但看不到玻璃本身。

这种模式带来的一个巨大好处是**后端架构可以随时变化而不影响用户**——你今晚把 Python 后端换成了 Go，用户毫不知情；你从 3 台服务器扩充到 10 台，用户只感觉网站变快了。Nginx 作为唯一的对外入口，隔离了内外，让内部变化对外部透明。

## 动手试试

1. 想一想你日常访问的网站。比如淘宝、微信网页版——它们把后端直接暴露给你了吗？你是怎么访问的？你在地址栏输入的是域名还是 IP 加端口号？
2. 如果你有运行中的后端程序（比如之前 Docker 教程中学的），试想一下：如果不用 Nginx 反向代理，直接让用户访问 `http://你的IP:3000`，会有什么问题？（提示：安全问题、端口号记忆问题、无法做负载均衡等）
3. 在你的 Nginx 配置文件中搜索 `proxy_pass`，现在还没有的话没关系，下一节我们就要手写第一个反向代理配置了。

## 本节小结

正向代理替客户端访问外部资源（像代购），反向代理替服务端接收外部请求（像公司前台总机）。反向代理是现代 Web 架构的标配入口，Nginx 在其中扮演了核心角色。
