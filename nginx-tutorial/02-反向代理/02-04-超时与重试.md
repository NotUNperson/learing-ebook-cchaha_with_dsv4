# 02-04 超时与重试

## 本节你会学到什么

- 理解三种代理超时参数的含义和区别
- 学会根据场景设置合理的超时时间
- 掌握 `proxy_next_upstream` 的重试机制
- 理解超时设置对用户体验的影响

## 正文

### 为什么要设置超时

想象你去一家快餐店点了一份外卖。你坐在店里等：

- 如果你等了 5 分钟还没做好，你可能会问一下
- 如果等了 30 分钟还没好，你肯定要发火
- 如果等了一个小时，你大概已经走了

HTTP 请求也是一样。Nginx 作为反向代理，在和后端服务器通信时，不能无限等下去。如果后端挂了、卡了、慢了，Nginx 需要一个明确的"最长等待时间"——超时。到了超时还不返回，Nginx 就放弃，返回错误给客户端。

不设超时或者超时设得太长，会导致连接堆积——所有请求都在等，新的请求进不来，服务器资源被耗尽。超时设得太短，则正常但稍慢的请求也会被误杀。

### 三种核心超时

Nginx 代理到后端时，有三个关键的超时参数，分别对应 TCP 连接过程的不同阶段。用**点外卖**来类比最直观：

**1. proxy_connect_timeout（建立连接超时）**

```nginx
proxy_connect_timeout 60s;  # 默认 60 秒
```

这是 Nginx 尝试和后端服务器建立 TCP 连接的最长等待时间。对应到你点外卖的流程，就是"外卖平台接单，联系餐厅"的阶段——电话打过去，餐厅接电话需要多久。如果电话一直占线打不通（后端服务器宕机了），等多久应该放弃。

这个值通常不需要设置很大。TCP 三次握手是很快的（毫秒级），如果几秒钟都建立不了连接，说明后端大概率已经挂了。推荐值：3-10 秒。

**2. proxy_read_timeout（读取响应超时）**

```nginx
proxy_read_timeout 60s;  # 默认 60 秒
```

连接建立后，Nginx 向后端发送请求，然后等待后端返回响应数据的超时时间。对应"餐厅已经接单，开始做菜"的阶段——你下了单，厨房在炒菜，但你不可能无限等下去，得有个底线。

这里有一个容易忽略的细节：`proxy_read_timeout` 不是"整个响应返回完"的总时间，而是**两次收到数据之间的间隔时间**。也就是说，只要后端每隔一段时间发一点数据过来（哪怕是 1 字节），计时器就会被重置。这个设计是为了支持"慢慢生成内容"的场景——比如后端在做复杂的数据库查询，边查边输出结果。

但对于普通 API 请求，后端通常会一次性返回整个响应。这种场景下，超时设置应该是一个合理的"整个响应的最长时间"。

**3. proxy_send_timeout（发送请求超时）**

```nginx
proxy_send_timeout 60s;  # 默认 60 秒
```

Nginx 向后端发送请求体时，两次发送数据之间的最大间隔。对应"你向餐厅说明你的口味偏好和特殊要求"的阶段。如果客户端上传一个大文件（比如 100MB 的视频），Nginx 需要把这个文件转发给后端，这个过程中如果网络卡顿导致长时间发不出数据，超时会触发。

对于普通 API 请求（请求体很小），这个值基本不会触发。

### 三个超时的对比总结

| 参数 | 作用的阶段 | 类比 | 推荐值（普通 API） |
|------|-----------|------|-------------------|
| `proxy_connect_timeout` | 建立 TCP 连接 | 餐厅接电话 | 3-10 秒 |
| `proxy_read_timeout` | 等待响应数据 | 等厨房做菜 | 30-120 秒 |
| `proxy_send_timeout` | 发送请求数据 | 说明口味要求 | 30-60 秒 |

### proxy_next_upstream：失败了换一个

这个指令用于负载均衡场景（下一章会详细讲）。当 Nginx 有多个后端服务器时，如果第一个后端出了问题，可以自动重试下一个。

```nginx
location /api/ {
    proxy_pass http://backend_pool;

    # 在以下情况下，自动重试下一个后端服务器
    proxy_next_upstream error timeout http_502 http_503;

    # 最多重试 2 个后端
    proxy_next_upstream_tries 2;

    # 重试等待超时（所有重试加起来不能超过这个时间）
    proxy_next_upstream_timeout 10s;
}
```

各项参数的含义：

| 条件 | 含义 |
|------|------|
| `error` | 和后端建立连接时出错 |
| `timeout` | 向后端发送请求或等待响应超时 |
| `invalid_header` | 后端返回了不合法的响应头 |
| `http_500` | 后端返回 500 内部错误 |
| `http_502` | 后端返回 502 网关错误 |
| `http_503` | 后端返回 503 服务不可用 |
| `http_504` | 后端返回 504 网关超时 |
| `http_403` | 后端返回 403 禁止访问 |
| `http_404` | 后端返回 404 未找到 |
| `http_429` | 后端返回 429 请求过多 |
| `non_idempotent` | 允许重试非幂等请求（POST 等） |

用**外卖取餐**来类比：你点了一份外卖，默认分配给最近的餐厅 A（第一个后端）。30 分钟后，你的超时设置触发了——餐厅 A 电话打不通（`error`），或者接了单但很久没出餐（`timeout`），或者告诉你做不了（`http_503`）。系统自动把订单转给餐厅 B（下一个后端）。这就是 `proxy_next_upstream` 在做的事。

**重要提醒：** POST 请求默认不会被重试，因为 POST 通常涉及数据写入（创建订单、提交表单等），重复执行可能导致重复写入。除非你明确加了 `non_idempotent`，Nginx 不会对 POST 请求尝试下一个后端。

### 完整配置示例

```nginx
server {
    listen 80;
    server_name api.example.com;

    location /api/ {
        proxy_pass http://backend_pool;

        # 请求头设置
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

        # 超时设置
        proxy_connect_timeout 5s;      # 连接后端，5 秒超时
        proxy_read_timeout 30s;        # 读取响应，30 秒超时
        proxy_send_timeout 15s;        # 发送请求，15 秒超时

        # 失败重试
        proxy_next_upstream error timeout http_502 http_503;
        proxy_next_upstream_tries 2;
        proxy_next_upstream_timeout 10s;
    }

    # 文件上传接口 —— 超时设长一些
    location /upload/ {
        proxy_pass http://backend_pool;
        proxy_connect_timeout 10s;
        proxy_read_timeout 300s;       # 上传大文件可能需要 5 分钟
        proxy_send_timeout 300s;       # 接收大文件也需要时间
    }
}
```

注意 `upload` 接口的超时设得特别长——因为文件上传可能需要很久，不能用普通 API 的超时标准。

## 动手试试

1. 修改你之前配置的反向代理 location，添加超时参数：
   - `proxy_connect_timeout 5s;`
   - `proxy_read_timeout 10s;`
   - `proxy_send_timeout 10s;`
2. 在你的后端代码中添加一个 15 秒延迟的接口（用 `setTimeout` 延迟响应），重载 Nginx 后访问这个慢接口。你会看到 Nginx 在 10 秒后返回 504 Gateway Timeout（因为 `proxy_read_timeout` 设了 10 秒）。
3. 把 `proxy_read_timeout` 改成 20 秒，重载后再访问同一个慢接口。这次应该能正常返回了。
4. 尝试把 `proxy_connect_timeout` 设成 1 毫秒（`1ms`），重载后访问任意接口。因为连接建立不可能在 1 毫秒内完成，你会发现所有请求都报 502。

## 本节小结

`proxy_connect_timeout` 控制连接建立等待，`proxy_read_timeout` 控制响应接收等待，`proxy_send_timeout` 控制请求发送等待。`proxy_next_upstream` 在后端失败时自动重试下一个。合理的超时设置是服务稳定性的基石。
