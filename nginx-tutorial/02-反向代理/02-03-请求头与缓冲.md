# 02-03 请求头与缓冲

## 本节你会学到什么

- 理解反向代理为什么要修改请求头
- 掌握 `proxy_set_header` 的正确用法
- 理解 X-Real-IP 和 X-Forwarded-For 的作用和区别
- 了解 `proxy_buffering` 缓冲机制及其影响

## 正文

### 为什么要改请求头

当 Nginx 作为反向代理转发请求时，它相当于帮客户端重新向后端发了一个 HTTP 请求。在这个过程中，如果不做特殊处理，后端收到的请求头会是 Nginx 自己构造的，而不是原始客户端的信息。

用**翻译官传话**来类比：一个外国人（客户端）问了一个服务员（Nginx），服务员转头用中文（新建 HTTP 连接）跟后厨（后端）说。如果不额外说明，后厨只知道是"一个服务员在问"，根本不知道后面坐着的是哪一桌的外国客人，用的是什么语言，从哪个国家来的。

这就是为什么要在反向代理中正确设置请求头——让后端"看到"原始客户端的真实信息。

### proxy_set_header

`proxy_set_header` 指令用来设置或覆盖转发给后端的 HTTP 头：

```nginx
location /api/ {
    proxy_pass http://backend;
    proxy_set_header 字段名 值;
}
```

三个最关键的配置：

```nginx
location /api/ {
    proxy_pass http://localhost:3000;

    # 1. 传递客户端的真实 Host（域名）
    proxy_set_header Host $host;

    # 2. 传递客户端的真实 IP
    proxy_set_header X-Real-IP $remote_addr;

    # 3. 传递完整的代理链（经过多个代理时记录所有 IP）
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

    # 4. 告诉后端用户是通过什么协议访问的（HTTP 还是 HTTPS）
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

逐条解释：

**Host 头（`$host`）**

客户端访问的是 `www.example.com`，Nginx 不设置的话，后端收到的是 `localhost:3000`（因为 Nginx 是向这个地址发的请求）。后端应用可能需要根据域名判断逻辑——比如多租户系统，不同域名返回不同内容。正确的 Host 头至关重要。

**X-Real-IP（`$remote_addr`）**

`$remote_addr` 是 Nginx 内置变量，代表直接连接 Nginx 的客户端 IP。当 Nginx 转发给后端时，这个头就告诉后端："这个请求最初是从这个 IP 来的。"

这就像外卖平台在给餐厅的订单上标注"下单人是张三，电话 138xxxx"。餐厅虽然是通过外卖平台接的单，但知道最终客户是谁。

**X-Forwarded-For（`$proxy_add_x_forwarded_for`）**

`$proxy_add_x_forwarded_for` 比 `$remote_addr` 更厉害。它记录的是完整的代理链：如果请求经过了多层代理（比如 CDN -> Nginx -> 应用服务器），每一层都会把自己的上一级 IP 追加到这个头后面。

格式是这样的：
```
X-Forwarded-For: 客户真实IP, 第一层代理IP, 第二层代理IP
```

用**快递盖章**来类比：一个包裹从上海发出，到杭州中转站，再发往北京。杭州中转站盖一个章"来自上海"，北京中转站盖一个章"来自杭州"。X-Forwarded-For 就是这些章的集合——"上海, 杭州"。

**X-Forwarded-Proto（`$scheme`）**

`$scheme` 的值是 `http` 或 `https`。因为 Nginx 处理后端通信时通常用 HTTP（内部网络不需要加密），所以需要这个头告诉后端"用户那边走的是 HTTPS"，后端才能正确处理重定向和安全策略。

### 关于 $host vs $http_host

初学者经常搞混 `$host` 和 `$http_host`：

- `$http_host`：严格等于请求头中的 Host 字段，原样输出
- `$host`：Nginx 处理过的版本，按优先级取：请求中的 Host 头 -> server_name 的值 -> 第一个匹配的 server_name

大多数场景用 `$host` 就行。`$http_host` 可能在含有端口的请求中带有 `:80`，不是所有后端都能正确处理。

### 代理缓冲

Nginx 在和后端通信时，默认会启用缓冲（`proxy_buffering on`）。这意味着 Nginx 会先把后端的响应收齐了，再一次性发给客户端。而不是收到一点发一点。

缓冲的好处：

- **保护后端**：后端慢慢生成响应，Nginx 替你扛着客户端的连接等待
- **断连友好**：如果客户端网络不稳定，Nginx 帮你守着连接，后端不知道也不关心

缓冲的弊端：

- **实时性差**：对于需要实时推送数据的场景（比如 SSE、WebSocket），缓冲会导致客户端等很久才收到数据
- **大文件问题**：后端返回一个很大的文件时，Nginx 需要先把整个文件缓存到内存或磁盘

```nginx
location /api/ {
    proxy_pass http://backend;

    # 关闭缓冲（用于实时数据场景）
    proxy_buffering off;

    # 或者调整缓冲参数
    proxy_buffering on;
    proxy_buffer_size 4k;           # 响应头的缓冲区大小
    proxy_buffers 8 4k;             # 响应体的缓冲区（数量 x 大小）
    proxy_busy_buffers_size 8k;     # 忙碌缓冲区的上限
    proxy_max_temp_file_size 1024m; # 临时文件最大大小（超出的写磁盘）
}
```

用**传达室收快递**来类比：

- 开启缓冲：传达室大爷收到你的大包裹后，不立刻叫你下楼，而是等包裹全部到齐、整理好了再通知你一次性取走。你省得跑多次，但包裹在传达室"等"了多久你不知道。
- 关闭缓冲：每到一个快递，大爷立刻喊你下楼取。你能看到快递"实时到达"，但可能要跑很多趟。

对于普通的 API 请求，默认缓冲通常是最佳选择。对于服务器推送事件（SSE）或流式响应，需要关闭缓冲。

### 完整配置示例

```nginx
server {
    listen 80;
    server_name api.example.com;

    location /api/ {
        proxy_pass http://localhost:3000/;

        # 传递客户端真实信息
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 关闭 Nginx 自己的 Connection 头，用后端的
        proxy_set_header Connection "";

        # 缓冲设置
        proxy_buffering on;
        proxy_buffer_size 4k;
        proxy_buffers 16 32k;
    }
}
```

## 动手试试

1. 启动上一节的后端程序（`node backend/app.js`），它已经在返回的 JSON 中包含请求头信息。
2. 在 Nginx 配置中，分别测试：不设置 `proxy_set_header`、只设置 `Host`、完整设置所有请求头。每次重载后用 curl 访问，观察后端返回的 headers 字段有什么不同。
3. 把 `X-Forwarded-For` 改成 `$remote_addr`（而不是 `$proxy_add_x_forwarded_for`），想想如果前面还有一层 CDN 代理，会出现什么问题？
4. 尝试在 location 中添加 `proxy_buffering off;`，重载后访问。感觉不到区别很正常（单次 API 请求差异很小），重点是理解它的存在和使用场景。

## 本节小结

`proxy_set_header` 让后端能看到真实客户端的 Host、IP 和协议信息，就像翻译官向对话的另一方说明说话人的真实身份。X-Real-IP 是直接来源 IP，X-Forwarded-For 是完整代理链。
