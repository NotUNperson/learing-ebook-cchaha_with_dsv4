# 00-01 什么是 Nginx

## 本节你会学到什么

- 认识 Nginx 在真实网站架构中处于什么位置

## 正文

### 餐厅前台接待员的比喻

想象你去一家非常火爆的餐厅吃饭。这家餐厅有一个前台接待员，他的工作很简单：客人来了，他负责招呼；客人要点菜，他告诉后厨；客人要结账，他通知收银台。后厨有很多厨师，但你作为客人，从头到尾只需要和前台打交道，根本不用知道后面到底有几位厨师、谁在炒哪道菜。

Nginx 在网站架构里，扮演的就是这个"前台接待员"的角色。当你用浏览器打开一个网站时，你的请求首先到达的就是 Nginx。Nginx 看看你要什么，然后决定：这个请求是直接给你一个静态文件（比如图片、CSS、JS），还是转发给后面的"厨师"（后端应用服务器，比如用 Java、Python、Node.js 写的程序）去处理。

### Nginx 到底是什么

Nginx（发音 "engine-x"）是一个高性能的 Web 服务器和反向代理服务器。2004 年由俄罗斯程序员 Igor Sysoev 开发，最初是为了解决俄罗斯某大型门户网站日均数亿次请求的性能问题。

说人话就是，Nginx 能做三件核心的事：

1. **直接提供静态文件**。你网站的 HTML 页面、图片、CSS 样式、JS 脚本，Nginx 可以直接把它们从硬盘里读出来发给浏览器，速度极快。
2. **作为反向代理**。把请求转发给背后的应用服务器（比如你的 Node.js 程序跑在 3000 端口），收到响应后再返回给浏览器。浏览器并不知道背后有个 Node.js 在干活，它只知道自己在和 Nginx 对话。
3. **做负载均衡**。当你网站流量大了，一台后端服务器扛不住，你启动了三台。Nginx 像一个聪明的分配员，把进入的请求均匀地分给这三台服务器。

### Nginx 解决了什么问题

回到 2000 年代初期，互联网正在爆发式增长。当时的 Web 服务器霸主是 Apache，它采用"一个请求一个进程/线程"的模型——每来一个用户就开一个进程去服务，就像餐厅给每位客人单独安排一个服务员全程跟随。人少的时候没毛病，但一旦同时来几千上万个客人，餐厅根本雇不起也装不下那么多服务员。

这就是著名的 **C10K 问题**（同时处理 10000 个客户端连接）。Apache 的进程模型下，一万个并发连接意味着要开一万个进程，操作系统的内存和 CPU 调度都承受不了。

Nginx 的解决办法完全不同。它采用**事件驱动**模型——一个 master 进程管理几个 worker 进程，每个 worker 可以同时处理成千上万个连接。这就好比餐厅的前台接待员不是一对一服务的，而是快速扫视全场，看到哪位客人举手就去处理，处理完立刻看下一位。一个接待员可以同时照看几十桌客人。

### Nginx vs Apache 对比

用交通管理来类比：

- **Apache（进程/线程模型）**像是一个路口给每辆车都配一个交警。10000 辆车就需要 10000 个交警，光交警站在一起就把路口堵死了。
- **Nginx（事件驱动模型）**像是智能红绿灯系统。不管来多少车，始终就那么几个红绿灯在规律切换，高效地指挥所有车辆通过。

具体对比：

| 特性 | Nginx | Apache |
|------|-------|--------|
| 并发处理 | 事件驱动，少量 worker 处理万级并发 | 进程/线程模型，高并发下内存消耗大 |
| 静态文件 | 极快，专为此优化 | 较快 |
| 动态内容 | 通过反向代理交给后端 | 可通过模块直接处理（mod_php 等） |
| 配置方式 | 集中式配置文件，简洁 | 支持 .htaccess 分布式配置 |
| 模块系统 | 编译时静态链接模块 | 可动态加载模块 |

简单来说：Nginx 强在高并发和静态文件，Apache 强在灵活性和动态内容处理。在现代架构中，两者经常搭配使用——Nginx 在前面处理静态文件和做反向代理，Apache（或 Node.js、PHP-FPM）在后面处理动态逻辑。

### Nginx 在真实架构中的位置

一个典型的现代网站架构长这样：

```
用户浏览器
    |
    | (HTTPS 请求)
    |
  Nginx (反向代理 + 静态文件 + 负载均衡)
    |      |      |
    |      |      +--- 静态资源（图片/CSS/JS）--- 直接从硬盘读取
    |      |
    |      +--- 动态请求转发 --- 应用服务器 1 (端口 8080)
    |      |                     应用服务器 2 (端口 8081)
    |      |                     应用服务器 3 (端口 8082)
    |
    +--- 缓存层（Redis/Memcached）--- 数据库
```

Nginx 站在最前面，就像大楼门口的前台和保安，所有来访者都要先经过它。它决定哪些人可以直接进去（静态文件），哪些需要通知内部员工来接（反向代理），以及当来访者太多时分流到不同的接待点（负载均衡）。

## 动手试试

1. 打开你的终端或命令行，输入 `nginx -v`（如果你还没安装 Nginx，会看到"命令未找到"的提示，这很正常，下一节我们就安装）。
2. 如果你已经安装了 Nginx，试试输入 `nginx -V`（大写 V），看看 Nginx 编译时带了哪些模块。
3. 思考一个问题：你平时访问的网站（比如淘宝、B站），为什么能同时服务几百万用户而不卡死？试着用本节学到的"前台接待员"类比来解释。

## 本节小结

Nginx 是一个高性能的 Web 服务器和反向代理，它用事件驱动模型解决了高并发连接问题，在现代互联网架构中扮演着"前台接待员"的角色，负责接收所有用户请求并进行智能分发。
