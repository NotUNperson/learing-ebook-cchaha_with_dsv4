### 回顾：三句话解释 Nginx

恭喜你完成了导论部分的学习。如果你现在需要向一个不懂技术的朋友解释 Nginx 是什么，你可以这样说：

> Nginx 就像一个餐厅的前台接待员。不管来多少客人，它都能快速接待、问清需求，然后把任务分配给后面真正干活的人。客人从头到尾只觉得前台在服务自己，实际上后面有一整个团队在协作。正是因为这种高效的工作方式，一个 Nginx 就能同时接待成千上万个客人。

如果你要对一个做技术的同事说：

> Nginx 是一个高性能的 Web 服务器和反向代理。它采用事件驱动模型，用少量 worker 进程处理极高的并发连接（C10K 甚至 C100K），同时支持静态文件服务、反向代理和负载均衡，是现代互联网架构中最常用的流量入口组件。

这两段话，前者是"让普通人听懂"，后者是"让同行听懂"。能把复杂的东西用简单的话说清楚，说明你真的理解了。

### C10K 问题：为什么 Nginx 诞生了

"C10K" 是 "Concurrent 10000 Connections" 的缩写，直译就是"同时处理一万个连接"。在上世纪 90 年代末到 2000 年代初，这是一个让无数后端工程师头疼的问题。

用**医院挂号窗口**来类比：

- **Apache 的模式**像传统的排队窗口——给每个病人安排一个接待员全程跟进。来了 10000 个病人，就需要 10000 个接待员，医院根本没有这么多接待窗口和工作人员。
- **Nginx 的模式**像现代的叫号系统——一个护士在分诊台快速判断每个病人的情况，轻症去 A 区，重症去 B 区，缴费去 C 窗。一个分诊护士就能处理大量病人，因为她不需要全程跟着每个病人。

Nginx 是如何做到的？核心是操作系统的 **epoll**（Linux 下的 I/O 多路复用机制）。简单说就是：一个 worker 进程不是"等在那里直到一个请求处理完"，而是"看谁有数据可读了就去处理谁"，处理完一个立刻切到下一个。这种模式叫做**异步非阻塞**。

总结一句话：Apache 是"多招人"的思路，Nginx 是"让每个人效率更高"的思路。事实证明，后者的上限高得多。

### Nginx 进程模型回顾

```
                    +-----------+
                    |  master   |  <- 读取配置、管理 worker
                    +-----------+
                        |   |   |
              +---------+   |   +---------+
              |             |             |
        +---------+   +---------+   +---------+
        | worker0 |   | worker1 |   | worker2 |   <- 真正处理请求
        +---------+   +---------+   +---------+
              |             |             |
        ==========客户端请求===============
```

这个模型有三个关键点：

1. **master 不干活，只管理**：它负责读取配置文件，启动和关闭 worker，以及接收 `nginx -s` 信号。如果 master 挂了，整个 Nginx 就瘫了。
2. **worker 之间是平等的**：每个 worker 都能处理任何请求，互不干扰。一个 worker 出问题不会影响其他 worker。
3. **worker 数量约等于 CPU 核心数**：这是最佳实践。比如 4 核 CPU 就设 4 个 worker。再多就是浪费（争抢 CPU 反而不如少开几个高效）。

### 常用命令速查表

| 命令 | 作用 |
|------|------|
| `nginx` | 启动 Nginx |
| `nginx -s stop` | 快速停止 |
| `nginx -s quit` | 优雅停止（等当前请求处理完） |
| `nginx -s reload` | 重载配置文件（不停机） |
| `nginx -t` | 测试配置文件语法是否正确 |
| `nginx -v` | 查看版本号 |
| `nginx -V` | 查看版本号和编译参数 |

### Nginx 在真实世界中的位置

让我们把视野放大一些。一个典型的网站请求经历了这样的旅程：

```
你的浏览器
    |
    | 1. DNS 解析：www.example.com => 服务器 IP
    |
    v
Nginx（负载均衡 & 反向代理）
    |
    | 2. 如果是静态文件（图片、CSS、JS），Nginx 直接从硬盘读取返回
    | 3. 如果是动态请求，Nginx 转发到后端应用服务器
    |
    v
后端应用服务器（Node.js / Java / Python / PHP-FPM）
    |
    | 4. 执行业务逻辑，读写数据库
    |
    v
数据库（MySQL / PostgreSQL / MongoDB）
    |
    | 5. 返回数据给应用，应用处理后返回给 Nginx，Nginx 返回给浏览器
    v
你的浏览器看到完整的页面
```

在整个链条中，Nginx 是"门面担当"——它决定了请求能不能进来、进来后去哪里、回来后又怎么返回给用户。没有 Nginx 在前面挡着，你的后端应用直接暴露在互联网上，就像你家大门直接对着马路，安全性和效率都无从谈起。

## 动手试试

这是一次综合小练习，目标是复习本章学到的所有核心操作：

1. **检查 Nginx 状态**：确保 Nginx 正在运行。如果没运行，启动它。用 `nginx -v` 确认版本号。
2. **修改欢迎页**：打开 `html/index.html`（Linux 上可能在 `/usr/share/nginx/html/`），把页面标题改成"我的 Nginx 学习之旅——导论完成！"，保存后刷新浏览器验证。
3. **查看进程**：用 `ps aux | grep nginx`（或 Windows 上的 `tasklist`）查看 master 和 worker 进程，数一数有几个 worker。
4. **查看日志**：打开 `access.log`，刷新几次浏览器页面，观察日志的实时变化。找到你自己请求的那几行，理解每一段的含义。
5. **测试与重载**：运行 `nginx -t` 测试配置语法，然后执行 `nginx -s reload` 重载配置（尽管你这次没有改配置，但可以感受一下这个过程）。
6. **停止与重启**：用 `nginx -s quit` 停止 Nginx，确认浏览器无法访问 localhost 了。然后重新启动 Nginx，再次确认能访问。

做完这些操作，你对 Nginx 的基本操作就熟练了。这些命令在后面每一章的学习中都会反复用到。

## 本节小结

Nginx 用事件驱动模型解决了 C10K 高并发问题，以一个 master 管理多个 worker 的架构高效处理海量请求。掌握基本的启停和日志查看命令是后续一切学习的基础。
