## 正文

### Session 困境：一个经典问题

在开始之前，先理解 Session 是什么。Session（会话）是 Web 应用中用来"记住用户状态"的机制。比如你登录淘宝、往购物车加商品、填写收货地址——这些操作跨越了多个 HTTP 请求，但服务器需要知道"这几个请求是同一个人发的"。Session 就是那张"临时身份卡"。

Session 通常的工作方式：

1. 你登录后，服务器生成一个 Session 数据（用户 ID、登录时间等），存在后端的内存或 Redis 中
2. 服务器给你发一个 Cookie，内容是一个 Session ID（比如 `SESSIONID=abc123`）
3. 你接下来发的每个请求都带着这个 Cookie
4. 服务器看到 Cookie 里的 Session ID，去查对应的 Session 数据，知道"哦，这是张三"

**问题来了：** 在负载均衡环境中，你的第一个请求（登录）被分配到了服务器 A，服务器 A 创建了 Session 存在自己的内存里。但你的第二个请求（查看订单）被分配到了服务器 B——服务器 B 的内存里没有你的 Session 数据，它不认识你。

用**银行柜台**来类比：你去银行取一笔分期付款的钱。第一次去，3 号柜台的小王已经了解了你的全部情况，帮你填了表格、复印了身份证。第二次你去，引导员把你分到了 5 号柜台。小李完全不知道你是谁，你还得重新解释一遍来龙去脉，重新填表复印。体验极差。

这就是"Session 困境"——负载均衡在分散压力的同时，也分散了用户的"上下文"。

### 解决方案概览

解决 Session 问题有两大类思路：

**思路一：把 Session 提取出来，所有服务器共享**
- 用 Redis 或 Memcached 存储 Session，所有后端都去同一个地方查
- 把 Session 加密后存在 Cookie 中（JWT 就是这个思路）
- 这是最"现代"的做法，但对老系统改动大

**思路二：确保同一个用户的请求始终打到同一台服务器**
- IP 哈希（基于客户端 IP）
- Cookie 黏连（基于 Cookie）
- 这就是本节重点

### IP 哈希

`ip_hash` 是 Nginx 内置的最简单的会话保持方案。在 upstream 块中加一行 `ip_hash;`，Nginx 就会根据客户端 IP 地址的哈希值来分配请求——同一个 IP 的所有请求永远打到同一台后端。

```nginx
upstream backend {
    ip_hash;

    server 192.168.1.10:3000;
    server 192.168.1.11:3000;
    server 192.168.1.12:3000;
}

server {
    listen 80;
    server_name app.example.com;

    location / {
        proxy_pass http://backend;
        proxy_set_header Host $host;
    }
}
```

**工作原理：** Nginx 取客户端 IP 的前三段（IPv4 中 C 类地址段），通过哈希函数计算出一个值，映射到某台服务器。只要 IP 不变，映射就不变。

**ip_hash 的优点：**
- 零配置零依赖，一行 `ip_hash;` 搞定
- 不需要改后端应用代码
- 不需要额外的服务（Redis 等）

**ip_hash 的局限（这是重点）：**

1. **同一局域网用户全打到同一台：** 公司里的 200 个同事共享同一个公网 IP，所有人的请求都打到同一台服务器——负载根本不均衡。
2. **移动网络 IP 会变：** 你坐地铁用手机刷网站，4G 基站切换时 IP 就变了，Session 就会丢。
3. **后端增减时重新分布：** 你在 upstream 中加一台新服务器或移除一台，哈希映射就会重新计算，大量用户的 Session 会"漂移"到不同的服务器。
4. **服务器故障时空缺：** 如果一台服务器挂了，本来打到这台的用户会被重新分配到其他服务器，Session 全丢。

### Cookie 黏连（Sticky Cookie）

Cookie 黏连是一种更精细的方案：Nginx 在第一次响应时往客户端种一个 Cookie（比如 `route=serverA`），后续请求带着这个 Cookie 回来，Nginx 根据 Cookie 的值把请求路由到对应的服务器。

```
第一次请求：Client -> Nginx -> 分配到 Server A
Nginx 给 Client 种 Cookie: route=A

后续请求：Client -> Nginx（Cookie 带着 route=A）-> Nginx 看到 route=A -> 转发到 Server A
```

遗憾的是，Cookie 黏连是 Nginx Plus（商业版）的功能，开源版不原生支持。不过有一个第三方模块 `nginx-sticky-module` 可以实现类似功能，需要重新编译 Nginx。

商业版的配置看起来像这样（仅供参考）：

```nginx
upstream backend {
    server 192.168.1.10:3000;
    server 192.168.1.11:3000;

    sticky cookie srv_id expires=1h domain=.example.com path=/;
}
```

### 三种方案的对比

| 方案 | 实现难度 | 负载分布 | Session 安全 | 适用场景 |
|------|---------|---------|-------------|---------|
| ip_hash | 极低（一行配置） | 不均匀 | 一般（IP 变了就丢） | 内部系统、流量不大 |
| Shared Session（Redis） | 中高（需要改后端） | 非常均匀 | 高（服务器挂了 Session 还在） | 生产环境、大流量 |
| Sticky Cookie | 中（需 Nginx Plus 或编译） | 均匀 | 较高 | 生产环境但不想改后端 |
| JWT/Token | 中（改登录逻辑） | 非常均匀 | 高（无状态） | 现代 SPA 应用、API 服务 |

### 实际生产中的选择

在实际项目中，现代架构越来越倾向于"思路一"——把状态从后端服务器中剥离出来。为什么？

因为**无状态服务**可以随意扩缩容。你今天 3 台服务器，明天流量暴增要加到 10 台，直接启动新服务器就行，不需要担心 Session 迁移问题，不需要重新计算 IP 哈希。Kubernetes 这类容器编排系统能自动帮你做到弹性伸缩，前提就是你的后端是无状态的。

但如果你在维护一个老系统，后端用 PHP 把 Session 存在本地文件里，改造成本太高——那就老老实实用 `ip_hash`，能解决问题就行。技术选型不是选最炫的，是选最合适的。

## 动手试试

1. 如果你在 03-02 中启动了 3 个后端进程，现在给 upstream 加上 `ip_hash;`，重载 Nginx。
2. 用同一台电脑多次访问代理地址（`curl http://api.local/`），观察每次返回的 `port` 字段是否一致（应该是同一个端口）。
3. 换一台设备访问（如果有的话，比如用手机访问你电脑的 IP），看是否打到了不同的后端。
4. 把 `ip_hash;` 注释掉（变回轮询），再访问多次，对比区别。
5. 思考：如果你的网站 80% 的用户来自同一个公司网络（共享 IP），用 `ip_hash` 会有什么问题？你会怎么解决？

## 本节小结

负载均衡会导致用户的 Session 数据分散在不同服务器上。`ip_hash` 让同一 IP 始终访问同一台后端，实现简单但分布不匀且 IP 变化时会丢失 Session。生产环境中，使用 Redis 共享 Session 或 JWT 无状态方案是更健壮的选择。
