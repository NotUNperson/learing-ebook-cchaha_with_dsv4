## 正文

### 什么是负载均衡

负载均衡（Load Balancing），从字面上拆解就是"把负载（工作量）均衡地分配出去"。它的本质是一个"工作分配器"——当有一大堆活儿要干的时候，不是让一个人累死，而是合理地分给多个人一起干。

在 Web 架构中，负载均衡器（比如 Nginx）站在最前面，把用户请求分发到后面多台服务器上。每台服务器只承担一部分请求，整体对外看起来就像一个巨能扛的服务。

用**超市收银排队**来类比最直观：

一家小超市只有一个收银员。平时还好，但到了下班高峰，20 个顾客推着购物车排队，收银员累得手忙脚乱，队伍排到了货架尽头。顾客体验极差，有人等不及就走了。

解决办法：增加收银台，并且安排一个引导员站在入口，根据各条队伍的长度来引导顾客去不同的收银台。顾客不知道也不关心自己被分到了几号台，只知道结账速度变快了。

- 超市 = 你的网站
- 收银员 = 后端应用服务器
- 排队顾客 = 用户请求
- 引导员 = 负载均衡器（Nginx）

### 为什么需要负载均衡

**1. 高可用（High Availability）**

只有一个收银员时，如果他突然肚子疼去厕所了（服务器宕机），整个超市的结账就瘫痪了。有了多个收银员，一个请假另一个顶上，顾客几乎感觉不到中断。

技术层面：如果某台后端服务器挂了，负载均衡器自动把流量切换到健康的服务器上，服务不受影响（至少不受致命影响）。

**2. 水平扩展（Horizontal Scaling）**

超市生意越来越好，一条队伍怎么调整都不够用了。怎么办？不是换一个更快的收银员（垂直扩展，换更好的服务器——但硬件有天花板），而是加更多的收银台（水平扩展，加更多服务器——理论上可以无限加）。

技术层面：流量从每天 1000 次请求涨到 100000 次，你不需要升级那台服务器到顶配，而是从 1 台加到 10 台普通机器。成本更低，弹性更大。

**3. 性能提升**

就算所有服务器都健康，负载均衡本身也是一种性能优化。Nginx 可以缓存后端的响应、对请求做预处理（比如过滤非法请求）、卸载 SSL 加密——这些都是后端不用操心的事，可以让后端专注处理业务逻辑。

### 常见负载均衡算法

算法就是负载均衡器"如何分配请求"的规则。不同的规则适用于不同的场景。

**1. 轮询（Round Robin）—— 最简单公平的分配**

轮询就是"挨个来"：请求 1 给服务器 A，请求 2 给服务器 B，请求 3 给服务器 C，请求 4 又回到服务器 A……循环往复。

用**幼儿园老师发点心**来类比：老师拿着一盘饼干，按座位顺序依次发给小明、小红、小刚、小丽……发完一轮再从头开始。每个小朋友拿到的一样多，公平公正。

这是 Nginx 的默认算法，适合后端服务器性能差不多的情况。

**2. 加权轮询（Weighted Round Robin）—— 能者多劳**

不是所有服务器性能都一样。A 服务器是 8 核 32G，B 服务器是 2 核 4G，让他们平均分配工作量显然不合理。加权轮询给每台服务器分配一个权重，权重高的多分到一些请求。

用**建筑工地分活**来类比：工头分配搬砖任务，壮汉一次能搬 20 块，瘦子一次能搬 10 块。工头给壮汉每次分 2 份（权重 2），给瘦子分 1 份（权重 1），让每个人的能力得到合理利用。

```nginx
upstream backend {
    server 192.168.1.10:3000 weight=3;  # 这台性能好，承担 50% 流量
    server 192.168.1.11:3000 weight=2;  # 承担 33%
    server 192.168.1.12:3000 weight=1;  # 承担 17%
}
```

**3. IP 哈希（IP Hash）—— 固定客户找固定的人**

IP 哈希根据客户端 IP 地址计算一个哈希值，同一个 IP 的所有请求始终打到同一台服务器。

用**银行柜台办理复杂业务**来类比：你去银行开户，第一次找的是 3 号柜台的小王。你的业务比较复杂，需要多次来银行补充材料。IP 哈希确保你每次来都去 3 号柜台找小王——他已经了解你的情况了，不用每次重新解释。

```nginx
upstream backend {
    ip_hash;
    server 192.168.1.10:3000;
    server 192.168.1.11:3000;
}
```

**局限：** 如果客户在外面用 WiFi 或 4G，IP 可能会变。而且公司里所有同事共用同一个公网 IP，全都打到同一台服务器，分布不够均匀。

**4. 最少连接（Least Connections）—— 谁闲找谁**

不是看"轮到谁"，而是看"谁当前最空闲"。Nginx 把请求发给当前活跃连接数最少的服务器。

用**医院挂号**来类比：不是固定让你去 3 号窗口，而是哪个窗口排队最短就去哪个。有人看小感冒 1 分钟搞定，有人要咨询复杂病情需要 10 分钟——只按"轮到谁"分的话，有的人忙死有的人闲死。

```nginx
upstream backend {
    least_conn;
    server 192.168.1.10:3000;
    server 192.168.1.11:3000;
}
```

这个算法适合请求处理时间差异较大的场景。

### 算法选择指南

| 场景 | 推荐算法 |
|------|---------|
| 后端服务器性能相同 | 轮询（默认） |
| 后端服务器性能差异大 | 加权轮询 |
| 需要会话保持（用户粘性） | IP 哈希 |
| 请求处理时间差异大 | 最少连接 |

## 动手试试

1. 打开你的 Nginx 配置文件，找到或新建一个 `upstream` 块（下一节会详细讲语法，先看看长什么样）。
2. 思考你日常使用的某个 Web 应用（比如淘宝、B站、Gmail），分析一下：它有多少并发用户？大概有多少台服务器？你猜它用的是什么负载均衡算法？为什么？
3. 如果你有编程基础，可以尝试写一个简短的伪代码：假设有 3 台服务器和 100 个请求，模拟轮询算法是如何分配的。

## 本节小结

负载均衡是把请求合理分配给多台服务器以提升可用性和性能的技术。轮询公平分配，加权让强者多干，IP 哈希保证同一用户固定访问同一服务器，最少连接则优先交给空闲的服务器。
