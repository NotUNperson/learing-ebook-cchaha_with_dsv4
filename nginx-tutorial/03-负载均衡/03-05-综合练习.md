# 03-05 综合练习

## 本节你会学到什么

- 综合运用 upstream 定义、权重分配、健康检查、故障转移
- 搭建一个完整的加权轮询负载均衡集群
- 模拟后端故障，观察 Nginx 的自动切换行为
- 验证会话保持和普通轮询的区别

## 正文

### 练习目标

搭建一个完整的负载均衡演示环境。场景设定：

一个外卖平台的 API 服务部署了 3 台后端服务器。各服务器的配置不同：服务器 A 是最新采购的高性能机器，服务器 B 是标准配置，服务器 C 是一台老旧的备用机。你需要：

- 设置加权轮询，让 A 处理 50% 流量，B 处理 30%，C 处理 20%
- 配置健康检查，确保故障时自动切换
- 准备一台备用服务器 D，在所有主服务器都挂掉时顶上
- 验证整个系统的容错能力

### 架构图

```
                Nginx (80 端口)
                负载均衡 + 健康检查
                    |
        +-----------+-----------+-----------+
        |           |           |           |
    Server A    Server B    Server C    Server D
    3001端口    3002端口    3003端口    3004端口
    weight=5    weight=3    weight=2    backup
    (50%)       (30%)       (20%)       (备用)
```

### 步骤一：准备后端程序

使用我们之前写好的后端程序（`examples/03-02/backend/server.js`），这个程序接受端口号作为命令行参数，返回自己的端口号和请求计数。

在 4 个终端窗口中分别启动 4 个后端进程：

```bash
# 终端 1：服务器 A（高性能）
node server.js 3001

# 终端 2：服务器 B（标准）
node server.js 3002

# 终端 3：服务器 C（老旧）
node server.js 3003

# 终端 4：服务器 D（备用）
node server.js 3004
```

每个终端都会显示"后端服务已启动，端口 300X"。保持所有终端运行。

### 步骤二：编写 Nginx 配置

```nginx
# ---------- 外卖平台后端组 ----------
upstream delivery_backend {
    # 加权轮询：总权重 = 5 + 3 + 2 = 10
    # A 得到 5/10 = 50% 流量
    server localhost:3001 weight=5 max_fails=2 fail_timeout=20s;
    # B 得到 3/10 = 30% 流量
    server localhost:3002 weight=3 max_fails=2 fail_timeout=20s;
    # C 得到 2/10 = 20% 流量
    server localhost:3003 weight=2 max_fails=2 fail_timeout=20s;

    # 备用服务器：所有主服务器都不可用时启用
    server localhost:3004 backup max_fails=1 fail_timeout=10s;
}

server {
    listen 80;
    server_name delivery.local;

    location / {
        proxy_pass http://delivery_backend;

        # 完整请求头
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Connection "";

        # 超时设置
        proxy_connect_timeout 3s;
        proxy_read_timeout 5s;
        proxy_send_timeout 5s;

        # 失败重试
        proxy_next_upstream error timeout http_502 http_503;
        proxy_next_upstream_tries 2;
    }
}
```

### 步骤三：测试正常负载分配

所有的后端都正常运行时，连续发送 30 个请求，观察分配比例：

```bash
# 在终端中运行这个循环（或写一个简单脚本）
for i in $(seq 1 30); do
  curl -s http://delivery.local/ | grep '"port"' | tr -d ' ,\n'
  echo ""
done
```

统计各端口出现的次数。理论上：
- 3001 出现约 15 次（50%）
- 3002 出现约 9 次（30%）
- 3003 出现约 6 次（20%）

实际值会有一些偏差，因为轮询不完全等于随机分配，但在 30 次请求下大体的比例应该接近 5:3:2。

### 步骤四：模拟后端故障

```bash
# 1. 找到 3003 端口的进程 PID
# Linux/Mac:
lsof -i :3003 | grep LISTEN
# 或者：
ps aux | grep "server.js 3003"

# 2. 杀死这个进程
kill <PID>

# 3. 确认进程已停止（没有进程在 3003 端口监听了）

# 4. 立即连续发请求
for i in $(seq 1 10); do
  curl -s http://delivery.local/ | grep '"port"' | tr -d ' ,\n'
  echo ""
done
```

观察：
- 前 1-2 个请求可能失败（被分配到了 3003，该端口已关闭）
- Nginx 检测到连续失败后，3003 被标记为不可用
- 后续请求全部打到 3001 和 3002
- 3004（备份）仍然不接收请求，因为还有主服务器（3001、3002）活着

### 步骤五：模拟全部主服务器故障

```bash
# 继续杀死 3002 和 3001
kill <3002的PID>
kill <3001的PID>

# 现在所有主服务器都挂了，只有备用服务器 3004

# 发请求测试
curl -s http://delivery.local/ | grep '"port"'
```

现在所有请求都应该打到 3004（备用服务器）。Nginx 在检测到所有主服务器不可用后，自动启用了备用服务器。

### 步骤六：恢复服务，观察健康检查

```bash
# 重新启动三台主服务器
node server.js 3001 &
node server.js 3002 &
node server.js 3003 &

# 等待 20 秒（fail_timeout 时间）后，再发请求
for i in $(seq 1 10); do
  curl -s http://delivery.local/ | grep '"port"' | tr -d ' ,\n'
  echo ""
done
```

Nginx 在冷却期结束后，试探性地向恢复的主服务器发请求。一旦确认它们健康，就会恢复正常分配，备用服务器 3004 退回"休眠"状态，不再接收请求。

### 验证清单

完成整个练习后，确认你观察到了以下行为：

1. 正常状态下，请求按权重比例（5:3:2）分布到三台主服务器
2. 一台主服务器宕机后，流量自动转移到剩余健康的主服务器
3. 所有主服务器都宕机后，备用服务器自动启用
4. 宕机的服务器恢复后，Nginx 自动检测到并将其重新加入轮转
5. 故障转移过程中，`proxy_next_upstream` 自动重试下一个后端，部分请求可能出现短暂延迟或失败，但整体服务不中断

## 动手试试

这个"动手试试"就是完成上面六个步骤。整个练习预计 20-25 分钟。这是对负载均衡所有核心知识点的综合检验。

如果遇到问题：
- `error.log` 是第一优先级排查入口
- 确认所有后端进程确实在对应端口监听（用 `lsof -i :300X` 或 `netstat -an | grep 300X`）
- 确认 hosts 文件中有 `127.0.0.1 delivery.local`
- 确认 `nginx -t` 没有语法错误

## 本节小结

负载均衡的核心是 upstream 组定义（服务器列表+权重）、健康检查（自动发现和隔离故障）、故障重试（proxy_next_upstream）。三者配合，构建出高可用的后端服务体系。
