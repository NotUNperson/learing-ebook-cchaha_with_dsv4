# 03-04 健康检查

## 本节你会学到什么

- 理解健康检查的概念和重要性
- 掌握被动健康检查的配置（max_fails + fail_timeout）
- 了解主动健康检查的原理和限制
- 理解后端故障时 Nginx 的具体行为

## 正文

### 什么是健康检查

健康检查就是 Nginx 定期"体检"后端服务器，确认它们还能正常工作。如果发现某台服务"不健康"了，就暂时不给它分配请求，把所有流量导向健康的服务器。等它恢复了，再重新加入工作队列。

用**公司体检**来类比：公司每年组织员工体检，发现谁有健康隐患就安排休养治疗（标记为不可用），让健康的人多分担一些（自动切换）。等康复了再回来正常上班。

没有健康检查的话，Nginx 会继续向已经挂掉的后端发请求——每个请求都得等它超时，严重的拖慢整个系统的响应速度。用户看到的是"网站好慢"，实际上只是其中一台后端慢而已。

### 被动健康检查

"被动"的意思是 Nginx 不会主动去探测后端，而是在正常的用户请求过程中观察后端的表现。如果连续几次和某个后端的通信出了问题，就标记它为不可用。

这就是我们在 03-02 中提到的 `max_fails` 和 `fail_timeout`：

```nginx
upstream backend {
    server 192.168.1.10:3000 max_fails=3 fail_timeout=30s;
    server 192.168.1.11:3000 max_fails=3 fail_timeout=30s;
    server 192.168.1.12:3000 max_fails=3 fail_timeout=30s;
}
```

**工作流程：**

1. Nginx 正常向 10 号后端发送请求
2. 10 号后端连续 3 次在 `fail_timeout`（30 秒）内响应失败
3. Nginx 将 10 号标记为"不可用"，接下来的 30 秒不再向它发请求
4. 30 秒"冷却期"结束后，Nginx 试探性地向 10 号发一个请求
5. 如果这次成功了，10 号恢复"健康"状态，重新加入正常轮转
6. 如果还是失败，继续冷却 30 秒

#### 什么是"失败"

触发 `max_fails` 计数的"失败"包括：

- 连接后端超时（`proxy_connect_timeout` 触发）
- 读取响应超时（`proxy_read_timeout` 触发）
- 后端返回 `proxy_next_upstream` 中列出的错误码
- 后端拒绝连接（端口没开、进程挂了）

**一次普通用户的 404 不算失败**——除非后端的正常业务逻辑就是针对某些请求返回 404。所以 `proxy_next_upstream` 要谨慎选择重试条件，不要把所有状态码都列为"失败"。

#### 被动检查的局限

被动健康检查有一个矛盾：它需要真实请求才能判断后端是否健康。但后端可能间歇性故障——刚才还好好的，突然不行了，导致部分用户的请求被浪费。

打个比方：就像你让食客去试菜来判断厨师在不在——食客点完菜等半天才知道厨师今天请假了。食客白白等了一顿（用户体验受损）。

### 主动健康检查

"主动"的意思是 Nginx 定期向后端发送专门的健康检查请求，不依赖真实用户流量来判断后端状态。

```nginx
upstream backend {
    server 192.168.1.10:3000;
    server 192.168.1.11:3000;

    # 主动健康检查（仅 Nginx Plus 支持）
    health_check interval=5s fails=3 passes=2 uri=/health;
    match statusOK {
        status 200;
        header Content-Type = text/html;
        body ~ "OK";
    }
}
```

这段配置的意思是：Nginx 每 5 秒主动向后端的 `/health` 路径发一个请求。连续 3 次失败就标记为不可用；连续 2 次成功就恢复为健康。

主动检查的参数：

| 参数 | 含义 |
|------|------|
| `interval` | 检查间隔时间 |
| `fails` | 连续失败多少次标记为不可用 |
| `passes` | 连续成功多少次标记为健康 |
| `uri` | 请求的路径 |
| `match` | 匹配规则块（判断什么算"健康"） |

主动健康检查是 Nginx Plus（商业版）的功能。开源版用户通常通过以下方式弥补：

**方案一：用第三方模块**
- `nginx_upstream_check_module`（淘宝开源的模块，需要重新编译 Nginx）

**方案二：后端自己实现健康检查端点**

```javascript
// 在 Node.js 后端中
const server = http.createServer((req, res) => {
    if (req.url === '/health') {
        // 这里可以检查数据库连接、缓存状态等
        res.writeHead(200, { 'Content-Type': 'text/plain' });
        res.end('OK');
        return;
    }
    // 正常业务逻辑
});
```

但仅靠被动检查+健康检查端点还不够——Nginx 不知道要去访问这个端点。部署时通常配合外部监控系统（如 Prometheus、存活检测脚本）或在容器编排中利用健康检查端点。

### 实际场景：模拟后端故障

让我们通过一个具体例子来理解健康检查的行为。

```
场景：3 台后端运行中，第 2 台突然挂了

T=0s:   3 台后端都健康，Nginx 正常分发请求
T=10s:  第 2 台后端进程崩溃（端口关闭）
T=11s:  一个用户请求被分配到第 2 台，连接失败（第1次失败）
T=13s:  又一个请求到第 2 台，连接失败（第2次失败）
T=16s:  第三个请求到第 2 台，连接失败（第3次失败）
        -> max_fails=3 达到，第 2 台被标记为不可用
T=16s ~ T=46s:  冷却期，第 2 台不接收任何请求
                第 1 和第 3 台承担所有流量
T=46s:  冷却结束，Nginx 试探性地发一个请求
        还是失败 -> 继续冷却 30 秒
```

在整个过程中，大部分用户的请求没有受到影响（失败的那几次除外），因为 Nginx 可以配合 `proxy_next_upstream` 自动重试下一个后端。

### 完整配置示例

```nginx
upstream backend {
    # 后端服务器列表，带健康检查参数
    server localhost:3001 max_fails=3 fail_timeout=30s;
    server localhost:3002 max_fails=3 fail_timeout=30s;
    server localhost:3003 max_fails=3 fail_timeout=30s;

    # 备用服务器（所有主服务器都不可用时启用）
    server localhost:3004 backup max_fails=1 fail_timeout=10s;
}

server {
    listen 80;
    server_name api.local;

    location / {
        proxy_pass http://backend;

        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

        proxy_connect_timeout 3s;
        proxy_read_timeout 10s;

        # 失败重试下一个后端（配合健康检查）
        proxy_next_upstream error timeout http_502 http_503;
        proxy_next_upstream_tries 2;
    }
}
```

## 动手试试

1. 启动 3 个后端进程（见 examples/03-02/backend/server.js），端口分别为 3001、3002、3003。
2. 配置 Nginx（参考 examples/03-04/nginx.conf），启用 `max_fails=2 fail_timeout=15s`。
3. 用 curl 连续发几个请求，确认正常的轮询工作。
4. 手动停掉其中一个后端（比如 kill 掉 3002 端口的进程），再连续发请求。
5. 观察：刚开始可能有 1-2 次请求失败（因为被分配到了挂掉的后端），但很快 Nginx 就会检测到异常，把该后端标记为不可用，后续请求全部走健康的后端。几分钟后把停掉的后端重新启动，观察 Nginx 什么时候恢复向它发请求。

## 本节小结

健康检查让 Nginx 自动规避故障后端。被动检查（max_fails + fail_timeout）基于真实请求判断后端状态，是开源版的标配。主动健康检查（Nginx Plus）能更早发现故障但需商业版。
