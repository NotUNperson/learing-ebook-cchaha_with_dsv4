# 07 综合练习：从本地到云端

## 本节你会学到什么

- 把本模块所有技能串成一个完整的工作流
- 从零开始走完"本地项目 → GitHub 仓库 → SSH → 推送 → Issues → PR → Actions"全流程
- 理解每个步骤在整个软件开发生命周期中的位置
- 建立"写代码 → 推送 → 自动检查 → 协作管理"的完整心智模型

## 这不是新知识，这是"串珍珠"

前面六节，我们分别学了六颗"珍珠"：SSH 密钥、推送仓库、Issues、Actions、Fork、多平台。每一颗都很亮，但放在一起才是一串完整的项链。

本节不教新知识。我们模拟一个真实项目的完整流程——从"这个项目只在我电脑上"到"全世界都能看到、别人能提意见、机器自动检查"。

## 情景设定

**你决定开发一个"待办事项管理"小项目，叫 `todo-app`。你想把它开源到 GitHub，让朋友帮忙提建议，并且每次改代码时让 Actions 自动帮你检查格式。**

这是你要做的事：

1. 本地创建项目，用 Git 管理
2. 在 GitHub 上创建仓库
3. 用 SSH 连接配置好
4. 把本地代码推送到 GitHub
5. 用 Issues 管理开发计划
6. 为一个 Issues 写代码修复，用 `fixes #编号` 自动关闭
7. 配置 Actions 自动检查
8. （可选）推送到 Gitee 作为国内镜像

## 完整流程一步步走

### 阶段一：本地初始化（5分钟）

```bash
# 1. 创建项目目录
mkdir todo-app
cd todo-app

# 2. 初始化 Git
git init

# 3. 创建项目文件
echo "# Todo App" > README.md
echo "一个简单的待办事项管理应用，用于学习 Git 平台联动。" >> README.md

mkdir src
cat > src/index.js << 'EOF'
// Todo App 主入口
console.log('Todo App 启动！');

const todos = [];

function addTodo(text) {
    todos.push({ text, done: false });
    console.log(`添加待办：${text}`);
}

function listTodos() {
    console.log('待办列表：');
    todos.forEach((t, i) => {
        console.log(`  ${i + 1}. [${t.done ? 'x' : ' '}] ${t.text}`);
    });
}

addTodo('学习 Git 平台联动');
addTodo('完成综合练习');
listTodos();
EOF

# 4. 第一次提交
git add .
git commit -m "初始化 todo-app 项目骨架"
```

### 阶段二：GitHub 网页操作（3分钟）

1. 打开浏览器，登录 [GitHub](https://github.com)
2. 点击右上角 **"+"** → **New repository**
3. Repository name 填 `todo-app`
4. Description 填 "一个简单的待办事项管理应用"
5. 选择 **Public**（公开）
6. **不要勾选** README、.gitignore、license
7. 点击 **Create repository**
8. 在 "Quick setup" 页面选择 **SSH**，复制仓库地址
   格式：`git@github.com:你的用户名/todo-app.git`

### 阶段三：连接并推送（1分钟）

```bash
# 5. 添加远程地址
git remote add origin git@github.com:你的用户名/todo-app.git

# 6. 首次推送（用 -u 建立追踪）
git push -u origin main
```

去 GitHub 网页刷新，你的 `README.md` 和 `src/index.js` 应该都在了。

### 阶段四：用 Issues 管理开发计划（3分钟）

在 GitHub 上创建三个 Issue：

**Issue #1** — "完善 README 文档"
- 标签：`documentation`
- 内容：现在的 README 太简单了，需要补充项目功能介绍、安装方法、使用说明

**Issue #2** — "增加删除待办事项的功能"
- 标签：`enhancement`
- 内容：目前只能添加待办，不能删除。需要增加一个 `deleteTodo()` 方法

**Issue #3** — "增加标记完成的功能"
- 标签：`enhancement`
- 内容：需要增加一个 `markDone()` 方法，能把某个待办事项标记为已完成

创建一个 Milestone：**"v0.1 基础功能"**，把这三个 Issue 都分配进去。

### 阶段五：写代码修复 Issue（5分钟）

现在回到本地，处理 Issue #2（增加删除功能）：

```bash
# 7. 创建功能分支（好习惯：不在 main 上直接改）
git checkout -b feature-delete-todo
```

编辑 `src/index.js`，添加 `deleteTodo` 函数：

```javascript
function deleteTodo(index) {
    if (index >= 0 && index < todos.length) {
        const removed = todos.splice(index, 1);
        console.log(`删除待办：${removed[0].text}`);
    } else {
        console.log('无效的待办编号');
    }
}
```

提交时用关键词自动关闭 Issue：

```bash
# 8. 提交，提交信息中引用 Issue #2
git add src/index.js
git commit -m "增加删除待办事项的功能，closes #2"

# 9. 推送到 GitHub
git push -u origin feature-delete-todo
```

### 阶段六：创建 Pull Request（2分钟）

1. 打开你的 GitHub 仓库页面
2. 你会看到一条黄色提示："feature-delete-todo had recent pushes..."
3. 点击 **Compare & pull request**
4. 写 PR 标题："增加删除待办事项功能"
5. 写 PR 描述："增加了 deleteTodo() 方法，支持删除指定编号的待办事项。\n\nCloses #2"
6. 点击 **Create pull request**

在 PR 页面的右侧，将 Issue #2 链接到这个 PR（在 Development 区域）。

为了方便练习，你可以直接 Merge 这个 PR（因为这是你自己的仓库）：

7. 点击 **Merge pull request** → **Confirm merge**
8. 合并后，去 Issues 页面确认 Issue #2 是否自动关闭了

### 阶段七：配置 Actions 自动检查（3分钟）

回到本地，创建 Actions 工作流：

```bash
# 10. 切回 main 并拉取最新代码（因为刚才在 GitHub 上合并了 PR）
git checkout main
git pull origin main

# 11. 创建工作流目录
mkdir -p .github/workflows
```

创建 `.github/workflows/check.yml`：

```yaml
name: 代码检查

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - name: 签出代码
        uses: actions/checkout@v4

      - name: 安装 Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: 检查文件结构
        run: |
          echo "正在检查项目文件结构..."
          ls -la
          echo ""
          echo "检查 src/ 目录..."
          ls -la src/
          echo ""
          echo "所有文件检查完毕！"

      - name: 运行项目
        run: node src/index.js
```

提交并推送：

```bash
git add .github/workflows/check.yml
git commit -m "添加 Actions 自动检查工作流"
git push
```

去 GitHub Actions 页面查看运行结果。你应该能看到绿色的勾，表示工作流运行成功。

### 阶段八：（可选）推送到 Gitee 作为国内镜像

```bash
# 添加 Gitee 远程仓库（假设你已经在 Gitee 上创建了同名仓库）
git remote add gitee git@gitee.com:你的用户名/todo-app.git

# 先推到 GitHub（主仓库）
git push origin main

# 再推到 Gitee（国内镜像）
git push gitee main
```

## 整个流程的心智模型

至此，你拥有了一个完整的"平台联动"开发循环：

```
 本地写代码                    远程协作                    自动化
┌──────────┐               ┌──────────┐               ┌──────────┐
│          │               │          │               │          │
│ 1.git init│              │ Issues   │               │ Actions  │
│ 2.写代码  │── git push ─→│ - 记录Bug │─ 触发 ──────→│ - 自动检查│
│ 3.commit │              │ - 提建议  │               │ - 自动测试│
│          │              │          │               │          │
│          │←─ git pull ──│ PR       │               │ 结果通知  │
│          │              │ - 审查代码│               │          │
│          │              │ - 讨论修改│               │          │
└──────────┘              └──────────┘               └──────────┘
      │                         │                          │
      │                         │                          │
      └─────────────────────────┴──────────────────────────┘
                         SSH 密钥串联一切
```

把这个循环印在脑子里：

1. **本地 git init → 写代码 → commit**（你早就熟练了）
2. **SSH 密钥**让你不用频繁输密码（本模块第1节）
3. **git remote add + git push**把代码送上云端（本模块第2节）
4. **Issues**用来管理任务、跟踪 Bug（本模块第3节）
5. **Actions**在后台自动帮你检查代码（本模块第4节）
6. **Fork + PR**是参与开源项目的标准方式（本模块第5节）
7. **多 remote**把一份代码同时托管在多个平台（本模块第6节）

## 动手试试

这就是你的"期末考试"了——请在 30 分钟内独立完成（允许翻前面的章节查阅）：

1. 在本地创建一个新项目（任意名字、任意内容，至少有2个文件）
2. `git init` → `git add` → `git commit`
3. 在 GitHub 上创建一个空仓库
4. 用 `git remote add origin` 连接，`git push -u origin main` 推送
5. 在 GitHub 上创建至少2个 Issue，并用 Labels 分类
6. 创建一个 Milestone，把 Issue 放进去
7. 创建一个功能分支，修改代码解决其中一个 Issue，提交时用 `fixes #编号`
8. Push 分支，创建 Pull Request，自己 Merge 掉
9. 确认 Issue 自动关闭
10. 创建 `.github/workflows/hello.yml`，配置一个简单的 Actions
11. Push 配置文件，在 Actions 页查看运行结果

完成上述全部 11 个步骤，你就真正掌握了 Git 平台联动的所有核心技能。
