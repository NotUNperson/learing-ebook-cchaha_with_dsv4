# 03 Issues 和项目管理

## 本节你会学到什么

- 理解 Issue 是什么以及它和 Bug 报告的关系
- 在 GitHub 网页上创建、回复、关闭 Issue
- 使用标签（Labels）和里程碑（Milestones）给任务分类
- 在提交信息中引用 Issue 以自动关闭它
- 理解 Issues 如何帮助团队协作

## 一个生活类比：小区物业报修系统

想象你住在一个小区。某天走廊的灯坏了，你走到物业办公室，在报修本子上写：

> "3栋5楼走廊灯不亮，晚上走路很危险。报修人：张三，日期：4月27日"

物业看到后派人修好，然后在你的报修条目上盖一个"已处理"的章。

GitHub 的 **Issue** 就是这个"报修本子"的电子版——只不过除了报修（报 Bug），还可以提建议、讨论新功能、记录待办任务。任何和项目相关的事情，都可以开一个 Issue 来讨论和跟踪。

## Issue 到底是什么

**Issue 不是代码，是讨论。** 每个 Issue 就像一个"话题帖子"，有标题、正文、评论、状态（开启/关闭）。它可以用来：

- **报告 Bug**："点击登录按钮没反应"
- **提功能建议**："能不能加一个暗黑模式？"
- **记录待办任务**："首页的响应式适配还没做"
- **发起讨论**："这个 API 设计成 REST 还是 GraphQL？"

每个 Issue 会自动获得一个编号（如 `#1`、`#42`），这个编号在整个仓库里是唯一的，你可以用它来引用某个 Issue。

## 在 GitHub 上创建 Issue

### 第一步：进入 Issues 页面

打开你 GitHub 仓库的页面，点击顶部的 **Issues** 标签页。

> 如果你看到 "Issues" 标签页不存在，去仓库 Settings 里启用 Issues 功能（默认是开启的）。

### 第二步：点击绿色按钮 "New issue"

你会看到一个编辑页面，类似于写一封邮件：

- **Title（标题）**：用一句话说清楚是什么事。好标题的例子：
  - "登录按钮在 Safari 浏览器上点击无效"
  - "希望增加按日期筛选的功能"
  - 不好的标题："有个问题"、"修一下"

- **Leave a comment（正文）**：详细描述情况。对于 Bug 报告，建议包含：
  1. **发生了什么**（实际结果）
  2. **应该发生什么**（期望结果）
  3. **如何重现**（操作步骤）
  4. **你的环境**（操作系统、浏览器版本等）

正文支持 Markdown 格式——你可以用 `#` 写标题、用反引号 `` ` `` 包裹代码片段、用 `-` 写列表。还可以直接拖拽图片到编辑框来插入截图。

### 第三步：提交

点击绿色按钮 **Submit new issue**，Issue 就创建好了。它现在是"Open"（开启）状态。

## 用标签（Labels）给 Issue 分类

### 类比：邮件文件夹

你收件箱里可能有上百封邮件。如果都堆在一起，找一封特定邮件会非常困难。于是你用标签分类——"工作"、"个人"、"紧急"、"稍后处理"。

GitHub 的 **Labels** 就是给 Issue 贴的彩色标签。常用的标签有：

| 标签 | 含义 | 颜色 |
|------|------|------|
| `bug` | 这是一个 Bug 报告 | 红色 |
| `enhancement` | 这是一个功能建议 | 浅蓝色 |
| `documentation` | 关于文档的改进 | 紫色 |
| `good first issue` | 适合新手参与的任务 | 深绿色 |
| `help wanted` | 需要帮助，欢迎贡献 | 绿色 |
| `question` | 这是一个提问 | 橙色 |

### 怎么贴标签

1. 打开一个 Issue
2. 在右侧边栏找到 **Labels** 并点击齿轮图标
3. 勾选合适的标签，标签会出现在 Issue 标题旁边

你也可以在仓库的 Issues 列表页对多个 Issue 批量打标签——勾选左边的复选框，然后在上方出现的菜单里操作。

### 创建自定义标签

GitHub 的默认标签不够用？你可以创建自己的：

1. 在 Issues 页面点击 **Labels** 按钮
2. 点击 **New label**
3. 填写标签名、描述、选择颜色
4. 点击 **Create label**

例如，你可以创建一个 `优先级-高` 标签，颜色选醒目的红色；再创建一个 `后端` 标签来标记跟后端相关的问题。

## 用里程碑（Milestones）规划版本

### 类比：新房装修的工程进度表

装修师傅不会一口气把所有活干完。通常分为几个阶段：
- **第一期**：水电改造（第1周）
- **第二期**：墙面地面（第2-3周）
- **第三期**：安装家具（第4周）

每个阶段都有明确的目标和截止日期。GitHub 的 **Milestone** 就是项目的"阶段进度表"。

### 创建 Milestone

1. 在 Issues 页面点击 **Milestones**
2. 点击 **New milestone**
3. 填写信息：
   - **Title**：如 "v1.0 正式版"
   - **Due date**：截止日期（可选）
   - **Description**：对这个版本/阶段的描述
4. 点击 **Create milestone**

创建后，你可以把 Issue 分配到某个 Milestone 下——在 Issue 右侧边栏的 **Milestone** 处选择即可。这样你打开 Milestone 页面时，就能看到这个版本下有多少个 Issue、完成了百分之多少。

## 用提交信息自动关闭 Issue

这是最"魔法"的一招——GitHub 能识别你提交信息中的关键词，自动关闭对应的 Issue。

### 类比：报修单的回执

物业修好了走廊的灯，直接在报修本子上你的那条记录旁边写："已修好，4月27日"。你一看就知道了。

在 Git 里，你的提交信息就是那个"回执"。

### 在提交信息中引用 Issue

当你在本地修复了一个 Bug，提交时这样写：

```bash
git commit -m "修复登录按钮在 Safari 上无响应的问题，fixes #3"
```

当这次提交推送到 GitHub 后，GitHub 会自动把 `#3` 这个 Issue 关闭。

### 支持的关键词

GitHub 识别多种关键词来关闭 Issue：

- `fixes #编号`
- `closes #编号`
- `resolves #编号`
- `fix #编号`

大小写都行，中间的 `#` 和编号之间可以有空格也可以没有：

```
fixes #3        # ✓ 可以
Fixes #3        # ✓ 可以（大写）
fix #3          # ✓ 可以（简写）
closes #3       # ✓ 可以
resolves #3     # ✓ 可以
fixes: #3       # ✗ 不可以（多了个冒号）
```

### 也可以只引用不关闭

如果你只是想引用某个 Issue 而不是关闭它，就在提交信息中写 `#编号` 即可（不加关键词）：

```bash
git commit -m "调整了按钮样式，参见 #3 中的讨论"
```

这样 Issue `#3` 里会出现一条引用记录，但不会自动关闭。

### 跨仓库引用

你甚至可以引用其他仓库的 Issue（例如这是组织名下的仓库）：

```
fixes org名/仓库名#3
```

这在大型项目中非常有用——你在 A 仓库提交的代码解决了 B 仓库提出的问题。

## Issue 模板（进阶）

如果你发现团队每次创建的 Issue 内容格式乱七八糟，可以设置 Issue 模板。在仓库中创建一个 `.github/ISSUE_TEMPLATE/bug_report.md` 文件：

```markdown
---
name: Bug 报告
about: 创建一个 Bug 报告帮助我们改进
title: ''
labels: bug
assignees: ''
---

## 问题描述
（清晰描述你遇到的问题）

## 复现步骤
1. 打开...
2. 点击...
3. 看到错误...

## 期望行为
（你本来希望发生什么）

## 截图
（如果有，拖拽到此处）

## 你的环境
- 操作系统：[如 Windows 10]
- 浏览器：[如 Chrome 120]
```

这样以后团队成员创建 Issue 时，可以选择这个模板，内容结构自动就有了。

## 动手试试

1. 打开你上一节推送到 GitHub 的仓库，进入 Issues 页面
2. 创建三个 Issue：
   - 一个 Bug 报告（如 "README 中有个错别字"）
   - 一个功能建议（如 "希望加一个贡献指南"）
   - 一个待办任务（如 "完成首页样式"）
3. 给它们分别打上合适的标签（bug、enhancement 等）
4. 创建一个 Milestone，叫 "v0.1"，把这三个 Issue 都分配进去
5. 查看 Milestone 页面，看看进度条的样子
6. 回到本地，修改 README（假装修了那个 Bug），提交时写 `fixes #编号`
7. 推送后去 GitHub 确认那个 Bug Issue 是否自动关闭了

## 本节小结

Issue 是项目的"报修本"——记录 Bug、建议和待办任务；标签是分类标签；Milestone 是版本计划表；在提交信息里写 `fixes #编号` 可以自动关闭对应的 Issue。
