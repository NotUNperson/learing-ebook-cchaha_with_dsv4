# 01-什么是版本控制

## 本节你会学到什么

- 用自己的话解释"版本控制"是什么
- 理解为什么程序员离不开版本控制
- 用游戏存档的类比记住版本控制的核心概念
- 认识没有版本控制时常见的混乱场景
- 动手模拟一次最原始的"版本控制"

---

## 从一个生活中的场景说起

你有没有遇到过这样的情况：你在写一份重要的文档，改了很多次，桌面上留下了这样的文件：

```
毕业论文.doc
毕业论文_修改版.doc
毕业论文_最终版.doc
毕业论文_最终版2.doc
毕业论文_真的最终版.doc
毕业论文_导师修改后.doc
毕业论文_导师修改后_我改的.doc
```

光是看文件名，你已经分不清哪个是哪个了。如果导师突然说："我还是觉得你第三版那个开头比较好"，你翻遍整个文件夹也找不回来。

这不是你的问题。这是因为**人类在做任何创造性工作的时候，天然就需要"回头看看"和"比较不同版本"的能力**。

## 游戏里的存档系统

玩过 RPG 游戏吗？在打 Boss 之前，你一定会先存个档。为什么？

因为你知道：
- 万一打输了，可以读档重来
- 如果想尝试不同的打法，可以回到存档点分叉
- 如果新装备不如旧装备好用，你还能换回来

**版本控制，就是给代码做的"存档系统"。**

| 游戏存档 | 版本控制 |
|---------|---------|
| 打 Boss 前存档 | 改一个大功能前保存一版 |
| 打输了读档重来 | 代码改坏了回退到之前的版本 |
| 尝试不同打法，从同一存档分叉 | 尝试不同方案，从同一版本分叉（这就是"分支"的雏形） |
| 存档里有当时的所有装备和等级 | 版本里记录了当时所有文件的状态 |

游戏存档让你可以从容地探索地下城，版本控制让你可以从容地探索代码的各种可能性。

## 没有版本控制的日子有多难受

想象你和两个朋友一起写一本小说。你们三个人分别负责不同的章节。

- 你把你的章节发给朋友 A
- 朋友 A 改了几段，发回来
- 同时朋友 B 也在改，发回来另一个版本
- 你打开两个文件，一行一行地找："这里改了啥？那里又改了啥？"
- 更可怕的是，你改了第三章的开头，但忘了告诉朋友 B——他还在根据旧版本写衔接段落

这还没完。当你终于把所有人的修改合并到一个文件里，发现有一段精彩的描写不见了——你根本不知道是哪个"版本"弄丢的，也不知道怎么找回来。

**写作、设计、编程——只要是多人协作、需要反复修改的工作，都会遇到上面的问题。**

版本控制系统就是为了解决这些问题而生的。

## 什么是版本控制（一句话版）

**版本控制就是：系统性地记录文件的每一次修改，让你随时可以查看历史、比较差异、回到过去的任意一个版本。**

- "系统性"意味着不是靠改文件名（`最终版2.doc`）
- "记录每一次修改"意味着你知道谁在什么时候改了什么
- "回到过去"意味着你有一台时光机

## 动手试试

我们来做一个 5 分钟的小实验，感受一下"没有工具的版本控制"是什么感觉。

1. 在桌面新建一个文件夹，叫 `my-story`
2. 在里面新建一个文本文件 `第一章.txt`，写一句话："从前有一座山。"
3. 把这个文件复制一份，重命名为 `第一章_v2.txt`，把内容改成："从前有一座山，山上有一座庙。"
4. 再复制一份，重命名为 `第一章_v3.txt`，把内容改成："从前有一座高山，山巅有一座古庙。"
5. 现在闭上眼（开玩笑的，别闭），回答一个问题：**v2 和 v3 之间，到底改了什么？**

你会发现，你不得不一行一行地对。这就是每次改代码都要手工比较的感觉。几十个文件、几百行代码的时候，这就成了噩梦。

## 本节小结

版本控制就是工作的"存档 + 时光机"，让你在改代码时拥有随时回退、查看历史和安心尝试的底气。

## 下一节预告

了解了版本控制"是什么"，我们来认识当今最流行的版本控制系统——Git。它不是唯一的工具，但它有一个非常特别的故事和设计理念，这直接决定了它是怎么工作的。
