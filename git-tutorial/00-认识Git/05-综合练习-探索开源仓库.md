# 05-综合练习：探索一个开源仓库

## 本节你会学到什么

- 独立完成从克隆到浏览的完整流程
- 初步使用 `git log` 翻阅一个项目的修改历史
- 对比"用 Git 看项目"和"用文件夹看项目"的本质区别
- 建立信心：你能读懂任何一个 Git 仓库的基础信息

---

## 本节目标

这一节不再是"学新概念"，而是一次综合练习。你要像一个开发者一样，找到一个真实的开源项目，把它下载下来，翻看它的历史，感受 Git 在你指尖运转的感觉。

**不要追求全懂。** 你看到的每一条命令、每一段输出，也许现在只理解三成，但没关系。这一节的目的就是让你"摸一摸"——让抽象的概念变成真实的体验。

## 任务一：克隆一个真实的开源项目

这次我们不用样例仓库了。我们来克隆一个真实、活跃、而且对新手友好（代码量不大、结构清晰）的项目。

推荐以下仓库之一（任选一个你感兴趣的）：

- **free-programming-books**：一个收集免费编程书籍链接的项目，文件以 markdown 为主，阅读友好
  ```
  https://github.com/EbookFoundation/free-programming-books.git
  ```
- **the-art-of-command-line**：一本命令行使用技巧的开源书
  ```
  https://github.com/jlevy/the-art-of-command-line.git
  ```
- **gitignore**：收集各种项目类型的 .gitignore 模板
  ```
  https://github.com/github/gitignore.git
  ```

你也可以自己去 https://github.com 随便找一个你感兴趣的项目。GitHub 上有几百万个开源仓库。

选定一个后，在命令行里执行克隆（以 free-programming-books 为例）：

```bash
cd ~/git-practice
# 回到我们之前创建的练习文件夹

git clone https://github.com/EbookFoundation/free-programming-books.git
# 克隆这个开源仓库
```

等待下载完成。这个仓库大约几十 MB，可能需要一到两分钟，取决于网速。

## 任务二：浏览项目结构

进入克隆下来的文件夹，看看里面有什么：

```bash
cd free-programming-books
# 进入仓库目录（如果你选了别的仓库，改成对应的文件夹名）

dir
# 列出文件。Windows 用 dir，Mac/Linux 用户用 ls
```

你会看到这个项目里的文件列表。有可能是一堆 markdown 文件（`.md` 后缀），也可能是各种编程语言的源代码文件。这就是成千上万人协作贡献的一个知识库。

**你现在看到的就是这个项目的"最新状态"。** 如果没有 Git，你只能看到这些——一个静止的快照。

## 任务三：翻阅历史记录

这才是 Git 真正发挥作用的地方。输入：

```bash
git log
# 查看完整的提交历史
```

你会看到类似这样的输出开始滚动：

```
commit a1b2c3d4e5f6... （一大串看起来像乱码的东西）
Author: 某人 <某人的邮箱>
Date:   某年某月某日

    修改了什么什么

commit b2c3d4e5f6a7...
Author: 另一个人 <另一个邮箱>
Date:   更早的某天

    添加了什么什么
```

**这些是什么意思？**

- `commit` 后面那串长长的英数字组合：这个版本的"身份证号"。每个版本都有一个全球唯一的编号，叫"提交哈希"（hash）。你可以把它理解为：Git 用一套数学算法，根据这个版本的内容算出了一个独一无二的指纹
- `Author`：谁做了这次修改
- `Date`：什么时候改的
- 空行下面的那段文字：**提交说明**（commit message）——修改者写的，解释这次改了什么、为什么改

如果历史很长，`git log` 会进入翻页模式（你看到底部有个冒号或者 `END` 字样）。用方向键上下翻，看够了按 `q` 退出。

如果历史太长看不过来，用简洁模式：

```bash
git log --oneline -20
# --oneline：每条记录只显示一行（只有哈希号前几位 + 提交说明）
# -20：只显示最近 20 条
```

你会看到类似这样的输出：

```
a1b2c3d 修复了搜索功能的 bug
e4f5g6h 添加了 Rust 书籍的链接
i7j8k9l 更新 README 文件
...
```

**现在你正在翻阅这个项目的"时间线"。** 从项目诞生到现在，每一个改动、每一次增删、每一位贡献者的名字，都在这里。

## 任务四：查看某次具体改了什么

从 `git log --oneline` 的输出里随便挑一行，复制它开头的那串短编号（比如 `a1b2c3d`），然后执行：

```bash
git show a1b2c3d
# git show 后面跟一个提交编号，会展示那次修改的详细内容
# 把 a1b2c3d 换成你复制的那串编号
```

你会看到：
- 作者、日期
- 提交说明
- **具体改了哪些文件的哪些行**（绿色开头带 `+` 的行是新增的，红色开头带 `-` 的行是删除的）

这个功能的意思是：**任何一次历史修改，Git 都能精确地告诉你改了哪里。** 想象一下，如果没有 Git，有人问你"三个月前是谁把那个 bug 引入的"，你只能对着文件夹茫然无措。

按 `q` 可以退出查看。

## 任务五：对比"用 Git"和"不用 Git"的区别

趁热打铁，我们做一个小对比。

1. 在文件资源管理器里，打开你克隆下来的这个文件夹（比如 `free-programming-books`）
2. 浏览一下里面的文件——这就是不用 Git 的人看到的全部东西：一堆文件
3. 回到命令行，你可以做这些文件资源管理器做不到的事：
   - 翻看历史（`git log --oneline`）
   - 查看某次修改的细节（`git show <编号>`）
   - 知道每个文件是谁、什么时候、为什么加进来的
4. 感受到了吗？Git 不是给你"一份文件"，而是给你"这个项目的完整传记"

## 动手试试

这一整节都是动手练习。作为总结，请完成以下检查清单：

- [ ] 我成功克隆了一个开源仓库到本地
- [ ] 我用 `dir`（或 `ls`）浏览了它的文件
- [ ] 我用 `git log --oneline` 看了它的修改历史
- [ ] 我用 `git show <编号>` 查看了某一次修改的详细内容
- [ ] 我理解了：文件夹里看到的只是"最新一集"，Git 里有整个"连续剧"

如果有任何一步卡住了，回到命令行，仔细看看报错信息——它通常已经告诉你怎么解决。最常见的错误是路径不对（你不在仓库文件夹里），用 `cd` 切进去就好。

## 关于这个模块的结语

现在你已经：
- 理解了版本控制是什么（游戏存档）
- 知道了 Git 的分布式哲学（每人一本完整相册）
- 在电脑上安装和配置好了 Git
- 亲手克隆了远程仓库，翻看了项目的历史

你有没有发现，到目前为止我们还没有创建过一个"自己的版本"？没错——**在学走路之前，我们先把地图看了一遍。**

从下一个模块开始，你将学习 Git 最核心的操作：初始化仓库、提交修改、查看状态。那才是你真正开始"用 Git 管理自己的项目"的时刻。

但在那之前，你已经比很多"会用 Git 但是不知道为什么这么用"的人站得更稳了——因为你从根本上理解了 Git 是什么以及它为什么被设计成这样。

## 本节小结

你亲手触摸了一个真实开源仓库的时间线。Git 从一个抽象概念变成了一个真实的、能翻阅历史、能看到每一次修改细节的工具——现在你真正"看见"Git 了。
