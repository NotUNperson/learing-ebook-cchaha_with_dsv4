## 正文

### 冲突是什么：两个人改了同一句话

想象你和朋友合作写一篇作文。你们各自拿了一份原始稿回去修改。第二天碰头一看：

- 你把第一段的"天气很好"改成了"阳光灿烂"
- 你朋友把第一段的"天气很好"改成了"乌云密布"

现在两份稿子放在一起，**同一句话被两个人改成了不同的样子**。该听谁的？这就是冲突。

Git 处理合并时非常智能：如果两个人改了不同的文件，或者改了同一个文件的不同位置，Git 都能自动合并，不需要你操心。**只有一种情况 Git 搞不定——两个人修改了同一个文件的同一行。** 这时 Git 不知道该保留谁的版本，于是它停下来，把这个决定权交给你。

### 冲突发生时的场景

```
你在 main 分支上，把 config.txt 的第一行改成了：
  主题颜色 = 蓝色

小明的 feature 分支上，也把 config.txt 的第一行改成了：
  主题颜色 = 红色

现在你把 feature 合并到 main：
  git merge feature

Git 输出：
  CONFLICT (content): Merge conflict in config.txt
  Automatic merge failed; fix conflicts and then commit the result.
```

Git 在说："config.txt 里有冲突，我不知道该留'蓝色'还是'红色'。你自己来决定，决定好了再提交。"

### 冲突标记长什么样

打开有冲突的文件，你会看到这样的标记：

```
主题颜色 = <<<<<<<< HEAD
蓝色
========
红色
>>>>>>>> feature
```

这些标记的含义：

| 标记 | 含义 |
|------|------|
| `<<<<<<< HEAD` | "从这里开始是 HEAD 的版本"（即你当前分支的版本） |
| `=======` | "上面是 HEAD 版本，下面是合并进来的版本"（分隔线） |
| `>>>>>>> feature` | "feature 分支的版本到这里结束" |

所以这张图翻译成人话就是：

- HEAD（你当前所在的分支，这里是 main）说：主题颜色 = 蓝色
- feature 分支说：主题颜色 = 红色
- 你来决定：保留哪个？还是两个都要？还是重新写一个？

### 如何手动解决冲突

解决冲突分四步。我们用一个完整的实战场景来演示。

**场景**：你和同事都修改了 `README.md` 的"作者"行。

#### 第一步：查看哪些文件有冲突

```bash
git status
```

输出会显示有冲突的文件，标记为 `both modified`。

#### 第二步：打开文件，找到冲突标记

用任意编辑器打开 `README.md`，找到 `<<<<<<<`、`=======`、`>>>>>>>` 这些标记。

#### 第三步：决定最终内容，删除标记

你要做的就是把冲突区域**编辑成你想要的最终样子**，然后删除所有 Git 标记。

比如，你决定两个名字都保留：

```
# 删除所有标记后的最终内容：
作者：张三、李四
```

注意：**必须删除所有冲突标记**（`<<<<<<<`、`=======`、`>>>>>>>`），一个都不能留。留着的话 Git 不会让你提交。

#### 第四步：标记为已解决并提交

```bash
# 把修改好的文件加入暂存区
git add README.md

# 提交这次合并
git commit -m "合并 feature 分支，解决了 README.md 的作者冲突"
```

`git add` 这个动作在告诉 Git："我搞定了这个文件，可以继续了。" 如果有多个文件有冲突，每个都要修改并 `git add`，最后一次性 `git commit`。

### 如果解决到一半想放弃

有时候冲突比较复杂，你暂时不想处理，或者处理错了想重来。可以用以下命令放弃合并：

```bash
git merge --abort
```

这条命令会把所有改动回退到合并之前的状态，就像合并从没发生过一样。**这是安全的后悔药**。

### 两种常见的冲突解决策略

| 策略 | 做法 | 适用场景 |
|------|------|----------|
| 保留一方的版本 | 删除另一边的内容 | 明显有一方的改动是错的或过时的 |
| 两边都保留（手动整合） | 把两边的内容都留下来，重新组织 | 两边的改动都有道理，需要融合 |

### 减少冲突的小技巧

1. **频繁合并**：不要把分支晾太久才合并。分支越久不合并，差异越大，冲突越多。
2. **小步提交**：每次提交改动的范围越小，冲突的可能性越小。
3. **沟通**：和队友约定好谁负责哪块代码，尽量避免两个人同时改同一个文件的同一区域。

## 动手试试

创建一个必然会冲突的场景（5 分钟）：

1. 初始化一个新仓库，创建 `config.txt`，写入 `主题=默认`，提交
2. 创建 `feature-blue` 分支，把 `config.txt` 的内容改为 `主题=蓝色`，提交
3. 切换回 `main`（或原来的分支），把 `config.txt` 的内容改为 `主题=红色`，提交
4. 在 `main` 上运行 `git merge feature-blue`
5. 观察 Git 报告冲突
6. 用 `cat config.txt` 查看冲突标记
7. 手动编辑文件解决冲突（比如改成 `主题=紫色`）
8. `git add config.txt` 然后 `git commit`
9. 用 `git log --oneline --graph --all` 查看合并结果
10. （额外挑战）重复步骤 2-5，然后尝试 `git merge --abort` 放弃合并

## 本节小结

冲突是 Git 在说"我不知道该听谁的"——你只需要打开文件，删除标记，决定最终内容，然后提交即可。
