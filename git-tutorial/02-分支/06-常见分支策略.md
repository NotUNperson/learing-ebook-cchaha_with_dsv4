## 正文

### 为什么要有一套"分支策略"

前面几节你学会了分支的创建、切换、合并、解决冲突。这些是"操作技能"——你知道了每个按钮怎么按。

但在真实项目中，只学会按按钮还不够。你还得知道**什么时候该创建分支、什么时候该合并、分支该怎么命名**。就像你学会了开车（操作技能），还需要交通规则（策略）才能安全上路。

**分支策略就是团队协作的"交通规则"**——它规定了谁在什么时候创建什么分支，以及分支之间如何整合。

目前业界最流行的两套策略是 **Git Flow** 和 **GitHub Flow**。它们解决的问题不同，没有谁更好，只有谁更适合你的项目。

> **类比**：Git Flow 像是大型交响乐团的指挥体系——有第一小提琴、第二小提琴、中提琴、大提琴等等，分工明确，流程严格。GitHub Flow 像是爵士乐小乐队——三四个人，谁有灵感谁先即兴，试好了就合进来，灵活轻快。

---

### Git Flow：为"有计划发布周期"的项目设计

Git Flow 由 Vincent Driessen 在 2010 年提出，它的核心思想是：**项目有两种"节奏"——日常开发和准备发布**。

#### 分支结构

Git Flow 定义了 5 种分支，各有明确的职责：

```
                        feature-A
                       /
                      /  (开发完成后合并回 develop)
                     /
main  ---- ---- ---- ---- ---- ----
       \                          \
        \                          \
         develop ---- ---- ----    \  release/1.0
                 \         \  \     \
                  \         \  \     (bug 修复合并回 develop 和 main)
              feature-B   (日常开发)
```

#### 各分支的职责

**1. main 分支（主分支）**

- 永远处于"可发布"状态
- 每一个提交都对应一个可以上线的版本
- 除了从 release 或 hotfix 合入，不直接在上面修改
- 类比：商店货架上陈列的商品——必须是成品

**2. develop 分支（开发分支）**

- 日常开发的主战场
- 所有功能分支都从这里分出去，完成后合并回来
- 类比：工厂车间——在这里制造、组装、测试

**3. feature 分支（功能分支）**

- 从 develop 分出，开发完后合并回 develop
- 命名示例：`feature/login`、`feature/search`
- 生命周期：几天到几周不等
- 类比：工厂里的一个独立工位——只做一件事，做完就撤

**4. release 分支（发布分支）**

- 从 develop 分出，用于准备一个新版本的发布
- 只做最后的测试、修 Bug、写文档，不添加新功能
- 完成后合并到 main（发布）和 develop（把修复的 Bug 同步回来）
- 命名示例：`release/1.0.0`
- 类比：质检线——做最后的检查和包装，然后出厂

**5. hotfix 分支（紧急修复分支）**

- 从 main 分出，修复线上紧急 Bug
- 完成后合并到 main（上线修复）和 develop（把修复同步回开发线）
- 命名示例：`hotfix/login-crash`
- 类比：售后维修——已经卖出的商品出了问题，紧急召回修理

#### Git Flow 的完整工作流程

一个新功能从想法到上线的旅程：

```
1. 从 develop 分出 feature/login
2. 在 feature/login 上开发登录功能
3. 开发完成，合并回 develop
4. 删除 feature/login（它已经完成了使命）

5. 积累了足够多的功能后，从 develop 分出 release/1.0.0
6. 在 release/1.0.0 上做最后的测试和 Bug 修复
7. 合并 release/1.0.0 到 main（正式发布）
8. 同时在 main 上打一个版本标签：v1.0.0
9. 把 release/1.0.0 的改动也合并回 develop（Bug 修复要同步回来）
10. 删除 release/1.0.0

紧急情况：
11. 线上出了登录崩溃的 Bug
12. 从 main 分出 hotfix/login-crash
13. 修复 Bug
14. 合并到 main（紧急上线）和 develop（同步修复）
15. 删除 hotfix/login-crash
```

**适用场景**：
- 项目有明确的版本号和发布周期（如：每两周发一个版本）
- 团队较大（5 人以上），需要严格的分工
- 有专门的测试环境和测试阶段
- 同时维护多个线上版本（如：v1.0 还在维护，v2.0 已在开发）

### GitHub Flow：为"持续部署"设计

GitHub Flow 是 GitHub 团队在实践中总结出来的简化策略，它的核心思想是：**main 分支永远可部署，任何改动都通过分支 + Pull Request 来做**。

#### 分支结构

```
main ---- ---- ---- ---- ----
   \           \           \
    feature-A   feature-B   feature-C
    (短命分支)  (几天内合并) (合并后删掉)
```

GitHub Flow 只有一种长期分支：`main`。其他都是临时的功能/修复分支。

#### 核心规则

**1. main 分支永远可部署**

- 任何合并到 main 的代码都必须经过测试，可以直接部署到生产环境
- 不允许多人直接向 main 提交——所有改动都通过 Pull Request（PR）
- 类比：main 就像发射台上的火箭——随时可以点火升空

**2. 从 main 创建描述性的分支**

- 命名要有意义：`fix-login-bug`、`add-search-bar`
- 分支从 main 的最新提交分出来
- 类比：从主路上开一条临时施工便道

**3. 持续推送并尽早创建 Pull Request**

- 即使代码还没写完，也可以创建一个 Draft PR（草稿 PR），让团队看到你在做什么
- 这鼓励了透明协作和早期反馈
- 类比：在公共白板上画出你的设计草图，同事们路过就能给建议

**4. 合并前需要代码审查（Code Review）**

- 至少有一位同事审查你的代码后才能合并
- 这不仅是为了发现 Bug，也是知识共享——让更多人了解这段代码
- 类比：重要文件需要第二个人过目，就像飞行员起飞前的"双人核对清单"

**5. 合并后立即部署**

- 合并到 main 后，自动或手动部署到生产环境
- "部署"不一定是"上线给用户看"，也可以是"部署到测试服务器"

**6. 合并后删除分支**

- 功能分支的使命完成了，及时清理
- 保持仓库整洁

#### GitHub Flow 的完整工作流程

```
1. 从 main 分出 add-search-bar
2. 在 add-search-bar 上开发和提交（可以多次提交）
3. 把 add-search-bar 推送到远端仓库
4. 创建一个 Pull Request，请求把 add-search-bar 合并到 main
5. 同事审查代码，提出修改意见
6. 根据意见修改，提交到 add-search-bar（PR 会自动更新）
7. 所有检查通过（测试通过、审查通过）
8. 合并 add-search-bar 到 main
9. 部署 main 到生产环境
10. 删除 add-search-bar 分支
```

**适用场景**：
- 持续部署项目（一天可能发布多次）
- 团队较小到中等（2-10 人）
- Web 应用、SaaS 服务（用户总是使用最新版本）
- 没有"多版本并行维护"的需求

### Git Flow vs GitHub Flow：一张表说清楚

| 对比维度 | Git Flow | GitHub Flow |
|----------|----------|-------------|
| 长期分支数 | 2 个（main + develop） | 1 个（main） |
| 发布方式 | 通过 release 分支 | 从 main 直接部署 |
| 版本管理 | 有明确的版本号 | 通常不需要版本号 |
| 适合团队 | 较大，分工明确 | 较小，沟通密切 |
| 适合项目 | 有固定发布周期的软件（App、硬件） | Web 服务、持续部署项目 |
| 分支生命周期 | 功能分支可能存活数周 | 功能分支尽量在 1-3 天内合并 |
| 学习成本 | 较高，需要理解 5 种分支 | 较低，流程简单 |
| 核心哲学 | "谨慎发布" | "快速迭代" |

### 你应该选哪个？

**选择 Git Flow 的信号**：
- 你的产品有明确的版本号（如 v1.0、v2.0）
- 你需要同时维护多个线上版本
- 你有严格的测试和发布流程
- 你的团队有专门的 QA（测试）人员

**选择 GitHub Flow 的信号**：
- 你做一个网站或 Web 服务，用户总是访问最新版
- 你每天或每几天就发布一次
- 你是一个人开发，或者只有 2-3 人的小团队
- 你希望流程越简单越好

**选哪个不重要，选了并坚持才重要**。很多团队也不是严格照搬某一套，而是取其思想、做适当简化。

## 本节小结

Git Flow 像交响乐团（分工精细、流程严格），GitHub Flow 像爵士乐队（灵活轻巧、快速迭代）——选对"交通规则"比选对工具更重要。
