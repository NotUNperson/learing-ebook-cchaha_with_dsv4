## 正文

### 用"搬家"类比理解 rebase

你已经学会了 `git merge`——把两个分支的内容"缝合"在一起，产生一条分叉再汇合的历史。

但有时候，你希望历史看起来**更干净**——就像你的代码从来没有分过叉一样，所有提交都是一条直线。

这时候就轮到 `git rebase`（变基）了。

看一个简单的例子。假设你有这样的分支结构：

```
A --- B --- C --- D  (main)
       \
        E --- F  (feature)
```

`feature` 分支是从 `B` 开始分叉出来的。现在 `main` 往前走了两步到了 `D`。

用 `git merge` 的话，结果会多出一个合并提交：

```
A --- B --- C --- D --- M  (main)
       \               /
        E --- F ------/
```

如果你用 `git rebase`，思路完全不同：**把 `feature` 分支的"基座"从 B 搬到 D**，也就是让 E 和 F "嫁接到" D 的后面。

```
变基前：
A --- B --- C --- D  (main)
       \
        E --- F  (feature)

变基后：
A --- B --- C --- D  (main)
                    \
                     E' --- F'  (feature)
```

注意 E' 和 F' 不是原来的 E 和 F——它们是"重新生成"的提交，内容相同但提交 ID 不同。因为提交的"父母"变了（原来是 B，现在变成了 D），所以哈希 ID 必须重新计算。

> **类比一（搬家）**：你在上海租了一套房子，地址是"XX路1号"。后来你搬到了北京，把家具原封不动搬过去，但地址变成了"YY路2号"。东西还是那些东西，但"基座"（地址）变了。
>
> **类比二（嫁接）**：你把一棵树的枝条剪下来，嫁接到另一棵树的树干上。枝条还是那根枝条（改动内容不变），但它现在长在另一棵树上了（基座变了）。

### rebase 的基本操作

假设你在 `feature` 分支上，想把 `main` 的最新改动拿过来：

```bash
# 第一步：确保在 feature 分支上（你要"搬"的那个分支）
git switch feature

# 第二步：变基到 main
git rebase main
```

这行命令的语义是："把我当前分支（feature）上的提交，**重放到** main 分支的最新提交之后"。

工作过程如下：

1. Git 找到 feature 和 main 的共同祖先（B）
2. Git 把 feature 上从共同祖先之后的所有提交（E 和 F）"暂存"起来
3. Git 把 feature 的指针移到 main 的最新提交（D）
4. Git 把暂存的提交（E 和 F）一个一个地"重放"到 D 之后
5. 如果一切顺利，得到一条直线历史

### 一次完整的变基示例

假设初始状态：

```
A --- B --- C (main)
       \
        D --- E (feature)
```

执行：

```bash
git switch feature
git rebase main
```

结果：

```
A --- B --- C (main)
              \
               D' --- E' (feature)
              (变基后的新提交)
```

现在如果你想合并 feature 到 main，就变成了一个简单的**快进合并**：

```bash
git switch main
git merge feature
```

结果是一条完全直线的历史：

```
A --- B --- C --- D' --- E' (main, feature)
```

对比 merge 的分叉历史，rebase 的历史看起来更整洁。

### rebase vs merge：什么时候用哪个

| 维度 | git merge | git rebase |
|------|-----------|------------|
| 历史记录 | 保留完整的分叉和合并轨迹 | 历史变成一条直线 |
| 操作复杂度 | 简单，一条命令 | 稍复杂，需要处理潜在冲突 |
| 冲突解决 | 只在合并时解决一次 | 可能需要多次解决（每个提交重放时都可能冲突） |
| 安全性 | 非常安全，不改变已有提交 | 会重写提交历史（已有的提交ID会变） |
| 适用场景 | 公共分支（多人共享的分支） | 个人分支（你自己在用的分支） |

**简单原则**：
- 你想把自己的改动"贴"到主线最新版本的末尾，用 rebase
- 你想把两个人的工作成果"缝合"起来，用 merge
- 如果分支已经推送到远程并且有其他人基于它工作，**不要 rebase**

### rebase 的黄金法则

**永远不要对已经推送到公共仓库的提交执行 rebase。**

为什么？因为 rebase 会改变提交的 ID（哈希值）。如果你 rebase 了一个已经推送的分支，其他人的仓库里的提交 ID 还是旧的，这会导致严重的不一致——就像你偷偷改了历史书的页码，别人翻书时翻到了完全不同的内容。

> **类比**：merge 像是给历史书写"附录"——原章节不动，追加新内容。rebase 像是"改写"——把章节重新编排，内容不变但页码全变了。如果书已经出版（推送到远端），你再改写会让已经买了书的人困惑。但如果你还没出版（只在本地），随便怎么改都行。

### 变基时遇到冲突

rebase 过程中每个提交被"重放"时都可能遇到冲突。处理方式类似 merge：

```bash
# rebase 过程中遇到冲突，Git 会暂停
# 手动解决冲突文件后：
git add <解决了冲突的文件>
git rebase --continue   # 继续 rebase

# 如果想放弃 rebase：
git rebase --abort      # 回到 rebase 之前的状态
```

## 动手试试

1. 创建一个仓库，在 `main` 上做两次提交（A 和 B）
2. 创建 `feature` 分支，在上面做两次提交（C 和 D）
3. 回到 `main`，做一次新提交（E）——模拟 main 有新进展
4. 切到 `feature`，运行 `git rebase main`
5. 用 `git log --oneline --graph --all` 观察结果——看看历史是否变成了一条直线？（注意，main 还在 E，feature 在 E 之后）
6. 切到 `main`，运行 `git merge feature`——这次合并应该是快进合并，因为 feature 已经在 main 的"下游"了
7. 再用 `git log --oneline --graph --all` 看看完整的直线历史

---

## 本节小结

rebase 就是"把你的提交搬到别人最新提交的末尾"——让历史变成一条直线，代价是会重写提交 ID。
