# 01 什么是分支

## 本节你会学到什么

- 理解"分支"到底是什么，以及它为什么存在
- 用日常生活中的类比理解"分叉"和"合并"
- 看懂分支的 ASCII 结构图
- 了解分支在实际开发中的基本应用场景

---

## 正文

### 从一个游戏说起

想象你在玩一个角色扮演游戏（RPG）。你的主角在村庄里接到一个主线任务：去山洞里打败巨龙。但是，在出发之前，路边一个村民告诉你："勇敢的冒险者，村外的森林里有一只捣乱的巨鼠首领，你能帮我解决它吗？"

这是个支线任务。你有两个选择：

1. **拒绝支线，只做主线**——直奔山洞打巨龙。
2. **接下支线，先打巨鼠，再打巨龙**——但等你打完巨鼠回来，发现主线任务的难度可能变高了，或者你的等级提升了，打巨龙更容易了。

现在，把这个场景换成写代码：

**主线任务 = 你的"主分支"（main/master 分支）**
**支线任务 = 你创建的"功能分支"（feature branch）**

你在主分支上正在开发网站的登录功能。突然领导说："能不能先加一个搜索框？很简单，两小时搞定。"你不想暂停登录功能的开发，于是你"分出一条支线"——创建一个新分支，在这个分支里写搜索框。写完测试没问题了，再把这个支线内容"合并"回主分支。

这就是 Git 分支的核心思想：**在不影响主线的情况下，开辟一个独立空间去做新的事情**。

### 分支的本质：一个"时间线分叉"

想象你写日记。每天晚上你写下当天发生的事情：

```
周一 -- 周二 -- 周三 -- 周四
```

这是你的"主时间线"。到了周三晚上，你突然想："如果周三我选了另一条路会怎样？"于是你从周三那一页开始，另写一个"IF线"版本：

```
周一 -- 周二 -- 周三 -- 周四（原版）
                   \
                    ---- 周四（IF线版）-- 周五（IF线版）
```

这就是分支！**分支就是代码历史的"分叉点"**。两个分支可以各自往前走，互不影响。等你想把 IF 线的内容合并回主时间线时，也可以做到。

### 为什么需要分支

**场景一：同时做多个功能**

你和小明一起开发一个网站。你要做登录功能，小明要做注册功能。如果两个人都在同一份文件上改，你的代码会覆盖小明的，小明的也会覆盖你的——一团糟。

有了分支：你开一个 `login` 分支，小明开一个 `register` 分支，各写各的，互不干扰。写完再合到一起。

**场景二：实验新想法**

你想给网站加一个很酷的动画效果，但不确定好不好用，也不确定会不会把现有功能搞坏。在主分支上直接改？万一搞坏了，整个项目都完蛋。

有了分支：你创建一个 `experiment` 分支，随便折腾。如果效果好，合并回主分支；如果效果差，删掉分支就行，主分支毫发无损。

**场景三：修复紧急 Bug**

你的网站突然出了个严重 Bug，必须立刻修复。但你正在主分支上开发一个新功能，代码只写了一半，不能提交（因为提交了会引入不完整的代码）。

有了分支：你创建一个 `hotfix` 分支修复 Bug，修好合并回主分支。然后回到你的功能分支继续写代码，后续再把 Bug 修复的代码合并到你的功能分支里。

### 用 ASCII 图看懂分支

假设你有一个主分支 `main`，上面有三次提交：

```
A --- B --- C  (main)
```

现在你创建一个新分支叫 `feature`：

```
A --- B --- C  (main, feature)
```

注意，刚创建新分支时，`main` 和 `feature` 指向同一个位置 C。它们还没有"分叉"。

接下来，你在 `feature` 分支上做一次新提交 D：

```
A --- B --- C  (main)
              \
               D  (feature)
```

同时，你在 `main` 上也做了一次新提交 E：

```
A --- B --- C --- E  (main)
              \
               D  (feature)
```

现在两条线分叉了！`main` 有 C 和 E，`feature` 有 C 和 D，它们从 C 之后分别走上了不同的路。

后面我们可以把 D 的内容合并回 `main`，这就是分支的"合流"：

```
A --- B --- C --- E --- M  (main)
              \         /
               D ------/
```

这里的 M 是一个"合并提交"，它把两条线的历史"缝合"在了一起。

### 分支在 Git 中的真实身份

前面的类比帮你建立了直觉。现在来看看 Git 中的分支到底是什么——其实非常简单：

**分支就是一个指向某个提交的"指针"**，它只是一个标签，贴在某个提交上。

就像书签。你读一本书，读到第 50 页，放了一个书签。下次打开书，直接翻到书签的位置继续读。分支就是一个可以移动的书签，你每做一次提交，书签就自动往后翻一页。

> 回顾：在"01-本地版本管理"中你学过，每次 `git commit` 会生成一个提交（commit），每个提交有一个唯一的哈希 ID（如 `a1b2c3d`）。分支就是这个哈希 ID 的"别名"——你不需要记住 `a1b2c3d`，只需要记住分支名 `main` 或 `feature`。

## 动手试试

1. 打开终端，找一个你之前练习过的 Git 仓库（或者新建一个），运行 `git branch`。你应该会看到 `* main` 或 `* master`——星号标记的是你当前所在的分支。
2. 运行 `git log --oneline --graph --all`。这个命令会用 ASCII 图展示整个仓库的提交历史——虽然现在可能只有一条直线，但后面学完分支再回来看这个图会非常直观。

---

## 本节小结

分支就是代码历史的"分叉点"，让你能在不影响主线的情况下独立开发新功能或做实验。
