## 正文

### 合并就像拼接故事线

还记得上一节的分叉图吗？你从同一个起点出发，在 `main` 和 `feature` 上分别写了不同的代码。现在，功能开发完了，你想把 `feature` 的内容"搬"回 `main`。

这个过程就叫**合并（merge）**。

用写小说的类比：

```
主线故事：第一章 -- 第二章 -- 第三章（主角出发冒险）
                               \
支线故事：                       第四章A（主角遇到精灵）-- 第五章A（获得魔法剑）

现在，你想把支线故事的内容放进主线。于是你得到：

主线故事：第一章 -- 第二章 -- 第三章 -- 合并章（精灵 + 魔法剑的故事被整合进来了）
```

Git 的合并做的就是这样的事——把两条线的改动"缝合"到一起。

### 合并的基本操作

合并的步骤很简单：

```bash
# 第一步：切换到你想"接收"改动的分支（目标分支）
git switch main

# 第二步：把另一个分支合并进来
git merge feature
```

这等价于："把 `feature` 分支的内容合并到 `main` 里"。

记住这个口诀：**先跳到目标分支，再把源分支合进来**。就像你要把一杯水倒进一个杯子里——先拿稳被倒的杯子（目标分支），再倒水（源分支）。

### 两种合并方式

Git 的合并分两种情况：**快进合并**和**三路合并**。用两张图来说清楚。

#### 情况一：快进合并（Fast-forward Merge）

场景：你从 `main` 分出 `feature` 之后，`main` 没有任何新提交，只有 `feature` 在往前走。

```
合并前：
A --- B --- C  (main)
              \
               D --- E  (feature)

合并后：
A --- B --- C --- D --- E  (main, feature)
```

因为 `main` 没有"自己走一步"，所以 Git 只需要把 `main` 的指针"快进"到 `feature` 的位置就行——没有冲突，也不需要创建新的提交。这就像你看视频时按快进键，直接跳到最新位置。

#### 情况二：三路合并（Three-way Merge）

场景：从 `main` 分出 `feature` 之后，**两个分支都有新提交**。

```
合并前：
A --- B --- C --- F --- G  (main)
              \
               D --- E  (feature)

合并后：
A --- B --- C --- F --- G --- M  (main)
              \               /
               D --- E ------/
```

这种情况下，Git 需要把两边各自的改动"缝合"起来。Git 会找到三个关键节点：

1. **共同祖先**（C）：两条分支分叉之前的最后一个共同提交
2. **main 的顶端**（G）：主线的最新状态
3. **feature 的顶端**（E）：支线的最新状态

Git 比较这三者，计算出最终的合并结果，并自动创建一个新的**合并提交 M**（merge commit）。

> **类比**：这就像两个人同时编辑同一篇文档的不同部分。A 改了开头，B 改了结尾。合并时，系统找到原始文档（共同祖先），比较 A 和 B 各自改了哪里，然后把两边的修改整合到一起。

### 合并不会删除源分支

注意：合并完成后，`feature` 分支**仍然存在**。`git merge` 只是把它的内容"复制"到了 `main`，并不会删除它。如果功能已经完成，你可以手动删除它：

```bash
git branch -d feature
```

### 查看合并历史

合并后，用图形模式查看历史特别直观：

```bash
git log --oneline --graph --all
```

你会看到分支线的汇合点，清楚地展示了代码历史的"分分合合"。

### 撤销一次合并

如果合并之后发现有问题，可以撤销（前提是还没有推送到远程）：

```bash
# 撤销最近一次合并，回到合并前的状态
git reset --hard HEAD~1
```

> `HEAD~1` 表示"HEAD 的前一个位置"，也就是合并之前的状态。这个在"01-本地版本管理"的撤销章节学过。

## 动手试试

1. 创建一个新仓库，在 `main` 上提交两次
2. 创建一个 `feature` 分支，在上面新建一个文件并提交
3. 切换回 `main`，运行 `git merge feature`，观察 Git 报告了什么（应该会看到 "Fast-forward" 字样）
4. 再做一次实验：这次在 `main` 上也做一个新提交（模拟两边都有改动），然后再合并另一个分支，观察是否出现了"Merge made by the 'ort' strategy" 的提示和一个自动生成的合并提交
5. 运行 `git log --oneline --graph --all` 观察合并后的图形

---

## 本节小结

`git merge` 把两个分支的改动"缝合"到一起；如果目标分支没有新提交就是"快进"，如果有新提交就需要"三路合并"。
