# 03 常见问题排查

## 本节你会学到什么

- 识别日常开发中最容易踩的五个坑，以及各自的"逃生路线"
- 学会在"忘记 add 就 commit"后快速补救
- 掌握"commit 到错误分支"的两种抢救方法
- 理解合并冲突为什么不可怕，以及一步步解决它的正确姿势
- 知道 push 被拒绝时的标准处理流程（pull --rebase vs merge）

---

## 正文

### 先给你一颗定心丸

学 Git 的人，99% 都遇到过以下问题。你不是一个人。而且好消息是：**Git 几乎从来不丢数据。** 只要你没有手动删除 `.git` 目录，你做的每一步操作 Git 都记着。所谓的"搞坏了"通常只是"看起来不对"，其实有办法恢复。

**类比：** Git 就像一个有详细存档功能的文字处理软件——你误操作之后不是文件毁了，而是界面看起来跟你预想的不一样。找到正确的"撤销"按钮就行。

下面这五个问题，按"你能遇到的概率"从高到低排序。

---

### 问题一：忘记 add 就 commit 了

**症状：** 你修改了一个文件，然后直接 `git commit -m "xxx"`。提交后发现这个文件的改动根本没包含在这次提交里。

**原因：** commit 只提交暂存区里的内容。你修改了文件但没有 `git add`，所以暂存区没变，commit 当然提交了个空。

**补救方案（推荐）：**

```bash
# 补上 add，然后用 --amend 把遗漏的文件"追加"到上一次提交里
git add 你忘记的文件
git commit --amend --no-edit
```

`--amend` 的意思是"修改上一次提交"。`--no-edit` 的意思是提交信息不用改。执行后，你忘记的文件就悄悄加进了上一次提交，就像它本来就在里面一样。

**类比：** 就像你寄出了一封信，突然想起有张照片忘了塞进信封。你赶紧追上邮递员（`--amend`），把照片塞进去，邮戳不变，外面的信封还是原来那个。

**注意：** 如果你上一次提交已经 push 到远程仓库了，不要用 `--amend`（原因见问题四）。

---

### 问题二：commit 到错误分支了

**症状：** 你本应该在新分支 `feature/login` 上开发，结果稀里糊涂地在 `main` 分支上写了代码还提交了。现在 main 上多了不该有的提交。

**补救方案 A（还没 push）：**

```bash
# 1. 先创建正确的分支（指向当前这个"错误"的提交）
git branch feature/login

# 2. 把 main 分支回退到"犯错之前"
git reset --hard HEAD~1
```

解释：`git branch feature/login` 在当前提交上创建了一个新的分支标签，相当于在"案发现场"插了一面旗子，"这以后的提交属于这个分支"。然后 `git reset --hard HEAD~1` 让 main 分支退回到上一个提交，就像什么都没发生过。

**补救方案 B（用 cherry-pick，更精细）：**

```bash
# 1. 先切换到正确的分支
git checkout feature/login

# 2. 把错误分支上那个提交"摘"过来
git cherry-pick <那个提交的哈希值>

# 3. 回到错误分支，回退
git checkout main
git reset --hard HEAD~1
```

`cherry-pick` 的意思是"把某一次提交复制一份，放到当前分支上"。就像从一棵树上摘一个果子，种到另一棵树上。

**类比：** 你在错误的日记本上写了一篇文章。方案 A 是"把这一页撕下来，贴到正确的日记本上"（是同一页纸）。方案 B 是"照抄一遍到正确的日记本上，然后把错误的那页撕掉"（是两份独立的内容）。

### 问题三：合并冲突，心慌意乱

**症状：** 你执行 `git merge` 或 `git pull` 时，终端出现了 `CONFLICT` 字样，Git 告诉你"自动合并失败"。

**这不是错误，这是 Git 在向你求助。**

**冲突发生的原因：** 两个人（或者你自己在两个分支上）修改了**同一个文件的同一行**。Git 不知道该听谁的，只能停下来问你。

**类比：** 两个人同时修改了同一份文档的同一句话。一个人把"蓝色"改成了"红色"，另一个人把"蓝色"改成了"绿色"。合并的时候，系统不知道最终该要哪个颜色，只能把两个都标出来让你决定。

**冲突时的文件长什么样：**

```
<<<<<<< HEAD
<h1>欢迎来到我们的网站</h1>
=======
<h1>欢迎访问官网</h1>
>>>>>>> feature/new-title
```

- `<<<<<<< HEAD` 和 `=======` 之间是你当前分支的内容
- `=======` 和 `>>>>>>> feature/new-title` 之间是你要合并进来的分支的内容

**解决冲突的标准步骤：**

```bash
# 1. 不要慌，先看看哪些文件有冲突
git status

# 2. 打开带冲突标记的文件，手工编辑
#    保留你想要的内容，删掉 <<<<<<<、=======、>>>>>>> 这些标记

# 3. 编辑完成后，把文件标记为"已解决"
git add 文件名

# 4. 完成合并
git commit -m "merge: 合并 feature/new-title，解决标题冲突"
```

**一个重要的心态：** 宁可手动解决冲突，也不要随便 `git merge --abort` 放弃合并。冲突是正常的，每次解决冲突你都在学习别人的代码。

### 问题四：push 被拒绝

**症状：** 你执行 `git push`，Git 报错：

```
! [rejected]  main -> main (fetch first)
error: failed to push some refs
```

**原因：** 在你上次 pull 之后，远程仓库有了新的提交（同事推的）。你现在是"基于旧版本改了东西"，Git 不让你直接覆盖远程的新内容。

**类比：** 你和同事同时修改了同一份共享文档。你先下载了一份到电脑上改，改完之后准备上传覆盖。但同事在你下载之后、上传之前已经上传了一个新版本。系统说："等等，你改的版本不是基于最新版的，如果直接覆盖会把同事的修改弄丢。"

**标准处理流程：**

```bash
# 方案 A（推荐日常使用）：用 pull --rebase
git pull --rebase origin main
# 如果有冲突，解决冲突后：
git add .
git rebase --continue
# 然后重新 push
git push

# 方案 B（更简单但会产生额外合并提交）：
git pull origin main
# 如果有冲突，解决冲突后：
git add .
git commit -m "merge: 合并远程更新"
git push
```

**`pull --rebase` 和 `pull` 的区别是什么？**

- `pull` = `fetch` + `merge`：把你的改动和远程改动"搅拌在一起"，产生一个合并提交。
- `pull --rebase` = `fetch` + `rebase`：把你的改动"挪到"远程改动之后，就像你是在最新版本上开发的一样，提交历史是一条直线，更干净。

大多数团队推荐 `pull --rebase`，因为它让提交历史更好看、更容易追溯。

### 问题五：不小心 git reset --hard，想反悔

**症状：** 你执行了 `git reset --hard`，发现有些代码不见了。

**补救方案：** Git 有一个"回收站"叫 `reflog`。你做的每一次修改 HEAD 的操作（包括 reset、commit、checkout）都被记录在这里，默认保存 90 天。

```bash
# 查看"回收站"里的记录
git reflog

# 输出类似：
# 1a2b3c4 HEAD@{0}: reset: moving to HEAD~1
# 5e6f7g8 HEAD@{1}: commit: 添加了一个重要功能
# ...

# 回到你 reset 之前的状态（用 HEAD@{1} 的那个提交哈希）
git reset --hard 5e6f7g8
```

**类比：** 你用 `git reset --hard` 就相当于把正在写的纸质笔记撕掉了一页扔进垃圾桶。但 `git reflog` 就像你有一个监控摄像头，能让你去垃圾桶里把撕掉的纸找回来重新粘上。

## 动手试试

下面的练习帮你模拟"合并冲突"这个最让人紧张的场景（5 分钟内完成）：

1. 创建一个新仓库：`mkdir conflict-practice && cd conflict-practice && git init`
2. 创建文件 `song.txt`，写入第一行：`第一句：月亮代表我的心`
3. `git add song.txt && git commit -m "添加第一句歌词"`
4. 创建新分支：`git checkout -b version-a`
5. 把 `song.txt` 的内容改为 `第一句：月亮代表我的心（摇滚版）`, 提交
6. 切回 main：`git checkout main`
7. 把 `song.txt` 的内容改为 `第一句：月亮代表我的心（抒情版）`, 提交
8. 执行 `git merge version-a` —— 冲突出现了
9. 打开 `song.txt`，你会看到冲突标记 `<<<<<<<` 和 `>>>>>>>`
10. 手动编辑，保留你想要的版本（或者合并两个版本），保存
11. `git add song.txt && git commit -m "merge: 合并歌曲版本"` —— 冲突解决！

---

## 本节小结

Git 的五个常见问题（忘 add、交错分支、合并冲突、push 被拒、误 reset）全都有解决方法。记住两句话：第一，冷静，别乱敲命令；第二，用 `git status` 先看清楚当前状态，再做决定。
