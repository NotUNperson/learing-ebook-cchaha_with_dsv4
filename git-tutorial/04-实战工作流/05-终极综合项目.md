## 正文

### 开场：一个真实的故事

恭喜你走到了这里。前面你学了 Git 的各种命令和技巧，现在是时候把这些技能串成一条完整的"项目开发链"了。

这一节不是讲新命令，而是让你**扮演一个团队里的开发者**，亲身体验"真正的团队是怎么用 Git 协作的"。

**项目背景：** 你和两个同事——小明、小红、小李——接了一个外包项目：为一家"猫猫咖啡馆"做一个简单的网页。页面要包含主页、关于我们和联系我们三个部分。

**你是小明，负责统筹和主页开发。** 小红负责样式美化。小李负责联系页面。

准备好了吗？故事开始了。

---

### 第一天上午：项目初始化（小明）

你（小明）坐在电脑前，第一件事是创建一个项目文件夹，初始化 Git 仓库，然后把基本结构推送到远程仓库。

```bash
# 创建项目文件夹
mkdir cat-cafe-website
cd cat-cafe-website
git init

# 创建主页的基本骨架
echo '<!DOCTYPE html>
<html>
<head><title>猫猫咖啡馆</title></head>
<body>
<h1>欢迎来到猫猫咖啡馆</h1>
<p>这里有好喝的咖啡和可爱的猫咪。</p>
</body>
</html>' > index.html

# 第一次提交
git add index.html
git commit -m "feat: 创建项目，添加主页基本骨架"

# 推送到远程仓库（假设已经在 GitHub 创建了同名仓库）
git remote add origin https://github.com/xiaoming/cat-cafe-website.git
git push -u origin main
```

**你为什么要做这些？** 因为团队需要一个"立足点"。就像打球需要一个球场，大家得知道球在哪里、往哪踢。项目初始化就是建这个"球场"。

---

### 同一天中午：小红加入，帮忙美化

小红打开电脑，第一件事是把项目克隆到自己的电脑上：

```bash
git clone https://github.com/xiaoming/cat-cafe-website.git
cd cat-cafe-website
```

然后她接了第一个任务："给主页加一些 CSS 样式，让它好看一点。"

```bash
# 创建分支
git checkout -b feature/add-styles

# 创建样式文件
echo 'body {
    font-family: "Microsoft YaHei", sans-serif;
    background: #fff8f0;
    color: #5d4037;
    max-width: 800px;
    margin: 0 auto;
    padding: 2rem;
}
h1 { color: #e65100; }' > style.css

# 在 index.html 中引入样式表（用 sed 插入一行）
sed -i 's|</head>|  <link rel="stylesheet" href="style.css">\n</head>|' index.html

# 提交并推送
git add style.css index.html
git commit -m "feat: 添加页面基础样式，主题色为暖橘色"
git push -u origin feature/add-styles
```

小红在 GitHub 上创建了一个 Pull Request，把她的分支合并到 main。小明看了看代码，觉得没问题，点了 "Merge"。

**关键点：** 小红没有直接在 main 上改，而是开了分支。她改完后没有"替你合并"，而是提 PR 请你 review。这就是团队协作的基石——**各干各的，用 PR 来"见面"。**

---

### 第二天上午：小李加入，同时小红继续改

小李克隆了项目，创建了他的分支，准备写"联系我们"页面。

```bash
git checkout -b feature/add-contact-page
```

与此同时，小红也在继续美化——她想把 `<h1>` 的颜色从橘色改成更深一点的棕色。

```bash
# 小红切回 main，拉取最新代码（包含了小李和之前合并的内容）
git checkout main
git pull

# 创建新分支
git checkout -b feature/deepen-heading-color

# 修改 style.css 中的 h1 颜色
sed -i 's/color: #e65100;/color: #bf360c;/' style.css

git add style.css
git commit -m "style: 将标题颜色从橘色改为深红棕色"
git push -u origin feature/deepen-heading-color
```

---

### 冲突爆发：小李也改了同一行！

小李在写"联系我们"页面时，发现标题的颜色跟整体风格不搭，也顺手把 `h1` 的颜色改了——改成了深蓝色：

```bash
# 小李在他的分支上
sed -i 's/color: #e65100;/color: #1a237e;/' style.css
git add style.css
git commit -m "style: 将标题颜色改为深蓝，配合联系页面风格"
```

小李提了 PR。小明先合并了小红的 PR（深红棕色），然后去合并小李的 PR 时，GitHub 显示："This branch has conflicts."

**冲突出现了！** 原因很简单：小红和小李都改了 `style.css` 中同一行。小红改成了深红棕色，小李改成了深蓝色。Git 不知道最终该用哪个颜色。

**这是整个团队协作中最经典的一幕。** 解决方法我们在第 03 节练过——但这次是在真实场景中：

```bash
# 小李在本地拉取最新的 main
git checkout main
git pull

# 切回自己的分支，变基到最新 main 上
git checkout feature/add-contact-page
git rebase main

# 冲突出现了！打开 style.css 你会看到：
# <<<<<<< HEAD
#   color: #bf360c;
# =======
#   color: #1a237e;
# >>>>>>> style: 将标题颜色改为深蓝

# 小李手动编辑 style.css，选了一个折中方案
# 改成：color: #4e342e;（温和的深棕色）

# 解决冲突
git add style.css
git rebase --continue
git push --force-with-lease
```

**`--force-with-lease` 是什么？** 因为 rebase 改写了提交历史，普通的 push 会被拒绝。`--force-with-lease` 是一种"安全的强制推送"——它只有在远程分支没有被别人更新过的情况下才会执行。比 `--force` 安全得多。

**团队会怎么决定颜色？** 现实中，小李会跟小红在 PR 评论里讨论："你觉得深红棕还是深蓝好？"他们可能达成一致后选一个，也可能让设计师拍板。Git 只负责"标记冲突"，怎么解决是团队沟通的事。

---

### 第三天：合并所有工作，发布

所有 PR 都合并后，项目的 main 分支上有完整的网站。小明在本地拉取最新代码，做最终检查：

```bash
git checkout main
git pull

# 查看完整的提交历史
git log --oneline --graph --all

# 给当前版本打一个标签（版本号）
git tag v1.0.0 -m "猫猫咖啡馆网站第一版"

# 推送标签
git push --tags
```

**标签** 是什么？就像给某个提交贴了一张"便签纸"，写上"这是 1.0 版本"。以后你随时可以回来看"1.0 版本长什么样"。

### 这个项目教会了你什么？

1. **一个人的项目**（前面所有章节）：学会 Git 的各种命令
2. **三个人的项目**（本节）：学会 Git 协调团队——什么时候建分支、什么时候提 PR、冲突怎么解决、怎么沟通

如果你能从"一个人在仓库里随便玩"升级到"三个人有条不紊地分工协作"，你就已经具备加入任何开发团队的基本能力了。

---

### 类比：项目就像合租

Git 仓库就像一个合租房：

- `main` 分支是客厅——大家共用的空间，必须保持整洁
- 功能分支是每个人的卧室——你在自己房间里怎么折腾都行，不会影响别人
- Pull Request 是"我收拾好了，大家来看看我的房间，准备让它并入客厅"
- Merge 是"OK，大家都同意，房间的门现在对客厅打开"
- 冲突是"你要在客厅摆一张桌子，我也要在同一个位置摆一张桌子——我们得商量一下"

## 动手试试

这个练习需要你运行配套的 Shell 脚本 `05-final-capstone.sh`。但如果你只想手动体验关键步骤，在 5 分钟内：

1. 创建一个临时仓库：`mkdir team-test && cd team-test && git init`
2. 创建 `index.html`，写一行 `<h1>Hello</h1>`，add + commit
3. 创建分支 `feature/change-title`：`git checkout -b feature/change-title`
4. 把 `<h1>Hello</h1>` 改成 `<h1>Hello World</h1>`，add + commit
5. 切回 main（或 master）：`git checkout main`
6. 把原始的 `<h1>Hello</h1>` 改成 `<h1>Hey There</h1>`，add + commit
7. 合并分支：`git merge feature/change-title`
8. 亲手制造一次冲突，亲手解决它

## 本节小结

团队的 Git 协作只有五件事：领任务建分支、在自己的分支上开发、推送到远程提 PR、解决冲突、合并到 main。你在这节里把五件事全做了一遍。以后进团队的那一刻，你不会问"我该做什么"，而会问"我们的 main 分支叫什么名字"。
