# 01 日常开发工作流

## 本节你会学到什么

- 用"上班打卡"的节奏理解一个开发者一天的标准流程
- 掌握早上拉取最新代码、晚上推送修改的完整循环
- 学会在写代码前先开分支，而不是直接在 main 上改
- 理解 add、commit、push 三个动作分别对应什么"生活场景"
- 完整模拟一次"上午接任务 -- 下午写代码 -- 晚上交作业"的流程

## 正文

### 一个开发者的"一天"长什么样

想象你是一个厨师，每天早上来到餐厅厨房。在开始做菜之前，你会做什么？

1. **先看看冰箱里有什么新食材**——别人可能昨天补充了东西，也可能用掉了一些。你得先拿到最新的库存状态。
2. **领一个任务**——"今天要开发一道新甜点"。
3. **在自己灶台上备菜**——你不会直接在大家共用的灶台上试来试去，万一搞砸了影响别人。
4. **尝味道，调整**——反复修改直到满意。
5. **拍照存档**——把这版配方正式记录下来。
6. **端到共用区域**——"我做好了，大家可以尝了。"
7. **告诉主厨我完工了**——申请正式合并到菜单上。

把这个厨房的流程翻译成 Git 的日常操作，就是下面这张表：

| 厨师的动作 | Git 操作 |
|-----------|---------|
| 看看冰箱里有什么新食材 | `git pull` 拉取最新代码 |
| 领一个新任务 | 从 main 创建新分支 `git checkout -b feature/xxx` |
| 在自己灶台上备菜 | 在分支上改文件，不影响 main |
| 尝味道，调整 | 反复修改、测试 |
| 拍照存档 | `git add` + `git commit` |
| 端到共用区域 | `git push` 推送到远程仓库 |
| 告诉主厨完工 | 在 GitHub 上创建 Pull Request（简称 PR） |

下面我们把每一步展开来讲。

### 第一步：早上来到工位 —— git pull

每天开始写代码之前，第一件事永远是**拉取最新代码**。因为昨晚或今天早上，你的同事可能已经推送了新内容。如果你不先 pull，你接下来写的代码可能跟别人冲突。

```bash
# 切到主分支
git checkout main

# 拉取远程最新代码
git pull
```

**为什么要先切到 main 再 pull？** 因为你可能昨天新建了一个分支，但还没切回来。`git pull` 只对你当前所在的分支生效。如果你的主分支叫 `main`，拉完后：

```bash
# 如果 main 有更新，顺便也更新你的功能分支（可选）
git checkout feature/my-task
git merge main
```

这样你的功能分支就能"继承"main 上最新的改动，减少后面合并时产生冲突的概率。

**类比：** 就像你早上到公司，先看一眼共享文件夹里的"最新周报"，确认今天要做的事有没有变动。

### 第二步：领任务 —— 创建功能分支

你肯定不想直接在 main 上改代码。为什么呢？我们来想象一个场景：

你把 main 当成草稿纸，在上面写了很多半成品代码。这时候突然线上出了个紧急 bug，老板让你立刻修。但你的草稿纸上全是半成品，根本没法直接发布——你被自己的"半成品"卡住了。

**正确做法：** 每接一个新任务，就建一个新分支。分支之间互不打扰，main 始终是"能随时发布的干净状态"。

```bash
git checkout -b feature/add-login-page
```

分支命名通常会带上分类前缀，比如：
- `feature/` —— 新功能
- `bugfix/` —— 修 bug
- `hotfix/` —— 线上紧急修复

这样一看名字就知道这个分支是干嘛的。

### 第三步：写代码

这一部分不用 Git 命令——就是你打开编辑器，吭哧吭哧敲键盘的过程。

**但有一个好习惯：** 写完一小块功能就 commit 一次，而不是攒到下班前一口气提交 50 个文件。为什么？

- **出问题好定位：** 如果每次只改一个小功能，出 bug 了很容易知道是哪次提交引入的。
- **后悔成本低：** 如果你提交了一个小功能后不满意，只想撤销这一个改动。如果一次提交混了 10 个功能，你只能全部撤销或者手动挑。
- **类比：** 就像你写论文，每写完一个小节就双击保存一次，而不是写到凌晨三点一次性保存——电脑死机你就哭吧。

### 第四步：拍照存档 —— git add + git commit

这两个命令经常一起出现，但它们各有各的职责。

#### git add：选择要拍进照片的内容

```bash
# 添加单个文件
git add index.html

# 添加所有改过的文件
git add .
```

**类比：** 你刚拍了一组照片，但拍之前你要选"哪些东西进入取景框"。`git add` 就是这个"取景"的动作——你告诉 Git："我这次要拍这几张，那几张先不拍。"

#### git commit：按下快门，正式存档

```bash
git commit -m "添加登录页面的表单结构"
```

提交信息（-m 后面的内容）一定要写清楚"做了什么"。不要写"改了一下"、"fix"这种模糊的东西。后面团队同事要看这个信息来决定要不要合并你的代码——你总不希望别人对着"改了改"三个字猜你干了什么对吧？

> 关于"好的提交信息到底该怎么写"，我们在 **第 04 节 Git 最佳实践** 里会展开讲。

### 第五步：上传到远程 —— git push

commit 只是把你的改动保存在了你自己的电脑上。队友还看不见。要把你的改动分享出去，就要 push：

```bash
git push -u origin feature/add-login-page
```

这里的 `-u` 是 `--set-upstream` 的缩写。它的作用是**把你本地的这个分支和远程的同名分支"绑定"起来**。绑定之后，你下次在这个分支上再 push，只需要写 `git push` 就够了，不用每次都写一长串。

**类比：** commit 是你把写好的信封装好了放在自己抽屉里。push 是把信封投进邮筒，让它飞到远程仓库，大家都看得见。

### 第六步：提 PR / Merge Request

代码推送到远程之后，你需要在 GitHub（或 GitLab）上创建一个 **Pull Request**。PR 的意思就是："我写了一些代码，请拉到主分支里看一下。"

在 GitHub 上操作：
1. 打开你的仓库页面
2. 点 "Pull requests" 标签
3. 点 "New pull request"
4. 选择你的分支（比如 `feature/add-login-page`），目标分支选 `main`
5. 写清楚你这个 PR 做了什么
6. 点 "Create pull request"

然后团队同事会 review 你的代码，提意见，你改，再 push，PR 会自动更新。最后一切 OK，点击 "Merge" 按钮，你的代码就正式合入 main 了。

### 完整的一天流程图

```
早上开工
  |
  v
git checkout main  ───>  切换到主分支
  |
  v
git pull           ───>  拉取最新代码（同事可能更新了）
  |
  v
git checkout -b feature/xxx  ───>  创建新功能分支
  |
  v
写代码、改代码、测试（反复循环）
  |
  v
git add .          ───>  选择要提交的文件
  |
  v
git commit -m "xxx"  ───>  正式存档
  |
  | （如果功能还没写完，回到"写代码"继续）
  |
  v
git push -u origin feature/xxx  ───>  推送到远程仓库
  |
  v
在 GitHub 上创建 Pull Request  ───>  请同事 review
  |
  v
根据意见修改 → add → commit → push（自动更新 PR）
  |
  v
合并成功！删除功能分支（可选）
```

## 动手试试

按照下面的步骤，完整走一遍"一天的工作流"。不需要真的写复杂的代码——创建一个空的 HTML 文件就够了。

1. 打开 Git Bash，进入你之前创建的任意一个本地仓库（如果没有，先用 `git init` 建一个，然后随便提交一个文件）
2. 确认当前在 main 分支上：`git branch`
3. 创建一个功能分支：`git checkout -b feature/add-heading`
4. 新建一个文件 `index.html`，内容写 `<h1>Hello</h1>`
5. 查看状态：`git status`
6. 暂存文件：`git add index.html`
7. 提交：`git commit -m "添加页面主标题"`
8. 切回 main：`git checkout main`
9. 查看 main 上有没有 `index.html` 文件（你会发现没有——因为改动还在分支上）
10. 切回功能分支，合并到 main：`git checkout main && git merge feature/add-heading`

（如果你的仓库关联了远程仓库，还可以试试 push）

## 本节小结

每一天的工作都沿着同一个节奏走：pull 拉最新 -- 建分支 -- 写代码 -- add -- commit -- push -- 提 PR。养成这个习惯后，你会发现自己很少再遇到慌乱的"代码丢了"或者"合并乱了"的情况。
