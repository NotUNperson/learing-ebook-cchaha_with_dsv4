# 04 Git 最佳实践

## 本节你会学到什么

- 掌握写出"一看就懂"的提交信息的七个字诀
- 理解多久提交一次最合理，以及"拆分提交"的艺术
- 学会分支命名的团队通用规范
- 养成保持提交历史干净的习惯（谁都会感谢你）
- 背熟一份"绝不提交清单"，避免安全事故

---

## 正文

你已经学会了 Git 的各种命令。但从"会用"到"用好"，中间还有一个台阶——就是"知道什么时候该做什么、为什么这样做"。这一节没有新命令，全是经验之谈。

### 一、提交信息怎么写

先看两个例子：

```
# 坏的提交信息
fix
改了一下
update
wip
啊啊啊终于修好了

# 好的提交信息
feat: 添加用户登录功能，支持邮箱和手机号两种方式
fix: 修复首页在 Safari 浏览器中图片错位的 bug
docs: 更新 README，添加项目启动说明
refactor: 提取公共的日期格式化函数到 utils/date.js
```

**好的提交信息有什么特征？**

1. **一看就知道干了什么。** 一两个月后你回头翻 git log，不需要点进去看代码就能知道每次提交做了什么。
2. **动词开头。** "添加"而不是"添加了"，"修复"而不是"修复了"——因为提交记录应该表达"这次提交做什么"。
3. **有分类前缀。** 很多团队用约定式提交（Conventional Commits）规范：

| 前缀 | 含义 | 例子 |
|------|------|------|
| `feat:` | 新功能 | `feat: 添加购物车页面` |
| `fix:` | 修复 bug | `fix: 修复订单金额计算错误` |
| `docs:` | 文档相关 | `docs: 更新 API 使用说明` |
| `style:` | 代码格式（不影响逻辑） | `style: 统一缩进为 2 空格` |
| `refactor:` | 重构（不改变外部行为） | `refactor: 提取公共校验逻辑` |
| `test:` | 测试相关 | `test: 添加登录模块的单元测试` |
| `chore:` | 杂务（构建工具、依赖等） | `chore: 升级 webpack 到 5.0` |

**类比：** 提交信息就像是给未来的自己和同事写的"便利贴"。你写的不是给 Git 看的（Git 不关心你写了什么），是给人看的。就像你在冰箱里的剩菜上贴一张"周二做的红烧肉"——一个月后打开冰箱，你一眼就知道这是什么、能不能吃。

#### 提交信息的结构

除了前缀，好的提交信息还有一个结构：

```
<前缀>: <简短描述（50 字以内）>

<详细说明（可选，如果需要解释"为什么"）>
```

为什么要有详细说明？因为"做了什么"往往从代码 diff 里能看出来，但"为什么这么做"只有你知道。比如：

```
fix: 将登录超时时间从 30 秒延长到 120 秒

原来的 30 秒对手机网络用户太短了。根据用户反馈和数据分析，
大约 15% 的用户在网络较差时超时。120 秒是一个平衡点，
既不会让用户等太久，也不会造成过高的服务器负载。
```

三个月后，你或者你的同事看到这条信息，马上能理解当时的决策背景——不需要去翻聊天记录或者猜。

### 二、多久提交一次？

**一句话答案：写完一个逻辑完整的小单元就提交。**

什么叫"逻辑完整的小单元"？

- 添加了一个表单的 HTML 结构 —— 一个单元
- 给这个表单加了 CSS 样式 —— 一个单元
- 给这个表单加了 JavaScript 验证逻辑 —— 一个单元

这三步应该分成三个提交，而不是攒在一起一个提交搞定。

**为什么不能攒到一起？**

**类比：** 你搬家打包。如果把书、碗、衣服全部塞进一个大箱子，到了新家拆箱的时候完全是灾难——你找不到东西，还容易打碎碗。如果把书装一箱、碗装一箱、衣服装一箱，每个箱子外面写上标签，拆箱的时候又安全又快。

提交也是同理：
- 出 bug 时能精确定位到是哪个提交引入的
- 如果需要回退某个功能，可以直接回退那一个提交，不影响其他
- 别人 review 你的代码时，可以一个提交一个提交地看，逻辑清晰

**但也不要太碎。** 不要每改一行就提交一次。如果你的提交信息写的全是"改了一行"、"又改了一行"、"又又改了一行"——该合并了。

**一个好的判断标准：** 如果你能用一句话说清楚这次提交做了什么，这句话里面没有"和"字（表示只做了一件事），那就可以提交了。

### 三、分支命名规范

你的项目里可能有几十个分支。一两个月后你看到一堆名叫 `test`、`new`、`try`、`aaa` 的分支，你怎么知道它们是干嘛的？

所以团队通常会约定一套分支命名规则。下面这套是最常见的：

```
<类型>/<简短描述>
```

具体来说：

| 类型 | 用途 | 例子 |
|------|------|------|
| `feature/` | 新功能 | `feature/user-login`、`feature/shopping-cart` |
| `bugfix/` | 修 bug | `bugfix/navbar-overlap`、`bugfix/order-total` |
| `hotfix/` | 线上紧急修复 | `hotfix/payment-crash` |
| `release/` | 发布准备 | `release/v2.0.0` |
| `chore/` | 杂务 | `chore/update-dependencies` |
| `refactor/` | 重构 | `refactor/extract-auth-module` |

**命名时注意：**
- 用小写字母和连字符（`-`），不要用空格和下划线
- 尽量简短，但要能看出干什么：`feature/login` 比 `feature/do-something` 好一百倍
- 如果有关联的任务编号，可以加在后面：`feature/login-JIRA-123`

**类比：** 分支名称就像快递单上的备注——"厨房用品"、"换季衣服"、"书籍"。你和室友收到快递后不用拆开就知道哪个箱子该放哪里。

### 四、保持提交历史干净

一个人的代码是一个人看，但 Git 历史是全团队共享的。脏乱的历史会让所有人心累。

**三条黄金法则：**

#### 法则一：不要提交半成品

如果你今天的代码写到一半，明天还要继续，怎么办？可以提交，但在提交信息前加上 `WIP:`（Work In Progress，工作进行中），比如：

```
WIP: 登录功能开发中，表单部分已完成
```

等到功能完成后，可以用 `git rebase -i` 把多个 WIP 提交合并成一个干净的提交。

#### 法则二：push 之前先整理

你已经学过了 `git commit --amend` 和 `git rebase -i`。这两个工具的典型使用场景是：

- 你做了三个提交：A（添加按钮）、B（调整按钮颜色）、C（又调了一下颜色）
- B 和 C 其实是在做同一件事（调整按钮样式），应该合并成一条

在 push 之前，用 `git rebase -i` 把这些碎提交压缩（squash）成一个：

```bash
git rebase -i HEAD~3
# 在编辑器中，把后两个提交的 pick 改成 squash（或 s）
```

#### 法则三：不要改写已推送的历史

`git commit --amend` 和 `git rebase -i` 都是**改写历史**的操作。它们只能在"还没 push 的时候"使用。一旦你的提交已经 push 到了远程仓库，就不要改了。

为什么？因为别人可能已经基于你的提交做了开发。你改写历史后，他们的工作会乱成一团。

**类比：** push 之前的提交像是你还没寄出的信——你可以拆开修改、合并几封信的内容。但 push 就是把信寄出去了——发出去的信就别追回来改了，除非你通知每一个收到信的人"旧的作废"（force push），这在大团队里是要挨骂的。

### 五、绝对不要提交的文件清单

这个清单在 02 节讲过，但值得在这里再强调一遍。因为**安全问题不是"出了再说"，而是"出了就晚了"。**

| 类型 | 应该写入 .gitignore 的文件 | 出事的后果 |
|------|--------------------------|-----------|
| 密码和密钥 | `.env`、`secrets.json`、`*.pem`、`*.key` | 密码泄露，数据库可能被删 |
| 依赖包 | `node_modules/`、`vendor/`、`venv/` | 仓库几百 MB，clone 一次要半小时 |
| 编译产物 | `dist/`、`build/`、`*.exe`、`*.o` | 每个人都提交编译产物，冲突满天飞 |
| 系统垃圾 | `.DS_Store`、`Thumbs.db` | 没危害但烦人，徒增仓库体积 |
| 编辑器配置 | `.vscode/`、`.idea/`、`*.swp` | 各人配置不同，互相覆盖 |
| 临时文件 | `*.tmp`、`*.bak`、`*~` | 没用的东西占地方 |
| 大文件/二进制 | 视频、PSD 源文件、数据库文件 | Git 不适合管大文件，仓库跑不动 |

**特别提醒：** 即使你后来删掉了密码文件并提交了删除操作，密码**仍然**存在于 Git 历史中。任何人执行 `git log -p` 都能翻出那个密码。如果不小心提交了密码：

1. **立刻修改密码**（这是最重要的，Git 历史里的旧密码已经不安全了）
2. 然后才考虑清理 Git 历史

### 六、其他好习惯

**（1）每次开始工作前先 `git pull`**

就像你进厨房前先检查冰箱，确保你是在最新状态上操作。

**（2）经常 `git status`**

每当你准备执行下一个 Git 命令之前，先看一下 status。很多人出错的原因是"我以为我在那个分支"或者"我以为那个文件已经暂存了"。看好 status 再操作。

**（3）不小心改了不想改的文件？**

```bash
# 放弃对某个文件的修改，恢复到上次提交的状态
git checkout -- 文件名

# 放弃所有没暂存的修改
git checkout -- .
```

注意：这个操作**不可逆**。执行前确认你真的不需要这些改动了。

**（4）定期清理已合并的分支**

合并到 main 之后，功能分支就完成了它的使命。留着太多旧分支会让 `git branch` 的输出像菜市场。定期清理：

```bash
# 删除本地已合并的分支
git branch --merged | grep -v "main" | xargs git branch -d

# 删除远程已合并的分支（在 GitHub 上操作即可）
```

## 动手试试

这次不是敲命令，而是动脑思考。

1. 打开你之前练习用的任意一个仓库，看看 `git log --oneline` 的输出
2. 挑出三条提交信息，问自己：如果三个月后回来看，我能一眼看懂每次提交干了什么吗？
3. 挑出一条你写得最模糊的提交信息，重写一个更好的版本（假装你要 `git commit --amend`）
4. 想一想你自己的项目里，有哪些文件"应该在 .gitignore 里但你没有加"

---

## 本节小结

好的提交信息是对未来的自己（和同事）的善意；合理的提交频率让你永远不怕出 bug；规范的分支命名让团队一眼看清方向。这些习惯不需要任何新命令，却能让你的 Git 水平从"会用"变为"用得漂亮"。
