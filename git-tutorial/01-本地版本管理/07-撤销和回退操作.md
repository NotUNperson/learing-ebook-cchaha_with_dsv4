# 07 撤销和回退操作

## 本节你会学到什么

- 掌握三种"后悔"操作的适用场景：`git restore`、`git reset`、`git revert`
- 理解"撤销工作区修改"和"回退已提交版本"的区别
- 用"后悔药"类比记住三种操作的严重程度
- 知道什么时候该用哪种"后悔药"，不用担心把仓库搞坏

## 正文

### 人人都会犯错

写代码的过程中，犯错是家常便饭。删了不该删的代码、改了一个文件后发现思路不对、提交后才发现提交信息写错了……Git 给你准备了从轻到重的三种"后悔药"，帮你应对不同程度的"后悔"。

在学具体命令之前，先记住这个类比：

| 后悔程度 | Git 命令 | 类比 |
|----------|----------|------|
| 轻度后悔 | `git restore` | 你用橡皮擦擦掉刚写的几个字，重新写 |
| 中度后悔 | `git reset` | 你把文件夹里的纸张拿出来，甚至从档案柜里把最近的文件袋抽出来 |
| 重度后悔 | `git revert` | 你写了一份"更正说明"，贴在原来的文件袋上——"之前那份作废，以此为准"。原来的那份还在，但以后都看这份新的 |

### 轻度后悔药：git restore

**使用场景**：你改了一个文件，但还没 `git add`（没放入暂存区）。改完之后你后悔了——"算了，还是原来的好"。

```bash
git restore 文件名
```

这条命令会用仓库里最新版本的内容**覆盖**工作区中的修改。你的改动会丢失，文件恢复到上次提交时的状态。

**类比**：你在草稿纸上改来改去，最后觉得还是原稿好。`git restore` 就是把原稿重新打印一份，盖住你的草稿纸。

**注意**：`git restore` 会**不可逆地**丢弃工作区的修改。执行前请确认你真的不需要这些修改了。

还有两个变体：

```bash
# 把暂存区中的文件"拿出来"，放回工作区（取消暂存，但不丢弃修改）
git restore --staged 文件名

# 从仓库的某个历史版本恢复文件
git restore --source=<提交哈希> 文件名
```

**重要提醒**：在旧版本的 Git 中，`git restore` 不存在，是用 `git checkout -- <文件>` 来实现的。如果你看到网上的教程用 `git checkout --`，它和 `git restore` 是同一个功能。现在推荐用 `git restore`，语义更清晰。

### 中度后悔药：git reset

`git reset` 是功能最强大的后悔药，但它也是最容易让人困惑的——因为它有三种模式，作用范围不同。

**使用场景**：你已经 `git add` 了，甚至已经 `git commit` 了，但想回退。

#### 三种模式的形象理解

继续用我们的办公桌/文件夹/档案柜类比：

| 模式 | 命令 | 对区域的影响 | 类比 |
|------|------|-------------|------|
| --soft | `git reset --soft HEAD~1` | 只动档案柜 | 从档案柜里拿出最近的文件袋，放在桌上。文件夹不受影响 |
| --mixed (默认) | `git reset HEAD~1` | 动档案柜 + 文件夹 | 拿出文件袋，把文件夹也清空。文件袋里的东西回到办公桌上 |
| --hard | `git reset --hard HEAD~1` | 三个区域全动 | 拿出文件袋，清空文件夹，**连办公桌上的改动也扔掉** |

#### --soft：最温柔的回退

```bash
git reset --soft HEAD~1
```

`HEAD~1` 的意思是"往前数 1 个提交"。这条命令把最近一次提交"撤销"了，但提交里的内容会**留在暂存区**。

**使用场景**：你刚提交完，发现提交信息写错了，或者漏了一个文件。`--soft` 撤销提交，让你重新整理暂存区，然后重新提交。

**类比**：你把文件袋从档案柜里拿出来，里面的文件还在文件夹里，你可以整理一下再放回去。

#### --mixed（默认）：回到暂存前

```bash
git reset HEAD~1
# 等同于
git reset --mixed HEAD~1
```

撤销最近一次提交，并且把暂存区也清空。提交的内容回到**工作区**（文件本身的修改还在，但变成了"未暂存"状态）。

**使用场景**：你提交完了，但想重新组织这些修改——可能想拆成两次提交，或者想和别的修改合并。

**类比**：你把文件袋从档案柜拿出来，文件夹里的纸张也倒回办公桌上。纸张上的内容还在，但你需要重新挑选、重新整理。

#### --hard：最彻底的回退

```bash
git reset --hard HEAD~1
```

**这是最危险的后悔药。**它不仅撤销提交、清空暂存区，还会把工作区的文件**覆盖**成目标版本的样子。这意味着你所有未提交的修改都会**永久丢失**。

**使用场景**：你确定最近的提交完全是错的，想彻底回到之前的状态，并且不在意丢掉工作区里的任何修改。

**类比**：你把文件袋从档案柜拿出来，文件夹清空，办公桌上的所有纸张也都换回档案柜里那个版本的样子。所有后来的改动都消失了。

> **警告**：在使用 `--hard` 之前，请再三确认你真的不需要这些改动了。一旦执行，就找不回来了（除非你记得之前提交的哈希值）。

### 重度后悔药：git revert

**使用场景**：你已经把提交推到了远程仓库（比如 GitHub），或者你只想"抵消"某次提交的改动，而不想破坏提交历史。

```bash
git revert <提交哈希>
```

`git revert` 不会删除任何历史。它会创建一个**新的提交**，这个新提交的内容是"把之前那次提交的改动反着做一遍"。如果原来那次提交加了一行代码，`git revert` 就会创建一次把那行代码删掉的提交。

**类比**：你不去档案柜里修改或拿出旧文件袋。而是在文件袋外面贴一张**更正通知**："注意，第 3 号文件袋里的内容已作废，请以本通知为准。"旧的文件袋依然留在柜子里，记录是完整的。

### 三种后悔药对比总结

| 命令 | 改历史吗？ | 丢改动吗？ | 适用场景 |
|------|-----------|-----------|---------|
| `git restore` | 不改 | 丢工作区未暂存的修改 | 还没 add，改岔了，想回去 |
| `git reset --soft` | 改本地历史 | 不丢（回到暂存区） | 刚 commit 完，想改提交信息或补文件 |
| `git reset --mixed` | 改本地历史 | 不丢内容（回到工作区） | 想重新组织这次提交 |
| `git reset --hard` | 改本地历史 | **全丢** | 确定最近的提交完全错了 |
| `git revert` | 不改历史 | 不丢 | 改动已分享给别人，必须保留历史 |

### 一个重要原则

**如果改动还没推送到远程仓库（还在你的电脑上）**：你可以放心使用 `git reset` 来修改历史。改完之后再推送即可。

**如果改动已经推送到远程仓库（别人可能已经看到了）**：请用 `git revert`。不要用 `git reset` 去改已经公开的历史——这会让队友的历史混乱。

**类比**：如果你已经发了一条朋友圈，然后觉得写得不好，你可以删掉重发（reset）。但如果有朋友已经评论了好几条，你再删掉就会让大家的评论悬空。这时候更好的做法是发一条"更正说明"（revert）。

### 撤销 git add（从暂存区拿出来）

这是最常见的"轻度后悔"：

```bash
git restore --staged 文件名
```

这不会丢失文件的修改内容，只是把文件从暂存区"拿回"工作区。

**类比**：你从文件夹里把一张纸拿出来，放回办公桌上。纸上的内容没变，只是不再"待归档"了。

## 动手试试

1. 创建一个文件并提交（建立基线）
2. **练习 git restore**：修改文件但不 add，运行 `git restore 文件名`，观察文件恢复
3. **练习撤销暂存**：修改文件并 add，运行 `git restore --staged 文件名`，观察文件回到工作区（修改内容还在）
4. **练习 git reset --soft**：add 并 commit，然后运行 `git reset --soft HEAD~1`，用 `git status` 和 `git log` 观察变化——提交消失了，但修改在暂存区
5. **练习 git reset --mixed**：再次 commit，然后运行 `git reset HEAD~1`，观察修改回到工作区
6. （可选，在测试仓库中进行）**练习 git reset --hard**：commit 后运行 `git reset --hard HEAD~1`，观察修改彻底消失
7. **练习 git revert**：commit 两次，对第一次提交运行 `git revert <哈希>`，观察 Git 创建了一个新提交来"抵消"那次的改动

## 本节小结

三种后悔药从轻到重：`git restore` 擦掉办公桌上的涂改，`git reset` 从档案柜里拿回文件袋（有三种模式控制拿回的程度），`git revert` 不碰历史，只追加一条"更正说明"。核心原则：没推送的可以 reset，已推送的用 revert。
