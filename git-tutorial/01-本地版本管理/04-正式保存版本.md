# 04 正式保存版本

## 本节你会学到什么

- 理解 `git commit` 的含义——把暂存区的快照正式存档
- 写出清晰、有用的提交信息
- 理解提交的"原子性"——要么全存，要么全不存
- 了解 `git commit -a` 的快捷用法及其局限

## 正文

### 什么是提交？

在前面三节，我们一直在铺垫：工作区（办公桌） → 暂存区（文件夹） → 仓库（档案柜）。现在到了最后一步——**提交（commit）**，也就是把文件夹里的内容正式归档。

`git commit` 执行时，Git 会做以下几件事：

1. 把暂存区里的所有文件打包成一个"快照"
2. 给这个快照打上一个独一无二的编号（一个很长很长的哈希值，比如 `a1b2c3d4...`）
3. 附上你写的提交信息（说明这次提交做了什么）
4. 把快照永久存入 `.git/objects/` 目录

**类比**：提交就像是你在档案柜里放入一个**密封的文件袋**。文件袋上写着日期、编号、内容摘要（提交信息）。袋子里的文件被完整保存，以后你可以凭编号随时调出来看。

### 你的第一次提交

最简单的提交命令：

```bash
git commit -m "这里写你的提交信息"
```

`-m` 是 `--message` 的缩写，后面跟着的就是你对这次提交的描述。

例如：

```bash
git add .                              # 先把要存档的文件放入暂存区
git commit -m "搭建了项目的基本骨架"     # 正式存档
```

### 写信息用什么语言？

Git 默认情况下会是英文环境，提交信息推荐用**英文小写**。但如果你是个人项目，或者团队说好了用中文，那就用中文。

不管用什么语言，提交信息都应该遵循这个原则：一个**完全的陌生人**看到你的提交信息，能在 5 秒内明白这次改动做了什么。

### 提交信息的黄金法则

一个完整的提交信息通常长这样：

```
简短摘要（50 字以内）

详细描述（如果有必要的话）。解释为什么要做这个改动，
它解决了什么问题，有没有什么需要注意的副作用。
```

**好的提交信息**：
```
修复了登录页面密码验证不生效的问题

用户在输入含特殊字符的密码时，验证函数会错误地拒绝。
原因是密码在传入验证函数前被错误转义。现在移除了那个多余的转义步骤。
```

**不好的提交信息**：
```
改了一下
```
```
fix
```
```
update
```
```
.
```

**类比**：提交信息就像是你在文件袋上写的**标签**。如果你写"东西"，三个月后你翻档案柜，你不可能知道这个袋子里装的是什么。如果你写"2024年3月的客户合同"，一目了然。

### 提交信息的常用约定

业界有一种流行的约定叫"约定式提交"，格式为：

```
<类型>: <简短描述>
```

常见的类型包括：

- `feat:` —— 新增功能（feature）
- `fix:` —— 修复 bug
- `docs:` —— 只改了文档
- `style:` —— 只改了格式（空格、分号等，不影响代码逻辑）
- `refactor:` —— 重构代码（既没加功能也没修 bug）
- `test:` —— 添加或修改测试
- `chore:` —— 杂活（构建工具、依赖更新等）

例如：
```
feat: 增加用户注册功能
fix: 修复密码验证正则表达式错误
docs: 补充 API 使用说明
```

你现在不需要硬背这些，知道有这个习惯就好。以后参与团队项目时会经常遇到。

### 提交是"原子"的

**原子性**是一个很重要的概念。它的意思是：一次提交要么完全成功，要么完全不发生。不会出现"存了一半、丢了一半"的情况。

**类比**：你把文件袋放进档案柜的操作，要么袋子完整地放进去、柜门关上、标签写好——全部完成。要么袋子留在外面，什么都没发生。绝不会出现"袋子进去了但标签没写"这种半吊子状态。

这个特性保证了你的版本历史是干净、可靠的。你永远不需要担心某次提交"不完整"。

### git commit -a：一条捷径

如果你修改了一些已被跟踪的文件（不是新建的），可以跳过 `git add` 步骤，直接：

```bash
git commit -a -m "提交信息"
```

`-a` 是 `--all` 的缩写。它告诉 Git："把**所有已跟踪且被修改了**的文件自动加入暂存区，然后提交。"

**注意**：`-a` **不会**添加未跟踪的新文件。如果你创建了一个全新的文件，还是需要先 `git add` 它。

**类比**：`git commit -a` 就像是"把办公桌上所有已经入职的老员工的工作报告自动收进文件夹并归档"。但新来的员工（未跟踪文件）你还是要手动给他办入职。

### git commit 不加 -m

如果你直接输入 `git commit`（不加 `-m`），Git 会打开一个文本编辑器让你写提交信息。在 Windows 上通常是 Vim 或 Notepad。

对于新手来说，建议先用 `-m` 的方式，等到你对 Git 熟悉了再尝试用编辑器写长信息。

如果你不小心进入了 Vim 界面：
- 按 `i` 进入编辑模式
- 写完提交信息
- 按 `Esc` 退出编辑模式
- 输入 `:wq` 然后按回车（保存并退出）
- 或者输入 `:q!` 然后按回车（不保存，取消提交）

### 检查最后一次提交的内容

```bash
git show
```

这条命令会显示最近一次提交的详细信息：谁提交的、什么时候提交的、提交信息是什么、具体改了什么。

```bash
git show --stat
```

`--stat` 只显示摘要——改了哪些文件，每个文件改了多少行。

## 动手试试

1. 在一个 Git 仓库中修改或创建几个文件
2. 用 `git add` 把它们放入暂存区
3. 执行 `git commit -m "你的提交信息"`
4. 用 `git show` 看看这次提交的详细信息
5. 用 `git show --stat` 看看摘要
6. 再创建一个新文件，不 add 它，然后尝试 `git commit -a -m "测试 -a"` —— 你会发现新文件没有被提交
7. 使用 `git log` 看一眼提交历史（下一节会详细讲）

## 本节小结

`git commit` 是"正式存档"的操作，它把暂存区的内容变成仓库里的一个永久版本。提交信息是你的"标签"——写好它，未来的你（和你的队友）会感激现在的你。
