# 02 Git 的三个区域

## 本节你会学到什么

- 理解 Git 的三个"工作空间"——工作区、暂存区、仓库
- 用"办公桌 / 待办文件夹 / 档案柜"的类比记住三个区域
- 知道文件在三个区域之间流转的完整路径
- 理解为什么 Git 要设计三个区域而不是两个

## 正文

### 一个问题：为什么 Git 不直接保存？

很多新手会有这个疑问：我改了一个文件，Git 为什么不直接帮我把这个修改存起来？为什么要我分两步走——先 `git add`，再 `git commit`？

答案是：因为 Git 想给你**选择的自由**。你一天改了好几个文件，但不是每个都准备好了要存档。有些改动只是试验性的，有些可能还没写完。Git 的设计让你可以**挑着存**——只把你想存档的那部分内容放入下一份版本。

为了实现这个"挑着存"的功能，Git 把文件从"你改了"到"正式存档"之间的旅程分成了三个站点。

### 三个区域的类比：办公桌、文件夹、档案柜

这是最重要的一组类比，请先记住它，后面的内容都会反复用到：

| Git 概念 | 类比 | 说明 |
|----------|------|------|
| **工作区** | 你的办公桌 | 你在上面写写画画，改来改去，所有修改都在这里发生。别人看不见，乱不乱你自己知道。 |
| **暂存区** | 桌面上的待办文件夹 | 你把改好的纸张整理好，放进这个文件夹里，准备去归档。你可以挑挑拣拣——这张放进去，那张先留着。 |
| **仓库** | 档案柜 | 你打开文件夹，把里面的所有纸张打上编号和日期，郑重地放进档案柜里。一旦入柜，这份记录就永久保存了。 |

下面我们来一一认识这三个区域。

### 工作区（Working Directory）

工作区就是你正在使用的项目文件夹。你在这里创建文件、编辑代码、删除文件。你所做的一切修改，在你 `git add` 之前，都只存在于工作区。

**类比**：工作区就像你厨房里的料理台。你在上面切菜、调味、炒菜。锅里的菜是你正在加工的，可以随意调整——盐少了加盐，火大了关小火。但一旦你装盘上桌（存档），就很难再改了。

当你刚执行完 `git init`，或者刚执行完 `git clone`，工作区里的文件都已经"签到"过了，它们和仓库里最新版本的记录是一致的。但当你开始编辑文件，工作区的文件就和仓库里的记录产生了差异。

### 暂存区（Staging Area / Index）

暂存区是一个"中间地带"。你可以把它想象成购物网站里的**购物车**——你浏览商品（工作区），把想买的放进购物车（暂存区），最后一起结算（提交到仓库）。在结算之前，你可以随时从购物车里拿出不想要的商品。

暂存区的存在给了你两个重要的能力：

1. **选择性存档**：你改了 5 个文件，但只想把其中 2 个保存为一个版本。你可以只把那 2 个文件加入暂存区。
2. **分批存档**：你修改了一个文件的不同部分，一部分是修复 bug，另一部分是添加新功能——你可以把它们分别暂存、分别提交，让历史记录更清晰。

### 仓库（Repository）

仓库就是 Git 存放版本历史的地方。当你执行 `git commit` 时，暂存区里的所有内容会被打包成一个"版本快照"，永久存入仓库。

**类比**：仓库就像是快递公司的包裹仓库。每个包裹（提交）都有唯一的编号（哈希值），记录了什么时间、由谁寄出（提交信息）、里面装了什么（文件快照）。一旦包裹入库，就不会被偷偷调包。

仓库里的数据存储在 `.git` 目录中，理论上你永远不需要手动进去翻找。

### 文件流转的完整路径

一个文件从被创建到被存入仓库，完整路径是：

```
你在编辑器中创建/修改文件     -->  工作区
git add 文件名               -->  暂存区
git commit                   -->  仓库
```

反过来，当你需要调取历史时的路径：

```
git checkout 某个版本         -->  仓库 → 工作区（把档案柜里的旧版本拿出来摆在桌上）
git reset 某个版本            -->  仓库 → 暂存区 + 工作区（更多操作）
```

### 为什么不是两个区域？为什么不是四个？

为什么不直接两个区域——工作区和仓库，省掉暂存区？

**答案**：因为如果没有暂存区，你每次存档就必须把工作区里**所有改动**都存进去。你无法只存一部分。这就好比你整理房间时，必须把桌上所有东西一股脑塞进抽屉——有用的、没用的、还没看完的书全混在一起。暂存区让你先**筛选**和**分类**，再存档。

那为什么不搞四个区域？因为三个足够表达"修改中 → 准备存档 → 已存档"这个流程。加更多的中间状态反而会让操作变复杂，记不住。三个是一个很好的平衡点。

### 一张图总结

```
  工作区                   暂存区                    仓库
┌──────────┐    git add    ┌──────────┐  git commit  ┌──────────┐
│          │ ─────────────>│          │ ────────────>│          │
│ 你的文件  │              │ 准备存档  │              │  版本历史 │
│ 随便改   │              │ 挑好的文件 │              │  永久保存 │
│          │ <─────────────│          │              │          │
└──────────┘   git restore └──────────┘              └──────────┘
     ↑                                                     │
     │                    git checkout                     │
     └─────────────────────────────────────────────────────┘
```

（`git restore` 和 `git checkout` 我们会在后面章节详细讲，这里先看一眼它们的关系）

### 怎么查看三个区域的状态？

在后续章节中，你会学到：

- `git status` —— 告诉你三个区域分别有什么变化（"看看桌上哪些文件还没收好"）
- `git diff` —— 比较不同区域之间的具体差异（"看看这份文件改了什么"）
- `git log` —— 查看仓库里已有的版本历史（"翻翻档案柜里的存档记录"）

## 动手试试

1. 进入上一节创建的 Git 仓库（或者新建一个）
2. 创建一个文件，比如 `echo "hello" > test.txt`
3. 运行 `git status` —— 观察输出，Git 会告诉你 test.txt 在"工作区"（Untracked files）
4. 运行 `git add test.txt`，再运行 `git status` —— 现在 test.txt 到了"暂存区"（Changes to be committed）
5. 运行 `git commit -m "first file"`，再运行 `git status` —— 工作区和暂存区都干净了（nothing to commit, working tree clean），文件已经存档到仓库里了

每个步骤运行 `git status`，看看 Git 告诉你的不同信息——这能帮你感受文件在三个区域之间流转的过程。

## 本节小结

工作区是办公桌（你改文件的地方），暂存区是待办文件夹（你挑选要存档的内容），仓库是档案柜（永久保存的版本历史）。理解这个三层结构，你就理解了 Git 的核心设计思路。
