# 03 追踪改动

## 本节你会学到什么

- 理解"跟踪文件"的含义以及文件的生命周期
- 掌握 `git add` 的多种用法——添加单个文件、批量添加、选择性添加
- 了解 `git add` 对不同类型文件（新建、修改、删除）的行为
- 区分"跟踪文件"和"未跟踪文件"

## 正文

### "跟踪"是什么？

在上一节我们知道了三个区域。但还有一个前提概念需要理清：**跟踪（tracking）**。

"跟踪一个文件"的意思是：Git 在关注这个文件的变化。被跟踪的文件，Git 会记录它的每一次修改。未被跟踪的文件，Git 根本不管。

**类比**：跟踪文件就像是你在公司的人事系统里**入了职**。入职之后，你的考勤、绩效、调薪都被记录。没入职的访客，来去自由，没人记录。

那么怎么"入职"呢？答案就是 `git add`。

### 文件的生命周期

一个文件在 Git 仓库里的"一生"是这样的：

```
未跟踪（Untracked）
    │
    └── git add ──→ 已暂存（Staged）
                        │
                        └── git commit ──→ 已提交（Committed）
                                              │
                                              └── 修改文件 ──→ 已修改（Modified）
                                                                   │
                                                                   └── git add ──→ 已暂存...
```

**未跟踪（Untracked）**：新创建的文件，Git 还不知道它的存在。相当于一个还没入职的新员工。

**已暂存（Staged）**：文件已经被放入暂存区，等待下一次提交。相当于你已经填好了表格，等着按手印（commit）。

**已提交（Committed）**：文件已经存入仓库，成为历史的一部分。

**已修改（Modified）**：已提交过的文件，你在工作区做了修改。相当于老员工调了岗位，需要重新登记。

### git add 的四种姿势

#### 1. 添加单个文件

```bash
git add 文件名
```

最精确的方式。只把指定的这一个文件放入暂存区。

**类比**：你办公桌上有很多纸张，你只挑出其中一张，放进待办文件夹。

#### 2. 添加当前目录下所有文件

```bash
git add .
```

`.` 代表"当前目录"。这条命令会把当前目录里所有新增和修改过的文件都放入暂存区。

**类比**：你把办公桌上所有纸张一股脑儿拨进文件夹——省事，但有时候会把不够格的文件也放进去。

#### 3. 添加所有文件（不管你在哪个子目录）

```bash
git add -A
# 或者
git add --all
```

这个命令会把整个仓库里所有新增、修改、删除的文件都放入暂存区。不管你现在在哪个子目录。

#### 4. 交互式暂存（选择性暂存）

```bash
git add -p
# 或者
git add --patch
```

这是 `git add` 的"精细模式"。当你修改了一个文件的多个地方，但只想把其中一部分修改放入暂存区时，`git add -p` 会逐块（hunk）询问你："这一块要暂存吗？" 你可以选择 y（是）、n（否）、s（拆分成更小块）等。

**类比**：你在一张纸上改了三个地方——修了一个错别字、加了一段话、改了一个标题。你把这张纸递给助手（`git add -p`），助手逐段指给你看："这一段要放进去吗？这一段呢？"

### git add 对已删除文件

如果你删除了一个已经被跟踪的文件，`git add` 也能记录这个"删除"操作。

```bash
rm old_file.txt       # 在工作区删除文件
git add old_file.txt  # 告诉 Git："这个文件我要删了，记下来"
```

或者更简单：

```bash
git rm old_file.txt   # 一步完成删除和暂存
```

### .gitignore：让 Git 忽略某些文件

有时候你不想让 Git 跟踪某些文件——比如编译产生的临时文件、日志文件、包含密码的配置文件。

在项目根目录创建一个 `.gitignore` 文件，里面列出你不想跟踪的文件模式：

```gitignore
# 忽略所有 .log 文件
*.log

# 忽略 node_modules 目录
node_modules/

# 忽略 .env 配置文件（通常包含密码）
.env
```

**类比**：.gitignore 就像是人事部的**不招人名单**。名单上的"人"（文件），就算出现在办公室里，人事系统也不会给他办理入职。

### git add 的常见误解

**误解一**："`git add` 就是保存文件。"
不对。`git add` 只是把文件放入暂存区（文件夹），离正式存档（仓库）还差一步 `git commit`。就好比你把纸张放进文件夹，不代表已经归档到档案柜了。

**误解二**："每次修改文件后都要 `git add` 一次。"
这个也没错，但要注意：`git add` 执行的那一刻，它把文件**当时的快照**放入了暂存区。如果你 `git add` 之后又改了同一个文件，新修改不会自动进入暂存区——你需要再执行一次 `git add`。

## 动手试试

1. 在一个 Git 仓库中创建 3 个文件：`a.txt`、`b.txt`、`c.txt`
2. 只 `git add a.txt`，然后 `git status`，看看哪个在暂存区、哪个不在
3. 修改 `a.txt`，添加一行内容。再次 `git status` —— 你会发现 `a.txt` 同时出现在"待提交"和"未暂存的修改"两个区域！这是因为你 add 的是旧版本的快照，新修改还在工作区
4. 再执行一次 `git add a.txt`，新修改也被放入暂存区
5. 使用 `git add .` 把 b.txt 和 c.txt 也加入暂存区
6. 观察 `git status`，确认所有文件都在暂存区了

## 本节小结

`git add` 是"放入暂存区"的操作，它不直接存档，而是让你在提交前有挑选和整理的机会。文件在 Git 中有清晰的生命周期：未跟踪 → 已暂存 → 已提交 → 已修改 → 再暂存。
