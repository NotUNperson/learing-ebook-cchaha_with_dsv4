# 06 查看状态和改动

## 本节你会学到什么

- 用 `git status` 随时了解三个区域的当前状态
- 用 `git diff` 比较不同区域之间的具体差异
- 理解"尚未暂存"和"已暂存待提交"的区别
- 学会在提交前用 `git diff --staged` 做最后检查

## 正文

### 你的仪表盘：git status

如果说 `git log` 是翻档案柜的历史记录，那么 `git status` 就是你**办公桌上的仪表盘**。它告诉你：

- 哪些文件在工作区被修改了（办公桌上改了的纸张）
- 哪些文件在暂存区等待提交（文件夹里准备归档的纸张）
- 哪些文件 Git 还不认识（没入职的新员工）

你可以在任何时候运行 `git status`，它**不会改变任何东西**，只是给你汇报情况。所以请放心大胆地经常用它。

**类比**：`git status` 就像你下班前环顾办公桌的那一眼——"这个文件改完了放文件夹了，那个文件还在桌上摊着，还有个新文件完全没收拾。" 这一眼让你知道明天从哪儿开始。

### 读懂 git status 的输出

一个典型的 `git status` 输出长这样：

```
On branch master
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
        modified:   index.html

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        new-page.html
```

每一部分的意思：

- **On branch master** —— 你当前在 master 分支上（分支后面章节会讲）
- **Changes not staged for commit**（红色）—— 已跟踪的文件被修改了，但还没放入暂存区。这些修改在**工作区**（办公桌上）。
- **Untracked files**（红色）—— Git 完全不认识的新文件，在**工作区**。
- 如果你已经执行了 `git add`，还会看到 **Changes to be committed**（绿色）—— 这些文件已经进入**暂存区**（文件夹里），等待提交。

Git 还很贴心地在括号里给了**建议命令**——告诉你下一步该怎么做。比如：
- `(use "git add <file>..." to update what will be committed)` —— "想把这些修改也放进去？用 git add"
- `(use "git restore <file>..." to discard changes)` —— "想放弃这些修改？用 git restore"

### git status 的简写模式

```bash
git status -s
# 或
git status --short
```

简写模式用字母缩写表示状态，节省屏幕空间：

```
 M index.html       # M 在左边 = 暂存区里有修改
  M index.html      # M 在右边 = 工作区里有修改（暂存区没有）
MM index.html       # 两边都有 M = 暂存区和工作区都有修改（你没重新 add）
?? new-file.txt     # ?? = 未跟踪文件
A  new-file.txt     # A = 新添加到暂存区的文件
```

### 你的放大镜：git diff

`git status` 告诉你"哪里变了"，`git diff` 告诉你"具体变了什么"。

**类比**：`git status` 是办公桌上的**目录清单**（告诉你哪张纸被改了），`git diff` 是**放大镜**（让你凑近了看纸上具体改了什么字）。

### git diff 的三种比较模式

#### 1. 工作区 vs 仓库（默认）

```bash
git diff
```

比较**工作区**和**仓库里最新提交**之间的差异。也就是说："自从上次存档之后，你在办公桌上又改了什么？"

这不会显示暂存区里的内容——只显示还没被 `git add` 的修改。

#### 2. 暂存区 vs 仓库

```bash
git diff --staged
# 或
git diff --cached
```

比较**暂存区**和**仓库里最新提交**之间的差异。也就是说："文件夹里的内容和档案柜里的最新版本有什么不同？"

这个命令在提交前非常有用——帮你最后确认一次："我马上要提交的这些内容，确实是我想要的吗？"

#### 3. 工作区 vs 暂存区

```bash
git diff HEAD
```

比较**工作区**和**仓库最新的提交**之间的差异（包括暂存区的内容）。这是最全面的比较——"从上次存档到现在，所有改过的地方。"

### 读懂 git diff 的输出

一个典型的 `git diff` 输出长这样：

```diff
diff --git a/index.html b/index.html
index abc1234..def5678 100644
--- a/index.html
+++ b/index.html
@@ -1,5 +1,6 @@
 <h1>我的网站</h1>
-<p>旧的段落</p>
+<p>新的段落</p>
+<p>新增的段落</p>
```

不要被这些符号吓到。我们来逐一解释：

- `--- a/index.html` 和 `+++ b/index.html` —— 表示比较的是哪个文件
- `@@ -1,5 +1,6 @@` —— 表示从旧文件的第 1 行开始的 5 行，对比新文件第 1 行开始的 6 行
- `-` 开头（红色）—— 被删除的行
- `+` 开头（绿色）—— 被添加的行
- 不带符号的行 —— 上下文，没改的行

**类比**：diff 的输出就像是校对员在稿纸上的**红色批注**——划掉要删的，写上要加的。只不过 Git 用 `-` 和 `+` 而不是红笔。

### 比较两个具体提交

```bash
git diff 提交A 提交B
# 例如：
git diff a1b2c3d b2c3d4e
```

比较两次提交之间的差异。"档案柜里的文件袋 A 和文件袋 B，里面的内容有什么不同？"

### 查看某个文件的具体改动

```bash
git diff -- 文件名
```

只查看某一个文件的具体改动。这在改动了很多文件时非常有用——"先把注意力集中在这个文件上"。

### 只看改动了哪些文件，不看具体内容

```bash
git diff --stat
```

类似于 `git status`，但告诉你每个文件大概改了多少行。

### 实战工作流：提交前的两步检查

一个推荐的提交前检查流程：

```bash
# 第一步：看看全局状态
git status

# 第二步：看看还没暂存的改了什么（工作区 vs 仓库）
git diff

# 第三步：把想提交的内容暂存
git add 文件名

# 第四步：看看暂存区里的内容（暂存区 vs 仓库）
git diff --staged

# 第五步：确认无误，提交
git commit -m "清晰的提交信息"
```

**类比**：这就像你交作业前的最后检查——翻一遍作业本确认都写完了（`git status`），逐题检查答案（`git diff`），把作业本放进书包（`git add`），再确认一次书包里的作业没错（`git diff --staged`），最后交给老师（`git commit`）。

## 动手试试

1. 进入一个 Git 仓库，修改一个已有文件，创建一个新文件
2. 运行 `git status`，仔细读每一个区域的输出
3. 运行 `git diff`，观察修改的具体内容
4. 运行 `git add` 暂存一个文件，再运行 `git status` 和 `git diff` —— 你会发现 `git diff` 不再显示那个已暂存的文件了（因为默认比较工作区 vs 仓库）
5. 运行 `git diff --staged`，你会看到已暂存文件的改动
6. 运行 `git diff --stat`，看简洁的统计模式
7. 对某个文件使用 `git diff -- 文件名`，只看这一个文件的改动

## 本节小结

`git status` 是你的仪表盘（看全局状态），`git diff` 是你的放大镜（看具体改了什么）。两个命令配合使用，让你在任何时候都清楚自己的三个区域分别在发生什么。
