# 04 远程分支管理

## 本节你会学到什么

- 搞清楚 `origin/main` 和本地 `main` 的区别
- 理解本地分支和远程分支之间的关系
- 学会查看和创建跟踪远程分支的本地分支
- 掌握把本地分支推送到远程的方法
- 会用 `git fetch --prune` 清理过时的远程分支引用

## 正文

### 一个让新手困惑的画面

你 clone 了一个仓库，然后执行 `git branch`：

```bash
$ git branch
* main
```

只看到一个分支。但执行 `git branch -a`：

```bash
$ git branch -a
* main
  remotes/origin/main
  remotes/origin/dev
  remotes/origin/feature-login
```

你会想：`remotes/origin/main` 是什么？它和本地的 `main` 是同一个东西吗？为什么 `dev` 分支在远程有，在本地看不到？

这一节我们把这些事情彻底搞清楚。

### 本地分支和远程分支——"家里的书"和"图书馆的书"

想象一下：

- 你从图书馆借了一本书（《Git 入门》）回家。这本书现在在你家书架上——这便是你的**本地分支**。
- 图书馆的书架上还有那本书——这便是**远程分支**。
- 你在书上做了笔记（本地提交），你的书和图书馆的书就不一样了。
- 你把书还回图书馆（push），图书馆的书才更新了。
- 别人在图书馆的那本书上贴了便签，你不会自动知道。你得去图书馆看一眼（fetch），才知道有变化。

在 Git 里：

- `main` —— 你的本地分支，就是"你家的那本书"。你可以随便改、随便加笔记。
- `origin/main` —— 远程跟踪分支，就是"图书馆的那本书"。它反映的是**你上次和远程同步时**远程仓库的状态。它不是实时的——就像你不会每秒钟跑去图书馆看那本书有没有变化。

### `origin/main` 到底是什么

`origin/main` 是一个**本地存储的、远程仓库状态的快照**。

- 它存在你的电脑上（所以 `git branch -a` 能看到它）
- 但它不属于你，你不能直接 checkout 到 `origin/main` 上做修改
- 它只在两种情况下更新：你执行了 `git fetch` 或 `git pull`

用更精确的话说：`origin/main` 是"Git 帮你记住的，上一次你跟远程仓库联系时，远程仓库 `main` 分支长什么样"。

你可以这样理解：`origin/main` 就像你手机上的"天气预报缓存"。天气预报 APP 不会每秒钟都去查服务器，而是每隔一段时间刷新一次。在这之间，你看到的是上次刷新的结果。`origin/main` 就是你上次 fetch 时的"缓存"。

### 查看远程分支状态

```bash
# 查看所有分支（包括远程跟踪分支）
git branch -a

# 只查看远程跟踪分支
git branch -r

# 查看本地分支和它跟踪的远程分支的差异
git status
# 会显示类似: Your branch is ahead of 'origin/main' by 2 commits.
# 意思是：你的本地 main 比远程的 main 多了 2 个提交
```

### 跟踪分支——"配对关系"

在 03 节我们提到过 `-u` 参数和"跟踪关系"。现在深入讲一下。

当你执行 `git push -u origin main` 时，Git 做了两件事：
1. 把本地 `main` 推送到远程
2. 建立跟踪关系：让本地 `main` 分支**跟踪**远程 `origin/main`

建立了跟踪关系之后：

- `git push` （不带参数）—— 自动推到跟踪的远程分支
- `git pull` （不带参数）—— 自动从跟踪的远程分支拉取
- `git status` —— 会告诉你本地和远程的提交数差异

怎么查看跟踪关系？

```bash
git branch -vv
```

输出会像这样：

```
* main    a1b2c3d [origin/main: ahead 2] 添加了新功能
  dev     b2c3d4e [origin/dev] 修复了一个 bug
```

`ahead 2` 意思是本地比远程多 2 个提交（还没 push）。如果显示 `behind 3`，就是本地比远程少 3 个提交（需要 pull）。

### 从远程分支创建本地分支

远程仓库上有一个 `dev` 分支，你想在本地也创建一个对应的：

```bash
# 方法一：先 fetch，再创建跟踪分支
git fetch origin
git checkout -b dev origin/dev
# 意思是：创建一个本地 dev 分支，让它跟踪 origin/dev

# 方法二：一步到位（Git 新版本推荐）
git checkout dev
# 如果本地没有 dev 分支，但远程有一个 origin/dev，
# Git 会自动帮你创建本地 dev 并跟踪 origin/dev

# 方法三：用 switch（Git 2.23+ 推荐）
git switch dev
# 效果同上
```

### 把本地分支推送到远程

你在本地建了一个新分支 `feature-x`，想让队友也能看到：

```bash
# 创建并切换到新分支
git checkout -b feature-x
# 做修改...
git add .
git commit -m "完成了 X 功能"

# 推送到远程（第一次推送要建立跟踪）
git push -u origin feature-x
```

推送之后，远程仓库就有了 `feature-x` 分支，队友就可以 fetch 然后 checkout 了。

### 删除远程分支

如果 `feature-x` 已经合并了，不再需要：

```bash
# 删除远程分支（注意冒号语法）
git push origin --delete feature-x

# 或者老式语法（效果一样）
git push origin :feature-x
```

### 清理本地的过时远程引用

你 fetch 之后，发现远程上某个分支已经被删了，但本地 `git branch -a` 还能看到 `remotes/origin/old-branch`。这是因为 Git 不会自动删除本地的远程分支引用。

```bash
# 清理本地已经失效的远程分支引用
git fetch --prune

# 或者
git remote prune origin
```

就像你定期清理手机里的过期天气预报缓存一样。

### 常见场景速查表

| 你想做什么 | 命令 |
|-----------|------|
| 查看所有分支（含远程） | `git branch -a` |
| 查看跟踪关系 | `git branch -vv` |
| 从远程分支创建本地分支 | `git switch <remote-branch-name>` |
| 把本地新分支推到远程 | `git push -u origin <branch-name>` |
| 删除远程分支 | `git push origin --delete <branch-name>` |
| 清理过时的远程引用 | `git fetch --prune` |

## 动手试试

1. 进入你之前克隆的练习仓库：`cd /path/to/your-repo`
2. 执行 `git branch -vv`，看看你的本地分支跟踪了哪个远程分支。
3. 创建一个新分支：`git checkout -b test-remote`
4. 在这个分支上做一个提交（随便改点什么）。
5. 执行 `git push -u origin test-remote`，把分支推到远程。
6. 去 GitHub 网页上看看，是不是多了一个 `test-remote` 分支。
7. 回到终端，删除远程分支：`git push origin --delete test-remote`
8. 删掉本地分支：`git checkout main && git branch -d test-remote`
9. 执行 `git fetch --prune`，然后 `git branch -a`，确认远程分支引用也被清理了。

## 本节小结

远程分支（`origin/main`）是你本地的"远程状态缓存"，不是实时的；本地分支通过跟踪关系与远程分支配对，它们就像"你家的书"和"图书馆的书"——不主动同步就各有各的变化。
