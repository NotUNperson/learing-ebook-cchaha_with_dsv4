# 03 推送和拉取

## 本节你会学到什么

- 用"发信和收信"的类比理解 `git push` 和 `git pull`
- 掌握把本地修改推送到远程仓库的完整流程
- 掌握从远程仓库拉取最新代码的方法
- 理解"推送之前必须先拉取"这个常见场景
- 知道 fetch 和 pull 的区别

## 正文

### 发信和收信——一个生活类比

想象你和远方的朋友写信：

- 你写了一封信，装进信封，贴上邮票，**发出去**——这就是 `git push`。你把本地的内容"发送"给了远程。
- 朋友给你回了一封信，你打开信箱，**收到**了——这就是 `git pull`。你从远程"接收"了新的内容。

Git 的推送和拉取跟这个一模一样。你在本地写代码（写信），然后推送到远程仓库（发信）。别人推送了新的代码（回信），你拉取下来（收信）。

### `git push`——把你的版本上传到云端

回顾上一节的类比：你的手机相册（本地仓库）和云端相册（远程仓库）。

你在手机上拍了新照片（做了新提交），云端相册不会自己知道。你得主动点"同步"，照片才会传上去。

在 Git 里，步骤是这样的：

```bash
# 1. 先做本地提交（在本地"拍照"）
git add .
git commit -m "完成了新功能"

# 2. 推送到远程仓库（"同步到云端"）
git push origin main
```

解释一下这行命令的各个部分：

- `git push` —— 推送命令
- `origin` —— 推到哪个远程仓库（还记得吗，origin 是远程仓库的别名）
- `main` —— 推送哪个分支

完整的意思是：**把我本地 `main` 分支上的新提交，推送到 `origin` 这个远程仓库的 `main` 分支上**。

### 第一次推送要加 `-u`

第一次推送某个分支的时候，通常要加一个 `-u` 参数：

```bash
git push -u origin main
```

`-u` 是 `--set-upstream` 的缩写，意思是"建立跟踪关系"。

什么叫跟踪关系？你可以把它理解成"配对"。你告诉 Git："以后我本地的 `main` 分支，就对应远程 `origin` 的 `main` 分支，记住了。"

建立配对之后，以后只需要敲 `git push` 就行了，不用每次都写 `origin main`。就像你给朋友存了手机号之后，拨号时直接点名字就行，不用每次都拨完整的号码。

### 推送失败的常见情况——"你先收信，再发信"

有时你满怀信心地执行 `git push`，结果收到一个报错：

```
 ! [rejected]  main -> main (fetch first)
error: failed to push some refs to 'https://github.com/...'
```

这是什么意思？

**远程仓库上有你本地没有的新提交**。可能是你的队友先推了一些东西上去。Git 在说："嘿，我这边有你还没看过的东西，你先看看，别把我的覆盖了。"

用信箱类比：你的朋友趁你不在时，也在信箱里放了一封信。现在你想把你的信塞进去，但信箱不是空的。你总得先把别人的信拿走，再放你的。

解决办法：**先拉取，再推送**。

```bash
# 先收信（把远程的新内容拉到本地，和你的内容合并）
git pull origin main

# 再发信（把你的内容推上去）
git push origin main
```

### `git pull`——下载云端的更新

`git pull` 做的事情其实就是两步的合并：

1. **下载**：把远程仓库的新数据下载到本地（这一步叫 `git fetch`）
2. **合并**：把下载下来的新内容和本地内容自动合并（这一步叫 `git merge`）

所以 `git pull` = `git fetch` + `git merge`。

用收信类比：收到信（fetch）之后，你得打开看，把它放进你的知识体系里（merge）。

```bash
# 从 origin 的 main 分支拉取最新代码
git pull origin main

# 如果你已经建立了跟踪关系（用了 -u），直接：
git pull
```

### `git fetch` 和 `git pull` 的区别

| 命令 | 做了什么 | 类比 |
|------|---------|------|
| `git fetch` | 只下载远程的新数据，但不合并到你的工作目录 | 你去邮局取了一封信，但还没拆开 |
| `git pull` | 下载 + 自动合并 | 你去邮局取了信，直接拆开读了 |

什么时候用 `fetch`？当你想先看看远程有什么新东西，但还不想让它影响你当前的工作时。就像你想先看看信的封面，确认一下再拆。

```bash
# 先看看远程有什么新内容
git fetch origin

# 查看远程分支的提交历史（不改变本地代码）
git log origin/main

# 看了之后觉得可以合并，再手动合并
git merge origin/main
```

### 日常工作流——以一天为单位

假设你和同事小美一起开发一个项目，你们都有本地仓库，共享同一个 GitHub 远程仓库。

**早上到公司**：
```bash
# 先拉取小美昨晚推的代码
git pull origin main
```

**开始写代码**：
```bash
# 创建分支，愉快的编码
git checkout -b feature/new-button
# ... 写代码 ...
git add .
git commit -m "添加了新按钮"
```

**准备下班，推送到远程**：
```bash
# 切回 main，拉取今天的远程更新（万一小美白天推了东西）
git checkout main
git pull origin main

# 把自己的分支合并到 main（或者用后面要学的 PR）
git merge feature/new-button

# 推送
git push origin main
```

### 一个完整的 push/pull 示例

假设你已经在 GitHub 上创建了仓库 `my-project`，并且本地也有这个仓库。以下是完整流程：

```bash
# 进入你的仓库
cd my-project

# 确保自己和远程是"配对"的
git remote -v
# 应该看到 origin 指向你的 GitHub 仓库

# 做一些修改
echo "# 我的项目" > README.md
git add README.md
git commit -m "更新 README"

# 推送到远程
git push origin main

# --- 假设现在你换了一台电脑，或者别人推送了新内容 ---

# 拉取远程的最新代码
git pull origin main

# 如果 pull 的时候出现合并冲突，Git 会提示你手动解决
# （冲突处理在本地版本管理章节已经学过，这里完全一样）
```

## 动手试试

**前提**：你需要在 GitHub 上有一个仓库（用 01 节创建的 `my-first-remote` 就行）。

1. 进入你的本地仓库：`cd /path/to/my-first-remote`
2. 创建一个新文件，比如 `notes.txt`，写点内容进去。
3. 提交它：`git add notes.txt && git commit -m "添加笔记"`
4. 推送到远程：`git push origin main`（或者 `git push` 如果你设置了跟踪关系）。
5. 打开 GitHub 网页，刷新，看看 `notes.txt` 是不是出现了。
6. 在 GitHub 网页上，直接点击 `notes.txt` 文件，然后点编辑按钮（铅笔图标），修改一些内容，提交。
7. 回到终端，执行 `git pull origin main`。
8. 打开本地的 `notes.txt`，看看是不是出现了你在网页上改的内容。

重复 6-8 这个循环几次，感受一下"本地改 -> push" 和 "网页改 -> pull" 的双向同步。

## 本节小结

`git push` 是把本地提交上传到远程仓库（发信），`git pull` 是从远程仓库下载并合并新内容（收信），推送前如果远程有新内容需要先拉取再推送，就像信箱里有别人的信时要先取出再放自己的。
