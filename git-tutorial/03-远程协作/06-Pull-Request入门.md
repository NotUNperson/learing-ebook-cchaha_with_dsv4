# 06 Pull Request 入门

## 本节你会学到什么

- 理解 Pull Request 是什么——"请对方审查并合并我的代码"
- 用"作文修改建议"类比理解 PR 的整个流程
- 学会在 GitHub 网页端创建一个 Pull Request
- 了解 PR 审查的基本流程：讨论、修改、合并
- 知道 PR 在团队协作中的真实价值

## 正文

### 一个生活场景——作文修改

假设你是班长，老师让你收全班同学的作文，整理成一本作文集。

你写好了自己的作文。但小明发现你的作文里有个错别字，他想帮你改。

流程是这样的：
1. 小明把你的作文**复印了一份**（Fork），带回家
2. 他在复印件上**改正了错别字**（修改代码）
3. 他把改好的复印件交给你，附上一张便签：**"班长，我帮你改了这行，你看看可以吗？"**（提交 Pull Request）
4. 你看了看，觉得对，就**把他的修改抄到了原稿上**（Merge / 合并）
5. 如果觉得改得不好，你可以**让他再改改**（Request changes）

这就是 Pull Request（简称 PR）的核心流程。它是 GitHub 协作最重要的功能。

### Pull Request 是什么——用 Git 术语说清楚

PR 是一种**请求**：

- **谁请求谁**？贡献者（写代码的人）请求项目维护者（管项目的人）
- **请求什么**？"请看看我改的代码，如果没问题，请合并到你的主项目中"
- **为什么要请求**？因为你没有权限直接修改别人的仓库（就像你不能直接在小明的原稿上写字）

PR 的本质是一句话：

> **"我 fork 了你的项目，做了一些修改。请你审查一下，如果觉得可以，就把我的修改合并进去。"**

### PR 的完整生命周期——一张图说清楚

```
[原项目]                          [你的账号]                          [原项目]
  repo                              fork的repo                           repo
    |                                  |                                   |
    |  1. Fork（复制到名下）           |                                   |
    |--------------------------------->|                                   |
    |                                  |                                   |
    |                                  |  2. Clone 到本地、建分支、修改     |
    |                                  |  3. Push 到你的远程仓库            |
    |                                  |                                   |
    |  4. 创建 Pull Request            |                                   |
    |<---------------------------------|                                   |
    |                                  |                                   |
    |  5. 审查讨论、修改、完善          |                                   |
    |<-------------------------------->|                                   |
    |                                  |                                   |
    |  6. Merge（合并）                |                                   |
    |<--- 修改进入原项目 ---------------|                                   |
```

### 为什么要有 PR——不只是为了改代码

你可能会想：为什么不直接让我 push 到原仓库呢？为什么要绕一圈？

原因有几个：

1. **安全**。你不能随便往别人的仓库 push。就像你不能随便在别人家门口放东西。PR 是一种"敲门-征得同意"的机制。

2. **质量控制**。PR 是一个审查环节。维护者可以检查代码质量、风格、是否有 bug——然后再决定是否接受。这就像你的作文在发表之前有编辑帮你把关。

3. **交流和讨论**。PR 不是一个单向的提交，而是一个**对话**。你提了 PR，维护者可以在具体的代码行上留言讨论。这种"行级讨论"是 GitHub 的重要创新。

4. **记录和归档**。PR 本身就是一份文档。以后有人问"这个功能为什么这样实现"，可以翻出对应的 PR，看到当时的讨论、修改过程、设计决策。这才是最有价值的。

### 在 GitHub 网页端创建 PR（完整教程）

下面通过一个完整的例子来走一遍。

**前提**：你有一个 GitHub 账号，并且已经 Fork 了一个仓库（如果还没有，可以 Fork 官方的 `octocat/Spoon-Knife` 项目来练习）。

#### 第 1 步：Fork 原项目

1. 打开你想贡献的项目页面（比如 `https://github.com/octocat/Spoon-Knife`）
2. 点击右上角的 **Fork** 按钮
3. 选择你的账号（如果提示）

Fork 的结果：你的 GitHub 账号下出现了 `你的用户名/Spoon-Knife` 这个仓库。它是原项目的完整副本（包括所有提交历史和分支）。

类比：你把别人的云相册完整复制到了自己的云空间里。

#### 第 2 步：在你的 Fork 里做修改

我们这次用 GitHub 网页来操作，不用命令行（这样你可以在任何电脑上完成）。

1. 进入你自己的 `你的用户名/Spoon-Knife` 仓库
2. 找到 `index.html` 文件，点击它
3. 点击右上角的**铅笔图标**（Edit this file）
4. 在文件里做一些修改（比如加一行注释）
5. 修改完后，滚动到底部
6. 在 "Commit changes" 区域：
   - 第一栏写一个简短的改动说明（比如 "添加了页面描述注释"）
   - 第二栏可以写详细说明（可选）
   - 选择 "Create a new branch for this commit and start a pull request"
   - 给新分支起个名字（比如 `update-index`）
7. 点击 "Propose changes"

#### 第 3 步：创建 Pull Request

点击 "Propose changes" 之后，GitHub 会自动带你到创建 PR 的页面：

1. **标题**：自动填入你的提交信息，可以修改得更清晰
2. **描述**：写清楚你做了什么修改，为什么这么改。这是给审查者看的，要写认真一点
3. **审查**：下方会显示你修改了哪些文件，每个文件的 diff（差异对比）
4. 点击 "Create pull request"

#### 第 4 步：PR 创建之后

PR 创建后，你会进入一个对话页面。这里有：

- **Conversation（对话）** 标签：大家在这里留言讨论
- **Commits（提交）** 标签：这个 PR 包含的所有提交
- **Files changed（文件变更）** 标签：所有修改的文件和具体改动

原项目的维护者会收到通知，他们可以：
- 在某个代码行上留言（行级评论）
- 请求你修改某个地方
- 批准 PR（Approve）
- 直接合并 PR（Merge）

你收到反馈后，可以在自己的分支上继续修改、推送（PR 会自动更新），直到维护者满意并合并。

### 一个真实的 PR 对话示例

```
[你] 创建了 Pull Request #42
标题: 修复登录页面按钮样式错乱问题
描述: 在移动端 Safari 上，登录按钮的圆角显示不正确。
      原因是 -webkit-appearance 未重置。
      修改: 在 .login-btn 的 CSS 中加了 -webkit-appearance: none;

[维护者] 评论在 styles.css 第 15 行:
> 这个改动会影响所有按钮，不只是登录按钮。
> 能不能限定在 .login-btn 选择器上？

[你] 回复:
> 已经限定了，改动就在 .login-btn {} 里面。

[维护者] 回复:
> 啊对，我看漏了。看起来没问题，合并了！

[维护者] 合并了 Pull Request #42
```

这就是一次典型的 PR 过程——写了代码、交上去、讨论了一下、确认没问题、合并。

### 合并 PR 的三种方式

当审查通过后，维护者点击 Merge 按钮旁边的下拉，有三种合并方式：

| 方式 | 效果 | 什么时候用 |
|------|------|-----------|
| **Create a merge commit** | 保留所有分支历史，创建一个合并提交 | 默认方式，推荐 |
| **Squash and merge** | 把 PR 的所有提交压成一个提交 | PR 里有很多零碎的小提交时 |
| **Rebase and merge** | 把 PR 的提交"接"到目标分支头部 | 想要干净的线性历史时 |

新手不用纠结，用默认的 "Create a merge commit" 就好了。

### "Pull Request" 这个名字为什么奇怪

你可能会困惑：我明明是想把自己的代码"推"进去，为什么叫"Pull" Request（拉取请求）？

这是因为这个名字是从**项目维护者的视角**起的：

- 维护者看到你的 PR，他的操作是：**拉取（pull）你的分支代码，合并到主项目中**。
- 所以 PR 的意思是：**请维护者拉取你的代码**。

理解了这一点，就不觉得名字奇怪了。

## 动手试试

**前提**：你有一个 GitHub 账号。

1. 打开浏览器，访问 [github.com/octocat/Spoon-Knife](https://github.com/octocat/Spoon-Knife)
2. 点击右上角的 **Fork** 按钮，把项目复制到你的账号下。
3. 进入你自己的 `你的用户名/Spoon-Knife` 仓库。
4. 找到 `index.html` 文件，点击右上角的铅笔图标编辑它。
5. 在 HTML 注释 `<!--` 和 `-->` 之间添加一句话（比如 `<!-- 我学会了 PR -->`）。
6. 滚动到底部，选择 "Create a new branch for this commit and start a pull request"。
7. 给分支起个名字，点击 "Propose changes"。
8. 在 PR 创建页面，检查标题和描述，点击 "Create pull request"。
9. （可选，但推荐）在自己的 Pull Request 页面下，自己留一条评论，感受一下 "Conversation" 功能。
10. 点击 "Files changed" 标签，看看你的改动被展示成什么样的 diff。

完成后，你就拥有了第一个真实的 Pull Request！

注意：因为 `octocat/Spoon-Knife` 是 GitHub 官方的示例项目，你的 PR 可能不会被合并（维护者不一定在线）。这不重要——你体验了整个流程就好。

## 本节小结

Pull Request 是 GitHub 协作的核心机制：贡献者 fork 项目、修改代码、提交 PR，维护者审查讨论后决定是否合并，整个过程就像一个"带审查的修改提议"制度，保证了代码质量和团队交流。
