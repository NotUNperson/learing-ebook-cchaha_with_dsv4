## 正文

### 恭喜你，学到最后一节了

前面六节，你已经分别学习了远程仓库、克隆、推送拉取、远程分支、GitHub 操作和 Pull Request。这一节不做新概念的讲解，而是把所有这些串在一起，**完整地走一遍远程协作流程**。

这个流程，就是一个专业开发者每天的日常。

把这次练习当成一场**彩排**。以后你真正参与项目时，每个步骤都会觉得很熟悉。

### 完整流程概览

```
你（贡献者）                         项目维护者（原项目）
─────────                          ─────────
第1步: Fork 原项目到自己的账号
        └─> 你的账号/project
第2步: git clone 到本地
第3步: 创建功能分支
第4步: 写代码、提交
第5步: git push 到你的远程仓库
第6步: 在 GitHub 创建 Pull Request ──> 收到 PR 通知
                                      第7步: 审查代码
                                      第8步: 讨论/要求修改
        <── 收到反馈，继续修改 ──────────
        └─> 修改后 push，PR 自动更新
                                      第9步: 批准并合并 PR
        <── 你的代码进入原项目！──────────
```

### 详细步骤：跟着走一遍

#### 第 1 步：找一个练习项目

如果你还没有可以 Fork 的项目，这里有两个选择：

- **选项 A**：用 GitHub 官方的 `octocat/Spoon-Knife`（专为新手练习 PR 而设）
- **选项 B**：自己先创建一个测试项目（你当维护者），然后用另一个账号（或让朋友）来 Fork

本教程用选项 A 来演示。

#### 第 2 步：Fork 项目

1. 打开浏览器，访问 [github.com/octocat/Spoon-Knife](https://github.com/octocat/Spoon-Knife)
2. 点击右上角的 **Fork** 按钮
3. 等待几秒，GitHub 会把项目完整复制到你的账号下
4. 你现在会在 `https://github.com/你的用户名/Spoon-Knife` 看到自己的副本

**关键理解**：Fork 不是下载，是在 GitHub 服务器上创建了原项目的完整副本，包括所有分支和提交历史。这个副本完全属于你，你可以随意修改。

#### 第 3 步：Clone 到本地

打开终端，找一个合适的目录：

```bash
# 克隆你自己账号下的 Spoon-Knife（注意：是你 Fork 之后的地址！）
git clone https://github.com/你的用户名/Spoon-Knife.git

# 进入项目目录
cd Spoon-Knife

# 看看有什么
ls -la
git log --oneline
git remote -v
```

你会看到 `origin` 指向你 Fork 的仓库，这正是我们需要的。

#### 第 4 步：添加上游仓库（Upstream）

"上游"（upstream）是指原项目。添加它，是为了方便以后同步原项目的最新代码：

```bash
# 把原项目的地址添加为 upstream
git remote add upstream https://github.com/octocat/Spoon-Knife.git

# 验证
git remote -v
# 应该看到 origin 指向你的仓库，upstream 指向原项目
```

这个操作不是每次都要做的，但给 Fork 的项目加上 upstream 是好习惯。这样当原项目更新时，你可以用 `git pull upstream main` 来同步。

#### 第 5 步：创建功能分支

永远不要直接在 main 分支上做修改——这是项目的"黄金规范"：

```bash
# 确保 main 是最新的
git checkout main

# 创建并切换到新分支
git checkout -b add-index-description
```

分支名要有意义：`add-xxx`（添加功能）、`fix-xxx`（修复 bug）、`update-xxx`（更新内容）。

#### 第 6 步：写代码，做提交

```bash
# 打开编辑器，找到 index.html，做一些修改
# 比如在 HTML 文件的注释里添加一段描述

# 查看修改了什么
git diff

# 提交
git add index.html
git commit -m "在 index.html 中添加页面描述注释"
```

提交信息要具体一点——不是 "update" 或 "fix bug"，而是说清楚你做了什么："在 index.html 中添加页面描述注释"。

#### 第 7 步：推送到你的远程仓库

```bash
# 第一次推送这个分支
git push -u origin add-index-description
```

推送后，去 GitHub 你的 Spoon-Knife 仓库页面看看，应该能看到一个新分支的提示。

#### 第 8 步：创建 Pull Request

1. 在浏览器进入你的 `你的用户名/Spoon-Knife` 仓库
2. GitHub 通常会自动显示一个黄色的提示条："add-index-description had recent pushes"，旁边有 "Compare & pull request" 按钮——点击它
3. 如果没看到提示条，就切换到 `add-index-description` 分支，然后点击 "Contribute" 下拉，选 "Open pull request"
4. 在 PR 创建页面：
   - **标题**：简洁描述你的改动
   - **描述**：详细说明改了什么、为什么这么改。一个好的 PR 描述模板：

   ```
   ## 改了什么
   在 index.html 的 HTML 注释中添加了页面描述

   ## 为什么这样改
   原来的注释太简略，新开发者读到这个文件时不清楚用途

   ## 如何测试
   打开 index.html，在浏览器开发者工具中查看源码注释
   ```

5. 点击 "Create pull request"

#### 第 9 步：等待审查

PR 创建后，原项目的维护者会看到你的请求。他们可能会：

- 合并（Merge）—— 直接接受
- 评论（Comment）—— 有讨论的地方
- 请求修改（Request changes）—— 需要你改

如果收到修改请求，相应地更新代码、提交、推送，PR 会自动更新。

#### 第 10 步：PR 被合并后

当你的 PR 被合并后：

```bash
# 切回 main
git checkout main

# 从原项目拉取最新代码（因为你的 PR 已经被合并进去了）
git pull upstream main

# 推送到你自己的远程仓库（保持同步）
git push origin main

# 可以删除本地功能分支（远程的 GitHub 通常会帮你删）
git branch -d add-index-description
```

### 好习惯清单

在真实的协作中，以下几个习惯会让你的 PR 更容易被接受：

| 习惯 | 为什么重要 |
|------|-----------|
| **一个 PR 只做一件事** | 审查者不用在一堆无关改动里找你的核心修改 |
| **分支名清晰有意义** | 别人一看就知道这个分支是干嘛的 |
| **提交信息具体** | 不说 "fix"，而说 "修复登录按钮在 Safari 上的样式错乱" |
| **写好的 PR 描述** | 解释清楚"改了什么、为什么、怎么测"，节省审查者时间 |
| **PR 提交前自己先检查** | 用 `git diff` 自己看一遍，确保没有多余的改动 |
| **保持 PR 小** | 几百行的 PR 容易审查，几千行的 PR 让人头疼 |

### 从"学会 Git"到"参与开源"

恭喜你，当你走完这个流程，你就具备了参与开源项目的基本能力。以下是一些给新手的建议：

1. **从自己熟悉的小项目开始**。找你用过的库或工具，看看有没有标记为 "good first issue" 的 Issue（专为新手准备的简单任务）。
2. **先读 CONTRIBUTING.md**。大多数项目有这个文件，告诉你如何贡献。
3. **先讨论再写代码**。在 Issue 里说明你想做什么，避免做了半天发现维护者不想要。
4. **不要怕被拒绝**。PR 被要求修改甚至被拒绝都很正常。这是学习的过程，不是失败。

## 动手试试

这个"动手试试"就是走一遍本节的整个流程。

目标：给 `octocat/Spoon-Knife` 项目（GitHub 官方的 PR 练习仓库）提交一个 Pull Request。

**完整清单**（预计 15-20 分钟）：

- [ ] 在 GitHub 上 Fork `octocat/Spoon-Knife`
- [ ] `git clone` 你 Fork 的仓库到本地
- [ ] `git remote add upstream` 添加原项目
- [ ] `git checkout -b my-first-contribution` 创建分支
- [ ] 修改 `index.html`（比如在注释里加一句话）
- [ ] `git add` + `git commit` 提交修改
- [ ] `git push -u origin my-first-contribution` 推送到你的远程仓库
- [ ] 在 GitHub 上创建 Pull Request
- [ ] 在 PR 描述中写清楚你的修改
- [ ] 检查 "Files changed" 标签，确认改动正确
- [ ] （可选）如果维护者回复了，试着回复他

等你完成这些，你就是一名真正的开源贡献者了！

## 本节小结

远程协作的完整流程是：Fork 项目到名下、Clone 到本地、建分支做修改、Push 推送、创建 Pull Request 请求合并——这个流程是参与开源项目和团队协作的基础，掌握它就是掌握了 Git 远程协作的全部核心技能。
