# 01 什么是远程仓库

## 本节你会学到什么

- 用"云端相册"来理解远程仓库到底是什么
- 弄清楚本地仓库和远程仓库之间的关系
- 明白为什么要用远程仓库——不止是备份
- 认识 Git 里最常用的远程仓库托管平台
- 把本地仓库推送到远程仓库

## 正文

### 先回忆一下：你手里有什么

学完前面的内容，你的电脑上已经有了 Git 仓库——就是那个 `.git` 目录，里面存着你所有的提交记录。你在本地可以随意提交、建分支、回退版本，Git 把所有东西都管得好好的。

但现在有一个问题：**这些东西全在那台电脑上**。如果你的硬盘坏了、电脑丢了，版本历史也跟着没了。而且，如果你想跟朋友合作写代码，你总不能把 U 盘递过去说"给你我的仓库"吧？

这就是远程仓库要解决的问题。

### 远程仓库是什么——云端相册类比

想象一下你手机里的相册。你拍了照片，照片存在手机本地——这就像你的**本地仓库**。

但你怕手机坏了照片全丢，于是你把这些照片同步到了云相册（比如 iCloud、Google Photos）。云端的那个相册，保存着和你手机里一模一样的照片，而且还能帮你同步到其他设备——这就像**远程仓库**。

| 生活中的事 | Git 里的事 |
|-----------|-----------|
| 用手机拍照 | `git commit`，在本地保存一个版本 |
| 照片存在手机里 | 提交记录存在本地仓库（`.git` 目录） |
| 把照片同步到云相册 | `git push`，把本地仓库的提交推送到远程仓库 |
| 从云相册下载照片到新手机 | `git clone`，从远程仓库把整个项目拉下来 |
| 云相册帮你同步到平板 | 团队其他成员 `git pull` 获取你的更新 |

**核心理解**：远程仓库就是放在服务器上的一份"备份"，但它不只是备份——它是一个可以被多人访问、同步的**共享版本库**。

### 为什么不能只用 U 盘或网盘

你可能会想："那我用百度网盘或者 QQ 传文件不就行了？"

不行。因为直接传文件有几个致命问题：

1. **你不知道对方改了什么**。你收到一个项目文件夹，里面可能有对方修改的好几个文件。你得一个一个打开看才知道改了哪，这太痛苦了。
2. **两个人同时改怎么办**。你改了 A 文件发给我，我也改了 A 文件发给你——两份修改怎么合并？传统做法是互相发消息："你别动那个文件，我在改！"——这根本没法协作。
3. **没有版本历史**。网盘里的文件只是"当前状态"。你想知道三个月前这行代码是谁写的、为什么这么写？网盘不会告诉你。

Git 远程仓库解决了所有这些问题：每个人的修改都有记录，合并有机制，版本历史完整保留。

### 本地仓库和远程仓库的关系——"镜子和人"类比

把远程仓库想象成一面**公共的镜子**，每个人都是站在镜子前的人。

- 你打扮好了（本地提交了），走到镜子前照一照——这就是 `git push`，把你的状态反映到远程。
- 别人在镜子里看到了你，也看到了自己的样子——这就是大家都共享同一个远程仓库。
- 你想看看别人打扮成什么样了，去照一下镜子——这就是 `git pull`，把远程的新变化同步到本地。

关键点：**你本地的操作（提交、建分支）不会自动跑到远程去**。就像你换了一身衣服，如果不照镜子，镜子里的人不会自己变。你得主动推送（push），远程仓库才会更新。

### 远程仓库长什么样

说白了，远程仓库就是一个放在服务器上的 Git 仓库。它和你本地的 `.git` 目录本质上是同一套东西——一堆对象文件存储着版本历史。

目前最流行的 Git 远程仓库托管平台有这些：

- **GitHub**：全球最大的开发者社区，几乎所有开源项目都在上面。你可以把它理解成"代码界的微博"——不只是存代码的地方，更是社交平台。
- **GitLab**：功能类似 GitHub，很多公司用它搭建内部代码服务器。
- **Gitee（码云）**：国内的代码托管平台，访问速度快。

你在这些网站上创建的每一个"项目"（repository，简称 repo），本质上就是一个远程仓库。你的本地仓库可以和它关联起来，互相推送和拉取代码。

### 给你的本地仓库添加一个远程仓库

远程仓库通常用 URL 来标识。一个远程仓库的 URL 可以长这样：

```
https://github.com/你的用户名/你的项目名.git
```

或者用 SSH 的方式：

```
git@github.com:你的用户名/你的项目名.git
```

要把本地仓库和远程仓库关联起来，用 `git remote add` 命令：

```bash
# 给本地仓库添加一个远程仓库，"origin" 是远程仓库的别名
git remote add origin https://github.com/你的用户名/你的项目名.git
```

这里的 `origin` 是什么意思呢？它只是远程仓库的一个**别名**，就像你给手机通讯录里存的名字。你可以给远程仓库起任何名字，但约定俗成，第一个远程仓库大家几乎都叫它 `origin`——就像每个人的第一个好朋友常常叫"小明"一样。

你可以查看当前关联了哪些远程仓库：

```bash
# 查看远程仓库列表
git remote -v
```

### 把本地仓库推送到远程

关联好远程仓库后，就可以把本地的提交推送到远程了：

```bash
# 把 main 分支推送到远程（-u 表示建立跟踪关系，后面会详细讲）
git push -u origin main
```

执行成功后，你的提交记录就上传到远程仓库了。打开 GitHub 网页，你就能看到你的代码和完整的提交历史。

## 动手试试

1. 打开终端，进入你之前练习用的任何一个本地 Git 仓库（如果没有，新建一个并做几次提交）。
2. 执行 `git remote -v`，看看有没有已经关联的远程仓库。
3. 读一读输出的内容，试着理解每一部分的意思。

现在，我们去 GitHub.com 上实际操作一下：
1. 打开 [github.com](https://github.com)，注册一个账号（如果还没有的话）。
2. 点击右上角的 "+" 号，选择 "New repository"，创建一个新仓库（名字随意，比如 `my-first-remote`）。
3. 创建后，GitHub 会给你一段命令提示。找到 `git remote add origin ...` 那一行。
4. 把这行命令复制到你的终端执行。
5. 然后执行 `git push -u origin main`（如果你的默认分支叫 master，就改成 master）。
6. 刷新 GitHub 页面，看看你的代码是不是出现在网页上了！

## 本节小结

远程仓库就是放在服务器上的共享版本库，本地仓库是你的私有工作区，它们通过 push（上传）和 pull（下载）来同步——就像你的手机相册和云端相册的关系。
