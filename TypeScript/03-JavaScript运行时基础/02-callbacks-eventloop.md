## 正文

### 什么是回调函数

你在 C++ 里可能用过函数指针或 `std::function`——把一个函数当作参数传给另一个函数，让它在合适的时机调用。JavaScript 里这个概念叫做**回调函数（callback）**，用法类似但更常见，几乎无处不在。

**一个最简单的类比：** 你去朋友家做客，朋友说"我烧壶水，水开了叫我一声"。这里"叫我一声"就是一个回调——你朋友（被调方）知道水开之后该做什么（回调函数），而不是你一直站在水壶旁边盯着。

转成代码思路：

```typescript
// 烧水函数：接受一个回调，水开后调用它
function boilWater(callback: () => void): void {
  console.log("开始烧水...");
  // 模拟烧水需要 3 秒
  // setTimeout 是 JS 内置的定时器，等指定毫秒数之后执行回调
  setTimeout(() => {
    console.log("水开了！");
    callback(); // 调用你传进来的"叫我一声"
  }, 3000);
}

boilWater(() => {
  console.log("主人，水开了，可以泡茶了！");
});
```

关键理解：`boilWater` 不会"阻塞"等待 3 秒——它注册了回调之后就返回了，你的程序可以接着做别的事。3 秒后，回调被触发。

### setTimeout 详解

`setTimeout` 是 JavaScript 内置的定时器函数：

```typescript
setTimeout(callback, delayInMilliseconds);
```

- 第一个参数：到时间后要执行的函数（回调）
- 第二个参数：等待多少毫秒（1000 毫秒 = 1 秒）
- 返回值：一个数字 ID，可以用 `clearTimeout(id)` 取消定时

看一个带时间戳的例子：

```typescript
console.log("1. 程序开始");

setTimeout(() => {
  console.log("3. 两秒后执行这里");
}, 2000);

console.log("2. 程序继续");

// 输出顺序：
// 1. 程序开始
// 2. 程序继续
// 3. 两秒后执行这里
```

注意输出的顺序！不是 1-3-2，而是 1-2-3。因为 `setTimeout` 是**非阻塞**的——它说"两秒后干这个"然后立刻返回，程序继续往下走，两秒后回调才被触发。

如果把延迟改成 0，会怎么样？

```typescript
console.log("A");

setTimeout(() => {
  console.log("C");
}, 0);

console.log("B");

// 输出顺序依然是：A - B - C
// 即使延迟是 0，回调也不会立即执行，而是要排队
```

即使延迟是 0 毫秒，回调也不会"插队"。这是因为 JavaScript 有一个精妙的机制：**事件循环**。

### 事件循环 —— 餐厅点餐模型

JavaScript 是**单线程**的，意味着同一时间只能干一件事。如果发一个网络请求时程序傻等着，整个页面（或服务器）就卡死了。JavaScript 的解决方案是事件循环（Event Loop）。

**餐厅点餐类比：**

想象你走进一家人气很旺的面馆：

1. 你点了碗牛肉面（相当于发起一个网络请求 / 读文件 / 设定时器）
2. 服务员把订单撕下来交给厨房（相当于把回调函数注册到"任务队列"）
3. 服务员不会站在厨房门口等面做好——她立刻去服务下一位客人（主线程继续执行后面的代码）
4. 厨房做好了面，打铃（相当于异步任务完成）
5. 服务员手上没活时，听到铃声，去端面送到你桌上（相当于事件循环从任务队列取出回调，在合适的时机执行）

转换到代码世界：

- **主线程** = 服务员。一直干活，一件事一件事处理。
- **异步任务**（读文件、网络请求、定时器）= 厨房。在后台慢慢做，不占用服务员。
- **任务队列**（callback queue）= 出餐口。厨房做完的菜（完成的任务）放在这里排队。
- **事件循环** = 一个永不停止的"检查器"。每处理完手头的事，就去任务队列看看有没有新做好的菜。如果有，取出对应的回调执行。

用代码演示这个模型：

```typescript
console.log("客人进店");              // ① 主线程执行

setTimeout(() => {                    // ② 注册定时器，交给"厨房"
  console.log("面好了，端给客人");   // ⑤ 2 秒后回调被事件循环取出执行
}, 2000);

console.log("服务员去服务下一位客人"); // ③ 主线程继续，不等定时器

// ④ 2 秒内主线程可能在做别的事（或空闲），定时器到期后回调排队
// ⑤ 事件循环发现主线程空了，从队列取出回调执行
```

### 回调的嵌套问题

回调很好用，但如果你有很多"等 A 做完做 B，B 做完做 C"的任务，代码会变成：

```typescript
readFile("a.txt", (dataA) => {
  readFile("b.txt", (dataB) => {
    readFile("c.txt", (dataC) => {
      // 代码越来越往右缩进……
      console.log(dataA, dataB, dataC);
    });
  });
});
```

这个现象叫**回调地狱（Callback Hell）**——代码层层嵌套，像金角大王的葫芦越来越深，难以阅读和排错。后面章节我们会看到 Promise 和 async/await 如何优雅地解决这个问题。

## 动手试试

1. 写一个 `setTimeout`，在 3 秒后打印"三秒到了"。
2. 在同一段代码里，在 `setTimeout` 之前和之后各写一个 `console.log`。
3. 运行代码，观察输出顺序是否跟你想的一样。
4. 尝试把延迟改成 0，观察顺序变化。
5. 思考：如果把 10 个 `setTimeout(fn, 1000)` 一个接一个写（不是嵌套），它们是在同一秒后同时触发，还是每隔一秒触发一个？

## 本节小结

回调函数让你把"等会再做的事"塞给系统，系统到时调用它；事件循环则是 JavaScript 单线程模型下的调度中心，用"服务员和厨房"的协作方式让程序既不会卡死，又能高效完成异步任务。
