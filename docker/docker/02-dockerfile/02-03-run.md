# 02-03 RUN 指令：在构建时执行命令

## 本节你会学到什么

- 掌握 RUN 指令的两种写法：shell 形式和 exec 形式
- 理解镜像层的概念以及为什么需要合并 RUN 指令
- 会用 `&&` 和 `\` 把多条命令合并到一层
- 了解构建缓存对 RUN 的影响

## 正文

### RUN 是什么

RUN 指令让你在镜像构建的过程中执行命令。注意，是**构建时**，不是运行时。RUN 的结果会固化在镜像里，容器启动后这些结果已经在了。

打个比方，RUN 就像装修队在施工时做的事。房子还没住人呢，工人在里面铺电线、贴瓷砖、刷墙。等这些活干完了，房子才算装修好（镜像构建完成）。你搬进去住（启动容器）的时候，电线已经在墙里了，瓷砖已经在地上了。

而 CMD/ENTRYPOINT（下节会讲）是你住进去以后每天做的事——比如每天早上泡咖啡。

### 两种写法：shell 与 exec

RUN 有两种写法，区别在于谁来执行你的命令：

**Shell 形式**（通过 `/bin/sh -c` 执行）：

```dockerfile
RUN apt-get update
RUN apt-get install -y nginx
```

**Exec 形式**（直接调用可执行文件，不用 shell）：

```dockerfile
RUN ["apt-get", "update"]
RUN ["apt-get", "install", "-y", "nginx"]
```

Shell 形式的好处是可以用 shell 的特性：管道 `|`、环境变量替换 `$HOME`、通配符 `*`。大部分时候你用这个就够了。

Exec 形式适用于基础镜像没有 shell 的情况（比如 scratch），或者你想精确控制命令执行的场景。

大多数时候，用 shell 形式就好，可读性更好。

### 每行 RUN 就是一个层

还记得 02-01 说的"多层蛋糕"吗？Dockerfile 的每一行指令都会生成一个新的镜像层。每行 RUN 也不例外。

这意味着你写：

```dockerfile
RUN apt-get update
RUN apt-get install -y nginx
RUN apt-get install -y curl
RUN apt-get clean
```

实际上产生了 4 个层。这不仅让镜像变大，还会因为每一层都是独立的，导致 `apt-get clean` 没法清理前几层产生的临时文件。

### 合并 RUN：用 && 把命令串起来

正确的做法是把相关的命令合并到一条 RUN 里：

```dockerfile
RUN apt-get update && \
    apt-get install -y nginx curl && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*
```

注意几个技巧：
- `&&` 确保前一条命令成功才继续（不用 `;`）。
- `\` 是 shell 的续行符，让你可以换行写，保持可读性。
- 把清理命令放在同一层，才能真正释放空间。
- 安装完立刻清理 apt 缓存，否则缓存会固化在层里，删不掉。

### 为什么缓存有讲究？

Docker 构建时是按行缓存的。如果某一行或之前的行没变，Docker 就直接用缓存而不重新执行。来看一个例子：

```dockerfile
FROM node:20-alpine
RUN apk add --no-cache git
RUN git clone https://github.com/some/large-repo.git /app
RUN npm install -g some-tool
```

第二次构建时，Docker 发现第一行没变，用缓存；第二行 `apk add` 没变，用缓存；第三行 `git clone` 没变，用缓存。但如果你把 `git clone` 和 `npm install` 分开了，一旦 git clone 那一层失效（比如仓库更新了），npm install 也得重跑——哪怕它的命令没变。

所以：**稳定不常变的命令放前面，经常变的放后面**。这个原则在 COPY 指令中更加重要（详见 02-04 和 02-09）。

### 一个实用的 RUN 示例

以下是一个为 Python 应用准备环境的 RUN 指令，展示了层合并的最佳实践：

```dockerfile
FROM python:3.12-slim

RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        gcc \
        libpq-dev \
    && apt-get clean && \
    rm -rf /var/lib/apt/lists/* && \
    pip install --no-cache-dir \
        django \
        psycopg2 \
        gunicorn
```

要点解析：
- `--no-install-recommends` 只安装必需的依赖，不安装"推荐"的额外包，减小体积。
- `apt-get clean` 和 `rm -rf /var/lib/apt/lists/*` 清理包管理器的缓存。
- `pip install --no-cache-dir` 不让 pip 缓存下载的包。
- 全部合并到一条 RUN 里，清理才有效。

## 动手试试

找一个已有的 Dockerfile，或者新建一个，试试以下操作：

1. 用 `docker build --no-cache` 构建，观察构建时间。
2. 再次构建（不加 `--no-cache`），观察哪些层被缓存了（输出里会显示 `CACHED`）。
3. 尝试把两条 RUN 合并成一条，构建后对比 `docker images` 的 SIZE 变化。

## 本节小结

RUN 在**构建时**执行命令并固化到镜像层。用 shell 形式（普通命令）即可，关键是把相关命令用 `&&` 合并到一条 RUN 中，既减小体积又确保清理生效。
