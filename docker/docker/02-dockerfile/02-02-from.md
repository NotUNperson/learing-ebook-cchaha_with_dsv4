# 02-02 FROM 指令：选择基础镜像

## 本节你会学到什么

- 理解 FROM 指令的作用和基础镜像的概念
- 区分 Alpine、Slim、完整版镜像的特点和适用场景
- 掌握镜像版本选择策略（固定版本 vs latest）
- 了解 scratch 和 distroless 这样的特殊镜像

## 正文

### 从"地基"说起

FROM 是每个 Dockerfile 的第一条指令（注释除外）。它告诉 Docker："我要在谁的基础上接着盖房子"。

盖房子不能凭空盖，你得先有块地、打个地基。FROM 指定的基础镜像，就是你这栋房子的地基。你选的不是 alpine 就是 ubuntu，不是 debian 就是 centos——它们都是"地基"。地基选得好不好，直接影响房子的安全性、大小和施工难度。

比如你只用 Node.js 写个简单 API，选 ubuntu:22.04 作为基础镜像就像为了盖个小木屋打了十米深的地基——不是不行，但太重了。

### 三大流派：Alpine、Slim、完整版

Docker 官方镜像（以及很多第三方镜像）通常提供多个变体。我们以 Node.js 为例，看看这三个版本的差异：

| 版本 | 大小 | 内置工具 | 适用场景 |
|------|------|----------|----------|
| node:20 | ~1.1 GB | Debian 完整系统 | 本地开发、调试 |
| node:20-slim | ~250 MB | Debian 精简版 | 生产环境常用 |
| node:20-alpine | ~130 MB | Alpine Linux | 追求极致体积 |

**完整版（默认标签）**：基于 Debian/Ubuntu 完整系统。自带 shell、包管理器、编译工具链。好处是"啥都有"，坏处是"太胖了"。适合本地开发，因为你需要进容器调试、装工具。

**Slim 版**：同样是 Debian/Ubuntu，但砍掉了文档、不常用的工具、编译器等。体积大概是完整版的四分之一。对于大多数生产环境来说，这是一个很好的平衡点。

**Alpine 版**：基于 Alpine Linux，一个专为容器设计的安全轻量发行版。体积超级小（基础系统才 5MB），但因为用的是 musl libc 而不是 glibc，某些依赖原生 C 扩展的应用（比如编译 Python 的某些科学计算包）可能会踩坑。如果你追求极致的小镜像，Alpine 是首选。

```dockerfile
# 三个版本的对比 —— 三种地基
FROM node:20           # 精装大平层，啥都有，1.1GB
FROM node:20-slim      # 简装公寓，够用但不奢侈，250MB
FROM node:20-alpine    # 极简胶囊房，麻雀虽小五脏俱全，130MB
```

### 选版本就像买机票

选基础镜像版本跟买机票有点像：

- **latest 标签**像买"浮动票价"机票——每次买价格都可能不一样，你没法确定明天构建出来的是同一个镜像。Node:20 今天可能是 20.11.0，过两个月就变成 20.12.0 了。
- **固定版本**像买"锁定票价"机票——`node:20.11.0-alpine3.19` 就是铁定的，今天和明天构建出的一样。

所以生产环境一定要用**精确版本**：

```dockerfile
# 不好：谁知道今天拉下来的是什么
FROM node:latest

# 还行：大版本固定了，小版本会自动更新
FROM node:20-alpine

# 最佳：精确到小版本和 Alpine 版本
FROM node:20.11.0-alpine3.19
```

### scratch：从零开始

Docker 还提供了一个叫 `scratch` 的特殊镜像——它什么都不是，完全空白，连 shell 都没有。只有在构建**静态编译的二进制程序**时才会用到它。

```dockerfile
# 适用于 Go、Rust 等编译为单一二进制文件的语言
FROM scratch
COPY myapp /myapp
ENTRYPOINT ["/myapp"]
```

用 scratch 做出来的镜像极小（就是你二进制文件的大小），但也极度受限：你不能 `docker exec` 进去调试，因为没有 shell。这也是为什么后来出现了 distroless（Google 出品），它在 scratch 之上加了一个最小化的 Linux 用户态，有 CA 证书、时区数据等基础文件，但依然没有 shell。

### Alpine 的坑：musl vs glibc

Alpine 虽然小，但有个常见的坑。Alpine 使用 musl libc 而不是大多数 Linux 发行版使用的 glibc。对于纯脚本语言（Python、Node.js、Ruby 的纯 JS 包），这基本没问题。但如果你用到了一些编译 C 扩展的 Python 包（比如 pandas、numpy），用 Alpine 可能会编译失败或运行时报错。

解决方案有两个：一是用 slim 版（基于 Debian，有 glibc）；二是在 Alpine 里安装编译依赖（gcc、musl-dev 等），但这就违背了"小"的初衷。选择前一定要测试。

### distroless：更安全的"裸"镜像

Google 出品的 distroless 镜像是一种特殊选择。它在 scratch 之上加了最小化的 Linux 用户态文件（CA 证书、时区数据、passwd 文件等），但依然没有 shell、没有包管理器。

```dockerfile
# 使用 distroless 运行 Node.js 应用
FROM node:20-alpine AS build
WORKDIR /app
COPY . .
RUN npm ci && npm run build

FROM gcr.io/distroless/nodejs20-debian12
COPY --from=build /app/dist /app
CMD ["/app/index.js"]
```

distroless 的优点：最小攻击面（没 shell 没法 exec 进去执行命令），体积也很小。缺点：调试极其困难——基本只能靠日志。适合对安全性要求极高的生产环境。

### 镜像选择决策树

遇到一个新项目，选择基础镜像时可以参考这个思路：

```
你的应用是编译型语言（Go/Rust/Java）吗？
├── 是 → 最终运行时用 scratch 或 distroless
│        构建阶段用对应语言的官方镜像
│
└── 不是（Python/Node.js/Ruby）
    ├── 依赖包含编译 C 扩展的包（pandas/grpc 等）吗？
    │   ├── 是 → 用 slim 版（基于 Debian，有 glibc）
    │   └── 不是 → Alpine 版是首选
    │
    └── 需要进容器调试吗（开发环境）？
        ├── 是 → 用完整版或 slim 版
        └── 不是 → Alpine 版
```

| 镜像 | 大小 | 包管理器 |

| alpine:3.20 | ~7 MB | apk |

## 动手试试

在空目录里创建两个 Dockerfile，一个用 `FROM node:20-alpine`，一个用 `FROM node:20`，都在最后加一行 `CMD ["node", "-e", "console.log('Hello from', process.platform)"]`。分别构建成 `test-alpine` 和 `test-full`，运行一下，然后用 `docker images` 比较两者的体积差。

## 本节小结

FROM 指定了镜像的"地基"，选择 Alpine/Slim/完整版要在体积、兼容性和调试便利性之间做权衡，生产环境务必使用精确版本号。
