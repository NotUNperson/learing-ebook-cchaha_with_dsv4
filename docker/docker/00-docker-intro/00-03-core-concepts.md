# 00-03 镜像、容器、仓库 —— Docker 世界的三块基石

## 本节你会学到什么

- 精准理解镜像（Image）、容器（Container）、仓库（Registry）三个核心概念
- 搞懂镜像和容器之间的关系（不是一回事！）
- 了解 Docker Hub 是什么以及它能为你做什么
- 用蛋糕模具的类比牢牢记住这三个概念

---

## 蛋糕模具、蛋糕和模具商店

我们来做蛋糕。

你去商店买了一个心形的蛋糕模具。这个模具是金属做的，上面有精确的形状和花纹——只要把面糊倒进去，烤出来的蛋糕就是心形的。你可以用这个模具烤一个蛋糕，也可以烤十个。每个蛋糕的形状都一样，因为共用一个模具。但每个蛋糕是独立的——你可以给一个蛋糕加草莓，给另一个加巧克力。

**在这个类比里：**

- **镜像（Image）= 蛋糕模具**。它是一个只读的模板，定义了"这个容器应该长什么样"——包含操作系统基础文件、应用代码、依赖库、环境变量、启动命令等等。镜像是静态的，你不去修改它。
- **容器（Container）= 烤出来的蛋糕**。它是镜像的一个运行实例。你可以从同一个镜像启动多个容器，每个容器都是独立的、可读可写的。你在容器里创建文件、修改配置、写入数据——这些改动只影响当前这个容器，不会影响镜像本身。
- **仓库（Registry）= 模具商店**。它是存放和分发镜像的地方。最知名的公共仓库是 Docker Hub，上面有成千上万个别人打包好的镜像，你可以免费下载使用。你也可以把自己的镜像上传到仓库，分享给别人，或者在公司内部搭建私有仓库。

## 用面向对象编程来类比

如果你有编程背景，这个类比会让你恍然大悟：

- **镜像 = 类（Class）**。一个定义了属性和方法的蓝图。
- **容器 = 对象（Object/Instance）**。用 `new` 关键字创建出来的具体实例。你可以 `new` 很多个对象，每个都有自己的状态。
- **仓库 = 包管理器（npm/PyPI/Maven）**。存放和分发包的地方。`docker pull` 就像 `npm install`。

这两种类比本质上说的是同一件事：**镜像是模板，容器是实例，仓库是分发平台。**

## 镜像的分层结构

镜像还有一个重要特性：**分层（Layers）**。

还拿蛋糕模具打比方。想象这个模具不是一整块金属，而是由几层叠起来的——底层是最基础的海绵蛋糕模具，中间是奶油层模具，上面是水果装饰模具。每一层是独立的，你可以单独替换中间那一层。

Docker 镜像就是这样分层的：

```
+-------------------+
|  可写容器层        |  <-- 容器运行时创建，记录你的修改
+-------------------+
|  应用代码层        |  <-- 你的 Python/Node.js 代码
+-------------------+
|  依赖库层          |  <-- pip install / npm install 的包
+-------------------+
|  基础系统层        |  <-- Ubuntu/Alpine 等基础镜像
+-------------------+
```

分层的好处是复用和节省空间。假如你有 10 个基于 Ubuntu 的镜像，Ubuntu 的基础层只需要存一份，所有镜像共享它。就像所有蛋糕共用底层的海绵蛋糕模具。

## 实际看一看

我们用一条命令来验证这些概念。拉取一个镜像，看看它长什么样：

```bash
# 从 Docker Hub（仓库）拉取 nginx 镜像
docker pull nginx

# 查看本地的镜像列表
docker images

# 基于这个镜像启动一个容器
docker run -d --name my-web nginx

# 查看正在运行的容器
docker ps

# 基于同一个镜像再启动第二个容器
docker run -d --name my-web-2 nginx

# 再查看一次容器列表——现在有两个容器，基于同一个镜像
docker ps
```

你会看到：
- `docker images` 列出了 `nginx` 这个镜像，这就是你的"模具"。
- `docker run` 基于同一个镜像创建了两个容器 `my-web` 和 `my-web-2`，这就是两个独立的"蛋糕"。
- `docker ps` 显示两个容器各自的状态——它们有各自的 ID、各自的名字、各自的状态，互不影响。

## 一个容易犯的错

新手最常犯的错误：**把镜像和容器当成同一回事**。

当你运行 `docker run nginx` 时，Docker 做的是：找到 `nginx` 这个镜像，以它为模板，创建一个全新的容器实例。镜像不会"变成"容器——镜像还是那个镜像，容器是新创建的独立实体。

就好比你用同一个模具又烤了一个蛋糕——模具还是那个模具，蛋糕是新烤出来的。你把蛋糕吃了（删除容器），模具还在（镜像还在）。

## 动手试试

打开终端，运行以下命令，边做边对照本节的概念：

```bash
# 1. 拉取一个镜像
docker pull alpine

# 2. 查看镜像
docker images

# 3. 从镜像启动一个容器，运行一条命令
docker run --rm alpine echo "我是一个容器实例"

# 4. 再启动一个容器，运行不同的命令
docker run --rm alpine echo "我是另一个容器实例，但我用的是同一个镜像"

# 5. 验证镜像还在
docker images alpine
```

两个容器都运行完了（`--rm` 会自动删除它们），但 `alpine` 镜像还在——模具还在，蛋糕已经吃完了。

如果还没装 Docker，这节先读概念，装完 Docker 后再回来做这个练习。

## 本节小结

镜像（Image）是模板，容器（Container）是实例，仓库（Registry）是分发平台——记住这个三角关系，Docker 的基本概念你就掌握了。
