# 00-01 什么是 Docker —— 把软件装进"集装箱"

## 本节你会学到什么

- 理解 Docker 到底是什么，它解决什么实际问题
- 认识"在我机器上能跑"这个让所有开发者头疼的经典难题
- 掌握 Docker 的核心思想：一次构建，到处运行
- 了解 Docker 在日常开发中的几个典型使用场景

---

## 那个让开发者崩溃的瞬间

想象这样一个场景：你在自己的电脑上吭哧吭哧写了一个 Web 应用，本地跑得一切正常。你把代码发给同事，同事一跑——报错。你检查了半天，发现同事的 Python 版本跟你不一样。你让同事升级 Python，结果他系统里还有另一个老项目依赖旧版 Python，一升级那个项目就挂了。

这还不算完。开发环境调通了，你要把应用部署到服务器上。服务器是 Linux，你是 Windows。服务器上的数据库版本、系统库版本、文件路径分隔符……统统跟你的开发机不一样。你又花了两天时间配环境。

这就是软件开发中的经典难题——**环境不一致**。代码一样，但运行环境不一样，结果就是跑不起来。

## 搬家：从手忙脚乱到标准化集装箱

我们拿搬家来打个比方。

**传统方式（没有 Docker 的时候）**：你要搬家，得把家里的沙发、桌子、冰箱、锅碗瓢盆一件一件搬上卡车。到了新家，因为新家格局跟旧家不一样，沙发可能放不进客厅，冰箱可能对不上插座位置。你得重新调整、重新接线、重新安装。每次搬家都是一场噩梦。

**Docker 的方式**：你把所有东西装进一个标准化的集装箱。沙发、桌子、冰箱全部固定好，集装箱的尺寸是标准的——不管你是搬到隔壁小区还是搬到另一个城市，不管接收方是什么类型的房子，只要是"能接收集装箱"的地方，叉车一叉，放下来就能用。你不用管新家格局怎么样，集装箱里面的一切都是自包含的。

Docker 就是这个"集装箱"。你的应用、应用的依赖（比如 Python 解释器、第三方库）、配置文件、运行脚本——全部打包在一起，形成一个标准化的单元。这个单元在任何装了 Docker 的机器上都能一致地运行，不管是你的 Windows 笔记本、同事的 MacBook，还是云上的 Linux 服务器。

## 不仅仅是打包，更是标准化

Docker 带来的不只是打包的便利。因为集装箱是标准尺寸的，码头、卡车、货轮、仓库都按这个标准来建。这就形成了一个生态系统。

同理，Docker 的标准化带来了：

- **统一的运行方式**：不管容器里面跑的是 Python、Node.js 还是 Java，你启动它们的方式都一样——`docker run`。
- **统一的日志输出**：不管什么应用，你都用 `docker logs` 看日志。
- **统一的资源管理**：不管什么应用，你都用同样的方式限制 CPU 和内存。
- **丰富的公共镜像**：Docker Hub 上有人家帮你打包好的 MySQL、Redis、Nginx、Node.js……你直接拿来用就行，不用自己从头配。

## 先感受一下

在后面的章节里我们会详细解释每一条命令的含义。现在你可以先感受一下，运行一个 Nginx Web 服务器有多简单：

```bash
# 一条命令，Nginx 就跑起来了
docker run -d -p 8080:80 --name my-nginx nginx
```

打开浏览器访问 `http://localhost:8080`，你会看到 Nginx 的欢迎页面。这台机器上不需要你手动安装 Nginx，不需要你写配置文件，不需要你处理各种依赖——Docker 帮你搞定了。

当你不需要这个 Nginx 的时候：

```bash
# 停掉并删除，机器上不留痕迹
docker stop my-nginx
docker rm my-nginx
```

干净利落，不会在你机器上留下一堆配置文件和无用的依赖包。

## 动手试试

回想一下你最近做过的项目。有没有遇到过"在我机器上能跑，在别人机器上不行"的情况？如果有，写下那个让你抓狂的环境差异是什么（Python 版本？Node 版本？操作系统差异？）。然后想一想：如果当时用了 Docker，这个问题还会存在吗？

如果你还没遇到过这种情况，试着在你的电脑上运行这个命令（确保已经装了 Docker，安装方法见后续章节）：

```bash
docker run --rm hello-world
```

看到输出了吗？这就是一个容器在运行。我们会在第 6 节详细解释这段输出的含义。

## 本节小结

Docker 就是把你的应用和它需要的环境打包成一个标准化的"集装箱"，让它在任何地方都能一致地运行，彻底告别"在我机器上能跑"的问题。
