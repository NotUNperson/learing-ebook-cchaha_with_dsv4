# 04-03 host 网络与 none 网络

## 本节你会学到什么

- 理解 host 网络模式：容器"借用"宿主机网络栈
- 理解 none 网络模式：彻底断网
- 对比 bridge、host、none 三种网络的使用场景
- 知道什么时候该用哪种网络模式

---

你已经学会了 bridge 网络：容器有自己的内部 IP，通过 Docker 交换机互联。现在来认识另外两种"极端"模式。

---

## host 网络：搬进宿舍用同一个 WiFi

想象你和室友住在同一个房间，你们共用同一个路由器、同一个公网 IP。你在电脑上跑了一个服务，室友直接连 `localhost:8080` 就能访问——因为你们本来就"站在同一块网络地板上"。

**host 网络**就是这样。容器不再有独立的网络命名空间，而是直接使用宿主机的网络栈。容器的端口就是宿主机的端口，不需要 `-p` 做映射。

```bash
# 使用 host 网络启动 nginx
$ docker run -d --name web --network host nginx:alpine

# 直接访问宿主机的 80 端口，不需要端口映射！
$ curl localhost:80
<!DOCTYPE html>
<html>
...
```

注意这里**没有 `-p 80:80`**，因为容器已经在宿主机网络上裸奔了，容器的 80 端口直接就是宿主机的 80 端口。

---

## host 模式的特点

| 方面         | bridge 模式                   | host 模式                        |
| ------------ | ---------------------------- | -------------------------------- |
| 端口映射     | 需要 `-p` 手动映射            | 不需要，容器端口=宿主机端口        |
| 网络隔离     | 有独立 IP，安全隔离            | 无隔离，容器直接暴露在宿主机网络上  |
| 性能         | 有 NAT 转发开销，微小延迟       | 无 NAT 开销，网络性能最优          |
| 端口冲突     | 多容器可用同一内部端口        | 容器端口不能冲突                   |
| DNS 解析     | 自定义网络支持                | 不适用，因为网络就是宿主机的       |

---

## 什么时候用 host 模式？

**高性能网络场景**：比如你正在运行一个网络密集型应用（例如 Redis 缓存集群、实时音视频传输），bridge 模式下的 NAT 转发会成为瓶颈。用 host 模式，容器直接走宿主机的网卡，性能几乎无损耗。

**端口不冲突的场景**：你只跑一个该服务的实例，不担心端口冲突。由于 host 模式下 80 端口只能被一个进程占用，跑两个 nginx 是做不到的。

下面是一个实际例子——跑 Redis 用 host 模式以获得最佳性能：

```bash
$ docker run -d --name redis --network host redis:alpine

# 直接通过宿主机 localhost 连接 Redis，零 NAT 开销
$ redis-cli -h 127.0.0.1 ping
PONG
```

---

## none 网络：离线模式

none 模式最简单——容器**没有任何网络接口**。就像你把电脑的网线拔了、WiFi 关了、蓝牙也关了。

```bash
$ docker run -it --network none alpine:latest sh
/ # ifconfig
lo        Link encap:Local Loopback
          inet addr:127.0.0.1
# 除了 lo（本地回环），没有任何网卡
/ # ping google.com
ping: bad address 'google.com'
```

容器里只有 `lo` 回环接口，连不上任何外部网络，其他容器也连不上它。

---

## none 模式的使用场景

这种"与世隔绝"的模式有什么用？两个典型场景：

**场景一：纯计算任务**。假如你在写一个批处理脚本，从宿主机挂载的文件里读数据、处理后写回挂载目录。整个过程不需要网络，用一个 none 网络模式能杜绝任何潜在的安全风险——不联网就不会被攻击。

**场景二：安全敏感操作**。比如你正在容器里生成 SSL 私钥，你绝对不希望这个密钥在任何情况下通过网络被传出去。none 模式确保物理隔离。

```bash
# 一个不需要网络的批处理容器
$ docker run -it --rm --network none \
  -v $(pwd)/data:/data \
  alpine:latest sh
/ # cat /data/input.txt | wc -l > /data/count.txt
/ # exit
```

---

## 三种网络一言以蔽之

| 模式   | 一句话                             | 类比                         |
| ------ | --------------------------------- | ---------------------------- |
| bridge | 容器有自己的 IP，通过 Docker 交换机通信 | 公司内部电话系统               |
| host   | 容器直接使用宿主机网络                 | 室友共用一个 WiFi 路由器       |
| none   | 容器没有任何网络                     | 拔了网线的电脑               |

---

## 动手试试

1. 分别用三种网络模式启动 alpine 容器：`--network bridge`（默认）、`--network host`、`--network none`
2. 进入每个容器，运行 `ifconfig`（需先 `apk add net-tools`）或 `ip addr`，观察网络接口数量
3. 在 host 模式的容器里 `curl localhost`（如果你有服务在跑的话），感受"零映射"的便利
4. 在 none 模式的容器里尝试 ping 任何地址，确认完全无法联网

---

## 本节小结

host 模式把容器直接"焊"在宿主机网络栈上，适合高性能和单实例场景；none 模式则彻底断网，适合安全敏感和纯计算任务。
