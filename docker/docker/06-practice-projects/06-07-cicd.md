## CI/CD 中 Docker 的角色

在没有 Docker 之前，CI/CD 的痛点是什么？

你在 Mac 上写代码，CI 跑在 Ubuntu 上，生产环境是 CentOS。三套环境，三套依赖，三套配置。"在我机器上能跑"成了程序员的万能借口。

Docker 解决了这个问题：**CI 里跑的是同一个镜像，生产环境跑的也是同一个镜像。** 代码推送到 GitHub，GitHub Actions 用你的 Dockerfile 构建镜像，跑测试，推到镜像仓库，然后生产环境拉下来部署。

**类比：** 以前的 CI/CD 像是在三个不同的厨房（Mac、Ubuntu、CentOS）里做同一道菜，每个厨房的灶台、调料、火候都不一样，味道自然不同。Docker 则是把厨房标准化了——无论你在北京、上海还是深圳的厨房，都是一模一样的设备配置。

## GitHub Actions 全流程示例

下面是一个完整的 GitHub Actions 工作流，覆盖了代码推送后自动构建、测试、推送镜像的全流程。

**.github/workflows/docker-ci.yml**

```yaml
name: Docker CI/CD Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

env:
  # 镜像仓库地址，根据你的实际情况修改
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  # ===================== 第一步：构建并测试 =====================
  build-and-test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Build image
        uses: docker/build-push-action@v5
        with:
          context: .
          # 只在 PR 阶段构建，不推送
          push: false
          load: true
          tags: myapp:test
          # 启用 BuildKit 缓存
          cache-from: type=gha
          cache-to: type=gha,mode=max

      - name: Run tests inside container
        run: |
          docker run --rm myapp:test npm test

  # ===================== 第二步：镜像漏洞扫描 =====================
  security-scan:
    needs: build-and-test
    runs-on: ubuntu-latest
    # 只在推送到 main 分支时扫描
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Build and scan
        uses: docker/scout-action@v1
        with:
          command: quickview,cves
          image: myapp:${{ github.sha }}
          # 如果有 high/critical 漏洞，让流水线失败
          only-severities: critical,high
          exit-code: true

  # ===================== 第三步：推送镜像 =====================
  push-image:
    needs: security-scan
    runs-on: ubuntu-latest
    # 只在推送到 main 分支时推送
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Log in to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          # 打多个标签：版本号 + latest
          tags: |
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

这个流水线分为三道闸门：

1. **构建+测试**：PR 提交就会触发，先构建、再跑测试。测试不过，代码不让合。
2. **安全扫描**：推送到 main 后自动扫描镜像漏洞。发现高危漏洞，流水线直接标记失败，阻止部署。
3. **推送镜像**：安全扫描通过后，镜像推送到 GitHub Container Registry，打好版本标签。

## Docker Compose 在 CI 中的使用

前端章节的练习里我们用过 docker-compose。在 CI 中，docker-compose 的主要价值是**提供真实的服务依赖**，而不是 mock。

假设你的后端依赖 PostgreSQL 和 Redis。传统做法是在测试里 mock 数据库连接，但 mock 意味着你的测试跑在一个"假环境"里——假数据库的行为和真数据库不一样，很多 bug 测不出来。

**docker-compose.ci.yml**

```yaml
version: "3.9"

services:
  app:
    build: .
    image: myapp:test
    environment:
      DB_HOST: db
      DB_USER: testuser
      DB_PASSWORD: testpass
      DB_NAME: testdb
      REDIS_HOST: redis
      NODE_ENV: test
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_healthy
    command: ["npm", "test"]
    networks:
      - test-net

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: testuser
      POSTGRES_PASSWORD: testpass
      POSTGRES_DB: testdb
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U testuser -d testdb"]
      interval: 3s
      timeout: 3s
      retries: 5
    networks:
      - test-net

  redis:
    image: redis:7-alpine
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 3s
      timeout: 3s
      retries: 5
    networks:
      - test-net

networks:
  test-net:
```

在 GitHub Actions 中这样使用：

```yaml
- name: Run integration tests with Docker Compose
  run: |
    docker compose -f docker-compose.ci.yml up --build --abort-on-container-exit --exit-code-from app
```

`--abort-on-container-exit` 保证一旦 app 容器跑完测试退出，所有依赖服务也跟着停止。`--exit-code-from app` 保证 app 容器的退出码就是整个步骤的退出码（测试失败则步骤失败）。

## 多环境部署策略

"一次构建，到处运行"是容器的核心价值。但不同环境（开发、测试、生产）的配置不一样，怎么办？

**核心原则：同一个镜像，不同的配置。**

```bash
# 开发环境
docker run -e NODE_ENV=development -e LOG_LEVEL=debug myapp:1.0.0

# 生产环境
docker run -e NODE_ENV=production -e LOG_LEVEL=warn myapp:1.0.0
```

同一个 `myapp:1.0.0` 镜像，换一套环境变量就从"开发模式"变成了"生产模式"。不用重新构建，不用重新测试。

**多环境 docker-compose 文件组织：**

```
docker-compose.yml          # 基础配置（所有环境共用）
docker-compose.dev.yml      # 开发环境覆盖（挂载代码、开启调试）
docker-compose.prod.yml     # 生产环境覆盖（资源限制、日志配置）
```

使用方式：

```bash
# 开发环境：合并基础配置 + 开发覆盖
docker compose -f docker-compose.yml -f docker-compose.dev.yml up

# 生产环境
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d
```

## CI 缓存策略：别让流水线每次都从零构建

CI 环境中，每次运行都是全新的虚拟机，没有 Docker 层缓存。如果不做特殊处理，每次构建都要重新下载所有依赖，流水线跑 20 分钟是常有的事。

**GitHub Actions 的 BuildKit 缓存：**

```yaml
- name: Build and push
  uses: docker/build-push-action@v5
  with:
    context: .
    push: true
    tags: myapp:latest
    # 关键：使用 GitHub Actions Cache 作为 BuildKit 缓存后端
    cache-from: type=gha
    cache-to: type=gha,mode=max
```

`type=gha` 把 BuildKit 的构建缓存存储在 GitHub Actions 的缓存服务中。下次构建时，相同层直接从缓存拉取，不用重新下载。

**效果对比：**

| 场景 | 无缓存 | 有 GitHub Actions 缓存 |
|------|--------|----------------------|
| 首次构建 | 5 分钟 | 5 分钟（无差别） |
| 只改了源码 | 5 分钟 | 30 秒 |
| 只改了依赖 | 5 分钟 | 2 分钟（只重装依赖） |

**类比：** CI 缓存就像你每次考试前不用重新学一遍所有课程。你之前学过的知识（缓存的层）直接拿来用，只需要复习今天新学的内容（变更的层）。

## 本节小结

Docker + CI/CD 的组合拳让"在我机器上能跑"成为历史：同一份 Dockerfile 在本地、CI、生产环境构建出完全一致的镜像，环境差异被彻底消灭。
