# 07-02 基本使用

## 本节你会学到什么

- 使用 `ssh` 命令连接到远程服务器（用户名、地址、端口）
- 理解 `known_hosts` 的作用和首次连接的安全确认
- 在远程服务器上直接执行命令而不进入交互式 Shell
- 使用 `ssh -v` 调试连接问题
- 掌握最常见的连接问题排查方法

---

上一节我们讲了 SSH 的原理。这节我们真正上手，从连上第一台远程服务器开始。

---

## 类比：走进一栋陌生大楼

SSH 连接远程服务器，就像是第一次去拜访一个朋友住的大楼：

1. 你知道地址（IP 或域名）-- 这是"大楼在哪里"
2. 你知道他的房号（端口，默认 22）-- 这是"哪个房间"
3. 你需要证明身份（用户名 + 密码/密钥）-- 这是"你是谁"
4. 保安看看你的身份证，确认无误后放你进去 -- 这是"认证通过"

整个过程就是：**找到 -> 敲门 -> 自报身份 -> 获得通行**。

## 最基本的连接

```bash
ssh 用户名@主机地址
```

拆开来看：
- `ssh`：调用 SSH 客户端程序
- `用户名`：你要远程登录的那个账号名
- `@`：分隔符，念作"at"
- `主机地址`：可以是 IP 地址（如 `192.168.1.100`）或域名（如 `server.example.com`）

```bash
# 完整的样子
ssh root@192.168.1.100
ssh john@myserver.example.com
ssh ubuntu@10.0.0.5
```

如果你的本地用户名和远程用户名一模一样，可以省略用户名：

```bash
# 假设你本地用户名是 "john"，你直接写：
ssh 192.168.1.100
# 等价于：
ssh john@192.168.1.100
```

SSH 默认连接的是 **22 端口**。如果服务器管理员把 SSH 端口改成了别的（这是一项常见的安全措施，可以避开大量自动化的暴力破解流量），你需要用 `-p` 指定：

```bash
ssh -p 2222 user@host
ssh -p 22222 admin@192.168.0.50
```

**类比**：`-p` 就像是告诉你的导航软件："这个朋友住的小区有点特殊，正门不在主干道上，你需要从侧门（端口 2222）进。"

## 首次连接：那个让人紧张的提示

第一次连接一台服务器时，你会看到这样的提示：

```
The authenticity of host '192.168.1.100 (192.168.1.100)' can't be established.
ED25519 key fingerprint is SHA256:xvVvxQyOZJ8kQbHpH7CMn2F1pK3wRrXsTtYyUuIiOoP.
Are you sure you want to continue connecting (yes/no/[fingerprint])?
```

很多人看到这个就直接敲 `yes` 了。让我们停下来理解一下这是在干什么。

这个提示的意思是：

1. "我不认识这台服务器，我从来没有连接过它。"
2. "这是它的身份证指纹（fingerprint），SHA256那一长串。"
3. "你确认这台服务器就是你要连接的那台吗？"

如果你敲 `yes`，SSH 客户端会把这台服务器的公钥保存到 `~/.ssh/known_hosts` 文件中。下次再连接时，客户端会自动检查服务器的公钥是否和之前保存的一致，不一致就报警。

**类比**：这就像是银行 App 的"是否信任此设备"提示。第一次在你家电脑上登录网银时，网银会让你确认"这是一台新设备，你信任它吗？" 确认后，下次直接从这台设备登录就不需要再验证了。

### 如何验证指纹？

理论上你应该通过一个安全的渠道（提前电话联系管理员、查看物理服务器的显示屏、或从已加密的文件中获取）确认指纹是否匹配。实际操作中，对于你自己的测试服务器，直接敲 `yes` 通常没问题。

```bash
# 查看已知主机的指纹列表
ssh-keygen -lf ~/.ssh/known_hosts

# 删除某台主机的记录（删除后下次连接会重新提示）
ssh-keygen -R 192.168.1.100

# 手动编辑 known_hosts 文件也可以
vim ~/.ssh/known_hosts
```

## known_hosts：你的"老朋友通讯录"

`~/.ssh/known_hosts` 是一个文本文件，每台你信任过的服务器在这里都有一条记录。它看起来像是这样（部分内容可能被哈希）：

```
192.168.1.100 ecdsa-sha2-nistp256 AAAAE2VjZH...
myserver.com ssh-ed25519 AAAAC3NzaC1lZDI...
```

当你在 SSH 连接过程中看到这个警告时：

```
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
@    WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!     @
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
```

这意味着服务器的公钥和你之前保存的不一样了。可能的原因有两个：

1. **服务器重装系统或重新生成了 SSH 密钥**（正常情况）
2. **有人在冒充服务器进行中间人攻击**（危险情况）

你必须先确认是哪种情况，才能采取行动。如果是第1种情况，删除旧记录即可：

```bash
ssh-keygen -R 192.168.1.100
```

**类比**：你的朋友张三换了手机号。下次你接到"张三"的来电，来电显示却是陌生号码。你会怀疑："这真的是张三吗？" SSH 也一样 -- 它发现服务器的"身份特征"变了，提醒你注意。

## 远程执行命令：不进门也能办事

SSH 不一定要"登录进去"。很多时候你只是想在远程服务器上执行一两条命令，就像让快递员帮你把包裹放在门口，不需要请快递员进家门。

```bash
# 查看远程服务器的磁盘使用情况
ssh user@server "df -h"

# 查看远程服务器的运行时间
ssh user@server "uptime"

# 重启远程服务器上的 Nginx
ssh user@server "sudo systemctl restart nginx"

# 执行多条命令
ssh user@server "cd /var/log && ls -la && tail -20 syslog"
```

一些重要选项：

```bash
# -t：强制分配伪终端
# 当你需要交互式操作（比如 sudo 要输密码）时使用
ssh -t user@server "sudo systemctl restart nginx"

# -T：明确不要伪终端
# 当你只是抓数据时，加 -T 可以避免不必要的终端输出
ssh -T user@server "cat /etc/hostname"
```

### 把远程命令的输出存到本地变量

```bash
# 获取远程主机名
remote_hostname=$(ssh user@server "hostname")
echo "远程主机名：$remote_hostname"

# 获取远程服务器上某个文件的内容
remote_config=$(ssh user@server "cat /etc/hostname")
echo "远程主机名配置：$remote_config"
```

### 把本地脚本传到远程执行

```bash
# 本地脚本 local_check.sh 不需要先 scp 过去，直接重定向执行
ssh user@server 'bash -s' < local_check.sh
```

这在批量部署时非常有用 -- 你的脚本不需要到处复制，直接从本地"投喂"给远程的 bash。

## 调试连接问题：ssh -v

连接出问题时，千万不要盲目重试。SSH 提供了三个级别的调试输出：

```bash
ssh -v user@host    # 基本信息（一级）
ssh -vv user@host   # 详细信息（二级）
ssh -vvv user@host  # 最详细（三级，调试专用）
```

`-v` 是"verbose"（啰嗦模式）的缩写。调试时一般用 `-vvv`，它会输出连接过程的每一步：

1. 版本交换（双方说"我支持 SSHv2"）
2. 密钥交换算法协商
3. 服务器公钥验证
4. 用户认证方式协商
5. 认证成功或失败的原因

**类比**：`ssh -vvv` 就像是飞机上的黑匣子数据 -- 它记录了每个环节发生了什么。如果"飞机坠毁"（连接失败），你翻看黑匣子就能知道是哪一步出了问题。

## 常见连接问题排查手册

### 问题1：Connection timed out（连接超时）

```
ssh: connect to host 192.168.1.100 port 22: Connection timed out
```

原因和排查顺序：
1. 目标 IP 写错了？`ping` 一下试试
2. SSH 端口对吗？`nc -zv host 22` 检查（`nc` 就是 netcat 的简称，在本系列 09 模块会讲）
3. 防火墙拦了？检查本地和服务器双边的防火墙
4. sshd 服务在运行吗？在服务器上执行 `systemctl status sshd`

### 问题2：Permission denied（权限拒绝）

```
Permission denied (publickey,password).
```

这是最常见的错误，可能的原因非常多：
- 用户名写错了
- 密码打错了
- 服务器禁止密码登录，必须用密钥
- 密钥文件权限不正确（应该是 `chmod 600`）
- 公钥没放到服务器的 `~/.ssh/authorized_keys` 中

### 问题3：WARNING: UNPROTECTED PRIVATE KEY FILE

```
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
@         WARNING: UNPROTECTED PRIVATE KEY FILE!          @
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
Permissions 0644 for '/home/user/.ssh/id_rsa' are too open.
```

这是因为你的私钥文件权限太"宽松"（比如 `644` 即所有人都可读）。SSH 拒绝使用这样的私钥，因为别人可能偷看了你的私钥。解决方法很简单：

```bash
chmod 600 ~/.ssh/id_rsa
```

`600` 的意思是：只有文件的拥有者可以读写，其他任何人都没有任何权限。

### 问题4：连接卡在"Connecting to..."很久

这通常是因为 SSH 服务端在做 DNS 反向解析 -- 它想从你的 IP 反查你的域名。在服务器上编辑 `/etc/ssh/sshd_config`，加上：

```
UseDNS no
```

然后重启 sshd：

```bash
sudo systemctl reload sshd
```

exit        # 最常用

Ctrl+D      # 发送 EOF（文件结束）信号

## 动手试试

1. 如果你的系统安装了 sshd（`sudo apt install openssh-server` 或 `sudo yum install openssh-server`），试着 SSH 到自己：`ssh localhost`。观察 `known_hosts` 文件的变化。
2. 用 `ssh user@localhost "date"` 在远程（其实是本机）执行一条命令，观察输出
3. 运行 `ssh-keygen -lf ~/.ssh/known_hosts` 查看你信任的服务器指纹列表
4. 用 `ssh -vvv localhost` 观察完整的连接过程（注意那些 DEBUG 行都说了什么）
5. 故意 `ssh -p 9999 localhost` 连接一个不存在的端口，观察报错信息
6. 如果有多台机器（或者虚拟机），从一台 SSH 到另一台，体验真正的"远程"连接

---

## 本节小结

`ssh user@host` 是最基本的远程连接命令，`-p` 指定端口，`-v` 开启调试。首次连接时需要确认服务器指纹（防止中间人攻击），确认后的指纹保存在 `~/.ssh/known_hosts` 中。`ssh user@host "命令"` 可以在远程执行命令而不进入交互式 Shell。
