# 07-01 SSH 是什么

## 本节你会学到什么

- 理解对称加密和非对称加密的核心概念
- 掌握 SSH 混合加密的工作流程
- 了解 SSH 相比 Telnet 的安全优势
- 用"加密对讲机"的类比记住 SSH 的特点
- 确认 SSH 是否已安装在你的系统上

---

想象你现在坐在家里的电脑前，需要管理一台远在数据中心的服务器。你必须登录进去、修改配置、重启服务。问题来了：你和服务器之间的网络不是私有的，数据包要经过几十个路由器才能到达。如果有人在你和服务器之间的某个路由器上"偷听"，直接看到你输入的 root 密码，那服务器就不是你的了。

SSH（Secure Shell）就是为解决这个问题而生的。它在你和远程服务器之间建立了一条"加密隧道"。你和服务器之间传输的所有数据 -- 包括密码、文件、命令输出 -- 都经过了加密。偷听者看到的只是一堆乱码，就像你对着加密对讲机说话，没有解密器的旁人只能听到杂音。

---

## 类比：加密对讲机 vs 普通扩音器

小时候玩对讲机，一按按钮说话，附近所有同频率的对讲机都听得到。你说"今晚 8 点操场见"，张三李四全知道了。这就像是 **Telnet** -- 一种老旧的远程连接协议，数据都是明文传输的。任何能抓到网络包的人都能看到你在做什么。

但**SSH**不一样。它就像是给双方各配了一个加密芯片。你说的话先被芯片加密成噪音，只有对方的芯片才能解密还原。即使有人截获了信号，听到的也只是"滋滋滋"的杂音。

这就是 Telnet 和 SSH 的本质区别：一个是对着全村喊话，一个是戴着加密耳麦说悄悄话。

---

## 两种加密方式：对称和非对称

要理解 SSH 的工作原理，首先得理解两种加密方式。这就像我们日常生活中的两种"传达秘密"的方法。

### 对称加密：一把钥匙开一把锁

**概念**：加密和解密用同一把钥匙。

**生活类比**：你和朋友共享一个密码本的复印本。你写信时，用密码本把"你好"替换成"ab12"，朋友收到后用同一本密码本查回来。速度快，但有个致命问题 -- 你怎么把密码本安全地交给朋友？如果你们第一次见面时有人偷看到了密码本，那之后的所有通信都不安全了。

```bash
# 对称加密的典型例子：用 openssl 加密一个文件
echo "这是秘密信息" > secret.txt
openssl enc -aes-256-cbc -salt -in secret.txt -out secret.enc -pass pass:mykey
# 解密用同一把"钥匙"（密码 mykey）
openssl enc -aes-256-cbc -d -in secret.enc -out decrypted.txt -pass pass:mykey
```

对称加密的优点是**快**，非常快。它适合加密大量数据（比如你的文件传输）。问题在于：双方怎么安全地商定这把"钥匙"？

### 非对称加密：两把不同的钥匙

**概念**：有一对钥匙 -- 公钥（可以公开，给所有人）和私钥（必须保密，只自己持有）。公钥加密的内容，只有私钥能解开。

**生活类比**：想象有一个特殊的邮箱。这个邮箱只有一个小缝（投递口），任何人都能把信塞进去。但要把里面的信拿出来，需要一把特殊的钥匙。这个投递口就是"公钥"（公开的，谁都能往里面塞信），那把特殊的钥匙就是"私钥"（只有你持有）。即使有人拍了投递口的照片，他也不可能用它把信弄出来。

```bash
# 非对称加密的例子：生成 RSA 密钥对
ssh-keygen -t rsa -b 2048 -f ~/.ssh/test_key
# 生成两个文件：
#   test_key      → 这是私钥（你自己保管好）
#   test_key.pub  → 这是公钥（可以给任何人）
```

非对称加密解决了"第一次怎么安全交换钥匙"的问题，但它的缺点是**慢**，比对称加密慢几百倍。所以 SSH 不只用一种方式。

---

## SSH 的混合加密：取长补短

SSH 聪明地结合了两种加密的优点：

**第1步：握手阶段（用非对称加密交换钥匙）**

客户端说："嗨，服务器，把你的公钥给我看看。"

服务器交出公钥。客户端生成一个随机会话密钥，用服务器的公钥加密后发给服务器。只有拥有私钥的服务器能解密，拿到这个会话密钥。

**第2步：通信阶段（用对称加密传输数据）**

双方用第1步商定的会话密钥，进行对称加密通信。速度快，且安全。

**生活类比**：

你想给住在另一座城市的朋友寄贵重物品，每天都在自家门口放一点东西。你不能把家门钥匙直接寄过去（万一快递路上被偷了）。

于是你这样做：先花大价钱请了最安全的快递公司（非对称加密），把一个便宜的小锁和钥匙寄给朋友。然后你每天在门口放东西时，就用这把便宜的小锁锁上盒子（对称加密），用普通快递寄出去。锁虽然便宜，但钥匙只有你们俩有，所以还是很安全。

SSH 就是这样的设计哲学：花大代价只做一次（建立安全通道），后续都省着来。

## SSH 的工作流程：从连接到通信

把上述加密原理放到实际的 SSH 连接中，完整的流程是这样的：

```
客户端                          服务器
  |                                |
  |---(1) TCP 连接请求:22端口------>|
  |                                |
  |<--(2) 服务器返回 SSH 版本信息---|
  |                                |
  |---(3) 密钥交换算法协商--------->|
  |---(算法：ECDH, Curve25519等)--->|
  |                                |
  |<--(4) 服务器发送主机公钥--------|
  |                                |
  |---(5) 生成会话密钥------------>|
  |    (用服务器公钥加密后发送)     |
  |                                |
  |<==(6) 加密通道建立 ===========>|
  |    (后续通信全部加密)           |
  |                                |
  |---(7) 客户端身份认证---------->|
  |    (密码 或 客户端密钥)         |
  |                                |
  |<==(8) 认证通过，开始会话 =====>|
```

第4步是一个关键的安全检查点：客户端收到服务器的公钥后，会检查它是否在 `~/.ssh/known_hosts` 文件中。如果是第一次连接，用户需要**手动确认这个公钥的指纹**，防止"中间人攻击"。

**中间人攻击的类比**：你在咖啡厅连上 Wi-Fi（其实是黑客伪造的 Wi-Fi），然后你"SSH 到你的服务器"，实际上连接的是黑客的电脑。黑客把你的数据解密后再转发给真正的服务器。你以为在和服务器对话，其实中间有个"翻译官"在偷听。

SSH 的指纹确认机制就是为了防止这种情况：第一次连接时，SSH 客户端会告诉你服务器的指纹，你可以打电话给管理员确认真实指纹是否一致。

## Telnet 对比：为什么不要用 Telnet

如果你在学 Linux 历史，可能会遇到 Telnet 这个词。简单说：Telnet 是 SSH 的前身，但因为不安全已经被淘汰了。

| 对比维度 | SSH | Telnet |
|----------|-----|--------|
| 数据传输 | 全程加密 | 明文传输 |
| 密码传输 | 加密，不会泄露 | 明文，可被截获 |
| 身份认证 | 密码/密钥/证书 | 仅密码 |
| 默认端口 | 22 | 23 |
| 中间人攻击防护 | 有（指纹验证） | 无 |
| 文件传输 | 内置 SCP/SFTP | 无，需要额外工具 |
| 生产环境使用 | 是，标准选择 | 否，仅历史遗留 |

如果你遇到有人说"我用 Telnet 调试端口很方便"，那是用 Telnet 测试某个端口是否开放（类似 `nc -zv` 的用法），不是真的用它来登录远程系统。

**类比**：SSH 就像是现代银行的安全 App，Telnet 则是明信片上的银行卡密码。

## 检查 SSH 是否安装

大多数 Linux 发行版默认安装了 SSH 客户端。你可以快速检查一下：

```bash
# 检查 SSH 客户端
which ssh
# 通常输出：/usr/bin/ssh

# 查看 SSH 版本
ssh -V
# 输出类似：OpenSSH_8.9p1, OpenSSL 3.0.2

# 如果没有安装
# Ubuntu/Debian:
sudo apt install openssh-client

# CentOS/RHEL:
sudo yum install openssh-clients
```

如果你需要**被别人 SSH 连接**（作为服务器），还需要安装 SSH 服务端：

```bash
# 安装 SSH 服务端
sudo apt install openssh-server    # Debian/Ubuntu
sudo yum install openssh-server    # CentOS/RHEL

# 启动并设置开机自启
sudo systemctl enable sshd
sudo systemctl start sshd

# 检查是否在运行
sudo systemctl status sshd
```

## 动手试试

1. 在你的 Linux 系统中执行 `ssh -V` 查看 OpenSSH 版本
2. 执行 `which ssh` 确认客户端的安装路径
3. 检查 `/etc/ssh/sshd_config` 是否存在（有的话说明服务端已安装）
4. 如果服务端已安装，用 `sudo systemctl status sshd` 查看运行状态
5. 阅读 `man ssh` 的第一段描述，了解 ssh 的基本定义
6. 思考：如果你在星巴克用公共 Wi-Fi SSH 到服务器，流量会被加密吗？答案是"会" -- 所有经过 SSH 连接的数据都端到端加密了，公共 Wi-Fi 上的窃听者只能看到加密后的乱码。

---

## 本节小结

SSH 是一个安全的远程连接协议，使用混合加密（非对称加密建立通道 + 对称加密传输数据）来保护通信安全。它就像一副加密对讲机，确保你和远程服务器之间的对话不被窃听。与 Telnet 的明文传输相比，SSH 是当前远程管理的唯一正确选择。
