# 07-05 SSH 配置文件

## 本节你会学到什么

- 创建和编辑 `~/.ssh/config` 文件，用别名替换长命令
- 掌握 HostName、User、Port、IdentityFile 等核心配置项
- 使用 `ProxyJump` 通过跳板机连接内网服务器
- 使用通配符批量配置多台服务器
- 用 `ssh -G` 验证配置是否生效

---

如果你管理着多台服务器，你的命令行历史记录里可能塞满了这种东西：

```bash
ssh -p 2222 -i ~/.ssh/prod_key deploy@10.0.0.50
scp -P 2222 -i ~/.ssh/prod_key deploy@10.0.0.50:/tmp/ ./local/
sftp -P 2222 -i ~/.ssh/prod_key deploy@10.0.0.50
```

每次都要记住：端口多少？用什么用户名？密钥在哪个文件？IP 是哪个？

这就像你每次打电话都要背号"张三 手机号 138xxxx1234 工作手机 139xxxx5678"，而不用通讯录保存联系人。SSH 配置文件就是你的"通讯录"。

---

## 类比：手机通讯录

打开你的手机通讯录，你会看到：
- 每个联系人有名字（比如"张三"）
- 名字下面有电话号、工作号等多个信息
- 你打电话只需要点联系人名字，不需要记号码

SSH 配置文件做的事一模一样：

```
通讯录                           SSH 配置文件
======                           ============
联系人：张三              →      Host zhangsan
电话号：138xxxx           →          HostName 192.168.1.100
备注：工作手机            →          User admin
                                    Port 2222
                                    IdentityFile ~/.ssh/work_key
```

配置好之后，你用 `ssh zhangsan` 就能连上张三的工作服务器，不用再记 IP、端口、用户名、密钥路径这些乱七八糟的东西。

---

## 配置文件的基本语法

`~/.ssh/config` 是一个纯文本文件（如果没有就创建一个）。它的基本格式是：

```
Host 别名
    配置项  值
    配置项  值
```

规则：
- `Host` 定义一个新的配置块
- 下面的配置项用缩进（2 个或 4 个空格都可以，但没有缩进也行）
- 空行分隔不同的 Host 块
- SSH 从上到下匹配，**第一个匹配的生效**
- `Host *` 匹配所有连接，通常放最后作为全局默认值

---

## 核心配置项

### 最常用的四个

```bash
Host prod-server              # 别名：你起的名字
    HostName 10.0.0.50        # 实际地址：IP 或域名
    User deploy               # 用户名
    Port 2222                 # SSH 端口
    IdentityFile ~/.ssh/prod_key  # 私钥路径
```

配置好后，下面这些等价：

```bash
# 没配置时
ssh -p 2222 -i ~/.ssh/prod_key deploy@10.0.0.50

# 有配置后
ssh prod-server

# SCP/SFTP/rsync 也会自动读取配置！
scp file.txt prod-server:/tmp/
rsync -avz ./dir/ prod-server:/dir/
```

### 连接管理类

```bash
ConnectTimeout 10
# 连接超时 10 秒。超过 10 秒连不上就放弃，不等了。

ServerAliveInterval 60
# 每 60 秒发送一个心跳包。长时间不操作时，防止防火墙断开连接。

ServerAliveCountMax 3
# 连续 3 次收不到心跳响应，断开连接。
# 配合上面：60秒 * 3次 = 180秒检测到断开
```

**类比**：`ServerAliveInterval` 就像是你在长途车程中和副驾驶每隔一段时间说句话，让对方知道你还没睡着。如果连续几次没回应，说明出问题了。

### 安全相关

```bash
StrictHostKeyChecking accept-new
# yes:        严格检查，首次连接必须手动确认
# no:         不检查（不安全！）
# accept-new: 新主机自动接受，已有主机如变更则报警（推荐）

PreferredAuthentications publickey
# 指定认证方式的顺序。只写 publickey 意味着只用密钥认证。
# 可以写多个：publickey,password

UserKnownHostsFile ~/.ssh/my_known_hosts
# 指定 known_hosts 文件的位置。默认是 ~/.ssh/known_hosts。
# 设为 /dev/null 表示不保存主机指纹（测试环境用）
```

## 通配符：批量配置多台服务器

如果你的公司有很多台服务器域名遵循相同的模式，通配符能省大量配置：

```bash
# 匹配所有 .example.com 的服务器
Host *.example.com
    User admin
    Port 22
    IdentityFile ~/.ssh/company_key

# 匹配所有 10.0.x.x 内网 IP
Host 10.0.*
    User root
    Port 22

# 全局默认值（放最后）
Host *
    ServerAliveInterval 60
    ConnectTimeout 10
```

匹配顺序：SSH 从上到下找，找到第一个匹配的 Host 块就用它的配置。所以原则是：

1. **具体配置放前面**（比如 `Host prod-web-01`）
2. **通配符放中间**（比如 `Host *.example.com`）
3. **全局默认 `Host *` 放最后**

## ProxyJump：跳板机配置

在 07-01 节我们提到了跳板机（堡垒机）的概念 -- 公司只暴露一台服务器给外网，要通过它跳转到内网的其他服务器。

不配置时，你需要：

```bash
# 手动跳两次
ssh bastion@gw.example.com
# 然后在跳板机上再 ssh：
ssh admin@10.0.1.50

# 或一行搞定但很丑：
ssh -J bastion@gw.example.com admin@10.0.1.50
```

配置 ProxyJump 后：

```bash
Host internal-web
    HostName 10.0.1.50
    User admin
    ProxyJump bastion@gw.example.com

Host internal-db
    HostName 10.0.1.51
    User dbadmin
    ProxyJump bastion@gw.example.com
```

连接时直接：

```bash
ssh internal-web     # SSH 自动处理跳板过程！
scp file.txt internal-web:/tmp/
```

**多层跳板**（某些金融/政务系统需要层层穿透）：

```bash
Host final-target
    HostName 10.0.2.100
    User admin
    ProxyJump bastion1@gate1.com,bastion2@gate2.com
```

**类比**：ProxyJump 就像是 SSH 帮你唱双簧。前台代理（你）对着观众（服务器）说话，但实际上后头的人（跳板机）在帮你传话。观众以为在和你对话，中间人悄悄完成了转发。

## 完整示例：一个专业级的 SSH 配置

```bash
# ==========================================
# ~/.ssh/config
# ==========================================

# ---------- 生产 Web 服务器 ----------
Host prod-web-01
    HostName web01.prod.example.com
    User deploy
    Port 2222
    IdentityFile ~/.ssh/keys/prod_ed25519
    ConnectTimeout 10
    ServerAliveInterval 60

Host prod-web-02
    HostName web02.prod.example.com
    User deploy
    Port 2222
    IdentityFile ~/.ssh/keys/prod_ed25519

# ---------- 生产数据库 ----------
Host prod-db
    HostName db.prod.example.com
    User dbadmin
    Port 22
    IdentityFile ~/.ssh/keys/prod_ed25519

# ---------- 内网机器（通过跳板机）----------
Host internal-*
    ProxyJump ops@bastion.example.com
    User admin
    IdentityFile ~/.ssh/keys/internal_ed25519

Host internal-web-01
    HostName 172.16.10.11

Host internal-web-02
    HostName 172.16.10.12

Host internal-api
    HostName 172.16.20.10

# ---------- GitHub ----------
Host github.com
    HostName github.com
    User git
    IdentityFile ~/.ssh/keys/github_ed25519
    PreferredAuthentications publickey

# ---------- AWS EC2 ----------
Host *.compute.amazonaws.com
    User ec2-user
    IdentityFile ~/.ssh/aws_key.pem
    StrictHostKeyChecking accept-new

# ---------- 全局默认 ----------
Host *
    ConnectTimeout 10
    ServerAliveInterval 120
    ServerAliveCountMax 3
    PreferredAuthentications publickey
```

## 验证配置

### 查看 SSH 对某个别名使用的完整配置

```bash
ssh -G prod-web-01
```

`-G` 选项会输出 SSH 解析配置后的所有参数，包括未在 config 中显式设置的系统默认值。输出很长，你可以用管道筛选：

```bash
# 只看端口相关
ssh -G prod-web-01 | grep -i port

# 只看主机名
ssh -G prod-web-01 | grep -i hostname

# 看密钥文件
ssh -G prod-web-01 | grep -i identityfile
```

### 用调试模式验证

```bash
ssh -v prod-web-01
```

在 DEBUG 输出中找类似这样的行：

```
debug1: Reading configuration data /home/user/.ssh/config
debug1: /home/user/.ssh/config line 5: Applying options for prod-web-01
```

这说明 SSH 确实读取了你的配置文件并应用了对应 Host 块的配置。

## 配置文件的权限

和私钥文件一样，SSH 对配置文件的权限也很挑剔：

```bash
chmod 600 ~/.ssh/config
```

如果权限太宽松（比如 `644`，所有人都可读），SSH 可能会忽略这个文件。因为配置文件里可能有敏感的服务器地址和用户名信息。

---

## 命令行覆盖配置文件

命令行传入的选项会覆盖配置文件中的设置：

```bash
# 配置文件中定义了 Port 2222
# 但这次临时用 22222 端口
ssh -p 22222 prod-web-01
```

优先级：
1. 命令行选项（最高）
2. `~/.ssh/config` 中对应 Host 的配置
3. `/etc/ssh/ssh_config` 系统级配置
4. SSH 的默认值（Port 22 等）

---

## 多个 IdentityFile

你可以为一个 Host 配置多个密钥文件，SSH 会依次尝试：

```bash
Host myhost
    HostName 10.0.0.50
    IdentityFile ~/.ssh/id_ed25519
    IdentityFile ~/.ssh/id_rsa
```

注意：SSH 有最大认证尝试次数（默认是 6）。如果配置太多密钥，可能会在找到正确密钥之前就用完尝试次数。所以一般配置 2-3 个就足够了。

---

## 动手试试

1. 创建 `~/.ssh/config` 文件，设置权限 `chmod 600`
2. 为你能访问的一台远程服务器添加一个 Host 块，至少要配置 HostName、User、Port
3. 用 `ssh 别名` 测试连接是否正常
4. 用 `ssh -G 别名` 查看 SSH 解析后的完整配置
5. 在文件末尾添加 `Host *` 全局默认值（至少设置 `ServerAliveInterval 60`）
6. 测试 scp：`scp 某个文件 别名:/tmp/`，确认 scp 也读取了配置文件
7. 如果有条件，配置一个跳板机场景并测试 ProxyJump

---

## 本节小结

`~/.ssh/config` 是 SSH 的"通讯录"，让你用简短的别名代替一长串连接参数。核心配置项包括 HostName、User、Port、IdentityFile。ProxyJump 让跳板机连接透明化，通配符让批量管理更高效。命令行选项可覆盖配置文件设置，调试时用 `ssh -G` 查看实际生效的配置。
