# 07-03 密钥认证

## 本节你会学到什么

- 使用 `ssh-keygen` 生成 Ed25519 和 RSA 密钥对
- 理解公钥和私钥各自的作用与安全要求
- 使用 `ssh-copy-id` 将公钥上传到远程服务器
- 掌握 `.ssh` 目录和文件的权限要求
- 在服务端禁用密码登录，只允许密钥认证

---

上一节我们用密码登录了远程服务器。但输入密码这件事有几个问题：

1. 每次都要输，麻烦
2. 密码可能太短、太简单、或被泄露
3. 如果密码在传输过程中被窃取（虽然 SSH 加密了，但密码本身可能在其他地方泄露）

密钥认证解决了这些问题。它是 SSH 最推荐的认证方式，安全且方便 -- 配置一次，之后登录不再需要密码。

---

## 类比：门禁卡 vs 密码锁

密钥认证和密码认证的对比，最贴切的类比就是：

**密码认证 = 密码锁**

- 你每次进门都要输入一串数字
- 如果有人站在你身后偷看，他就能记住密码
- 密码太简单的话，别人可以一个个试
- 密码被记录到某个地方（你的脑袋或笔记本），可能被泄露

**密钥认证 = 门禁卡**

- 你刷卡进门，不需要记忆任何东西
- 别人看到你刷卡也没用，他复制不了你的卡
- 门禁系统验证的是卡里的加密信息，不是表面数字
- 离职的时候，管理员直接在系统里吊销这张卡即可，不需要换锁

SSH 密钥认证就是电子版的"门禁卡机制"。你的私钥就是那张卡（绝对不能给别人），公钥放在服务器上是"读卡器里的授权名单"。

---

## 公钥和私钥：一对搭档

密钥对包含两个文件：

| 文件 | 作用 | 放在哪里 | 类比 |
|------|------|----------|------|
| 私钥 (private key) | 证明"我是我" | 你自己的电脑 `~/.ssh/` | 你的门禁卡 |
| 公钥 (public key) | 验证"你是你" | 远程服务器 `~/.ssh/authorized_keys` | 读卡器 |

私钥永远不会离开你的机器，远程服务器上只有你的公钥。认证过程是：

1. 你发起到服务器的 SSH 连接
2. 服务器用一个随机数挑战你的客户端："用我的公钥加密了这个随机数，你能解密吗？"
3. 你的客户端用私钥解密，返回正确的响应
4. 服务器确认响应正确，认证通过

整个过程中，私钥从未被传输出去。服务器只是"考验"你是否持有对应的私钥。

**类比**：就像你去酒吧，保安不检查你的身份证内容，而是问"你的身份证后4位是什么？" 只有真正的身份证持有者才能回答。私钥提供了这种"证明而不用泄露"的能力。

## 生成密钥对：ssh-keygen

### 选择密钥类型

当前（2026年），最推荐的密钥类型是 **Ed25519**。它安全、快速、密钥短小。如果遇到不兼容的老旧系统，再用 RSA 4096 位。

| 类型 | 推荐度 | 速度 | 密钥长度 | 兼容性 |
|------|--------|------|----------|--------|
| Ed25519 | 强烈推荐 | 最快 | 短 | OpenSSH 6.5+ |
| RSA 4096 | 兼容备选 | 较慢 | 长 | 几乎所有系统 |
| ECDSA | 中间方案 | 快 | 中等 | 较新系统 |
| DSA | 不推荐 | - | - | 已废弃 |

### 生成 Ed25519 密钥

```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
```

参数说明：
- `-t ed25519`：指定密钥类型为 Ed25519
- `-C "注释"`：添加注释（方便以后识别这个密钥的用途，通常放邮箱）

执行后会出现三个交互提示：

```
Generating public/private ed25519 key pair.
Enter file in which to save the key (/home/user/.ssh/id_ed25519):
```

第一个提示是问你把密钥保存在哪里。默认路径 `~/.ssh/id_ed25519` 就可以，直接回车。如果你有多组密钥（比如 GitHub 一组、公司服务器一组），可以在这里指定不同的文件名。

```
Enter passphrase (empty for no passphrase):
```

第二个提示是设置 passphrase（口令）。这是**给你的私钥加一层密码保护**。即使别人偷走了你的私钥文件，没有 passphrase 也不能使用。

**类比**：passphrase 就像是手机 SIM 卡的 PIN 码。别人捡到你的手机还不够，还得知道 PIN 码才能用。同样，别人复制了你的私钥文件，还得知道 passphrase 才能用它登录服务器。

```
Enter same passphrase again:
```

第三个提示是确认 passphrase，再输一遍。

### 生成 RSA 密钥（兼容性备选）

```bash
ssh-keygen -t rsa -b 4096 -C "your_email@example.com"
```

`-b 4096` 指定密钥长度为 4096 位。2016 年之后就不再推荐 2048 位的 RSA 密钥了，最低 3072 位，推荐 4096 位。

### passphrase 到底该不该设？

| 场景 | 建议 |
|------|------|
| 生产服务器 | 必须设置！配合 ssh-agent 使用 |
| 个人开发机 | 建议设置 |
| 自动化脚本（CI/CD） | 可以不设，但要严格保护私钥文件权限 |
| 学习/测试 | 可以先不设 |

如果你设置了 passphrase 又嫌每次都输入很麻烦，可以用 `ssh-agent`：

```bash
# 启动 ssh-agent（大多数桌面 Linux 已经自动启动了）
eval $(ssh-agent)

# 将私钥加入 agent，这次输入 passphrase 后，本会话内不再需要
ssh-add ~/.ssh/id_ed25519
```

ssh-agent 就像是一个"钥匙管家"。你把钥匙交给他保管，之后每次需要开门，他帮你处理，你不用再自己掏钥匙。

## 上传公钥：ssh-copy-id

生成密钥对之后，你需要把公钥复制到远程服务器：

```bash
ssh-copy-id user@remote_host
```

这个命令会：
1. SSH 登录到远程服务器（你需要输入一次密码）
2. 把公钥追加到远程服务器的 `~/.ssh/authorized_keys` 文件中
3. 自动设置正确的文件和目录权限

如果要指定特定的公钥文件：

```bash
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@remote_host
```

执行成功后，试试免密登录：

```bash
ssh user@remote_host
# 应该直接登录，不再提示密码！
```

---

## 手动上传公钥（当 ssh-copy-id 不可用时）

某些极简系统或嵌入式设备上可能没有 `ssh-copy-id` 命令。手动上传也不难，但需要仔细注意权限：

```bash
# 第一步：在远程服务器上创建 .ssh 目录
ssh user@host "mkdir -p ~/.ssh && chmod 700 ~/.ssh"

# 第二步：将本地公钥追加到远程的 authorized_keys
cat ~/.ssh/id_ed25519.pub | ssh user@host "cat >> ~/.ssh/authorized_keys"

# 第三步：设置严格的权限（SSH 对此非常挑剔）
ssh user@host "chmod 600 ~/.ssh/authorized_keys"
```

---

## 权限！权限！权限！

SSH 对文件权限非常严格。如果权限"太松"，SSH 会直接拒绝使用密钥，打印那个著名的警告：

```
WARNING: UNPROTECTED PRIVATE KEY FILE!
```

正确的权限设置：

| 文件/目录 | 权限 | 含义 |
|-----------|------|------|
| `~/.ssh/` | `700` (drwx------) | 只有你能进入 |
| `~/.ssh/id_*` (私钥) | `600` (-rw-------) | 只有你能读写 |
| `~/.ssh/id_*.pub` (公钥) | `644` (-rw-r--r--) | 你读写，别人可读（无所谓，公钥就是公开的） |
| `~/.ssh/authorized_keys` | `600` (-rw-------) | 只有你能读写 |
| `~/.ssh/config` | `600` (-rw-------) | 配置文件也不该让其他人看 |
| `~/.ssh/known_hosts` | `644` (-rw-r--r--) | 可读 |

**类比**：SSH 对权限的挑剔就像银行金库 -- 门没关好（权限太宽），系统就拒绝工作。这是一种强制性的安全保护：宁可不工作，也不能在不安全的条件下运行。

快速修复所有权限：

```bash
chmod 700 ~/.ssh
chmod 600 ~/.ssh/id_*
chmod 644 ~/.ssh/id_*.pub
chmod 600 ~/.ssh/authorized_keys
```

注意 `chmod 600 ~/.ssh/id_*` 会把公钥文件的权限也改成 600，所以需要单独把公钥调回 644。

## 禁用密码登录（安全加固）

一旦确认密钥认证正常工作，你就可以在服务器上禁用密码登录，彻底堵住暴力破解的可能性。

编辑 `/etc/ssh/sshd_config`：

```bash
sudo vim /etc/ssh/sshd_config
```

核心配置项：

```
# 禁止密码认证（关键！）
PasswordAuthentication no

# 禁止空密码
PermitEmptyPasswords no

# 禁止 root 直接登录（重要！）
PermitRootLogin no

# 确保密钥认证开启
PubkeyAuthentication yes

# 限制认证尝试次数（防暴力破解）
MaxAuthTries 3

# 只允许密钥认证方式
AuthenticationMethods publickey
```

修改后一定要验证配置语法：

```bash
sudo sshd -t
```

如果显示 "Syntax OK"，就可以重新加载：

```bash
sudo systemctl reload sshd
```

**关键提醒**：在禁用密码登录之前，一定要：
1. 先确认密钥登录已经成功
2. **保持当前已登录的会话打开**，另开一个终端测试
3. 如果新终端无法登录，用原来的会话修复配置

很多人就是因为先关了密码登录、然后发现密钥登录也不成功，结果自己被锁在服务器外面了。

## 密钥的安全管理

几个好习惯：

1. **不同的用途用不同的密钥**：GitHub 一组、公司服务器一组、个人云服务器一组
2. **定期轮换密钥**：每年更新一次生产环境的密钥
3. **离职或不再使用某台机器时，删除 authorized_keys 中对应的公钥**
4. **私钥绝不出本机**：不要用邮件/微信/U盘传私钥
5. **私钥文件备份到加密介质**：比如加密的 U 盘或密码管理器

---

## 动手试试

1. 生成一组 Ed25519 密钥对（如果已有，生成一组新的，放在不同路径）
2. 查看公钥的内容：`cat ~/.ssh/id_ed25519.pub`，理解各部分（类型、密钥数据、注释）的含义
3. 查看密钥指纹：`ssh-keygen -lf ~/.ssh/id_ed25519.pub`，再用 `-lvf` 看看随机艺术图像
4. 如果有远程服务器，用 `ssh-copy-id` 上传公钥，验证免密登录
5. 检查 `.ssh` 目录和各个文件的权限是否正确
6. 运行命令后故意改错私钥权限（`chmod 644 ~/.ssh/id_ed25519`），再尝试 SSH 连接，观察那个 WARNING
7. 用 `ssh-keygen -p -f ~/.ssh/id_ed25519` 修改或删除 passphrase

---

## 本节小结

SSH 密钥认证比密码认证更安全更方便。`ssh-keygen -t ed25519` 生成密钥对，私钥留本地（保密，权限 600），公钥放服务器的 `authorized_keys` 文件中。`ssh-copy-id` 一键完成公钥上传。确认密钥登录正常后，在 `sshd_config` 中设置 `PasswordAuthentication no` 禁用密码登录，封堵暴力破解的可能。
