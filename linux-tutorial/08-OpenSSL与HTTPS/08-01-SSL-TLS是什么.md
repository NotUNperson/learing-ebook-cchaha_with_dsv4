## 正文

### 一、从一封明信片说起

假设你给朋友寄一张明信片。明信片的内容写在背面，**任何人都可以在邮寄途中翻过来读**。邮递员、分拣中心的工作人员、甚至你邻居路过你门口的信箱——都能看到你写了什么。

这就是 **HTTP 协议**的工作方式。你访问一个 HTTP 网站时，你和服务器之间的所有数据（密码、银行卡号、聊天记录）都以明文形式在互联网上传输。任何在你网络路径上的中间节点——你的路由器、ISP 的网关、WiFi 热点提供者——理论上都可以看到甚至修改这些数据。

**类比**：HTTP 就是明信片。HTTPS 是上了火漆封印的信封。

HTTPS 在 HTTP 和 TCP 之间加了一层 **TLS（Transport Layer Security，传输层安全）**，这层的作用就是对数据进行加密。即使有人截获了数据包，看到的也是一堆无意义的乱码。

### 二、两种加密方式：对称 vs 非对称

加密的核心思想是"把原文变成只有拥有钥的人才能看懂的密文"。但"钥"这个东西，有不同的用法。

#### 对称加密（Symmetric Encryption）

**一把钥匙，既用来锁门，也用来开门。**

```
明文 + 密钥 --> [加密算法] --> 密文
密文 + 密钥 --> [解密算法] --> 明文
```

**生活类比**：你和暗号联络员约好了一本相同的密码本。你写"明天下午三点"，按照密码本替换成"XJQP#3PM"，他拿到后用同一本密码本翻回来。

对称加密的优点是**速度快**。缺点是：**怎么安全地把这把密钥交给对方**？如果你通过网络传密钥，那这个传输过程本身就可能被窃听——这就是"鸡生蛋蛋生鸡"的问题。

常用的对称加密算法：AES、ChaCha20。

#### 非对称加密（Asymmetric Encryption）

**两把不同的钥匙：一把公钥（公开），一把私钥（保密）。**

```
明文 + 公钥 --> [加密] --> 密文
密文 + 私钥 --> [解密] --> 明文
```

公钥加密的内容，只有对应的私钥能解开。公钥可以满世界分发，私钥必须死死守住。

**生活类比**：你在家门口放一个开口的投递箱（公钥），任何人都能往里面投信件，但只有你有投递箱底部的钥匙（私钥）才能取出里面的信。

非对称加密解决了密钥分发问题，但缺点是**速度慢**（比对称加密慢 100-1000 倍）。

常用的非对称加密算法：RSA、ECDSA、Ed25519。

> **思考题**：你大概已经发现了，上一节学的 SSH 也是基于非对称加密。这就是为什么 `ssh-keygen` 会生成一对密钥文件——私钥自己留着，公钥上传到服务器。HTTPS 的思路几乎一样。

### 三、HTTPS 的握手过程：两者结合

HTTPS 不单独用某一种加密，而是把两者**结合**起来：

```
1. 非对称加密（慢但安全）→ 用来传输"对称加密的密钥"
2. 对称加密（快）        → 用来传输实际的数据
```

这个过程叫 **TLS 握手（Handshake）**，简化版流程如下：

```
客户端（浏览器）                          服务器（网站）
    |                                        |
    | ---- (1) ClientHello: 我支持的加密套件 --->  |
    |                                        |
    | <-- (2) ServerHello: 就选这个 + 证书 ------ |
    |                                        |
    | (3) 验证证书是否可信                       |
    |                                        |
    | (4) 用证书里的公钥，加密一个"对称密钥种子"      |
    | ---- (5) 加密的种子 -------------------> |
    |                                        |
    |             双方各自用种子算出相同的对称密钥     |
    |                                        |
    | <== (6) 之后所有数据用对称密钥加密传输 ====> |
```

**类比**：你去一家高级餐厅吃饭。你用餐厅的公钥（投递箱）把点菜单（对称密钥种子）投进去。餐厅用私钥取出点菜单，厨房开始做菜。之后所有上菜、加点的沟通，都用你们约定的一个简单手势（对称加密）来传递。这个手势是你们通过投递箱秘密约定的，外人完全不知道。

### 四、证书是什么：网站的身份证

在上面的握手第(3)步，浏览器要"验证证书是否可信"。那证书到底是什么？

**证书（Certificate）= 网站的公钥 + 网站的身份信息 + CA 的数字签名**

拆开来看：

| 证书里包含什么 | 解释 | 类比身份证上的 |
|---------------|------|--------------|
| 域名（CN/SAN） | 这个证书是颁给哪个域名的 | 姓名 |
| 有效期 | 从哪天到哪天有效 | 身份证有效期 |
| 公钥 | 网站的公钥 | ——（身份证没有这个） |
| 颁发者（Issuer）| 谁签发的这个证书 | 签发机关（公安局） |
| 签名 | CA 用它的私钥对这个证书做的数字签名 | 公安局的防伪钢印 |

**类比**：证书就是网站的身份证。你访问 `www.baidu.com`，浏览器要求它出示身份证。浏览器检查：
1. 身份证上的名字是不是 "www.baidu.com"？（域名匹配）
2. 身份证有没有过期？（有效期检查）
3. 身份证上的钢印是不是真的公安局盖的？（CA 签名验证）

如果三项都通过，浏览器就在地址栏显示小锁图标。

### 五、CA 和证书链：信任的传递

**CA（Certificate Authority，证书颁发机构）** 是互联网的"公证处"。全球有几十个被浏览器和操作系统信任的根 CA，比如：

- DigiCert
- Let's Encrypt（免费！）
- GlobalSign
- Sectigo（原 Comodo CA）
- Google Trust Services

当 CA 给你颁发证书时，它用自己的私钥对你的证书做数字签名。浏览器用 CA 的公钥（预装在操作系统或浏览器中）验证签名。如果签名正确，浏览器就信任你的证书。

**证书链（Certificate Chain）**：

```
根 CA 证书（自签名，预装在你的系统中）
    │  用根 CA 的私钥签名
    ▼
中间 CA 证书
    │  用中间 CA 的私钥签名
    ▼
你的服务器证书（叶子证书）
```

浏览器会沿着这条链向上验证，直到找到一个它信任的根 CA。如果链条中任何一环断了（比如某一级过期或被吊销），整个链条就不可信。这就是 PKI（Public Key Infrastructure，公钥基础设施）的信任模型。

**类比**：证书链就像户口迁移证明。你要证明你的祖籍在某个村，你需要：
- 村户口簿（根 CA）
- 乡镇派出所开的证明（中间 CA）
- 你自己的身份证（你的证书）

每份材料都有上一级的盖章，形成一个完整的信任链条。

### 六、SSL vs TLS：名称的由来

你可能经常看到 "SSL 证书" 和 "TLS 协议" 混用。简单区分：

- **SSL（Secure Sockets Layer）**：由 Netscape 公司在 1990 年代发明，经历了 SSL 1.0（从未公开发布）、2.0、3.0
- **TLS（Transport Layer Security）**：SSL 3.0 的后继者，由 IETF 标准化。TLS 1.0 本质上就是 SSL 3.1

今天你在任何地方听到"SSL 证书"，实际上指的都是 TLS 证书。SSL 2.0 和 3.0 都已被废弃（有严重安全漏洞），现在最低要求是 **TLS 1.2**，推荐 **TLS 1.3**。

你可以这么记：
- 协议名字叫 TLS
- 证书大家习惯还叫 "SSL 证书"
- 就像大家说的"拍张照"其实是"用手机拍张照"——名字没变，技术变了

### 七、HTTPS 的完整 URL 长什么样

```
https://www.example.com:443/path/to/page?query=value
  │          │           │         │
  │          │           │         路径和参数
  │          │           端口（HTTPS 默认 443，HTTP 默认 80）
  │          域名
  协议（HTTP + TLS）
```

当你访问一个 HTTPS 网站时，浏览器首先在 443 端口建立 TCP 连接，然后进行 TLS 握手，握手成功后开始传输加密的 HTTP 数据。

## 动手试试

1. 在浏览器中访问一个 HTTPS 网站（如 `https://www.baidu.com`），点击地址栏的小锁图标
2. 查看证书详情：看颁发者（Issuer）、有效期、SAN（Subject Alternative Names）
3. 在终端用 `curl -v https://www.baidu.com` 查看 TLS 握手的详细输出
4. 用 `openssl s_client -connect www.baidu.com:443 -showcerts` 查看服务器证书链
5. 对比访问 HTTP 版本（如果网站支持的话），思考数据安全性的差异

## 本节小结

HTTPS = HTTP + TLS。TLS 用非对称加密交换对称密钥，然后用对称加密传输数据。证书就像网站的身份证，由 CA（公证处）签发，浏览器通过证书链向上验证到信任的根 CA。SSL 是老名字，今天大家口中说的"SSL 证书"实际上都是 TLS 证书。
