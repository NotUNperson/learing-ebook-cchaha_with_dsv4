# 08-03 自签名证书：搭建你自己的微型 CA

## 本节你会学到什么

- 理解自签名证书与 CA 签名证书的区别和使用场景
- 创建自己的根 CA 证书和私钥
- 用自建 CA 签发服务器证书
- 正确配置 SAN（Subject Alternative Names）支持多域名
- 让操作系统和浏览器信任你的自建 CA

---

## 正文

### 一、什么时候用自签名证书

CA 签发的证书（如 Let's Encrypt）是给**公网用户访问**用的。但很多场景下，你不需要也不应该用公网 CA：

- **开发环境**：`localhost`、`dev.mysite.local` 这些内网域名，公网 CA 根本签不了
- **内网服务**：公司的内部管理系统、API 网关、微服务之间的 TLS 通信
- **测试环境**：在正式上线前验证 HTTPS 配置是否正确
- **学习目的**：理解证书体系的运作机制

**类比**：CA 签发证书 = 找房管局登记房产证（有法律效力，要花钱/花时间）。自签名证书 = 你在自己笔记本上画一张"地契"（在家关起门来挺有用，拿出去没人认）。

自签名证书在浏览器里会显示"不安全"警告，这是因为它不是受信任的 CA 签发的。但你可以在自己的电脑上**手动信任你的自建 CA**，这样所有由它签发的证书都会显示绿色小锁。

### 二、搭建微型 CA：完整流程

一个完整的自签名方案需要三步：

```
步骤 1：创建根 CA（"公安局"的身份）
    └── 生成 CA 私钥 + CA 自签名根证书

步骤 2：用 CA 签发服务器证书（"做身份证"）
    └── 生成服务器私钥 + CSR → CA 签发 → 服务器证书

步骤 3：配置信任（"让浏览器认识这个公安局"）
    └── 把 CA 根证书安装到系统/浏览器信任库
```

### 三、步骤 1：创建根 CA

#### 3.1 生成 CA 私钥

```bash
# 创建 CA 的工作目录
mkdir -p ~/my-ca
cd ~/my-ca

# 生成 CA 的私钥（用 4096 位 RSA，根 CA 要用最好的）
openssl genrsa -out ca.key 4096
chmod 400 ca.key    # 根 CA 私钥权限要最严格！
```

**重要**：这个 `ca.key` 是整个信任体系的根基。如果它泄露了，攻击者可以签发任意域名的证书。在生产环境中，根 CA 私钥应该离线存储在硬件安全模块（HSM）里。

#### 3.2 创建 CA 根证书（自签名）

根 CA 的证书是自己给自己签名的——它就是信任链条的最顶端。

```bash
# 创建 CA 根证书（有效期 10 年）
openssl req -new -x509 -days 3650 -key ca.key -out ca.crt \
    -subj "/C=CN/ST=Beijing/L=Beijing/O=MyOrg/OU=Security/CN=MyOrg Root CA"
```

参数解释：

| 参数 | 含义 |
|------|------|
| `-new` | 生成新的 CSR（但在 `-x509` 模式下直接变为自签名证书） |
| `-x509` | 生成自签名证书，而不是 CSR |
| `-days 3650` | 有效期 3650 天（10 年） |
| `-key ca.key` | 用这个私钥签名 |
| `-out ca.crt` | 输出证书文件 |
| `-subj` | 证书主体信息 |

查看你的根 CA 证书：

```bash
openssl x509 -in ca.crt -text -noout | head -20
```

你会看到 `Issuer:` 和 `Subject:` 是一样的——这就是"自签名"的含义。

### 四、步骤 2：签发服务器证书

#### 4.1 生成服务器私钥

```bash
# 生成服务器的私钥（ECDSA 推荐，性能更好）
openssl ecparam -genkey -name prime256v1 -out server.key
chmod 600 server.key
```

#### 4.2 创建服务器 CSR

这里的关键是**配置 SAN（Subject Alternative Name）**。现代浏览器（Chrome 58+）已经不再信任仅用 CN 指定域名的证书，必须通过 SAN 扩展声明域名。

```bash
# 方式一：用 openssl.cnf 配置文件（推荐）
cat > server.cnf << 'EOF'
[req]
default_bits       = 2048
prompt             = no
default_md         = sha256
req_extensions     = req_ext
distinguished_name = dn

[dn]
C  = CN
ST = Beijing
L  = Beijing
O  = MyOrg
OU = IT
CN = www.example.com

[req_ext]
subjectAltName = @alt_names

[alt_names]
DNS.1 = www.example.com
DNS.2 = example.com
DNS.3 = localhost
DNS.4 = *.example.com
IP.1  = 127.0.0.1
IP.2  = 192.168.1.100
EOF

# 用配置文件生成 CSR
openssl req -new -key server.key -out server.csr -config server.cnf
```

SAN 字段说明：
- `DNS.1` `DNS.2` ...：声明证书支持的域名
- `*.example.com`：通配符，匹配所有 `xxx.example.com`
- `IP.1` `IP.2` ...：声明证书支持的 IP 地址（内网场景常用）

#### 4.3 创建签发用的扩展配置文件

CA 签发证书时也需要知道 SAN 信息，否则 SAN 会被丢弃。创建一个专门给签发用的扩展配置：

```bash
cat > server-ext.cnf << 'EOF'
basicConstraints       = CA:FALSE
keyUsage               = digitalSignature, keyEncipherment
extendedKeyUsage       = serverAuth, clientAuth
subjectAltName         = @alt_names
subjectKeyIdentifier   = hash
authorityKeyIdentifier = keyid,issuer

[alt_names]
DNS.1 = www.example.com
DNS.2 = example.com
DNS.3 = localhost
IP.1  = 127.0.0.1
EOF
```

扩展项解释：

| 扩展项 | 含义 | 重要性 |
|--------|------|--------|
| `basicConstraints = CA:FALSE` | 标记这不是 CA 证书 | 必须设置 |
| `keyUsage` | 限制公钥用途（签名、加密） | 建议设置 |
| `extendedKeyUsage = serverAuth` | 标记为服务器认证用途 | 浏览器要求 |
| `subjectAltName` | 域名列表 | Chrome 强制要求 |

#### 4.4 CA 签发证书

```bash
# 用 CA 签发服务器证书（有效期 365 天）
openssl x509 -req \
    -in server.csr \
    -CA ca.crt \
    -CAkey ca.key \
    -CAcreateserial \
    -out server.crt \
    -days 365 \
    -sha256 \
    -extfile server-ext.cnf
```

| 参数 | 含义 |
|------|------|
| `-req` | 输入是 CSR |
| `-CA ca.crt` | CA 证书 |
| `-CAkey ca.key` | CA 私钥（用来签名） |
| `-CAcreateserial` | 自动为证书分配序列号 |
| `-days 365` | 证书有效期 |
| `-sha256` | 签名哈希算法 |
| `-extfile server-ext.cnf` | 扩展信息文件 |

#### 4.5 验证签发的证书

```bash
# 查看证书信息
openssl x509 -in server.crt -text -noout | grep -A5 "Subject Alternative"
# 应该能看到你配置的 DNS 和 IP 列表

# 验证证书链
openssl verify -CAfile ca.crt server.crt
# 输出：server.crt: OK

# 确认证书和私钥匹配
openssl x509 -in server.crt -modulus -noout | md5sum
openssl ec -in server.key -pubout 2>/dev/null | openssl md5
# 或用 rsa 替代 ec（取决于你的密钥类型）
```

### 五、步骤 3：让系统信任你的 CA

#### Linux 系统

```bash
# Debian / Ubuntu
sudo cp ca.crt /usr/local/share/ca-certificates/myorg-ca.crt
sudo update-ca-certificates

# CentOS / RHEL / Fedora
sudo cp ca.crt /etc/pki/ca-trust/source/anchors/myorg-ca.crt
sudo update-ca-trust

# Arch
sudo cp ca.crt /etc/ca-certificates/trust-source/anchors/myorg-ca.crt
sudo trust extract-compat
```

#### macOS

```bash
# 将 CA 证书添加到系统钥匙串，并设为永远信任
sudo security add-trusted-cert -d -r trustRoot \
    -k /Library/Keychains/System.keychain ca.crt
```

#### Windows

```cmd
# 以管理员身份运行命令提示符
certutil -addstore Root ca.crt
```

#### Firefox（独立证书库）

Firefox 使用自己的证书库，不走系统信任。需要在 Firefox 中手动导入：
1. 打开 `about:preferences#privacy`
2. 滚动到底部，点击"查看证书"
3. "证书颁发机构" -> "导入" -> 选择 `ca.crt`
4. 勾选"信任该 CA 以标识网站"

#### Chrome / Edge

Chrome 和 Edge 在 Linux 上通常用系统的证书库（所以 `update-ca-certificates` 就够），但也可以单独管理：
1. `chrome://settings/certificates` -> "证书颁发机构" -> "导入"

### 六、快速自签名一行命令

如果你不需要完整的 CA 体系，只想给一个测试服务快速搞一张自签名证书：

```bash
# 一键生成自签名证书+私钥，支持 SAN
openssl req -x509 -newkey ec -pkeyopt ec_paramgen_curve:prime256v1 \
    -keyout server.key -out server.crt \
    -days 365 -nodes \
    -subj "/CN=localhost" \
    -addext "subjectAltName=DNS:localhost,DNS:*.local,IP:127.0.0.1"
```

| 参数 | 含义 |
|------|------|
| `-newkey ec` | 生成新的 ECDSA 密钥 |
| `-pkeyopt ec_paramgen_curve:prime256v1` | 指定曲线 |
| `-nodes` | 不加密私钥（no DES，即不需要密码） |
| `-addext` | 添加证书扩展（这里用来加 SAN） |

**注意**：`-nodes` 意思是 "no DES"，即不用 DES 给私钥加密。在服务器场景下这是必须的——否则每次重启服务都要手动输入私钥密码，不现实。

### 七、自签名证书的局限性

| 场景 | 自签名可行？| 备注 |
|------|------------|------|
| 本地开发 localhost | 可行 | 安装 CA 到系统即可 |
| 内网微服务间 TLS | 可行 | 服务端配置跳过证书验证 |
| 内网员工访问的后台 | 可行 | 需要每台电脑安装 CA |
| 公网用户访问 | 不可行 | 用 Let's Encrypt 等免费 CA |
| 微信小程序 API | 不可行 | 要求公网 CA 签发的证书 |
| API 对外的 HTTPS | 不可行 | 第三方调用你的 API 会失败 |

**底线**：自签名适合你控制的设备和网络。需要公网用户访问的，必须用公网 CA 签发的证书。

## 动手试试

1. 创建你的微型 CA：生成 `ca.key` 和自签名 `ca.crt`
2. 用 CA 给 `localhost` 签发一个支持 SAN 的服务器证书
3. 用 `openssl verify` 验证证书链是否完整
4. 把 CA 证书安装到你的系统信任库
5. 用 Python 快速起一个 HTTPS 服务测试：

```bash
# 创建一个简单的 HTTPS 测试页面
python3 -c "
import http.server
import ssl

server_address = ('localhost', 4443)
httpd = http.server.HTTPServer(server_address, http.server.SimpleHTTPRequestHandler)
ctx = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
ctx.load_cert_chain('server.crt', 'server.key')
httpd.socket = ctx.wrap_socket(httpd.socket, server_side=True)
print('HTTPS Server running on https://localhost:4443')
httpd.serve_forever()
"
```

6. 浏览器访问 `https://localhost:4443`，观察是否出现绿色小锁

## 本节小结

自建 CA 分为三步：创建根 CA（自签名）、签发服务器证书（CA 签名）、安装 CA 到信任库。SAN 扩展是现代证书的必需项，Chrome 不再信任仅用 CN 的证书。自签名适合内网和开发环境，公网服务必须用 CA 签名证书。
