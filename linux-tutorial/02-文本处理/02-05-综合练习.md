# 02-05 综合练习：分析 Nginx 访问日志

## 本节你会学到什么

- 把 grep、sed、管道、重定向组合成完整的数据分析流水线
- 从 Nginx 日志中统计访问量最高的 IP 地址
- 从日志中找出所有返回 404 错误的请求
- 提取并统计 User-Agent 信息
- 体会"管道思维"——把一个复杂任务拆成多个简单步骤

## 正文

### 一、任务背景

假设你是一个网站的运维工程师。老板甩给你一个 Nginx 的 access.log 文件，让你回答三个问题：

1. 哪些 IP 访问网站最频繁？（统计访问量）
2. 有哪些请求返回了 404 错误？（找问题页面）
3. 用户都用什么浏览器访问网站？（统计 User-Agent）

如果你没有学过命令行工具，你可能会把这个文件导进 Excel，然后拖拖拽拽半天。但学了 grep、sed 和管道之后，你在终端里敲几行命令就能搞定。

**生活类比**：这就好比你手里有一大叠快递签收单（日志），你需要回答"谁寄了最多快递"、"哪些快递丢了"、"快递都用什么包装"。你可以一张一张翻，翻到手酸；也可以用 grep（快速挑出相关单据）、sed（整理格式）、管道（把一个个步骤连起来），三分钟搞定。

### 二、认识 Nginx 的 access.log 格式

Nginx 的访问日志每一行长这样：

```
192.168.1.100 - - [15/May/2026:10:30:45 +0800] "GET /index.html HTTP/1.1" 200 612 "-" "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36"
```

每一行的结构是（空格分隔）：

| 字段 | 示例值 | 说明 |
|------|--------|------|
| 1 | 192.168.1.100 | 客户端 IP 地址 |
| 2 | - | 远程登录名（通常为空） |
| 3 | - | 认证用户名（通常为空） |
| 4 | [15/May/2026:10:30:45 +0800] | 请求时间 |
| 5 | "GET /index.html HTTP/1.1" | 请求行（方法 + URL + 协议） |
| 6 | 200 | HTTP 状态码 |
| 7 | 612 | 响应体大小（字节） |
| 8 | "-" | 来源页面（Referer） |
| 9 | "Mozilla/5.0 ..." | User-Agent（浏览器标识） |

把这个结构记在心里，后面分析时你才知道该"抓"哪个字段。

### 三、任务一：统计访问量最高的 IP

#### 3.1 思路分解

用管道思维拆解：

1. 用 `awk '{print $1}'` 提取每行的第一个字段（IP 地址）
2. 用 `sort` 排序（相同的 IP 会排在一起）
3. 用 `uniq -c` 统计每个 IP 出现的次数
4. 用 `sort -rn` 按次数从多到少排列
5. 用 `head` 取前几个

#### 3.2 完整命令

```bash
# 统计所有 IP 的访问次数，按访问量排序
awk '{print $1}' access.log | sort | uniq -c | sort -rn | head -10
```

#### 3.3 分解验证

一个好习惯是：不要把一整条命令直接敲完然后祈祷它正确。而是**逐步验证**。

```bash
# 第一步：先看看能提取出什么
awk '{print $1}' access.log | head -10
# 输出应该是一列 IP 地址

# 第二步：看看排序后长什么样
awk '{print $1}' access.log | sort | head -10

# 第三步：看看统计结果
awk '{print $1}' access.log | sort | uniq -c | sort -rn | head
```

这个"逐步验证"的习惯，是你从初学者变成老手的标志性转变。

### 四、任务二：找出所有 404 错误的请求

#### 4.1 思路分解

1. 用 `grep` 筛选包含 " 404 " 的日志行（注意 404 前后有空格，避免匹配到 4040 这种状态码）
2. 用 `awk` 提取有用的字段：IP、请求的 URL、时间

#### 4.2 完整命令

```bash
# 方法一：用 grep 筛选
grep '" 404 ' access.log

# 方法二：用 awk 按字段精确筛选（更精确）
awk '$9 == "404" {print $1, $7, $4}' access.log
# $9 是状态码字段，$7 是请求 URL，$4 是时间

# 方法三：统计每个 404 页面的被请求次数
awk '$9 == "404" {print $7}' access.log | sort | uniq -c | sort -rn
```

#### 4.3 进一步分析

```bash
# 看看哪些 IP 触发了最多的 404（可能是在扫漏洞？）
awk '$9 == "404" {print $1}' access.log | sort | uniq -c | sort -rn | head -10

# 看看 404 是如何分布的——按小时统计
awk '$9 == "404" {print substr($4, 2, 14)}' access.log | sort | uniq -c
```

### 五、任务三：提取并统计 User-Agent

#### 5.1 User-Agent 是什么

User-Agent 是浏览器在请求网页时发送的一段自我介绍。looks like this:

```
"Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/120.0.0.0 Safari/537.36"
```

不过这个字段很长，不便于统计。我们通常只提取有意义的部分，比如浏览器类型。

#### 5.2 思路分解

1. 用 `awk -F'"'` 提取 User-Agent 字段（它在第 6 个双引号段里）
2. 用 `sed` 清洗，只保留关键信息
3. 统计

#### 5.3 完整命令

```bash
# 提取 User-Agent（用双引号做分隔符）
awk -F'"' '{print $6}' access.log | sort | uniq -c | sort -rn | head -10

# 提取浏览器类型并统计
awk -F'"' '{print $6}' access.log | \
  grep -oE '(Chrome|Firefox|Safari|Edge|curl|python-requests|Wget)/[0-9.]+' | \
  sort | uniq -c | sort -rn

# 简化：把各种 Chrome 变体归为一种
awk -F'"' '{print $6}' access.log | \
  sed 's/.*Chrome.*/Chrome/; s/.*Firefox.*/Firefox/; s/.*Safari.*/Safari/; s/.*curl.*/curl/' | \
  sort | uniq -c | sort -rn
```

### 六、把分析结果保存下来

每一次的分析结果，最好都保存下来。这样下次需要查的时候不用重新跑。

```bash
# 创建一个分析报告
{
  echo "===== Nginx 访问日志分析报告 ====="
  echo "分析时间: $(date)"
  echo "日志文件: access.log"
  echo ""
  
  echo "--- 访问量 TOP 10 IP ---"
  awk '{print $1}' access.log | sort | uniq -c | sort -rn | head -10
  echo ""
  
  echo "--- 404 错误 TOP 10 URL ---"
  awk '$9 == "404" {print $7}' access.log | sort | uniq -c | sort -rn | head -10
  echo ""
  
  echo "--- User-Agent 统计 ---"
  awk -F'"' '{print $6}' access.log | sort | uniq -c | sort -rn | head -10
  
} > analysis_report.txt

echo "报告已生成: analysis_report.txt"
```

### 七、管道思维总结

通过这个综合练习，你应该体会到了"管道思维"的工作方式：

1. **拆解**：把一个大问题拆成几个小步骤。
2. **逐步验证**：每一步都用 `head` 或 `less` 预览输出，确认正确后才接下一步。
3. **组合**：用 `|` 把验证过的步骤串起来。
4. **保存**：最终结果输出到文件或通过 `tee` 留底。

**生活类比**：这就像做菜。你不会一口气把所有食材丢进锅里然后开火——你会先洗菜（第一步，验证水龙头出水正常），再切菜（第二步，验证刀工），再炒菜（第三步，验证火候）。每一步确认无误，才进行下一步。管道命令也是一样——每一步先单独跑通，再串联。

### 八、练习时如果没有真实日志怎么办

你可以自己生成一个模拟日志来练习：

```bash
# 用脚本生成模拟的 Nginx 日志
#!/bin/bash
IPS=("192.168.1.100" "10.0.0.5" "172.16.0.23" "10.0.0.5" "192.168.1.100" "8.8.8.8" "192.168.1.100" "10.0.0.5")
URLS=("/index.html" "/about" "/contact" "/api/users" "/nonexistent" "/images/logo.png" "/css/style.css")
CODES=(200 200 200 200 404 200 200 304)
AGENTS=("Mozilla/5.0 (Windows NT 10.0; Win64; x64) Chrome/120.0.0.0" "Mozilla/5.0 (Macintosh) Firefox/121.0" "curl/7.68.0" "Mozilla/5.0 (X11; Linux x86_64) Chrome/120.0.0.0" "python-requests/2.28.0" "Mozilla/5.0 (Windows NT 10.0) Edge/120.0.0.0" "Mozilla/5.0 (iPhone) Safari/17.0" "Mozilla/5.0 (Macintosh) Chrome/120.0.0.0")

for i in $(seq 1 20); do
    ip=${IPS[$((RANDOM % ${#IPS[@]}))]}
    url=${URLS[$((RANDOM % ${#URLS[@]}))]}
    code=${CODES[$((RANDOM % ${#CODES[@]}))]}
    agent=${AGENTS[$((RANDOM % ${#AGENTS[@]}))]}
    size=$((RANDOM % 5000 + 100))
    hour=$((RANDOM % 24))
    printf '%s - - [%02d/May/2026:%02d:%02d:%02d +0800] "%s %s HTTP/1.1" %d %d "-" "%s"\n' \
        "$ip" 15 "$hour" "$((RANDOM % 60))" "$((RANDOM % 60))" \
        "GET" "$url" "$code" "$size" "$agent"
done
```

运行这个脚本会生成 20 行模拟日志，足够你练习了。本节的示例文件 `05-capstone.sh` 里包含了完整的生成和分析脚本。

## 动手试试

1. 运行配套的 `05-capstone.sh` 脚本（它会自动生成模拟日志并进行分析）。
2. 针对生成的 `access.log`，手工执行以下命令，验证你理解每一步：
   - `awk '{print $1}' access.log | sort | uniq -c | sort -rn` — 统计 IP 访问量
   - `grep ' 404 ' access.log` — 找出所有 404
   - `awk -F'"' '{print $6}' access.log | sort | uniq -c | sort -rn` — 统计 User-Agent
3. 尝试修改脚本中的模拟数据（比如加入更多 IP 或 URL），重新运行，看分析结果如何变化。
4. 挑战：用 sed 把日志中的 IP 地址的前三段替换成 `xxx.xxx.xxx`（匿名化处理），提示：`sed -E 's/([0-9]{1,3}\.){3}/xxx.xxx.xxx./' access.log`

## 本节小结

通过把 grep（查找过滤）、sed（修改提取）、awk（按字段提取）、管道（串联）和重定向（保存结果）组合在一起，你可以在几分钟内完成对成千上万行日志的统计分析——这就是"管道思维"的力量：把一个复杂任务拆成多个简单步骤，每一步做好一件事，然后用 `|` 把它们串起来。
