# 02-01 标准输入输出：三条传送带

## 本节你会学到什么

- 理解 stdin、stdout、stderr 三个数据流的概念
- 使用 `>` 和 `>>` 把命令输出重定向到文件
- 使用 `2>` 和 `&>` 重定向错误信息
- 理解 `/dev/null` 的作用并知道什么时候用它
- 区分"重定向"和"管道"的根本差异

## 正文

### 一、先讲一个快递分拣站的故事

想象你走进一家快递分拣站。这个站里有三个闸口：

1. **一号闸口**是接货口。所有寄来的包裹都从这儿进来，工作人员从这里取货。这个闸口就是 **stdin（标准输入）**，默认连在你的键盘上。
2. **二号闸口**是正常出货口。分拣好的包裹从这里装车送走。这个闸口就是 **stdout（标准输出）**，默认连在你的屏幕上。
3. **三号闸口**是异常品通道。破损的、地址不清的包裹被单独分到这条线上。这个闸口就是 **stderr（标准错误）**，默认也连在你的屏幕上。

每一个 Linux 命令运行的时候，就好像在这个分拣站里干活。它从一号闸口拿原料（输入），把成果送到二号闸口（输出），遇到问题就从三号闸口发出警报（错误）。

你平时运行一个命令比如 `ls`，文件列表出现在屏幕上——那是因为二号闸口默认通向屏幕。屏幕上如果冒出一行 `Permission denied`，那是三号闸口在说话——它也被默认接在了屏幕上。

那如果我**不想让包裹去默认的地方**呢？我想把正常包裹直接装车运到仓库（文件），把破损包裹丢进黑洞（/dev/null）？这就是重定向要做的事。

### 二、三个数据流的正式名称

在 Linux 里，上面说的三个闸口都有编号，叫"文件描述符"（file descriptor，简称 FD）：

| 名称 | 编号 (FD) | 中文称为 | 默认目的地 |
|------|-----------|----------|-----------|
| stdin | 0 | 标准输入 | 键盘 |
| stdout | 1 | 标准输出 | 终端屏幕 |
| stderr | 2 | 标准错误 | 终端屏幕 |

记住这三个数字很重要：0 进、1 正常出、2 错误出。后面的重定向符号都要用到它们。

### 三、重定向 stdout：把正常输出存起来

最常用的操作是把命令的输出保存到文件。

#### `>` — 覆盖写入

```bash
# 把 date 命令的输出写入 time.txt（如果文件已有内容，会先清空）
date > time.txt

# 把 ls 结果写入文件
ls -la /home > home_list.txt

# > 会先清空文件再写入，哪怕命令执行失败也会清空！
ls /不存在的目录 > result.txt
# 此时 result.txt 的内容是空的（被清空了），错误提示仍然显示在屏幕上
```

**生活类比**：`>` 就像一个"先倒空杯子再倒水"的动作。不管杯子里之前有什么，先全倒掉，然后装新饮料。正因为有这个"先倒空"的特性，往同一个文件连续用 `>` 写多次，你只能看到最后一次的内容。

再看一个用 `>` 快速创建文件的技巧：

```bash
# 快速创建一个有内容的文件
echo "Hello, Linux!" > greeting.txt
```

#### `>>` — 追加写入

```bash
# 第一次写入
echo "第一行日志" >> log.txt

# 第二次写入——这次不会覆盖，而是追加在末尾
echo "第二行日志" >> log.txt

# 现在 log.txt 里有两行内容
```

**生活类比**：`>>` 就像往一本已经写了字的笔记本后面接着写。之前的内容完好无损，新内容加在最后一行的下面。

#### 一个区分 `>` 和 `>>` 的实验

请在终端里依次执行以下命令，观察结果：

```bash
# 先准备一个文件
echo "原始的苹果" > fruit.txt

# 用 > 覆写
echo "新的香蕉" > fruit.txt
# fruit.txt 现在只有一行："新的香蕉"

# 用 >> 追加
echo "追加的橙子" >> fruit.txt
# fruit.txt 现在有两行："新的香蕉" 和 "追加的橙子"
```

这就是 `>`（覆盖）和 `>>`（追加）的区别，一目了然。

### 四、重定向 stderr：把错误信息单独处理

很多初学者都踩过一个坑：命令明明报错了，可为什么文件里什么都没有？

```bash
ls /不存在的目录 > error.txt
# error.txt 是空的，但屏幕上还是有错误提示
# 为什么？因为 > 默认只重定向 stdout（FD=1），不管 stderr（FD=2）
```

**生活类比**：你告诉快递站"把二号闸口的正常包裹都送到仓库"，但你没说怎么处理三号闸口的破损包裹。所以破损包裹还是走原来的路线（直接送到你眼前）。

要重定向 stderr（FD=2），你需要在 `>` 前面加上那个编号：

```bash
# 把错误信息重定向到文件
ls /不存在的目录 2> error.txt
# 现在 error.txt 里有内容了，而屏幕上不再显示错误

# 把正常输出和错误分开保存
ls /home /不存在的目录 > normal.txt 2> error.txt
# normal.txt 里是 /home 的文件列表
# error.txt 里是 "/不存在的目录: No such file or directory"
```

**重要规则**：`2>` 中间**不能有空格**，必须紧挨着。`2 >` 是错误写法。

### 五、合流：把 stdout 和 stderr 输出到同一个地方

很多时候你不想分开处理——你只想让它们都进同一个文件。

```bash
# 方法一：&> （bash 最常用的方法）
ls /home /不存在的目录 &> all_output.txt

# 方法二：>& （>& 把 stderr 重定向到 stdout，再整体重定向到文件）
ls /home /不存在的目录 > all_output.txt 2>&1
```

**生活类比（合流）**：你在快递站加了一条合并传送带。正常包裹和破损包裹都先汇入同一条主传送带，然后再一起送进仓库。

`2>&1` 的含义是：**"把 FD=2 的内容，导向 FD=1 目前指向的地方"**。注意写法的顺序很重要，以下两种写法结果完全不同：

```bash
# 写法一：正确。先把 stdout 指向文件，再把 stderr 指向 stdout（此时 stdout 指向文件）
ls /x > out.txt 2>&1
# 结果：stdout 和 stderr 都进入 out.txt

# 写法二：错误。先把 stderr 指向 stdout（此时 stdout 还是屏幕），再改变 stdout 的指向
ls /x 2>&1 > out.txt
# 结果：stderr 仍然出现在屏幕上！因为 2>&1 执行时 stdout 还没被重定向
```

这个顺序问题非常容易搞混。如果你觉得绕，记住一条简单原则：**用 `&>` 就对了**，它是 bash 专门提供的简写，你不用思考顺序。

### 六、/dev/null：终极垃圾桶

Linux 里有一个特殊的"黑洞文件"叫 `/dev/null`。你丢进去的任何东西都会消失得无影无踪，而且它永远不会满。

```bash
# 场景一：只关心正常输出，丢弃错误信息
ls /home /x > files.txt 2>/dev/null

# 场景二：只关心是否成功（通过 $? 判断），完全不要输出
cat giant_file.txt > /dev/null 2>&1

# 场景三：在脚本里静默执行某个命令
wget https://example.com/file.tar.gz > /dev/null 2>&1
```

**生活类比**：`/dev/null` 就是垃圾粉碎机。你把不需要的包裹丢进去，它们就从这个世界上消失了，而且粉碎机永远不会满。

还有一个常见的骚操作：用 `/dev/null` 判断一个命令是否成功，而不在乎它的输出：

```bash
# ping 一次，只看是否成功，不看输出
ping -c 1 google.com > /dev/null 2>&1 && echo "网络通" || echo "网络不通"
```

### 七、重定向 stdin：让命令从文件读取输入

有些命令默认从键盘读取输入（比如 `cat` 不带参数时）。你可以用 `<` 让它从文件读取。

```bash
# 不用 <：cat 打开文件自己读
cat poem.txt

# 用 <：把 poem.txt 的内容从 stdin 喂给 cat
cat < poem.txt

# 上面两种写法结果一样，但原理不同。
# 第一个：cat 自己打开 poem.txt 文件来读
# 第二个：shell 把 poem.txt 的内容通过 stdin 管道送进 cat
```

`<` 在简单场景下似乎多余——`cat poem.txt` 不就行了吗？但它在某些场景很有用：

```bash
# 场景：把 SQL 文件通过 stdin 喂给数据库客户端
mysql -u root -p < backup.sql

# 场景：对文件内容做排序
sort < unordered.txt > ordered.txt

# 场景：统计文件的行数、词数、字符数
wc < report.txt
```

还有一个很帅的用法叫 **heredoc（`<<`）**，可以在脚本里直接写多行文本：

```bash
# heredoc：以自定义标记（这里是 EOF）作为结束符
cat << EOF > message.txt
你好，这是一封自动生成的信。

欢迎来到 Linux 的世界。
祝学习愉快！
EOF

# 输出：message.txt 里包含了上面三行文字（EOF 之间的内容）
```

**生活类比**：普通的交互就好像你在餐馆对着服务员点菜（从键盘输入）。而 `<` 重定向就相当于你提前打印了一张菜单交给后厨（从文件输入），不需要每次都在旁边等。

### 八、一张总览表

| 符号 | 作用 | 通俗解释 |
|------|------|---------|
| `cmd > file` | stdout 覆盖写入 file | 二号线上的货倒进 file，倒之前清空 file |
| `cmd >> file` | stdout 追加写入 file | 二号线上的货追加到 file 末尾 |
| `cmd 2> file` | stderr 覆盖写入 file | 三号线上的货倒进 file |
| `cmd 2>> file` | stderr 追加写入 file | 三号线上的货追加到 file 末尾 |
| `cmd &> file` | stdout+stderr 覆盖写入 file | 二号线 + 三号线合流后倒进 file |
| `cmd < file` | 从 file 读取作为 stdin | 一号线的货从 file 来，不从键盘来 |
| `cmd << MARKER` | heredoc，多行文本作为 stdin | 在一号线上贴一张多行便签 |
| `cmd > /dev/null 2>&1` | 丢弃所有输出 | 所有包裹进粉碎机 |

### 九、常见的错误写法

```bash
# 错误：数字和 > 之间不能有空格
ls /x 2 > error.txt      # ❌ shell 会把 2 当成命令的参数！

# 正确
ls /x 2> error.txt        # ✔

# 错误：2>&1 写在 > 前面，导致 stderr 没被重定向到文件
ls /x 2>&1 > all.txt      # ❌ stderr 仍然出现在屏幕

# 正确
ls /x > all.txt 2>&1      # ✔
# 或直接用 &>
ls /x &> all.txt          # ✔
```

## 动手试试

1. 在终端执行 `ls /home /definitely_not_exists`。观察屏幕上的输出。正常信息是什么？错误信息是什么？
2. 执行 `ls /home /definitely_not_exists > a.txt`。打开 a.txt 看内容，再观察屏幕上是否还有错误信息。
3. 执行 `ls /home /definitely_not_exists > b.txt 2> c.txt`。分别打开 b.txt 和 c.txt 看内容。
4. 执行 `ls /home /definitely_not_exists &> d.txt`。打开 d.txt，看正常信息和错误信息是不是都在里面。
5. 执行 `echo "第一次" > test.txt`，然后 `echo "第二次" >> test.txt`，用 `cat test.txt` 查看内容。体会 `>` 和 `>>` 的区别。
6. 执行 `ls /home /definitely_not_exists &> /dev/null`。确认屏幕上没有任何输出。
7. 用 heredoc 创建一个文件：`cat << END > myfile.txt`，然后输入几行文字，最后输入 `END` 回车。

## 本节小结

每个 Linux 命令运行时都有三条"传送带"——标准输入（fd=0，默认来自键盘）、标准输出（fd=1，默认到屏幕）、标准错误（fd=2，默认到屏幕）。你可以用重定向符号 `>`、`>>`、`2>`、`&>`、`<` 把传送带的目的地改成文件，或者用 `/dev/null` 丢弃不需要的信息。
