# 02-02 管道：让命令排成流水线

## 本节你会学到什么

- 理解管道 `|` 的工作原理——把一个命令的输出变成另一个命令的输入
- 使用 `tee` 在管道中途"分流"保存数据
- 使用 `xargs` 把管道输入转成命令行参数
- 掌握几个常用的管道组合技巧
- 区分"管道"和"重定向"的本质不同

## 正文

### 一、从一个做菜流水线讲起

假设你在一家快餐店的后厨。后厨有一条传送带，上面摆着各种工序站：

1. 洗菜工把蔬菜洗干净，放上传送带。
2. 切菜工从传送带上取菜，切好后放回传送带。
3. 炒菜工从传送带上取切好的菜，炒完后装盘。

每个人只做一件事，但大家连成一条线，原料从一端进去，成品从另一端出来。

**管道（pipe）**在 Linux 里就是这条传送带。每个命令就是一个"工序站"，它从前一个命令的输出拿原料，加工后把自己的输出交给下一个命令。

```
命令A | 命令B | 命令C
       ↑       ↑
     管道1    管道2
```

命令A 的标准输出 → 变成 → 命令B 的标准输入  
命令B 的标准输出 → 变成 → 命令C 的标准输入

注意：**管道只传递 stdout（标准输出），不传递 stderr（标准错误）**。这是新手最容易搞混的地方。错误信息会绕过管道，直接出现在你的屏幕上。

### 二、管道和重定向到底有什么区别

很多初学者刚学完上一节的重定向，转头学管道就困惑了：不是都能把数据送到别的地方吗？

**关键区别**：

| | 重定向 `>` | 管道 `|` |
|---|---|---|
| 数据流向 | 命令 → 文件 | 命令A → 命令B |
| 接收方 | 文件系统里的文件 | 另一个正在运行的程序 |
| 数据是持久的吗 | 是（写入磁盘） | 否（只在内存中传递） |

**生活类比**：重定向像写信——你写好一封信（命令的输出），装进信封存到抽屉里（文件）。管道像打电话——你嘴里说的话直接实时传进对方耳朵，中间不存盘，打了就没了。

再看一个对比：

```bash
# 这样会产生中间文件 temp.txt（不优雅）
ls -la > temp.txt
wc -l temp.txt
rm temp.txt

# 用管道一步到位（优雅）
ls -la | wc -l
```

管道的美妙之处在于：**没有中间文件，不占磁盘，速度更快，代码更干净**。

### 三、最简单的管道：把一个大输出分页查看

你肯定遇到过这种情况：`ls /usr/bin` 输出了好几百行，屏幕疯狂滚动，根本看不清。

```bash
# 用管道把 ls 的输出送给 less，一页一页地看
ls -la /usr/bin | less

# 更常见的场景：看一个巨大的日志文件
cat /var/log/syslog | less

# 或者是看一个命令的 man 页面（虽然 man 自己就分页了，但这是演示原理）
man ls | less
```

`less` 是一个非常实用的分页查看器。进入后：
- 按空格键翻一页
- 按 `j` 向下一行，`k` 向上一行
- 按 `/` 搜索，按 `q` 退出

### 四、管道的基础组合技

#### 4.1 `grep` + `wc -l` — 统计匹配行数

```bash
# 统计 /etc/passwd 里有多少个 bash 用户
grep bash /etc/passwd | wc -l
```

#### 4.2 `ls` + `sort` — 排序输出

```bash
# 按文件大小排序（-S 按大小，-r 反过来即从大到小）
ls -la | sort -k5 -n -r | head -10
# 解释：
#   sort -k5: 按第 5 列排序（ls -la 的第 5 列是文件大小）
#   -n: 按数字排序（而不是字典序）
#   -r: 从大到小
#   head -10: 只看前 10 行
```

#### 4.3 `ps` + `grep` — 查找特定进程

```bash
# 查找所有包含 "nginx" 的进程
ps aux | grep nginx

# 但是会有一个尴尬的问题——grep 命令自身也会出现在结果里！
# 解决方法：用 grep -v 排掉 grep 自己
ps aux | grep nginx | grep -v grep

# 或者用一个小技巧：把搜索字符放进中括号
ps aux | grep '[n]ginx'
```

#### 4.4 `history` + `grep` — 查找你以前执行过的命令

```bash
# 查找你以前用过的所有 git 相关命令
history | grep git

# 查找最常用的命令
history | awk '{print $2}' | sort | uniq -c | sort -rn | head -10
```

#### 4.5 `find` + `xargs` — 找到文件后执行操作

```bash
# 找到所有 .log 文件并删除
find /tmp -name "*.log" | xargs rm

# 找到所有 .txt 文件并统计每个文件的行数
find /tmp -name "*.txt" | xargs wc -l
```

**为什么需要 xargs？** 因为 `rm` 和 `wc` 这些命令不是从 stdin 读取数据的——它们从命令行参数读取文件名。`xargs` 的作用就是把管道送来的每一行数据，转换成命令行参数。

```bash
# 没有 xargs——wc 不知道要处理什么文件
find /tmp -name "*.txt" | wc -l    # 这只会统计 find 找到了多少个文件

# 有 xargs——wc 对每一个找到的文件执行行数统计
find /tmp -name "*.txt" | xargs wc -l  # 这才是你想要的效果
```

**生活类比**：管道送来的是一张写了文件名的纸条。`wc` 这个工人不看纸条，它只看"工单上的任务栏"（命令行参数）。`xargs` 就像一个勤奋的文书，把纸条上的内容抄到工单上，`wc` 一看工单就知道该处理哪些文件了。

### 五、tee：管道中途的分流阀

有时候你希望数据既顺着管道走下去，又在中途另存一份到文件。这就是 `tee` 的作用。

**生活类比**：tee 就像水管上的三通接头（T-junction）。水流从一头进来，分成两股——一股继续往前走（进下一个命令），另一股分到一个蓄水池（文件）。

```bash
# 常见场景：执行一个命令，既要看输出，又想保存日志
./run_analysis.sh 2>&1 | tee analysis.log

# 上面的命令做了两件事：
# 1. 所有输出正常显示在屏幕上
# 2. 同时把输出保存到 analysis.log

# 再看一个：管道中间的检查点
cat huge_file.txt | grep ERROR | tee errors_only.txt | wc -l
# 解释：
#   1. 从 huge_file.txt 读取内容
#   2. 筛选包含 ERROR 的行
#   3. tee 把筛选结果保存到 errors_only.txt 同时继续往下传
#   4. 统计有多少行
```

`tee` 还有一个 `-a` 选项（append），和 `>>` 一样是追加模式：

```bash
# 追加写入，不清空原文件
./build_project.sh 2>&1 | tee -a build.log
```

### 六、管道的高级技巧

#### 6.1 管道中处理 stderr

管道默认只传递 stdout。如果你想在管道里同时处理 stderr，需要先把它"导向"stdout：

```bash
# 把 stderr 也送进管道
command 2>&1 | grep ERROR

# 或者只把 stderr 送进管道，stdout 丢弃
command 2>&1 1>/dev/null | grep ERROR
```

#### 6.2 连续管道组合：一种思维方式

当你面对一个"需要多步骤才能完成"的任务时，学会用管道思维来分解它：

**任务**：找出当前目录下占用空间最大的 5 个文件。

管道分解法：

```bash
# 步骤拆解：
# 1. du -sh * : 列出每个文件/目录的大小（人类可读格式）
# 2. sort -hr : 按人类可读格式从大到小排序
# 3. head -5 : 取前 5 个

du -sh * 2>/dev/null | sort -hr | head -5
```

**任务**：统计一个源码目录里每种编程语言有多少个文件。

```bash
# 步骤拆解：
# 1. find . -type f : 找出所有文件
# 2. sed 's/.*\.//' : 提取扩展名
# 3. sort : 排序（让 uniq 能正确去重计数）
# 4. uniq -c : 统计每种扩展名出现次数
# 5. sort -rn : 从多到少排列

find . -type f | sed 's/.*\.//' | sort | uniq -c | sort -rn
```

这就是管道思维：**把一个复杂任务，拆成多个简单步骤，每一步用一个命令实现，然后用管道串起来。**

#### 6.3 awk 在管道中的威力

`awk` 是一个轻量级的文本处理工具，在管道中非常实用。虽然我们后面有专门的章节讲它，但这里先给你一个感觉：

```bash
# 取出 ls -la 输出里的文件名和大小
ls -la | awk '{print $5, $9}'

# 统计所有用户的 home 目录大小
du -sh /home/* | awk '{sum += $1} END {print "总计:", sum}'
```

### 七、管道常见错误与排错

#### 错误一：管道中某个命令报错但你不知道

```bash
# grep 没找到任何匹配时返回非零退出码，管道后面的命令收不到数据
cat data.txt | grep "稀有数据" | wc -l
# 如果 "稀有数据" 不存在，wc -l 会输出 0，但不会报错
# 管道里的每个命令是独立运行的，一个命令失败管道可能继续
```

#### 错误二：大文件卡死管道

```bash
# 如果 log.txt 有 10GB，sort 会把整个文件读进内存排序，可能卡死
cat log.txt | sort | uniq -c | sort -rn | less
# 更好的做法：用 sort 直接读文件（sort 针对大文件有外排序优化）
sort log.txt | uniq -c | sort -rn | less
```

#### 错误三：忘记 sed/awk 等命令支持直接从文件读取

```bash
# 不必要的管道
cat data.txt | grep "error" | sed 's/error/ERROR/g'

# 更简洁的写法——grep 和 sed 都能直接读文件
grep "error" data.txt | sed 's/error/ERROR/g'

# 或者更好的
sed -n 's/error/ERROR/gp' data.txt
```

### 八、管道与会话的交互

有时候你的管道很长，写到一半想换行怎么办？用反斜杠 `\`：

```bash
find /var/log \
  -name "*.log" \
  -mtime -7 \
  -type f \
  | xargs grep -l "ERROR" \
  | sort
```

每一行结尾的 `\` 告诉 shell："这行没完，下一行还是同一条命令。"这样你的管道可读性高了很多。

## 动手试试

1. 执行 `ls -la /usr/bin | less`。按空格翻页，按 `/` 搜索 "python"，按 `q` 退出。
2. 执行 `history | grep ls`，找出你过去用过的所有包含 "ls" 的命令。
3. 在你当前目录下创建几个不同后缀名的文件：`touch a.txt b.txt c.jpg d.jpg e.py`。然后执行 `ls | sed 's/.*\.//' | sort | uniq -c | sort -rn`，看输出结果。
4. 执行 `ps aux | grep bash | grep -v grep`，看当前有哪些 bash 进程（不含 grep 自己）。
5. 执行 `echo "hello" | tee save.txt | tr 'a-z' 'A-Z'`。观察屏幕输出（全大写 HELLO），然后 `cat save.txt` 看保存的内容（原样小写 hello）。体会 tee 的作用。
6. 执行 `find /tmp -name "*.txt" 2>/dev/null | xargs wc -l 2>/dev/null`，统计 /tmp 下所有 txt 文件的行数。

## 本节小结

管道 `|` 把前一个命令的标准输出直接送进后一个命令的标准输入，让你能把简单命令像积木一样拼装出复杂的数据处理流水线。配合 `tee`（中途分流存文件）、`xargs`（转成命令行参数）和 `2>&1`（把 stderr 也送进管道），你就有了一整套在命令行处理数据的能力。
