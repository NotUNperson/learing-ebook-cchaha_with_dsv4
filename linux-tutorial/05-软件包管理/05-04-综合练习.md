# 05-04 综合练习：从源码编译安装 Nginx

## 本节你会学到什么

- 完整经历从源码编译安装一个 C 语言程序的流程：下载 -> 配置 -> 编译 -> 安装
- 理解 `./configure` 的作用，以及如何自定义编译选项
- 理解 `make` 和 `make install` 的区别
- 理解源码编译安装和包管理器安装的优劣对比
- 将编译安装的程序封装为 systemd 服务，实现开机自启

---

## 本节概述

前面三节我们学了包管理器的概念、apt 的用法、yum/dnf 的用法。用包管理器安装软件确实方便（一条命令搞定），但它有一个短板：软件版本受限于发行版的官方仓库。如果官方仓库里的 Nginx 是 1.20 版，但你需要 1.26 版的新特性，怎么办？

答案是：**从源码编译安装。**

这一节，我们将从头编译安装 Nginx -- 世界上最流行的 Web 服务器之一。整个过程会让你理解"C 程序的经典安装三步曲"：`configure -> make -> make install`。

你会看到一个完整的项目是怎么从一堆源代码文件变成一个可以运行的服务，然后再被 systemd 管理起来的。配套的示例脚本 `examples/04-capstone.sh` 已经自动化了整个流程，但这节文字教程的目的是让你理解**每一步的原理**。

---

## 为什么选择 Nginx？

Nginx（发音 "Engine X"）是一个高性能的 HTTP 和反向代理服务器。全球约三分之一的网站使用它（包括 Wikipedia、Netflix、Dropbox）。它也是 C 语言项目编译安装的经典范例，因为它：

- 使用经典的 GNU Autotools 构建系统（`./configure && make`）
- 编译选项非常丰富，是学习自定义编译的好素材
- 支持以守护进程模式运行，天然适合 systemd 管理

但即使你以后不碰 Nginx，本节学到的 `./configure && make && make install` 流程适用于绝大多数 C/C++ 项目（包括 Redis、Python 源码编译、Node.js、各种 C 库）。

## 源码编译 vs 包管理器：先想清楚

在动手之前，你必须想清楚一个问题：**真的需要从源码编译吗？**

| | 包管理器安装 | 源码编译安装 |
|---|---|---|
| 安装时间 | 几秒到几十秒 | 几分钟到几十分钟 |
| 依赖处理 | 自动 | 手动（出错时你自己排查） |
| 版本选择 | 受限制于官方仓库 | 自由选择任意版本 |
| 编译优化 | 通用优化 | 可以针对你的 CPU 架构优化 |
| 功能定制 | 装什么就是什么 | 可以关闭不需要的模块，启用实验性功能 |
| 升级/卸载 | 一条命令 | 需要手动操作 |
| 安全补丁 | 自动推送 | 需要你自己关注并重新编译 |

**类比**：包管理器是去餐厅点菜，源码编译是自己买菜做饭。前者方便、安全、快速；后者自由度高但费时费力，而且做了没人帮你洗碗（升级卸载都得自己来）。

**建议**：除非你真的需要特定的版本或编译选项，否则优先使用包管理器。对于学习来说，做一次源码编译是非常好的经历，它能让你理解一个程序是怎么从代码变成可执行文件的。

## 第 1 步：安装编译依赖

编译 Nginx 需要一些基础工具和开发库。类比：你要做红烧肉，得先有锅（编译器 gcc）、炉灶（make 构建工具）、调味料（开发库头文件）。

```bash
# Debian/Ubuntu
sudo apt update
sudo apt install build-essential libpcre3-dev libssl-dev zlib1g-dev wget tar

# RHEL/CentOS/Fedora
sudo dnf install gcc make pcre-devel openssl-devel zlib-devel wget tar
# 或 sudo dnf group install "Development Tools"  （一次性安装全套开发工具）
```

每个依赖的作用：

| 依赖包 | 用途 | 类比 |
|--------|------|------|
| `build-essential` / `gcc make` | 编译器和构建工具 | 锅和炉灶 |
| `libpcre3-dev` / `pcre-devel` | 正则表达式库（Nginx 的 rewrite 需要） | 酱油（调味） |
| `libssl-dev` / `openssl-devel` | SSL/TLS 加密库（HTTPS 需要） | 盐（必不可少） |
| `zlib1g-dev` / `zlib-devel` | 压缩库（gzip 压缩响应需要） | 料酒（去腥增香） |

注意包名的差异：Debian 系叫 `libxxx-dev`（开发文件），Red Hat 系叫 `xxx-devel`。这些开发包中包含的是头文件（`.h`）和静态库（`.a`），它们是编译时需要的，运行时不需要（运行时只需要动态链接库 `.so`）。

## 第 2 步：下载和解压源码

```bash
# 下载
wget https://nginx.org/download/nginx-1.26.2.tar.gz

# 解压
tar xzf nginx-1.26.2.tar.gz

# 进入源码目录
cd nginx-1.26.2/

# 看看源码里有什么
ls -la
```

你会在目录中看到：
- `src/`：核心源码（C 语言文件）
- `configure`：配置脚本（一个 Shell 脚本）
- `auto/`：辅助构建脚本
- `conf/`：默认配置文件
- `man/`：手册页源文件
- `README`、`CHANGES`：说明文档

## 第 3 步：./configure -- 配置编译选项

这是最关键的一步。`configure` 是一个 Shell 脚本，它做三件事：

1. **检查环境**：你的系统有没有 gcc？openssl 的版本够不够？pcre 库在哪个路径？
2. **生成 Makefile**：根据你的配置选项，生成 `make` 命令要用的构建规则文件
3. **生成配置头文件**：有些编译时选项会写到 `nginx.h` 头文件中

**类比**：`configure` 就像一个建筑设计师在开工前做的前期勘测 -- 量土地尺寸（检查系统环境）、画施工图纸（生成 Makefile）、确定水电网走线（写配置头文件）。

```bash
./configure \
    --prefix=/usr/local/nginx \
    --sbin-path=/usr/local/nginx/sbin/nginx \
    --conf-path=/usr/local/nginx/conf/nginx.conf \
    --error-log-path=/var/log/nginx/error.log \
    --http-log-path=/var/log/nginx/access.log \
    --pid-path=/var/run/nginx.pid \
    --user=www-data \
    --group=www-data \
    --with-http_ssl_module \
    --with-http_v2_module \
    --with-http_realip_module \
    --with-http_gzip_static_module \
    --with-http_stub_status_module \
    --with-stream \
    --without-http_ssi_module \
    --without-http_autoindex_module
```

逐项解释这些选项：

### 路径选项（prefix/path 系列）

| 选项 | 含义 | 如果不指定会怎样 |
|------|------|----------------|
| `--prefix` | 所有文件安装的根目录 | 默认是 `/usr/local/nginx` |
| `--sbin-path` | 可执行文件（nginx 二进制）的路径 | 默认 `prefix/sbin/nginx` |
| `--conf-path` | 主配置文件的路径 | 默认 `prefix/conf/nginx.conf` |
| `--error-log-path` / `--http-log-path` | 错误日志 / 访问日志路径 | 默认 `prefix/logs/` |
| `--pid-path` | PID 文件的路径（systemd 需要） | 默认 `prefix/logs/nginx.pid` |

### 用户选项

```bash
--user=www-data --group=www-data
```

这是前面 04-01 用户管理章节学到的知识的实际应用。Nginx 的 worker 进程会以 `www-data` 用户身份运行，而不是 root。这样即使 Nginx 被攻击，攻击者也只能获得 `www-data` 用户的权限，不能控制系统。

### 模块选项：with 和 without

```bash
--with-http_ssl_module       # 启用 HTTPS 支持
--with-http_v2_module        # 启用 HTTP/2 协议支持
--with-stream                # 启用 TCP/UDP 代理支持
--without-http_ssi_module    # 禁用 SSI（服务端包含）功能，减小编译产物
```

`--with-xxx` 表示**启用**某个可选模块（默认可能关闭），`--without-xxx` 表示**禁用**某个默认开启的模块。禁用不需要的模块可以缩小 Nginx 二进制文件的大小，减少攻击面。

### configure 执行时的输出

运行 `./configure` 后，你会看到一长串输出：

```
checking for OS
 + Linux 5.15.0-91-generic x86_64
checking for C compiler ... found
 + using GNU C compiler
checking for PCRE library ... found
checking for OpenSSL library ... found
checking for zlib library ... found
...
Configuration summary
  + using system PCRE library
  + using system OpenSSL library
  + using system zlib library

  nginx path prefix: "/usr/local/nginx"
  nginx binary file: "/usr/local/nginx/sbin/nginx"
  ...

configuring additional modules
...
creating objs/Makefile
```

如果某个 `checking for xxx` 输出 `not found`，说明缺少对应的开发库。安装对应的 `-dev` 或 `-devel` 包后重新运行 configure 即可。

## 第 4 步：make -- 编译

```bash
# 编译（使用所有 CPU 核并行编译以加速）
make -j$(nproc)

# 如果没有 nproc，手动指定核数
make -j4
```

`make` 命令会读取 `configure` 生成的 `Makefile`，然后调用 gcc 将 `.c` 源文件编译成 `.o` 目标文件，再把所有 `.o` 文件链接成一个 `nginx` 可执行文件。

**类比**：`make` 就像建筑工人按照施工图纸（Makefile）砌砖、绑钢筋、浇筑混凝土。`-j4` 意味着 4 个工人同时干活，速度是 1 个工人的 4 倍。

### 编译时间估算

在一台现代机器上，编译 Nginx 通常需要 1-3 分钟。如果是在树莓派或低配虚拟机上，可能需要 5-10 分钟。这期间终端会输出大量的 gcc 编译信息：

```
gcc -c -pipe -O -Wall -Wextra ... src/core/nginx.c -o objs/src/core/nginx.o
gcc -c -pipe -O -Wall -Wextra ... src/core/ngx_log.c -o objs/src/core/ngx_log.o
...
```

这些输出可以忽略（除非出现 `error:` 开头的行）。编译成功后，`objs/` 目录下会生成 `nginx` 可执行文件。

## 第 5 步：make install -- 安装

```bash
sudo make install
```

`make install` 会根据 `--prefix` 和其他路径选项，把编译好的文件拷贝到目标目录：

```
/usr/local/nginx/
├── sbin/
│   └── nginx          # 可执行文件（约 5-10 MB）
├── conf/
│   ├── nginx.conf     # 主配置文件
│   ├── mime.types     # MIME 类型映射
│   └── ...
├── html/
│   ├── index.html     # 默认欢迎页面
│   └── 50x.html       # 错误页面
└── logs/              # 日志目录（可能是符号链接）
```

你需要手动创建日志目录（因为它不在 `make install` 的默认行为中）：

```bash
sudo mkdir -p /var/log/nginx
sudo chown www-data:www-data /var/log/nginx
```

## 第 6 步：验证安装

```bash
# 查看 Nginx 的版本和编译选项
/usr/local/nginx/sbin/nginx -v
/usr/local/nginx/sbin/nginx -V

# 测试配置文件语法
/usr/local/nginx/sbin/nginx -t

# 手动启动（测试用）
sudo /usr/local/nginx/sbin/nginx

# 检查是否在 80 端口监听了
sudo ss -tlnp | grep nginx
```

`nginx -V` 的输出特别有价值。它会列出编译时的所有 `configure` 选项和模块列表。以后如果你想重编译 Nginx（比如加个新模块），可以用 `nginx -V` 查看原来的选项，然后照搬再加上新选项即可。

## 第 7 步：创建 systemd 服务

现在 Nginx 可以手动启动了，但每次重启系统后都需要手动启动它。我们需要把它注册为 systemd 服务。

创建 `/etc/systemd/system/nginx.service`：

```ini
[Unit]
Description=Nginx - 高性能 Web 服务器（源码编译安装）
Documentation=http://nginx.org/en/docs/
After=network.target remote-fs.target nss-lookup.target

[Service]
Type=forking
PIDFile=/var/run/nginx.pid
ExecStartPre=/usr/local/nginx/sbin/nginx -t
ExecStart=/usr/local/nginx/sbin/nginx
ExecReload=/usr/local/nginx/sbin/nginx -s reload
ExecStop=/bin/kill -s QUIT $MAINPID
PrivateTmp=true
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
```

### 为什么 Type=forking？

Nginx 启动时采用"守护进程化"的方式：主进程先启动，完成初始化后 fork 出一个子进程作为真正的守护进程，然后主进程退出。这叫做"double-fork"。

systemd 看到 `ExecStart` 启动的进程退出了，但 `PIDFile` 指定的文件里出现了新的 PID，就会把新 PID 当作服务的主进程来监控。

如果写成 `Type=simple`，systemd 会以为 Nginx 在第一次 fork 后就"死"了，从而误报启动失败。

### ExecStartPre 的妙用

```ini
ExecStartPre=/usr/local/nginx/sbin/nginx -t
```

`-t` 参数让 Nginx 只测试配置文件的语法，不实际启动。如果配置文件有语法错误，这个命令会返回非 0 退出码，systemd 会阻止启动。这是一道"安全阀"：你不会因为改错了配置而让服务挂掉。

这是一个在生产环境中被证明非常有用的设计模式。即使你用别的服务（MySQL、Redis），也应该在 systemd unit 文件中加入类似的配置检查指令。

### 启用服务

```bash
# 重新加载 systemd 配置
sudo systemctl daemon-reload

# 启动 Nginx
sudo systemctl start nginx

# 查看状态
sudo systemctl status nginx

# 设置开机自启
sudo systemctl enable nginx

# 查看日志
sudo journalctl -u nginx -f
```

## 第 8 步：测试

```bash
# 用 curl 测试 HTTP 服务
curl http://localhost/

# 查看响应头
curl -I http://localhost/

# 用浏览器访问 http://你的服务器IP/
```

你应该看到 Nginx 的默认欢迎页面。如果看不到，检查：
1. 防火墙是否开放了 80 端口：`sudo firewall-cmd --add-service=http --permanent && sudo firewall-cmd --reload`
2. Nginx 是否在运行：`systemctl status nginx`
3. 是否有其他程序占用了 80 端口：`sudo ss -tlnp | grep :80`

## 源码编译的维护

与包管理器安装不同，源码编译的 Nginx **不会自动升级**。你需要自己定期关注 Nginx 官方的安全公告，需要升级时按同样的流程重做一遍：

```bash
# 备份旧版本
sudo cp /usr/local/nginx /usr/local/nginx.old -r

# 下载新版本、编译、安装（同样的 configure 选项）
# ...

# 升级
sudo systemctl stop nginx
sudo make install              # 会覆盖旧版本的文件
sudo systemctl start nginx
```

这就是为什么服务器上通常优先使用包管理器 -- 安全更新的自动化能省下大量运维精力。

## 动手试试

**练习：从源码编译安装 Nginx**

1. 打开终端，确保你有 root 权限。克隆或定位到教程目录后，运行配套脚本：
   ```bash
   cd /path/to/linux-tutorial/05-软件包管理/examples/
   ```

2. 脚本会自动执行所有步骤。仔细阅读每一步的输出。特别关注：
   - `./configure` 阶段的路径和模块选项
   - `make -j$(nproc)` 的编译输出
   - systemd unit 文件的内容

3. 验证 Nginx 正常运行：
   ```bash
   systemctl status nginx
   curl http://localhost/
   ```

4. 修改 Nginx 的欢迎页面（试试你自己的 HTML）：
   ```bash
   sudo vim /usr/local/nginx/html/index.html

   sudo bash -c 'echo "<h1>Hello from my own compiled Nginx!</h1>" > /usr/local/nginx/html/index.html'
   ```

   ```bash
   sudo systemctl reload nginx
   curl http://localhost/
   ```

6. 查看 Nginx 的访问日志，确认你的请求被记录了：
   ```bash
   sudo tail -f /var/log/nginx/access.log
   # 在另一个终端执行 curl，观察日志输出
   ```

7. （可选）手动重新编译一次。进入下载的源码目录：
   ```bash
   cd /tmp/nginx-install/nginx-1.26.2/
   # 加一个新的编译选项（比如启用 HTTP/3 的实验性支持）
   ./configure --prefix=/usr/local/nginx --with-http_v3_module ... (复制之前的选项 + 新增)
   make -j$(nproc)
   sudo systemctl stop nginx
   sudo make install
   sudo systemctl start nginx
   ```

## 本节小结

从源码编译安装软件遵循经典的 `./configure（配置选项）-> make（编译）-> make install（安装）` 三步骤，`configure` 检测系统环境并生成构建规则，`make` 调用编译器将源码变为可执行文件，`make install` 将产物拷贝到目标位置；编译安装给了你完全的版本和模块控制权，但代价是失去了包管理器提供的自动更新和便捷的卸载能力；将此编译产物封装为 systemd 服务后，你就拥有了一个生产级别的自定义 Web 服务器。
