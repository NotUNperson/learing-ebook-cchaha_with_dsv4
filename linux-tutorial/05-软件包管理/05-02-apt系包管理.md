# 05-02 apt 系包管理

## 本节你会学到什么

- 掌握 apt 的核心操作流程：`update` -> `install`/`upgrade` -> `remove`/`purge`
- 使用 `apt search`、`apt show`、`apt list` 搜索和查看软件包信息
- 理解 `apt` 和 `dpkg` 的分工关系，以及何时需要用到 `dpkg`
- 学会添加和管理 PPA（第三方软件仓库）
- 掌握 apt 缓存管理、锁文件问题等常见故障的排查

---

## apt 是什么？一个智能管家

Ubuntu 桌面版打开时，你会看到一个叫 **Ubuntu Software** 的图形界面商店。在这个商店背后，真正干活的命令行工具就是 **apt**（Advanced Package Tool，高级包工具）。

apt 是整个 Debian 系发行版的心脏。没有 apt，你连 `ls` 命令都装不上（因为 `ls` 也是 `coreutils` 包的一部分）。

**类比**：apt 就像一个智能管家。你说"我需要一个视频播放器"，他会自己去仓库挑选合适的播放器（VLC），检查你家的系统环境（依赖），顺便帮你把解码器、字体、图标这些配套的东西都装好，最后告诉你"装好了，桌面上有图标，你双击就能用"。

## 工作流程：apt 的四步舞步

使用 apt 有一个标准的工作流程，就像你去超市买东西的流程：

```
1. 更新购物清单（apt update）
2. 查找商品在哪个货架（apt search）
3. 把商品放进购物车（apt install）
4. 定期清理过期食品（apt upgrade）
```

### 第 1 步：apt update -- 同步菜单

```bash
sudo apt update
```

当你执行 `apt update` 时，apt 会做以下事情：
- 读取 `/etc/apt/sources.list` 中配置的所有仓库地址
- 访问每个仓库，下载最新的软件包索引文件（说白了就是"菜单"）
- 把菜单存到本地（`/var/lib/apt/lists/` 目录下）
- 告诉你"有哪些新菜上架了"

**如果不运行 `apt update`** 直接运行 `apt install`，你可能装不到最新版本，甚至可能因为包名变化了而找不到包。这就像你用一本 2023 年的菜单点菜，结果有些菜已经换名字了。

类比：你手机上打开外卖 APP，第一步是看到它在"加载中..."，这就是在拉取各商家最新的菜单数据。`apt update` 就是这个过程。

### 第 2 步：apt search -- 搜索包

```bash
# 搜索关键词
apt search nginx

# 只搜索包名（不含描述）
apt search --names-only nginx

# 搜索已安装的包
apt search --installed python
```

`apt search` 会在本地的包索引中查找匹配关键词的包。它不会联网搜索，而是查你上次 `apt update` 时下载到本地的索引。这个设计意味着即使你是离线状态，只要之前做过 `apt update`，你也能搜索包。

### 第 3 步：apt show -- 查看包的详细信息

```bash
apt show nginx
```

输出包括：版本号、优先级、所属分类（section）、维护者、原始大小和安装后大小、依赖列表、建议安装的包、冲突信息、以及详细的描述。

安装前看一下 `apt show` 是个好习惯。特别关注 **Depends** 行 -- 如果有些依赖你的系统版本不满足，安装会失败。

### 第 4 步：apt install -- 安装

```bash
sudo apt install nginx

# 同时安装多个包
sudo apt install nginx php-fpm mariadb-server

# 自动回答 yes（适合在脚本里用）
sudo apt install -y nginx

# 模拟安装（看会装什么但不真的装）
sudo apt install --dry-run nginx

# 只安装，不升级依赖
sudo apt install --no-upgrade nginx

# 重新安装（覆盖配置文件）
sudo apt install --reinstall nginx

# 安装指定版本
sudo apt install nginx=1.18.0-0ubuntu1
```

当你运行 `apt install` 时，apt 会：
1. 检查这个包在不在索引里
2. 计算所有依赖（包括依赖的依赖）
3. 列出一个清单："我将安装以下 NEW 包：nginx, nginx-common, nginx-core, libnginx-mod-xxx..."
4. 询问你是否确认（输入 `y` 或直接回车）
5. 下载所有需要的 .deb 包到本地缓存
6. 调用 `dpkg` 按依赖顺序逐个安装
7. 运行每个包的 post-install 脚本（比如自动启动 nginx 服务）

### 第 5 步：apt upgrade -- 升级

```bash
# 标准升级：先刷新索引，再升级
sudo apt update && sudo apt upgrade

# 完全升级（可以解决依赖冲突，可能需要删除一些包）
sudo apt full-upgrade

# 模拟升级（看看会升级什么）
sudo apt upgrade --dry-run
```

`upgrade` 和 `full-upgrade` 的区别：

- **`upgrade`**（温和）：只升级已安装的包，**不会**为了满足依赖而删除已安装的包，也不会新增包。类比：你把调料瓶的标签换新，但瓶子本身不增不减。
- **`full-upgrade`**（激进）：为了满足依赖关系，它可以删除旧的包，安装新的依赖。类比：你不仅换标签，还扔掉了过期的酱油，买了瓶新的蚝油来配新菜谱。

**服务器上推荐用 `upgrade`**，因为它的行为可预测，不会意外删掉什么东西。桌面系统可以用 `full-upgrade`，因为它能处理更复杂的升级场景（比如 Ubuntu 的大版本升级）。

## 卸载软件：remove vs purge

```bash
# 卸载软件，保留配置文件
sudo apt remove nginx

# 卸载软件 + 删除配置文件
sudo apt purge nginx

# 自动删除不再需要的依赖包
sudo apt autoremove

# 删除不再需要的依赖包 + 它们的配置文件
sudo apt autoremove --purge
```

三个命令的"深入"程度：

```
apt remove nginx
    -> 删除 nginx 的程序文件，但 /etc/nginx/ 下的配置还在

apt purge nginx
    -> 删除 nginx 的程序文件 + /etc/nginx/ 配置文件也全删

apt autoremove
    -> 当初为了 nginx 而安装的依赖包（如 libnginx-mod-xxx），
       现在 nginx 卸载了，它们也没用了，autoremove 帮你清理这些"孤儿"。
```

**类比**：你搬家时：
- `remove`：把家具搬走了，但地板上留下了地毯和钉子洞
- `purge`：连地毯也掀了，钉子洞也填了
- `autoremove`：顺便把你搬家时发现早已不用的健身器材也扔了

## dpkg：apt 背后的"打工人"

当你看 `apt install` 的输出时，经常会看到这样的信息：

```
Unpacking nginx (2.0.1-ubuntu1) ...
Setting up nginx-core (2.0.1-ubuntu1) ...
Processing triggers for man-db ...
```

这些"Unpacking"、"Setting up"、"Processing triggers"都是由 **dpkg** 执行的。apt 是"大脑"，dpkg 是"双手"。

### dpkg 的常用场景

```bash
# 安装本地 .deb 文件（apt 不会装本地文件，只能找仓库里的）
sudo dpkg -i /path/to/package.deb

# 列出所有已安装的包
dpkg -l | grep nginx

# 查看某个包安装了哪些文件到系统里
dpkg -L nginx

# 查看某个文件属于哪个包
dpkg -S /usr/sbin/nginx

# 查看包的详细状态
dpkg -s nginx

# 修复被中断的 dpkg 操作（如安装到一半强制关机了）
sudo dpkg --configure -a
```

`dpkg -S` 尤其有用。假设你在 `/usr/bin/` 里发现一个不认识的可执行文件，你可以用 `dpkg -S /usr/bin/xxx` 查出它是哪个包安装的。这就像一个"反向查找"，从文件追溯到它的来源包。

### 什么时候必须用 dpkg 而不是 apt？

场景：你从某个项目的 GitHub Releases 页面下载了一个 `.deb` 安装包（比如 VS Code 的 .deb 文件）。apt 不认识本地文件，你需要用 `dpkg -i` 安装它。但 dpkg 不处理依赖 -- 如果这个 .deb 需要某些库，dpkg 只会报错。

```bash
# 先用 dpkg 安装 .deb
sudo dpkg -i vscode.deb
# 如果报依赖错误，再用 apt 修复
sudo apt install -f
```

`apt install -f`（`-f` 是 fix-broken 的简写）会扫描系统中未满足的依赖，并尝试自动安装缺失的包。这个组合是必学的实用技巧。

## PPA：第三方软件仓库

### 什么是 PPA？

PPA（Personal Package Archive，个人包存档）是 Ubuntu 提供的一项服务。它允许任何人在 Launchpad 平台上创建自己的软件仓库，发布自己打包的软件。

类比：官方仓库是大型连锁超市（沃尔玛、家乐福），PPA 就像社区菜市场里的摊贩。摊贩的东西超市可能没有（比如某个小众的开源工具），但质量参差不齐，需要你自行鉴别。

### 添加 PPA

```bash
# 添加 PPA
sudo add-apt-repository ppa:deadsnakes/ppa
sudo apt update  # 必须刷新索引

# 查看已添加的 PPA
ls /etc/apt/sources.list.d/

# 删除 PPA
sudo add-apt-repository --remove ppa:deadsnakes/ppa
```

添加 PPA 后，`add-apt-repository` 会在 `/etc/apt/sources.list.d/` 目录下创建一个 `.list` 文件，内容就是 PPA 的仓库地址和签名密钥。

### 使用 PPA 的风险

- PPA 的维护者可能停止更新，导致你失去安全补丁
- PPA 里的包可能和官方仓库的包版本冲突（比如你同时装了两个版本的 Python）
- PPA 的包没有经过 Ubuntu 安全团队的审查

**最佳实践**：服务器上尽量用官方仓库，只在桌面或个人开发机上用 PPA。如果必须在服务器上用，选择维护活跃、社区口碑好的 PPA。

## apt 缓存管理：清理磁盘空间

apt 会把下载的 .deb 文件缓存到 `/var/cache/apt/archives/` 目录。一台长期使用的机器，这个目录可能膨胀到几个 GB。

```bash
# 查看缓存大小
du -sh /var/cache/apt/archives/

# 清理所有缓存的 .deb 包（释放最多空间）
sudo apt clean

# 只删除旧版本/无用的缓存包（保留最新版本的缓存）
sudo apt autoclean
```

类比：
- `clean`：把冰箱里所有东西都扔了，无论好坏。第二天你要重装某个软件时，需要重新下载。
- `autoclean`：只扔过期的食物，新鲜的食物留着。如果你重装最新版，直接用缓存，不用下载。

## apt 常见故障与排查

### 锁文件问题

```
E: Could not get lock /var/lib/dpkg/lock-frontend
E: Unable to acquire the dpkg frontend lock
```

这个错误的意思是：另一个进程正在使用 apt/dpkg。可能是：
- 你开了另一个终端窗口正在运行 apt
- 系统后台的自动更新服务正在运行
- 上次 apt 操作异常中断，锁文件没有被清理

解决方法：

```bash
# 第一步：检查是谁在占用
ps aux | grep -E "apt|dpkg"

# 如果看到 unattended-upgrades（自动更新），等它自己完成
# 如果看到自己之前的终端进程，切过去结束它

# 第二步：如果确定没有任何 apt 在运行（最危险的一步！）
sudo rm /var/lib/dpkg/lock-frontend
sudo rm /var/lib/dpkg/lock
sudo rm /var/cache/apt/archives/lock
sudo dpkg --configure -a
```

### 依赖破损修复

```bash
# 自动修复破损的依赖（最常用的救急命令）
sudo apt install -f

# 重新配置所有未完成配置的包
sudo dpkg --configure -a
```

## 动手试试

**练习：用 apt 安装并探索一个软件包**

1. 刷新包索引：
   ```bash
   sudo apt update
   ```

2. 搜索 `cowsay`（一个在终端打印说话的小牛的趣味程序）：
   ```bash
   apt search cowsay
   ```

3. 查看 `cowsay` 的详细信息：
   ```bash
   apt show cowsay
   ```
   注意观察它的依赖列表和包大小。

4. 模拟安装，看看会装些什么：
   ```bash
   sudo apt install --dry-run cowsay
   ```

5. 正式安装：
   ```bash
   sudo apt install cowsay
   ```

6. 运行它试试：
   ```bash
   cowsay "Hello, Linux!"
   ```

7. 用 dpkg 查看这个包安装了哪些文件：
   ```bash
   dpkg -L cowsay
   ```

8. 查找 cowsay 的可执行文件属于哪个包：
   ```bash
   which cowsay
   dpkg -S $(which cowsay)
   ```

9. 清理：
   ```bash
   sudo apt remove cowsay    # 或者 sudo apt purge cowsay
   ```

## 本节小结

apt 是 Debian/Ubuntu 系的软件包管理器，核心工作流程是 `update`（刷新索引）-> `install`（安装）-> `upgrade`（升级），`remove` 保留配置而 `purge` 彻底清除；`dpkg` 是 apt 的底层工具用于操作本地 .deb 文件；PPA 提供了第三方软件来源但需要谨慎使用；遇到锁文件问题先用 `ps aux` 确认没有 apt 进程在跑再手动清理。
