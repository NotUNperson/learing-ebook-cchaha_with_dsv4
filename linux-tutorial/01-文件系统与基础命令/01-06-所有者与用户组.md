# 01-06 所有者与用户组

## 本节你会学到什么

- 理解文件"所有者"和"所属组"的概念
- 用 `chown` 改变文件的所有者
- 用 `chgrp` 改变文件的所属组
- 知道 `chown` 一行命令同时改所有者和所属组
- 理解 Linux 多用户协作的权限模型

---

上一节我们讲了 rwx 三种权限，但权限是"谁的"权限？这就要搞清楚两个概念：**所有者** 和 **所属组**。

在你的 Windows 电脑上，通常只有一个用户（你用 Microsoft 账号登录的那个），所以你很少想"这个文件属于谁"的问题。但 Linux 天生就是多用户系统--几十个甚至几百个用户可以同时登录同一台服务器。每个人有自己的文件，也有共享的文件。文件的"所有权"决定了谁能对这个文件做什么。

## 文件的三层身份

回顾一下 `ls -l` 的输出：

```bash
-rw-r--r--  1  zhangsan  developers  4096 May 15 10:30  project.txt
              (所有者)    (所属组)
```

每一行都有两个名字：
- **zhangsan**（所有者 / Owner）：这个文件的"主人"
- **developers**（所属组 / Group）：这个文件属于哪个"用户组"

它们的意义是：
- 如果你是 zhangsan，系统用第一组权限（rw-）来对待你
- 如果你不是 zhangsan，但属于 developers 组，系统用第二组权限（r--）来对待你
- 如果你既不是 zhangsan，也不在 developers 组里，系统用第三组权限（r--）来对待你

## 生活类比：房产证上的名字

想象你买了一套房。房产证上写的是你的名字。这意味着这套房的**所有者**是你。

但你不是一个人住。你和家人（老婆/老公/孩子）一起住。你们组成了一个"家庭"（用户组）。你对这套房有完全的处置权（可以装修、可以卖掉），你的家人对这套房也有使用权（可以住、可以用厨房），但不能卖掉（因为没有写的权限）。

外人（邻居、快递员）呢？他们和你没有任何关系，最多在门口站一下（读一下文件名），不能进房间，更不能动里面的东西。

这个类比对文件权限模型：

| 房产概念 | 文件概念 | 说明 |
|---------|---------|------|
| 房产证上的名字 | 所有者（Owner） | 对文件的最高权限 |
| 家庭成员 | 所属组（Group） | 共享使用权，权限由组权限控制 |
| 外人 | 其他人（Others） | 最低权限，甚至完全没有 |

**类比**：Linux 的权限系统就像一个小区的门禁系统。每套房（文件）的房产证上写着户主（所有者）的名字。户主有全部钥匙（rwx）。户主可以给"家人圈"（所属组）配钥匙，家人可以进门（x）看看（r），但不能砸墙（没有 w）。至于小区外面的游客（其他人），最多在小区门口看看楼号，进不了单元门。

## 用户和用户组是怎么来的

在你的 Linux 上，运行以下命令：

```bash
# 看看你是谁
whoami

# 看看你属于哪些组
groups

# 看看系统上有哪些用户
cat /etc/passwd

# 看看系统上有哪些组
cat /etc/group
```

当你安装系统、创建第一个用户时（比如用户名是 `zhangsan`），系统自动做了两件事：
1. 创建了一个用户 `zhangsan`
2. 创建一个同名的组 `zhangsan`，并把 `zhangsan` 加入这个组

这就是为什么你看自己新建的文件，所有者和所属组都是你的用户名：

```bash
touch newfile.txt
ls -l newfile.txt
# -rw-r--r-- 1 zhangsan zhangsan 0 May 15 10:30 newfile.txt
#              ^^^^^^^^ ^^^^^^^^
#              所有者    所属组
```

**类比**：你去新公司入职。HR 在系统里创建了你的账号（用户名 `zhangsan`），同时把你加入了"技术部"（用户组 `developers`）。你自己有一个工位（家目录 `/home/zhangsan`），技术部的共享文件夹（组共享目录）是大家都能访问的。你现在在公司的"身份"就是 `zhangsan:developers`（用户 + 组）。

## chown -- 改变所有者

`chown` 是 "change owner" 的缩写。它的作用就是改房产证上的名字。

```bash
# 基本语法
chown 新所有者 文件名

# 把 file.txt 的所有者改为 lisi
chown lisi file.txt

# 只改变所有者，不改变所属组
# 注意：只有 root 用户才能使用 chown（或者有相应 sudo 权限）
# 普通用户不能把自己的文件"送"给别人（安全设计）
```

为什么普通用户不能随便 `chown`？想象一下：如果任何人都可以把文件的"房产证"上的名字随意改掉，那整个权限系统就形同虚设了。我创建了一个包含恶意代码的文件，把所有者改成 root，权限设成 777……所以 Linux 规定：只有 root 可以改变文件的所有权。

```bash
# 普通用户尝试 chown 会失败
touch myfile.txt
chown lisi myfile.txt
# chown: changing ownership of 'myfile.txt': Operation not permitted

# 需要用 sudo 提权
sudo chown lisi myfile.txt
# 成功
```

**类比**：你不能随便把自家房产证上的名字改成别人的（需要去房管局过户）。同样，你不能随便把文件"送人"，需要管理员（root）来做"过户"操作。

### chown 同时改所有者和所属组

```bash
# 语法: chown 所有者:所属组 文件名
chown zhangsan:developers project.txt

# 只改所属组（不改所有者）
chown :developers project.txt

# 等价于 chgrp（下面会讲）
```

## chgrp -- 改变所属组

`chgrp` 是 "change group" 的缩写。它只改变文件的"家庭"归属。

```bash
# 把 file.txt 的所属组改为 developers
chgrp developers file.txt

# 等价于
chown :developers file.txt
```

普通用户可以使用 `chgrp`，但有限制：你只能把文件改成你所属的组。也就是说，你不能把文件塞进一个你不在里面的组。

```bash
# 查看你属于哪些组
groups
# zhangsan developers docker

# 你可以把文件改成 developers 组（因为你在里面）
chgrp developers myfile.txt   # 成功

# 但不能改成你不在的组
chgrp admin myfile.txt        # 可能失败，因为你不是 admin 组的成员
```

**类比**：你可以选择把自己的房子挂在"家庭"名下还是"投资公司"名下，但前提是你必须是"投资公司"的成员。你不能随便把房子挂在一个你不属于的组织名下。

### 递归修改

```bash
# 把整个目录及其所有子文件的所有者/组都改掉
chown -R zhangsan:developers /home/zhangsan/projects/

# 只递归改所属组
chgrp -R developers /home/zhangsan/projects/
```

## 常见协作场景

### 场景一：个人项目（默认模式）

```bash
# 每个人在自己的家目录下工作，文件自己拥有
/home/zhangsan/
  ├── project_a/    # 所有者: zhangsan, 所属组: zhangsan
  └── notes.txt     # 所有者: zhangsan, 所属组: zhangsan

/home/lisi/
  ├── project_b/    # 所有者: lisi, 所属组: lisi
  └── todo.txt      # 所有者: lisi, 所属组: lisi
```

这个模式下，每个人管自己的文件，互不干涉。就像一个公司里每人有自己的工位，别人不会乱动你桌上的东西。

### 场景二：团队协作（共享目录模式）

假设你和同事共同维护一个 Web 项目，放在 `/var/www/website/` 下。你们都属于 `webdev` 组。

```bash
# 管理员创建共享目录
sudo mkdir -p /var/www/website

# 把目录的所有者和组设为 webdev
sudo chown root:webdev /var/www/website

# 设置权限：
# 所有者为 root（root 不需要在里面写文件，只是保证没有个人用户拥有它）
# 组有读写执行权限（webdev 组的成员可以自由操作）
# 其他人没有任何权限
sudo chmod 770 /var/www/website

# 设置 SGID 位（见下文）
sudo chmod g+s /var/www/website
```

这样，`webdev` 组的所有成员都可以在 `/var/www/website/` 中创建、修改、删除文件。但不在 `webdev` 组里的人（比如营销部的同事）完全看不到这个目录的内容。

### SGID 位：让新文件自动继承目录的组

有一个很有用的特殊权限叫 **SGID**（Set Group ID）。当一个目录设置了 SGID 位后，在这个目录中创建的所有新文件都会自动继承该目录的所属组，而不是使用创建者的默认组。

```bash
# 设置 SGID
chmod g+s /var/www/website

# 查看效果：注意权限字符串中的 s
ls -ld /var/www/website
# drwxrws--- 2 root webdev 4096 May 15 10:30 /var/www/website
#     ^
#     这个 s 就是 SGID 位

# 现在 zhangsan 在目录中创建文件
cd /var/www/website
touch zhangsan_file.txt
ls -l zhangsan_file.txt
# -rw-r--r-- 1 zhangsan webdev 0 May 15 10:30 zhangsan_file.txt
#                    ^^^^^^^^ ^^^^^^
# 所有者是 zhangsan，但所属组自动变成了 webdev！
# 如果没有 SGID，所属组会是 zhangsan
```

**类比**：SGID 就像公司的"默认项目归属"规则。技术部的共享文件夹上贴了个标签："在这个文件夹里创建的任何文件，默认标记为技术部项目"。不管你（张三还是李四）是谁，你在这里创建的文件都自动挂着"技术部"的标签（所属组 = webdev），这样技术部的其他人自然就有相应权限了。

## 动手试试

在你的 Linux 终端中完成以下操作：

1. 查看你的身份信息：
```bash
whoami
groups
id
```

2. 创建一个测试目录和文件：
```bash
mkdir ~/linux-practice/ownership
cd ~/linux-practice/ownership
touch test.txt
ls -l test.txt
```
观察所有者和所属组都是你的用户名。

3. 查看系统有哪些用户（只看前 5 个）：
```bash
head -5 /etc/passwd
```

4. 查看系统有哪些组（只看前 10 个）：
```bash
head -10 /etc/group
```

5. 查看 /etc 下一些重要文件的所有权：
```bash
ls -l /etc/passwd
ls -l /etc/shadow
ls -l /etc/hostname
```
注意哪些文件属于 root，哪些是只读的。

6. 查看你用户家目录下的隐藏配置文件的所有权：
```bash
ls -la ~ | head -20
```

7. 尝试用 `chown`（会提示权限不够，没关系，理解原因即可）：
```bash
sudo chown root test.txt
ls -l test.txt
sudo chown $USER test.txt
```
8. 清理：`cd ~ && rm -rf ~/linux-practice/ownership`

## 本节小结

每个文件有一个所有者（Owner，房产证上的户主）和一个所属组（Group，同住家庭成员）。`chown` 改所有者（需要 root 权限），`chgrp` 改所属组（需要你是目标组的成员）。多用户协作时，共享目录配合组权限是最常用的模式。
