## 前台还是后台？一场音乐会的比喻

你去音乐会听钢琴独奏。舞台上只放着一架三角钢琴，演奏者坐在琴前弹奏。全场观众都看着这一个人。

这时，后台有一支弦乐队在准备。他们拉琴的声音你能隐约听到，但他们不在聚光灯下，不影响钢琴家在舞台中央的表演。等钢琴部分结束，弦乐队就会被"调到前台"来。

Linux 的作业控制也是这样：

- **前台（Foreground）**：舞台中央的那个程序。它在你的终端上运行，占据你的屏幕和键盘输入。一次只能有一个前台程序。
- **后台（Background）**：后台伙计。它也在运行，但是不占你的屏幕，不需要你的输入。可以同时有多个后台程序。

你的终端像一个只有一台监视器的监控室。你只能盯着一个画面（前台），但其他的摄像头（后台）也在录像，你可以随时切过去看。

## 作业控制的基础操作

### 启动后台任务：在命令后面加 &

```bash
# 在后台启动 find
find / -name "*.log" > ~/log_files.txt 2>&1 &

# 在后台启动一个持续监控脚本
./monitor.sh &
```

`&` 符号就像跟 Shell 说："这条命令你放到后台去跑，别占着我的终端。"

运行后，Shell 会打印：
```
[1] 12345
```

- `[1]`：作业编号（job number），Shell 内部用来管理后台任务
- `12345`：进程 PID

### 查看后台作业：jobs

```bash
# 查看当前终端的所有后台作业
jobs

# 输出示例：
# [1]  - running    find / -name "*.log" > ~/log_files.txt 2>&1
# [2]  + running    ./monitor.sh
```

`+` 号标记的是"当前作业"（最近一个被放到后台的），`-` 号标记的是"上一个作业"。

### 暂停前台程序：Ctrl+Z

假设你正在用 `vim` 编辑一个文件，突然想回到终端查看一个命令的输出。你可以退出 `vim` 再重新打开，但更快的办法是：

- 按 `Ctrl+Z`：暂停当前前台程序，把它扔到后台，回到 Shell
- 干完你的事后，用 `fg` 把程序调回前台继续

```bash
$ vim important.txt
# 按 Ctrl+Z
[1]+  Stopped    vim important.txt
$ ls -la          # 现在你可以操作 Shell 了
$ fg              # 回到 vim
```

`Ctrl+Z` 让程序像被按下暂停键的 DVD 播放器 -- 它没被关掉，只是停在当前帧。

### 恢复后台作业：fg 和 bg

```bash
# 把后台作业调到前台运行
fg        # 恢复当前作业（+ 号标记的那个）
fg %1     # 恢复作业编号为 1 的作业
fg %vim   # 恢复命令名匹配 "vim" 的作业

# 让暂停的作业在后台继续运行
bg        # 让当前作业在后台继续
bg %1     # 让作业 1 在后台继续
bg %vim   # 让 vim 作业在后台继续
```

`fg` 就像把演员从后台叫到舞台上，`bg` 就是让后台演员继续排练，不打扰舞台的人。

### 总结：作业状态转换图

```
命令启动 -> [前台运行]
     |             |
     | 添加 &      | 按 Ctrl+Z
     v             v
 [后台运行] <-- [后台暂停]
     |        bg    |
     |              | fg
     v              v
 [前台运行] <-- [前台暂停]
```

## 信号：给进程发通知

### 什么是信号？

信号（Signal）是 Linux 内核提供的一种**进程间通信机制**。它非常轻量，本质上就是给进程发一个"通知"：

- "嘿，你该退出了" -- SIGTERM
- "马上被干掉！" -- SIGKILL
- "用户按了 Ctrl+C" -- SIGINT
- "你被暂停了" -- SIGSTOP / SIGTSTP

类比：信号就像是手机的**通知横幅**。你正在用微信聊天，屏幕顶部弹出一条横幅："您的快递已到菜鸟驿站"。这个通知不影响你继续聊天，但你看到了就知道去取快递。进程收到信号后可以选择：忽略它、处理它、或者按默认方式响应。

唯一的例外是 SIGKILL（信号 9）和 SIGSTOP（信号 19），这两个信号非常霸道，进程无法忽略也无法自定义处理，必须服从。

### 常用信号速查表

| 编号 | 信号名 | 触发方式 | 含义 | 类比 |
|------|--------|---------|------|------|
| 1 | SIGHUP | 终端关闭 | 挂断信号，通知进程父终端已关闭 | 电话线被掐断了 |
| 2 | SIGINT | Ctrl+C | 中断，请进程停止当前操作 | 观众喊"别弹了！" |
| 3 | SIGQUIT | Ctrl+\\ | 退出，并生成 core dump | 观众扔臭鸡蛋，还要你写检讨 |
| 9 | SIGKILL | `kill -9` | 强制杀死，进程无法捕获 | 直接拔电源 |
| 15 | SIGTERM | `kill`（默认） | 终止，进程可以捕获并做清理 | 老板说"下班前收拾好走人" |
| 18 | SIGCONT | `kill -CONT` | 让暂停的进程继续运行 | 解除暂停，继续播放 |
| 19 | SIGSTOP | 内核 | 暂停进程，无法捕获 | 强制按暂停（遥控器坏了也得停） |
| 20 | SIGTSTP | Ctrl+Z | 终端暂停，进程可以忽略 | 温和地按暂停 |

查看完整列表：`kill -l`

## 杀死进程的正确姿势

### 理解 kill 这个名字的误导性

`kill` 这个名字取得不好。它实际上不是"杀死"，而是**"发送信号"**。你能用 `kill` 发送任何信号，包括让进程暂停的 SIGSTOP 和让它继续的 SIGCONT。

```bash
kill -STOP 12345    # 暂停进程 12345
kill -CONT 12345    # 让进程 12345 继续运行
kill 12345          # 发送 SIGTERM（默认），请进程退出
```

这个名字就像把"邮递员"叫做"送讣告的"。他确实偶尔送讣告（SIGTERM），但大多数时候送的是普通信件。

### 杀人三步法（你应该遵守的礼仪）

当你需要停止一个进程时，不要上来就用 `kill -9`。这就像邻居音响开太大声，你先报警而不是先敲门。

**正确的三步法**：

```bash
# 第 1 步：礼貌地请它退出（SIGTERM）
kill 12345

# 等几秒，检查它还在不在
ps -p 12345

# 第 2 步：如果还在，再友好地提醒一次（SIGINT，等于 Ctrl+C）
kill -2 12345

# 再等几秒，再检查
ps -p 12345

# 第 3 步：如果还不走，强制执行（SIGKILL）
kill -9 12345
```

### 为什么不要一上来就用 kill -9？

因为进程收到 SIGTERM 后有机会做善后工作：

- 关闭打开的文件，把缓存数据写入磁盘
- 释放占用的资源（网络连接、共享内存等）
- 记录日志："我在 14:32 被 SIGTERM 终止"
- 通知相关进程："我要退了，你们注意"

而 SIGKILL 不给任何反应时间。就像你正在用 Word 写文档，突然有人把你的电脑电源拔了 -- 你还没来得及保存！

实际生产中，"`kill -9` 后发现数据库文件坏了"这类故事屡见不鲜。所以请把 `kill -9` 作为最后的手段。

### 批量杀进程：killall 和 pkill

```bash
# 杀死所有名为 nginx 的进程（发送 SIGTERM）
killall nginx

# 强制杀死所有 nginx 进程
killall -9 nginx

# pkill：按模式匹配进程名
pkill -f "python.*app\\.py"

# pkill 按用户杀
pkill -u zhangsan

# pkill 不要密码，用交互模式确认
pkill -i nginx
```

`killall` 是按进程的**精确名称**匹配，而 `pkill` 是按**模式**匹配。`pkill -f` 可以匹配完整的命令行，比 `killall` 更灵活。

**新手注意**：在不同的 Unix 系统上，`killall` 的行为可能完全不同。在 Solaris 上，`killall` 会杀死所有进程（相当于关机！）。好在 Linux 上的 `killall` 是安全的，按名称匹配。

## nohup：让你的程序在你离开后继续跑

### 问题的产生

你通过 SSH 登录到一台服务器，启动了一个机器学习训练脚本，预计要跑 8 小时。你关了笔记本电脑，去睡觉了。第二天起来发现，训练在凌晨 2 点就中断了。

为什么？因为当你关闭 SSH 连接时，终端会向它的所有子进程发送 **SIGHUP**（hangup，挂断）信号。你的训练脚本收到 SIGHUP 后，默认行为是退出。

类比：你是一条船队的旗舰。你和所有护卫舰用对讲机保持联系。当你关闭对讲机（退出终端）时，所有护卫舰的默认反应是"旗舰失联了，我们也解散回家吧"。

### nohup 的解决方式

```bash
# 基本用法
nohup ./train_model.sh &

# 推荐用法：指定输出文件
nohup ./train_model.sh > training.log 2>&1 &

# 查看输出
tail -f training.log
```

`nohup` 是 **no hang up** 的缩写。它让程序**忽略 SIGHUP 信号**。即使你关闭终端，程序也不会收到"挂断"通知，而是安安静静地继续运行。

`nohup` 默认会把程序的输出写入当前目录下的 `nohup.out` 文件中。如果你不重定向，这个文件可能会变得很大。

### 另一种方案：disown

你已经启动了一个程序但忘了加 `nohup`？别慌，还有补救办法。

```bash
# 启动了一个长任务
./long_task.sh &

# 突然想起来忘了 nohup！
# 补救步骤：
Ctrl+Z              # 先暂停它
bg                  # 放到后台继续
disown -h %1        # 放弃监护权（告诉 Shell：我不管它了）
# 现在可以安全退出终端了
```

类比：`disown` 就像对 Shell 说："这孩子从现在起独立了，它的事跟我无关了。"Shell 退出时不会给"已断绝关系"的子进程发 SIGHUP 信号。

## screen 和 tmux 的前瞻

`nohup` 和 `disown` 能解决"退出终端后程序继续运行"的问题，但它们有一个局限：你没法再重新"切回去"看程序的运行状态。

如果你需要既能保持程序运行，又能随时重新连接查看状态，就需要 **screen** 或 **tmux**。它们在 SSH 和远程连接章节中会详细讲，这里先留个印象。

类比：
- `nohup` / `disown`：你把孩子送去寄宿学校，偶尔通过信件（日志文件）了解情况，但不能当面看他。
- `screen` / `tmux`：你有一间"云端办公室"，不管从哪里登录，你都能回到那间办公室，看到所有窗口和正在运行的程序，就像从未离开过。

## 动手试试

**练习：前后台切换和信号发送**

1. 打开终端，启动一个长时间运行的命令（比如 `sleep`）：
   ```bash
   sleep 300
   ```

2. 按 `Ctrl+Z` 暂停它。观察 Shell 打印的作业信息。

3. 输入 `jobs` 查看暂停的作业。注意 `+` 号标记的是哪个作业。

4. 用 `bg` 让它在后台继续运行：
   ```bash
   bg
   jobs
   ```

5. 再启动一个后台任务（直接在命令后加 `&`）：
   ```bash
   sleep 200 &
   jobs
   ```
   现在应该有两个后台作业了。

6. 把第一个作业调回前台：
   ```bash
   fg %1
   ```
   它回到前台后继续 `sleep`，等待它结束或按 `Ctrl+C` 提前终止。

7. 信号实验（需要两个终端）：
   - 终端 A：运行 `sleep 600 &`，记录 PID（比如是 12345）
   - 终端 B：运行 `kill -TERM 12345`，观察终端 A 的反应
   - 终端 A：再运行 `sleep 600 &`（新 PID），终端 B 运行 `kill -KILL 新PID`，感觉一下区别

8. 尝试 `nohup`：
   ```bash
   nohup sleep 300 &
   cat nohup.out   # 应该是空的，因为 sleep 不输出任何东西
   ```

## 本节小结

Linux 通过前台/后台机制和信号系统让你灵活控制进程的执行方式：`Ctrl+Z` 暂停、`bg` 后台运行、`fg` 调回前台、`&` 直接后台启动；而 `kill` 本质上是一个"信号发送器"，`SIGTERM`（15）给进程体面退出的机会，`SIGKILL`（9）是最后手段，`nohup` 和 `disown` 让你的程序在你退出终端后继续存活。
