# 04-06 综合练习：创建自定义 systemd 服务

## 本节你会学到什么

- 将前面五节学到的知识串联成一个完整的实践项目
- 亲手创建一个自定义 systemd 服务：编写脚本、配置 unit 文件、启停管理
- 理解 `Type=simple` vs `Type=forking` 在实际中的应用
- 学会 `systemctl edit` 修改已有服务的配置
- 掌握从无到有搭建一个生产级服务的完整流程

## 本节概述

前三节我们学了用户管理、权限切换、进程和信号的查看控制。第四节我们学了 systemd 的概念和 systemctl 的用法。现在是时候把所有这些串起来，做一个真正的手工项目。

我们的目标是：**从零开始创建一个 systemd 管理的后台服务**。

这个服务会做一件简单但有意义的事 -- 一个微型的 HTTP 服务器，返回系统信息和访问计数器。通过这个项目，你会完整经历：写脚本 -> 选用户 -> 配置 unit 文件 -> 启动 -> 查日志 -> 设开机自启 -> 排错 的全过程。

你已经准备好了配套的示例脚本 `examples/06-capstone.sh`，它自动化了整个过程。但这节文字教程的目的是让你理解每一步**为什么要这样做**，而不仅仅是跑一个脚本。

## 设计思路：我们要做什么？

假设你是一家小公司的运维工程师。老板说："我需要一个简单的 Web 页面，显示服务器的基本信息和访问次数，用来监控我们的开发服务器。"

你没有现成的 Web 框架，但你知道 Linux 自带一切需要的零件：
- `nc`（netcat）：能监听 TCP 端口
- `bash`：能生成 HTML 文本
- `echo`、`date`、`hostname`：能获取系统信息
- `systemd`：能让这个脚本变成真正的服务

开发机器不会有太多并发请求，所以 bash + nc 的性能完全够用。这就叫**用对工具，而不是用重型工具**。

## 第 1 步：编写服务脚本

我们的脚本需要做什么？

1. 监听一个端口（比如 8080）
2. 当有人访问时，返回一个 HTML 页面
3. 记录总访问次数（持久化到文件）
4. 正确处理 `SIGTERM` 信号（以便 systemctl stop 能优雅关机）
5. 把日志输出到标准输出/错误（以便 systemd 收集到 journal）

### 信号处理的陷阱

在第 04-04 节我们学过，`kill` 默认发送 SIGTERM，而 `systemctl stop` 也是发送 SIGTERM。如果你的脚本不捕获 SIGTERM，它会被强制杀死，来不及保存计数器和清理资源。

所以在脚本里，我们务必加上：

```bash
cleanup() {
    echo "$(date) - 收到终止信号，正在退出..." >> /var/log/myservice.log
    exit 0
}
trap cleanup SIGTERM SIGINT
```

`trap` 是 bash 内建命令，用来捕获信号并执行指定的函数。类比：就像你手机上的"低电量模式" -- 当电量降到 20% 时，自动触发一系列省电操作。`trap cleanup SIGTERM` 就是"当收到 SIGTERM 时，自动执行 cleanup 函数"。

### 不使用 Python 或 Node.js 的原因

你可能会想：为什么不直接用 Python 的 `http.server` 或者 Node.js 的 `http` 模块？因为这节课的目的是让你理解**service 的本质**，而不是依赖某个特定语言或框架。用 bash + nc 做服务，你会更清楚地看到：

- 进程如何启动
- 如何绑定端口
- 如何响应请求
- 信号如何传递

当你理解了这些底层机制后，无论你以后用 Go、Python、Rust 还是 Java 写服务，systemd 的管理方式都是一样的。

## 第 2 步：决定服务以哪个用户运行

这是一个容易被忽略但非常关键的决策。服务应该以哪个用户的身份运行？

- **不要用 root**：如果服务有安全漏洞，攻击者就能以 root 身份执行任意代码。这是最严重的错误之一。
- **不要用你自己的用户**：如果某天你离职了，账号被禁用，服务就跟着挂了。
- **考虑用专用用户**：如果你的服务需要访问特定文件，最好创建一个专用的系统用户（无登录 Shell），只授予它最少必要的权限。

在我们的示例脚本中，我们用一个普通用户（运行脚本的 sudo 用户）来演示。在生产环境中，你应该创建专用用户：

```bash
# 创建系统用户（无家目录，无登录 Shell）
sudo useradd -r -s /usr/sbin/nologin myservice

# 在 unit 文件中使用这个用户
# User=myservice
# Group=myservice
```

`-r` 参数创建的是**系统用户**（UID < 1000），`-s /usr/sbin/nologin` 意味着这个用户不能登录 Shell，只能被 systemd 用来运行服务。就像你给外卖小哥配了一把只能开小区大门的临时卡，他不能进你家，不能坐电梯，只够走到快递柜。

## 第 3 步：编写 systemd unit 文件

unit 文件放在 `/etc/systemd/system/` 下，以 `.service` 结尾。下面是我们的完整 unit 文件，逐个指令解释：

```ini
[Unit]
Description=Linux 学习用 Demo HTTP 服务
After=network.target

[Service]
Type=simple
ExecStart=/usr/local/bin/linux-demo.sh
ExecStop=/bin/kill -SIGTERM $MAINPID
Restart=on-failure
RestartSec=5
User=nobody
Group=nogroup
WorkingDirectory=/var/lib/linux-demo
MemoryMax=50M
CPUQuota=20%
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

### 逐个指令解读

**`After=network.target`**
网络服务可能还没启动时，我们的 HTTP 服务去绑定端口就会失败。`After` 告诉 systemd："等我网络好了你再启动我。"注意这只是**顺序**约束，不是**依赖**约束。如果网络启动失败，我们的服务依旧会被尝试启动。如果你需要"没有网络我就不启动"，那应该用 `Requires=network.target`。

**`Type=simple`**
我们的脚本用 `nc -l` 在前台持续运行，所以它是 `simple` 类型。systemd 认为 `ExecStart` 启动的进程就是主进程，直接监控它。

如果换成 `Type=forking`（用于传统守护进程），systemd 会等待 `ExecStart` 启动的进程退出，然后把它的子进程视为真正的服务主进程。这让事情复杂很多，所以我们尽量选择 `simple`。

**`Restart=on-failure`**
只有在服务异常退出时（进程崩溃、被 kill 信号杀死）才自动重启。如果你用 `systemctl stop` 正常停止它，它不会自己活过来。`RestartSec=5` 表示重启前等 5 秒 -- 这是为了避免"崩溃-重启-崩溃-重启"的死循环把 CPU 打满。

**`MemoryMax=50M` 和 `CPUQuota=20%`**
这是 systemd 的资源限制功能。它利用 Linux 内核的 cgroup 机制来限制服务能使用的资源。如果一个服务出现内存泄漏，它最多吃掉 50MB 内存而不是把整台机器的内存吃光。

**`StandardOutput=journal` 和 `StandardError=journal`**
把服务的标准输出和标准错误都送到 systemd 的 journal 日志系统。这样你在脚本里用的 `echo` 语句会自动变成日志条目，可以用 `journalctl -u linux-demo` 查看。你也可以选择 `syslog`（传统的 syslog 系统）或 `file:/path/to/file`（写到指定文件）。

## 第 4 步：加载并启动服务

```bash
# 重新加载 systemd 配置（每次修改 unit 文件后都要执行）
sudo systemctl daemon-reload

# 启动服务
sudo systemctl start linux-demo

# 检查状态
sudo systemctl status linux-demo

# 设置开机自启
sudo systemctl enable linux-demo
```

`daemon-reload` 这一步新手经常忘记。systemd 不会自动感知你修改了 unit 文件，你需要明确告诉它"配置文件变了，重新读一下"。

---

## 第 5 步：查看日志和排错

```bash
# 查看服务日志
journalctl -u linux-demo

# 实时跟踪
journalctl -u linux-demo -f

# 只看错误
journalctl -u linux-demo -p err

# 最近 50 条
journalctl -u linux-demo -n 50
```

如果服务启动失败，`systemctl status linux-demo` 会显示退出码和最近的几行日志。常见的启动失败原因：

| 症状 | 可能原因 | 解决办法 |
|------|---------|---------|
| exit code 203 | `ExecStart` 指定的程序不存在或不可执行 | 检查路径和权限（`ls -la /path/to/script`） |
| exit code 1 | 程序运行了但报错退出了 | 查看日志（`journalctl -u 服务名`） |
| 端口被占用 | 有其他程序占用了你指定的端口 | `ss -tlnp` 查看端口占用，换个端口 |
| Permission denied | User 指定的用户没有权限访问某些文件 | 检查文件权限或换一个用户 |

## 第 6 步：测试服务

```bash
# 用 curl 测试
curl http://localhost:8080

# 或者用浏览器访问
# http://你的服务器IP:8080

# 多次访问后检查计数器
curl http://localhost:8080
curl http://localhost:8080
```

每次刷新页面，计数器应该会增加。这验证了我们的服务正常工作，数据持久化也正常。

## systemctl edit：给现有服务加配置

假设你想给已有的 `linux-demo` 服务添加一个环境变量，但不想修改原始的 unit 文件。你可以用 `systemctl edit` 创建覆盖文件：

```bash
sudo systemctl edit linux-demo
```

在弹出的编辑器中写入：

```ini
[Service]
Environment="APP_MODE=production"
Environment="LOG_LEVEL=debug"
```

保存后，`systemctl edit` 会创建一个文件：`/etc/systemd/system/linux-demo.service.d/override.conf`。这个文件的指令会和原始 unit 文件合并，你的修改覆盖原始值。

检查合并后的效果：

```bash
systemctl cat linux-demo
```

最后别忘了：

```bash
sudo systemctl daemon-reload
sudo systemctl restart linux-demo
```

## 完整的工作流程总结

汇总一下从零到生产使用一个 systemd 服务的标准流程：

```
1. 写程序/脚本（bash, python, go, ...）
2. 决定运行用户（不要用 root）
3. 创建 unit 文件（/etc/systemd/system/xxx.service）
4. systemctl daemon-reload
5. systemctl start xxx
6. systemctl status xxx  （确认运行正常）
7. journalctl -u xxx     （查看日志）
8. systemctl enable xxx  （设置开机自启）
9. 测试验证
10. 文档记录（把这个服务的用途、端口、依赖告诉团队）
```

## 动手试试

**练习：创建你自己的 systemd 服务**

1. 用 `cd` 进入你克隆的教程目录，找到配套脚本：
   ```bash
   cd /path/to/linux-tutorial/04-用户与进程管理/examples/
   ```

2. 用 `cat` 查看示例脚本的开头部分，理解它做了什么（或者直接阅读上文的说明）。

3. 以 root 权限运行它：
   ```bash
   sudo bash 06-capstone.sh
   ```

4. 脚本会打印完整的操作日志。仔细阅读每一步的输出。

5. 验证你的服务真的在运行：
   ```bash
   systemctl status linux-demo
   ```

6. 测试 HTTP 服务（打开浏览器或使用 curl）：
   ```bash
   curl http://localhost:8080
   # 多执行几次，观察计数器变化
   curl http://localhost:8080
   curl http://localhost:8080
   ```

7. 查看服务的实时日志：
   ```bash
   journalctl -u linux-demo -f
   # 在另一个终端执行 curl，观察日志实时更新
   # 按 Ctrl+C 退出日志跟踪
   ```

8. 重启服务，验证计数器还在（因为数据持久化到了文件）：
   ```bash
   sudo systemctl restart linux-demo
   curl http://localhost:8080
   ```

9. 清理（如果你不想保留这个服务）：
   ```bash
   sudo systemctl stop linux-demo
   sudo systemctl disable linux-demo
   sudo rm /etc/systemd/system/linux-demo.service
   sudo rm /usr/local/bin/linux-demo.sh
   sudo rm -rf /var/lib/linux-demo
   sudo systemctl daemon-reload
   ```
