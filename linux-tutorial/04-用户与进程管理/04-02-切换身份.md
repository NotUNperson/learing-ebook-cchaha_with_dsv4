# 04-02 切换身份：su 与 sudo

## 本节你会学到什么

- 理解为什么不应该直接用 root 登录系统
- 掌握 `su` 和 `su -` 的区别，理解它们对环境变量的影响
- 掌握 `sudo` 的各种用法：单条命令、开启 Shell、以其他用户身份执行
- 了解 `visudo` 的安全机制，学会查看和编辑 `/etc/sudoers`
- 理解 wheel 组和 sudo 组的作用，以及如何给用户授权

## 为什么不能一直用 root？

在上一节我们说过，root 就像公司里拥有所有万能钥匙的超级管理员。那你可能会问：为什么我不直接用 root 登录，省得每次执行命令还要在前面加一个 `sudo`？

答案是：**因为人是会犯错的。**

想象你手握万能钥匙，走进服务器机房。你本来想去 3 楼维护财务系统，但不小心按了 13 楼的按钮 -- 而 13 楼是放着公司核心数据库的地方。你用 root 权限随手敲了一个 `rm -rf / home/user/temp/`（注意 `/` 后面多了一个空格），然后整个系统就没了。

这种事在现实中发生过太多次。2017 年，GitLab 的工程师在维护数据库时，不小心用 `rm -rf` 删掉了生产环境的数据目录，导致网站挂了很长时间。哪怕是最有经验的工程师，在凌晨三点被叫起来处理故障时，都会犯低级错误。

所以 Linux 管理员社区有一条不成文的规矩：**平常工作时用普通用户身份，只在需要的时候临时提权。** 这就像你平时穿着拖鞋在办公室晃，需要进机房时再穿防静电服。多一道"换衣服"的步骤，就能让你多一次确认的机会。

这个"临时提权"的机制，在 Linux 里就是 `su` 和 `sudo`。

## su：切换用户身份

`su` 的全称是 **switch user**（切换用户），它让你从当前用户切换到另一个用户。

### su 的三种用法

```bash
# 方式1：切换到 root，同时加载 root 的环境变量（推荐）
su -

# 方式2：切换到 root，但保留当前用户的环境变量
su

# 方式3：切换到指定用户
su - zhangsan
```

这三种用法有什么不同？关键在于那个 `-`（短横线）。

### 破折号的魔法：su 与 su - 的环境差异

`su -` 中的 `-` 约等于 `-l`（login），意思是"模拟一次完整的登录"。它会：
- 切换到目标用户的家目录（`cd ~目标用户`）
- 加载目标用户的 `.bashrc`、`.profile` 等环境配置
- 使用目标用户的 `PATH` 环境变量（这样你才能顺利调用目标用户安装的软件）

而 `su` 不带 `-` 时，只是把身份换了，环境还留在原来的位置。打个比方：

- **`su`**：你借了同事的工牌去他工位上拿文件，但你自己的水杯、外套、键盘还是留在自己的工位上。你心里还是觉得自己在"借用"。
- **`su -`**：你完全坐到了同事的工位上，打开他的电脑，用他的水杯喝水。你现在就是在"作为他"工作。

来看一个实际的例子：

```bash
# 当前用户是 zhangsan，在 /home/zhangsan
$ pwd
/home/zhangsan

# su 不带横线：切换到 root，但还在 zhangsan 的家目录
$ su
Password:
# pwd
/home/zhangsan          # 没变！
# echo $HOME
/home/zhangsan          # 环境变量也没变！

# 退出，回到 zhangsan
# exit

# su 带横线：完全切换
$ su -
Password:
# pwd
/root                   # 已经到 root 的家目录了
# echo $HOME
/root                   # 环境变量也变成了 root 的
```

**建议**：养成习惯，需要 `su` 时就用 `su -`。不加横线的 `su` 容易导致环境变量混乱，比如你用 `su` 切换到 root 后，运行一个命令报"command not found"，因为它找的是普通用户的 PATH，而不是 root 的 PATH。

### su 的缺点

`su` 有一个不方便的地方：你需要知道**目标用户的密码**。如果你要切换到 root，就需要输入 root 的密码。这意味着：
- root 密码需要在多个管理员之间共享（安全风险）
- 如果有人离职了，得改 root 密码，然后通知所有人新密码（管理麻烦）
- 你没法精细化控制"这个人只能做某几件事"

于是就有了更好的方案：`sudo`。

## sudo：代理审批流程

`sudo` 的全称是 **superuser do**（以超级用户身份执行）。它的设计理念非常巧妙：

你不直接"成为" root，而是拿着自己的"审批单"去找系统审批，说"我要以 root 身份执行这条命令，这是我的申请"。系统一看，你的名字在许可名单（`/etc/sudoers`）里，就批准了。

**类比：公司的采购审批流程**

想象你在公司想买一台新显示器（1000 块钱）。你自己没权直接划公司的账 -- 你得填一张采购申请单，找你主管签字审批。主管签完，拿着这张"批条"去财务，财务才给你拨款。

在这个类比里：
- **你** = 普通用户
- **采购申请单** = `sudo` 命令
- **主管签字** = `/etc/sudoers` 文件授权
- **财务拨款** = 内核授予的 root 权限
- **整个流程的审计记录** = `/var/log/auth.log`（谁在什么时间申请了什么操作）

### sudo 的核心用法

```bash
# 以 root 身份执行单条命令（最常用）
sudo apt update

# 执行上一条命令，但加上 sudo（超级实用！）
sudo !!

# 开启一个 root 登录 Shell（等于 su - , 但用自己的密码）
sudo -i

# 开启一个 root Shell（不等于登录 Shell，不加载 root 的环境变量）
sudo -s

# 以其他用户的身份执行命令
sudo -u www-data whoami
# 输出：www-data

# 查看自己有哪些 sudo 权限
sudo -l
```

`sodu !!` 是我个人最爱的技巧。比如你敲了 `apt install nginx`，结果提示权限不够，你不用重新打一遍命令，直接敲 `sudo !!`，Shell 会把上一条命令自动补上，变成 `sudo apt install nginx`。省时省力。

### sudo -i 和 su - 的区别

| | `su -` | `sudo -i` |
|---|---|---|
| 需要谁的密码 | root 的密码 | 当前用户的密码 |
| 切换后身份 | root | root |
| 加载的环境 | root 的完整环境 | root 的完整环境 |
| 审计日志 | 不太清楚谁用了 su | 清楚记录是哪个用户使用的 sudo |

`sudo` 的审计优势很大。如果五个人都有 root 密码，用 `su -` 切换后执行了危险操作，你根本不知道是谁干的。而 `sudo` 会记录每一条命令是哪个用户发起的，出了问题可以追溯到具体的人。就像公司的采购审批系统会留下完整的签字链，而不是一笔糊涂账。

## sudoers 权限配置文件：审批名单

`sudo` 的权限配置集中在 `/etc/sudoers` 文件中。这个文件决定了"谁可以在什么机器上以谁的身份执行什么命令"。

### 为什么必须用 visudo

编辑 `/etc/sudoers`，必须使用 `visudo` 命令，不能直接用 `vim` 或 `nano`。

原因有二：

1. **语法检查**：`sudoers` 文件的语法很严格。如果写错了，`sudo` 整个系统可能崩溃，届时只有物理访问机器用 root 登录才能修复。`visudo` 会在保存前检查语法，发现问题时提示你"语法错误，改好了再保存"，而不是让你把错误语法写进文件。
2. **文件锁定**：`visudo` 使用文件锁防止两个人同时编辑，避免互相覆盖对方的修改。

类比一下：直接用 vim 编辑 sudoers，就像用手写直接修改法律条文 -- 一个笔误可能让整部法律失效。而 `visudo` 就像一个法律审核系统，任何修改在生效前都要经过自动审核。

### sudoers 的格式

```bash
# 格式：谁 在哪台机器=(以谁的身份:以哪个组的身份) 能执行什么命令

# root 可以在任何地方做任何事
root    ALL=(ALL:ALL) ALL

# sudo 组的所有成员拥有完整权限（Debian/Ubuntu）
%sudo   ALL=(ALL:ALL) ALL

# wheel 组的所有成员拥有完整权限（CentOS/RHEL/Fedora）
%wheel  ALL=(ALL:ALL) ALL

# 只允许 restartuser 免密码重启 nginx
restartuser ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart nginx

# 只允许 backupuser 免密码执行备份脚本
backupuser ALL=(ALL) NOPASSWD: /usr/local/bin/backup.sh
```

注意：组名前有一个 `%`。没有 `%` 的是用户名，有 `%` 的是组名。

### 推荐的配置方式

不要直接修改 `/etc/sudoers` 主文件，而是在 `/etc/sudoers.d/` 目录下创建独立的配置文件。系统会自动加载这个目录下的所有文件。

```bash
# 为某个用户创建专属的 sudo 配置
sudo visudo -f /etc/sudoers.d/zhangsan
```

文件中写入：

```
# 给 zhangsan 完整的 sudo 权限（需要输入密码）
zhangsan ALL=(ALL:ALL) ALL
```

这样做的好处就像公司的"分部门审批制度" -- 不用把所有审批规则写在同一个文件里，每个部门的审批流程有独立的文件，修改时只影响自己部门，不会误改别人的。

### 查看 sudo 使用日志

```bash
# Ubuntu/Debian
sudo grep sudo /var/log/auth.log

# CentOS/RHEL
sudo grep sudo /var/log/secure
```

你可以看到每一条 `sudo` 命令的记录：谁执行的、什么时间、执行了什么命令。这就是那个"审计签名"。

## wheel 组 vs sudo 组

你可能会在教程里同时看到 `wheel` 和 `sudo` 两个组名，它们是同一个东西在不同发行版里的不同叫法：

| 发行版家族 | 管理员组名 | 将用户加入管理员组 |
|-----------|----------|-----------------|
| Debian/Ubuntu | `sudo` | `sudo usermod -aG sudo 用户名` |
| CentOS/RHEL/Fedora | `wheel` | `sudo usermod -aG wheel 用户名` |
| Arch Linux | `wheel` | `sudo usermod -aG wheel 用户名` |

"wheel"这个词源自计算机史上一个古老的俚语，意思是"大轮子" -- 拥有"wheel 权限"的人就是能"推动大轮子转动"的人，也就是系统的核心管理者。虽然名字有点奇怪，但 CentOS/RHEL 家族一直沿用至今。

## 常见身份切换场景

### 场景1：我自己有 sudo 权限，想临时当一会儿 root

```bash
sudo -i
# 输入你自己的密码
# 你现在是 root 了
# 做完事别忘了 exit 退出
exit
```

### 场景2：我只想用 root 身份执行一条命令

```bash
sudo systemctl restart nginx
```

### 场景3：我想切换到另一个普通用户（比如管理 web 服务用的 www-data）

```bash
sudo -u www-data bash
# 或者
sudo -u www-data -i
```

### 场景4：我需要以 root 身份执行一串命令

```bash
# 方法1：用 sudo bash -c 包裹
sudo bash -c 'cd /etc/nginx && cp nginx.conf nginx.conf.bak && ls -la'

# 方法2：先用 sudo -i 开启 root Shell，再逐条执行
sudo -i
cd /etc/nginx
cp nginx.conf nginx.conf.bak
ls -la
exit
```

**新手常犯的错误**：

```bash
sudo cd /root   # 错误！cd 是 Shell 内建命令，sudo 管不到
sudo ls /root   # 正确，ls 是外部程序，sudo 可以给它提权
```

`cd` 是 Shell 自己实现的命令（内建命令），不是外部可执行文件。`sudo` 只能提权运行外部程序，管不了 Shell 内建命令。如果需要进 `/root` 目录，用 `sudo -i` 或者 `sudo bash -c 'cd /root && ...'`。

## 动手试试

**练习：探索 su 与 sudo 的环境差异**

1. 打开两个终端窗口，都在你自己的用户下工作。

2. 在第一个终端里，用 `su`（不带横线）切换到 root：
   ```bash
   su
   # 输入 root 密码
   pwd
   echo $HOME
   echo $PATH
   exit
   ```

3. 记录下 `pwd`、`$HOME`、`$PATH` 的值。

4. 在第二个终端里，用 `su -`（带横线）切换到 root：
   ```bash
   su -
   # 输入 root 密码
   pwd
   echo $HOME
   echo $PATH
   exit
   ```

5. 对比两种方式下这三个值的差异。特别观察 `pwd` 的不同 -- 一个是 `/home/你的用户名`，一个是 `/root`。

6. 如果你使用的系统有 sudo（大多数桌面发行版都有），再试一下：
   ```bash
   sudo -l          # 查看你的 sudo 权限
   sudo whoami      # 验证 sudo 确实以 root 身份执行
   echo "hello" | sudo tee /root/test.txt   # 用 sudo 往 /root 写文件
   sudo cat /root/test.txt
   sudo rm /root/test.txt
   ```

## 本节小结

`su` 是"变身"为另一个用户（需要知道目标用户的密码），`sudo` 是"借用" root 权限执行特定命令（用自己的密码 + 管理员事先授权），而 `visudo` 和 `/etc/sudoers` 构成了这套权限审批体系的核心配置，`sudo -i` 是日常管理中最安全便捷的提权方式。
