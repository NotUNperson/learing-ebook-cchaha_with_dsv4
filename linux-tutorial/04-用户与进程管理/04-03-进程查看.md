# 04-03 进程查看

## 本节你会学到什么

- 理解进程（Process）是什么，以及 PID 和 PPID 的含义
- 读懂 `ps aux` 输出中每一列的含义，并能按 CPU、内存等维度排序查看
- 使用 `top` 和 `htop` 进行交互式的实时进程监控
- 使用 `pstree` 查看进程之间的父子关系
- 了解 `/proc` 虚拟文件系统及其提供的信息

---

## 什么是进程？一条河里的船

打开你电脑的任务管理器，你会看到一排排正在运行的程序。在 Linux 里，每个正在运行的程序实例都叫一个 **进程（Process）**。

想象一条繁忙的运河。运河上有很多船同时在航行，每条船有：
- 一个独一无二的编号（船牌号）
- 装载的货物（程序代码和数据）
- 所属的公司（用户）
- 当前的航行状态（在行驶、在等待过闸、还是抛锚停靠）

Linux 里的进程基本上就是这样：

| 船的属性 | 进程的对应物 |
|---------|------------|
| 船牌号 | **PID（Process ID，进程ID）** |
| 所属公司 | **USER（运行该进程的用户）** |
| 货物 | 占用的内存（VSZ、RSS） |
| 航行状态 | 进程状态（R 运行、S 睡眠、Z 僵尸等） |
| 谁派你出来的 | **PPID（Parent PID，父进程ID）** |

每一个进程（除了 PID=1 的那个老祖宗）都是由另一个进程创建出来的。创建者叫**父进程**，被创建的叫**子进程**。就像现实世界中，每个人（除了亚当夏娃）都有父母。`PPID` 记录的就是"父进程的 PID"。

Linux 系统启动后的第一个进程（通常是 `systemd` 或旧的 `init`）的 PID 永远是 1。它是所有进程的"先祖"。如果 PID 1 进程崩溃了，整个系统就会内核恐慌（Kernel Panic），因为进程家族树断了根。

---

## 三个视角看进程：就像是看运河上的航运情况

Linux 提供了多种工具来"查船"，各有侧重：

| 工具 | 类比 |
|------|------|
| `ps` | 拍一张运河的**快照**，看此刻所有船的状态 |
| `top`/`htop` | **监控摄像头**，实时直播运河上的交通情况 |
| `pstree` | **家谱图**，显示哪些公司（父进程）派出了哪些船（子进程） |
| `/proc` | 每艘船的**档案盒**，里面有船的详细技术参数 |

## ps aux：给进程拍一张全家福

`ps` 是 "process status" 的缩写。它是查看进程最基础的工具。

```bash
# BSD 风格（带横线的是 Unix 风格，不带的是 BSD 风格）
ps aux

# Unix 风格（等价表达）
ps -ef
```

执行 `ps aux` 后，你会看到类似这样的输出：

```
USER       PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
root         1  0.0  0.1 169500 13248 ?        Ss   12:00   0:05 /sbin/init
root       523  0.0  0.3 324560 27136 ?        Ssl  12:00   0:02 /usr/lib/systemd/systemd-journald
zhangsan  2145  0.5  1.2 856432 98240 ?        Sl   12:15   0:30 /usr/bin/python3 /home/zhangsan/app.py
zhangsan  3201  0.0  0.1  12784  8320 pts/0    Ss   13:00   0:00 bash
zhangsan  3298  0.0  0.0  11544  3584 pts/0    R+   13:05   0:00 ps aux
```

逐列解释：

### USER -- 这条船属于哪家公司
进程是哪个用户运行的。注意，一个程序可能以不同用户的身份运行。比如 Web 服务器 Nginx 的主进程通常以 root 运行（需要绑定 80 端口），但工作进程会以 `www-data` 或 `nginx` 用户运行（出于安全考虑）。

### PID -- 船的编号
这个是进程的唯一标识符。后续你要停止某个进程（`kill`），或者查看它的详细信息（`/proc/PID/`），都要用 PID。注意 PID 是系统分配的，不是你指定的。

### %CPU -- CPU 使用率
这条"船"占用了多少引擎动力。如果某个进程的 %CPU 持续接近 100%，说明它可能进入了死循环或者正在进行高密度计算。

### %MEM -- 内存使用率
这条船占了多大一片水域。如果 %MEM 持续增长而不回落，可能是内存泄漏。

### VSZ（虚拟内存）和 RSS（物理内存）
这两个字段常让人困惑。用一个"仓库"的类比来理解：

- **VSZ（Virtual Size）**：公司租给这条船的仓库总面积。包括：实际放了货物的货架区 + 预留的过道 + 借来的临时堆场（交换空间）。总之就是"申请了多少地"。
- **RSS（Resident Set Size）**：目前真正堆了货物的面积。也就是"实际用了多少地"。

所以 VSZ 通常远大于 RSS。你真正需要关心的是 RSS，它反映了进程实际占用的物理内存。

### TTY -- 终端
这个进程关联了哪个终端（屏幕/键盘）。`pts/0` 表示一个 SSH 连接或终端窗口，`tty1` 表示第一个物理控制台（`Ctrl+Alt+F1`），`?` 表示这个进程没有关联任何终端，通常是后台服务（守护进程）。

类比：有些船有船长在和港口通话（有 TTY），有些无人货船自己在河里漂（TTY 为 `?`，就是守护进程）。

### STAT -- 进程状态
这是最重要的列之一。常见的状态码：

| 代码 | 含义 | 类比 |
|------|------|------|
| R | Running，正在跑活或在排队等着跑 | 在河上航行 |
| S | Sleeping，可中断睡眠（等待键盘输入、网络数据等） | 在码头停靠，等调度通知 |
| D | Disk Sleep，不可中断睡眠（等待磁盘 I/O） | 正在装货/卸货，不能打断 |
| Z | Zombie，僵尸进程（已结束但父进程未回收） | 船沉了但没注销编号 |
| T | Stopped，被暂停的进程 | 抛锚停船，不进不退 |

附加标志：
- `s`：这个进程是**会话领导者**（session leader），相当于"船队的旗舰"
- `+`：属于**前台进程组**（你在终端里能看到它运行）
- `<`：高优先级（nice 值为负）
- `N`：低优先级（nice 值为正）

### START -- 启动时间
进程什么时候开始运行的。如果显示的不是具体时间而是一个日期（如 `Apr15`），说明这个进程已经运行了好几天甚至好几周了。

### TIME -- 累计 CPU 时间
进程从启动到现在，加起来用了多少 CPU 时间。注意这不是自然时间（钟表时间），而是"真正的 CPU 干活时间"。一个运行了三天但大部分时间在睡觉的进程，TIME 可能只有几秒钟。

### COMMAND -- 命令
启动进程时用的完整命令行，包括参数。

## ps 的实用组合技

```bash
# 按内存使用排序，只看前 10 名
ps aux --sort=-%mem | head -11

# 按 CPU 使用排序，只看前 10 名
ps aux --sort=-%cpu | head -11

# 只看某个用户的进程
ps -u zhangsan

# 自定义输出列
ps -eo pid,ppid,user,cmd,%cpu,%mem --sort=-%cpu | head

# 显示线程（-L 参数）
ps -eLf | grep nginx

# 查看进程树（-H 显示层级关系）
ps -ejH
```

还有一个非常实用的命令：**`pgrep`**。它可以按进程名查找 PID，省去 `ps aux | grep xxx` 的麻烦：

```bash
# 查找所有名为 ssh 的进程
pgrep -a ssh

# 查找某个用户的所有 bash 进程
pgrep -u zhangsan bash
```

## top：实时监控摄像头

`ps` 是拍快照，但你想看动态画面怎么办？用 `top`。

```bash
top
```

运行 `top` 后，终端会进入一个不断刷新的界面。最上面几行是系统概览：

```
top - 14:23:15 up 3 days,  2:15,  2 users,  load average: 0.05, 0.08, 0.03
Tasks: 198 total,   1 running, 197 sleeping,   0 stopped,   0 zombie
%Cpu(s):  1.2 us,  0.5 sy,  0.0 ni, 98.2 id,  0.0 wa,  0.0 hi,  0.1 si
MiB Mem :   7856 total,   2345 free,   3210 used,   2301 buff/cache
MiB Swap:   2048 total,   1900 free,    148 used
```

逐行解读：

**第 1 行**：系统运行时间、登录用户数、**负载均值（load average）**。
- 负载均值有三个数字，分别代表过去 1 分钟、5 分钟、15 分钟的平均负载
- 负载可以简单理解为"等待 CPU 处理的进程数"。如果数字超过 CPU 核数，说明系统有点忙不过来了
- 类比：超市收银台前排队的人数。1 分钟前有 0.05 个人在排队（基本没人），5 分钟前有 0.08 个，15 分钟前有 0.03 个。都很低，说明 CPU 很闲。

**第 3 行**：CPU 使用率分解
- `us`：用户态（user space），你的程序在干活
- `sy`：内核态（system），内核在干活
- `ni`：被调整了优先级的用户态
- `id`：空闲（idle），CPU 在摸鱼
- `wa`：等待 I/O（wait），CPU 想干活但数据还没从磁盘读上来

**第 4 行**：物理内存使用情况

**第 5 行**：交换空间使用情况

`top` 运行时，你可以用键盘控制它：

| 按键 | 功能 |
|------|------|
| `q` | 退出 |
| `1` | 显示每个 CPU 核的使用率（在按一次就切换回来） |
| `M` | 按内存使用排序 |
| `P` | 按 CPU 使用排序 |
| `k` | 杀掉一个进程（会提示输入 PID 和信号） |
| `u` | 只看某个用户的进程 |
| `c` | 切换显示完整命令行 |
| `h` | 帮助 |

## htop：top 的豪华升级版

`htop` 是 `top` 的增强版，界面是彩色的，支持鼠标点击，还能横向纵向滚动查看完整的命令行。如果你用的是桌面 Linux，几乎一定应该安装 `htop`。

```bash
# 安装 htop
sudo apt install htop   # Debian/Ubuntu
sudo yum install htop   # CentOS/RHEL 7
sudo dnf install htop   # CentOS/RHEL 8+ / Fedora

# 运行
htop
```

`htop` 的额外好处：
- 彩色界面，一眼看出 CPU/内存/Swap 的用量
- 支持鼠标点击（比如直接点击列头排序）
- 可以直接用 F 功能键操作（F3 搜索，F9 杀进程，F10 退出）
- 不需要记住按键，底部有功能键提示栏
- 可以显示进程树视图（按 F5）

---

## pstree：画出进程的家族树

有时候你的程序卡住了，你想知道是谁启动的它 -- 它在进程家族树里处于什么位置。`pstree` 帮你画出来。

```bash
# 安装（通常默认安装）
sudo apt install pstree   # Debian/Ubuntu
sudo yum install psmisc   # CentOS/RHEL（pstree 在 psmisc 包里）

# 显示进程树
pstree

# 显示 PID
pstree -p

# 显示某个用户的进程树
pstree zhangsan

# 高亮当前进程
pstree -h
```

输出类似：

```
systemd─┬─NetworkManager───2*[{NetworkManager}]
        ├─cron
        ├─nginx───2*[nginx]
        ├─sshd───sshd───sshd───bash───pstree
        └─systemd-journal
```

从这个树形输出你能看到：
- `systemd`（PID=1）是一切进程的根
- `sshd` 启动了一个子进程 `sshd`，它又启动了一个 `sshd`（这是 SSH 登录的会话），最后启动了 `bash`，我们在 `bash` 里运行了 `pstree`
- `nginx` 主进程启动了 2 个工作进程（`2*[nginx]`）

## /proc：每个进程的档案盒

`/proc` 是一个特殊的目录。它不存在于硬盘上，而是内核在内存中虚拟出来的。你可以把它理解成"系统档案室" -- 每个进程在这里有一个以 PID 命名的文件夹，里面放满了该进程的各种技术档案。

```bash
# 查看某个进程的详细信息
ls /proc/1234/

# 启动命令
cat /proc/1234/cmdline

# 进程状态（内存、信号等）
cat /proc/1234/status

# 环境变量
cat /proc/1234/environ | tr '\0' '\n'

# 当前工作目录的软链接
ls -la /proc/1234/cwd

# 打开的文件描述符
ls -la /proc/1234/fd/

# 可执行文件的软链接
ls -la /proc/1234/exe
```

类比：就像每艘船都有一个档案盒，里面装着船的建造图纸（exe）、航行日志（cmdline）、当前运单（environ）、船上所有正在使用的工具的清单（fd/）。

`/proc` 里还不止有进程信息：

```bash
# CPU 信息
cat /proc/cpuinfo

# 内存信息
cat /proc/meminfo

# 系统运行时间和空闲时间
cat /proc/uptime

# 内核版本
cat /proc/version

# 当前已加载的内核模块
cat /proc/modules
```

## 动手试试

**练习：探索自己系统的进程**

1. 打开终端，运行 `ps aux | head -20`，找到你自己的 `bash` 进程。观察它的 PID、%MEM、STAT 和 START 值。

2. 用 `pgrep -a bash` 找到所有 bash 进程的 PID。

3. 有了自己的 bash 进程的 PID 后（假设是 5678），去 `/proc/5678/` 看看：
   ```bash
   cat /proc/5678/cmdline | tr '\0' ' '
   cat /proc/5678/status | grep -E "^(Name|Pid|PPid|VmRSS|State)"
   ls -la /proc/5678/fd/
   ```

4. 运行 `top`，试着按下列顺序操作：
   - 按 `1` 查看每个 CPU 核的使用率
   - 按 `M` 按内存排序
   - 按 `P` 恢复按 CPU 排序
   - 按 `u` 然后输入你的用户名，只看自己的进程
   - 按 `q` 退出

5. 运行 `pstree | less`，在进程树中找到你的 bash 进程，追溯它向上到 systemd 的整条路径。（提示：可以先用 `echo $$` 知道自己的 PID，然后在 `pstree -p` 中搜索）

## 本节小结

进程是 Linux 中正在运行的程序的实例，每个进程有唯一的 PID 和父进程 PPID；`ps aux` 提供进程快照，`top`/`htop` 提供实时监控，`pstree` 展示进程的家族关系，而 `/proc` 虚拟文件系统则是深入了解每个进程和系统状态的宝库。
