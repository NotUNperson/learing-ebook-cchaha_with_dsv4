## 我们要做什么

这个备份脚本的功能：

1. 接收命令行参数：源目录（`-s`）、目标备份目录（`-d`）、保留天数（`-k`）
2. 检查源目录是否存在、是否可读
3. 自动创建目标备份目录（如果不存在）
4. 用 `tar` 把源目录打包成 `.tar.gz`，文件名包含目录名和日期时间戳
5. 自动清理超过保留天数的旧备份
6. 所有操作记录日志，出错了有明确的错误信息
7. 用 `trap` 确保异常退出时也能写日志

---

## 类比：建筑工地上的施工流程

写一个完整脚本就像是盖一栋小房子。你不是把所有砖头随便堆起来，而是：

1. **打地基**：设置全局变量、日志目录、安全选项（`set -euo pipefail`）
2. **搭框架**：`main()` 函数负责调度，各子函数负责具体任务
3. **装门窗**：`parse_args()` 负责输入，`generate_report()` 负责输出
4. **通水电**：`log()` 函数负责信息流通，`trap` 负责安全兜底

一个结构良好的脚本，读起来应该像一篇有层次的散文，而不是一团乱麻。

## 主流程设计

先把"骨架"画出来，不着急填代码：

```bash
main() {
    parse_args "$@"        # 1. 解析命令行参数
    check_source           # 2. 检查源目录
    prepare_dest           # 3. 准备目标目录
    do_backup              # 4. 执行备份打包
    cleanup_old_backups    # 5. 清理过期备份
    generate_report        # 6. 生成备份报告
}
```

这个设计遵循了"一个函数只做一件事"的原则。如果你想加新功能（比如备份到远程服务器），只要加一个新函数，不影响其他部分。这就是模块化的好处。

## 参数解析：getopts

`getopts` 是 bash 内置的命令行参数解析器。它比你手动解析 `$1` `$2` 更专业、更标准化。

```bash
while getopts "s:d:k:h" opt; do
    case "$opt" in
        s) SOURCE_DIR="$OPTARG" ;;    # -s 后面跟参数
        d) DEST_DIR="$OPTARG" ;;      # -d 后面跟参数
        k) KEEP_DAYS="$OPTARG" ;;     # -k 后面跟参数
        h) show_help; exit 0 ;;       # -h 无参数，显示帮助
        ?) show_help; exit 2 ;;       # 非法选项
    esac
done
```

`getopts` 的格式字符串 `"s:d:k:h"` 中，字母后面有冒号表示该选项需要一个值。比如 `s:` 表示 `-s /var/www`，`h` 后面没有冒号表示 `-h` 是一个独立开关。

`getopts` 自动处理：
- 合并短选项（`-abc` = `-a -b -c`）
- 参数的缺失检测
- 非选项参数的处理

**类比**：`getopts` 就像是酒店的前台接待 -- 客人来就说"我需要办好入住（-c），这是我的身份证（-i）"，前台按流程检查文件、分配房间。如果没有 `getopts`，你只能自己挨个问"你是要入住吗？身份证呢？"很容易漏掉什么。

## 检查源目录：防御性编程

在操作任何文件或目录之前，先"探路"：

```bash
check_source() {
    if [ ! -e "$SOURCE_DIR" ]; then
        log "ERROR" "源目录不存在：$SOURCE_DIR"
        exit 1
    fi

    if [ ! -d "$SOURCE_DIR" ]; then
        log "ERROR" "源路径不是目录：$SOURCE_DIR"
        exit 1
    fi

    if [ ! -r "$SOURCE_DIR" ]; then
        log "ERROR" "源目录不可读：$SOURCE_DIR"
        exit 1
    fi
}
```

这三个检查对应三个问题：存在吗？是目录吗？能读吗？每个问题都给出了明确的错误提示，让用户知道怎么修正。

为什么要这么"啰嗦"？因为你不写这些检查，`tar` 命令失败时只会给你一个模糊的错误信息（"tar: Cannot stat: No such file or directory"），用户可能以为是自己的问题。好的错误信息能节省大量排查时间。

## 备份执行：tar 打包

```bash
do_backup() {
    local dir_basename=$(basename "$SOURCE_DIR")
    local backup_file="${DEST_DIR}/backup_${dir_basename}_${DATE_STAMP}.tar.gz"

    if tar -czf "$backup_file" -C "$(dirname "$SOURCE_DIR")" "$(basename "$SOURCE_DIR")"; then
        log "INFO" "备份完成：$backup_file"
    else
        log "ERROR" "备份失败（tar 命令出错）"
        exit 1
    fi
}
```

几个关键细节：

- `-C` 选项：让 tar 切换到源目录的**父目录**去执行，这样打包进去的路径是相对路径（如 `www/`），而不是绝对路径（如 `/var/www/`）。还原时更灵活。
- 文件名包含：前缀 + 目录名 + 时间戳。比如 `backup_www_20260515_143000.tar.gz`。一眼就能看出备份的是什么、什么时候备份的。
- 用 `if` 检查 `tar` 的退出码，失败了立即报错。

## 清理旧备份：自动回收空间

```bash
cleanup_old_backups() {
    local dir_basename=$(basename "$SOURCE_DIR")

    # 用 find 找出超过 KEEP_DAYS 天的备份文件
    find "$DEST_DIR" \
        -name "backup_${dir_basename}_*.tar.gz" \
        -mtime +"${KEEP_DAYS}" \
        -type f \
        -delete
}
```

`-mtime +7` 的意思是"修改时间超过 7 天"（精确说是 7 个 24 小时周期之前）。

我们用循环 + `rm` 而不是 `find` 的 `-delete`，是因为我们想记录每一条删除日志。但 `find -delete` 也能用，取决于你的需求。

**类比**：这就像是冰箱管理 -- 每周清理一次过期食物。你设定一个规则：超过 7 天的剩菜统统扔掉。这样冰箱（备份目录）就不会被塞爆。

## 日志系统：脚本的"黑匣子"

```bash
log() {
    local level="$1"
    local message="$2"
    mkdir -p "$LOG_DIR" 2>/dev/null || true
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] [$level] $message" | tee -a "$LOG_FILE"
}
```

`tee -a` 的作用是把消息同时输出到终端和日志文件。运行脚本时你能看到输出，事后也能查看日志。

## 完整脚本结构回顾

```
1. set -euo pipefail         # 安全选项
2. 全局变量 + 默认值          # 配置
3. log()                     # 日志工具
4. cleanup() + trap          # 清理守卫
5. show_help()               # 帮助信息
6. parse_args()              # 参数解析
7. check_source()            # 源目录检查
8. prepare_dest()            # 目标目录准备
9. do_backup()               # 核心备份
10. cleanup_old_backups()    # 旧备份清理
11. generate_report()        # 报告生成
12. main()                   # 主调度函数
13. main "$@"                # 脚本入口
```

这个结构是一个可复用的模板。你以后写其他脚本（日志清理、部署工具、监控检查）都能套用这个骨架：安全开头 -> 工具函数 -> 业务函数 -> main 调度 -> 入口调用。
