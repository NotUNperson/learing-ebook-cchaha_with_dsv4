# 06-07 调试与错误处理

## 本节你会学到什么

- 使用 `set -e` 让脚本在遇到错误时立即停止
- 使用 `set -u` 在引用未定义变量时报错
- 使用 `set -x` 追踪脚本执行的每一条命令
- 使用 `set -o pipefail` 检测管道中的失败
- 使用 `trap` 捕获信号并执行清理工作

---

写脚本的过程中，有两件事几乎必然发生：第一，脚本会出错。第二，你一开始不知道它错在哪里。

就像你写文章会有错别字一样，脚本也会有 bug。变量拼写错了、忘了某个文件可能不存在、管道中间的命令悄悄失败了... 这些 bug 轻则输出奇怪的结果，重则删错了文件、搞砸了服务器。所以学会调试和错误处理，和学会写正确的代码同样重要。

---

## 类比：开车时的安全系统

一辆车有多层安全保护：
- **刹车（set -e）**：遇到危险立刻停下来
- **仪表盘警报灯（set -u）**：有问题马上亮灯告诉你
- **行车记录仪（set -x）**：记录每一步操作，方便事后回放
- **安全气囊（trap）**：出了事故也有最后一层保护

Shell 脚本也需要这些"安全装置"。幸运的是，bash 内置了所有这些工具。

---

## set -e：一错就停

默认情况下，Shell 脚本是一个"心大"的司机。前面有命令执行失败了，它咕哝一声就继续开，好像什么都没发生。这非常危险！

```bash
#!/bin/bash
# 危险：即使 cd 失败，也会继续执行
cd /some/nonexistent/dir
rm -rf *        # 这可太可怕了！
```

如果 `cd` 失败了，你还在原来的目录，然后 `rm -rf *` 就删掉了不该删的东西。正确做法是：

```bash
#!/bin/bash
set -e            # 任何命令失败，立即停下来

cd /some/nonexistent/dir   # 如果失败，脚本在这里停止，不会执行下一行
rm -rf *
```

`set -e` 的规则是：**任何命令返回非零退出码，脚本立即退出**。

但 `set -e` 有一些"盲区"，它不会在意以下位置的失败：
- `if` 或 `while` 的条件判断里的命令
- 管道中最后一个命令之前的命令（除非开了 `pipefail`）
- `&&` 或 `||` 左边的命令

举例来说：

```bash
set -e

# 这个 if 条件里的 false 不会触发 set -e（合理，因为 if 本就要检查返回值）
if false; then
    echo "不执行"
fi
echo "脚本继续"
```

## set -u：变量必须先定义

`set -u` 是一个"强迫症"选项 -- 它要求你使用变量之前必须先定义。这会帮你发现很多拼写错误：

```bash
#!/bin/bash
set -u

config_file="/etc/myapp/config.ini"
# 不小心拼错了
echo "配置文件：$config_file"       # 报错：config_file 未定义
echo "配置文件：$config_file"       # 这才对
```

如果没有 `set -u`，拼写错误的变量会默默地变成空字符串，你很难发现。

**类比**：`set -u` 就像是一个严格的语文老师 -- 你不能用没学过的词语。必须先定义（学习）这个变量，才能使用它。

---

## set -x：每一步都记录在案

`set -x` 是调试时最有用的选项。它让 bash 在执行每条命令之前先把它打印出来，前面加上 `+` 号：

```bash
#!/bin/bash
set -x

name="小明"
echo "你好，$name"
```

输出：

```
+ name=小明
+ echo '你好，小明'
你好，小明
```

有了 `set -x`，你可以看到：
- 命令被展开后的样子（变量被替换成实际值）
- 实际执行了什么（而不是你想象中应该执行什么）

日常使用中，通常只在调试时开启，调完再关掉：

```bash
set -x     # 开始追踪

# 需要调试的代码...

set +x     # 停止追踪（注意是 + 号，不是 - 号）
```

你也可以在运行脚本时就开启：

```bash
# 方法一：bash -x
bash -x ./my_script.sh

# 方法二：只追踪脚本的一部分
bash -v ./my_script.sh    # -v 打印原始行（不做变量展开）
```

`-x` 和 `-v` 的区别：`-x` 显示展开后的命令，`-v` 显示原始的命令行。通常 `-x` 更有用，因为你能看到变量被替换成了什么。

## set -o pipefail：管道的"木桶效应"

木桶能装多少水，取决于最短的那块板。但默认情况下，Shell 管道的退出码只由最后一个命令决定 -- 前面的命令就算失败了，管道整体也算"成功"。

```bash
# 默认行为
grep "something" /path/to/nonexistentfile | wc -l
echo "退出码：$?"       # 输出 0（因为 wc -l 成功了！）
```

`grep` 因为文件不存在而失败，但 `wc -l` 收到空输入后还是成功了，管道整体返回成功！这是很多隐蔽 bug 的源头。

`set -o pipefail` 修复了这个问题：

```bash
set -o pipefail
grep "something" /nonexistent | wc -l
echo "退出码：$?"       # 输出 2（grep 的失败码）
```

现在管道中任何一个命令失败，整个管道的退出码就是非零。

## 推荐的"安全四件套"

生产环境脚本的标准头部：

```bash
#!/bin/bash
set -euo pipefail
# -e: 遇错即停
# -u: 未定义变量报错
# -o pipefail: 管道中间失败也报错
```

有时候你可能想再加入 `-x` 用于调试：

```bash
# 仅在设置了 DEBUG 环境变量时开启追踪
if [ "${DEBUG:-}" = "true" ]; then
    set -x
fi
```

这样你就能在需要时用 `DEBUG=true ./my_script.sh` 打开调试模式。

## trap：即使崩溃，也要优雅

脚本可能因为各种原因"非正常退出"：你按了 `Ctrl+C`、系统发送了 kill 信号、或者某个命令触发了 `set -e`。不管怎样，你应该确保临时文件被清理、锁被释放、数据库连接被关闭。

`trap` 就是干这个的。它让你在脚本退出（或者收到某个信号）时执行一段清理代码。

```bash
#!/bin/bash

# 创建临时文件
tempfile="/tmp/myscript_$$.tmp"
echo "work data" > "$tempfile"

# 注册清理函数
cleanup() {
    echo "正在清理..."
    rm -f "$tempfile"
    echo "清理完毕"
}

# trap：退出时（EXIT）、Ctrl+C（INT）、kill（TERM）都执行清理
trap cleanup EXIT INT TERM

# 主逻辑...
echo "脚本运行中，按 Ctrl+C 试试"
sleep 10
echo "正常结束"
```

无论脚本怎么结束，`cleanup` 函数都会被执行。这就是"优雅退出"。

常见的信号：

| 信号 | 编号 | 触发方式 | 用途 |
|------|------|----------|------|
| EXIT | - | 脚本退出时 | 清理临时文件 |
| INT | 2 | Ctrl+C | 中断当前操作 |
| TERM | 15 | `kill` 命令 | 优雅终止 |
| HUP | 1 | 终端关闭 | 重新读取配置 |
| ERR | - | 任何命令失败 | 错误日志记录 |

`ERR` 信号配合 `set -e` 特别有用：

```bash
#!/bin/bash
set -e

# 任何命令失败时记录日志
trap 'echo "[ERROR] 脚本在第 $LINENO 行失败" >&2' ERR

# 如果这里出错，上面那行会告诉你出错的是第几行
cd /nonexistent
```

`$LINENO` 是 bash 内置的变量，代表当前行号。

## 结构化日志：让输出有价值

当脚本在后台运行（比如 cron 定时任务）时，你不会盯着屏幕看输出。这时候需要把重要的信息记到日志文件里。

```bash
#!/bin/bash

LOG_FILE="/var/log/myapp/backup.log"

log() {
    local level=$1
    local message=$2
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] [$level] $message" | tee -a "$LOG_FILE"
}

log "INFO" "备份脚本开始"
log "INFO" "备份源：/var/www"

# ... 实际备份操作 ...

if [ $? -eq 0 ]; then
    log "INFO" "备份成功"
else
    log "ERROR" "备份失败！"
    exit 1
fi
```

`tee -a` 的作用是把输出同时写到终端和文件 -- 就像 T 型水管，水流分到两个出口。

## 退出码约定

退出码不只是"成功"或"失败"。约定俗成的退出码范围：

| 退出码 | 含义 |
|--------|------|
| 0 | 成功 |
| 1 | 一般错误 |
| 2 | 参数使用错误 |
| 126 | 文件不可执行 |
| 127 | 命令未找到 |
| 128+N | 收到了信号 N 而终止 |

你的脚本应该明确返回退出码，而不是"靠运气"。调用你脚本的人（或者其他脚本）会依赖正确的退出码来做判断：

```bash
#!/bin/bash

if [ $# -lt 1 ]; then
    echo "用法：$0 <文件名>"
    exit 2          # 参数错误
fi

if [ ! -f "$1" ]; then
    echo "文件不存在：$1"
    exit 1          # 一般错误
fi

# 正常处理...
exit 0              # 成功
```

## 动手试试

1. 写一个有"危险"的脚本：故意写错变量名，先用普通模式运行，再开启 `set -u` 运行，对比行为
2. 用 `bash -x` 运行任意一个你之前写的脚本，观察展开后的命令与原始命令的差异
3. 写一个创建临时文件的脚本，注册一个 `trap cleanup EXIT INT TERM`，试着在运行时按 `Ctrl+C`，看临时文件是否被清理
4. 写一个有管道的脚本，在其中故意让第一个命令失败。对比加和不加 `set -o pipefail` 时的退出码
5. 让你的脚本支持 `DEBUG=true ./script.sh` 开启调试输出

---

## 本节小结

错误处理和调试是写靠谱脚本的基础。"安全四件套" `set -euo pipefail` 应该在几乎每个脚本里使用。`set -x` 帮你追踪命令执行过程，`trap` 帮你优雅清理。好的日志和正确的退出码让脚本在无人值守时也能可靠运行。
