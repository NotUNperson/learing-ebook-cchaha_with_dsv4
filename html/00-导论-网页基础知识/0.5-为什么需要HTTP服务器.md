# 0.5 为什么需要 HTTP 服务器

## 本节你会学到什么

* `file://` 协议和 `http://` 协议的本质区别
* 为什么双击 HTML 文件打开的网页会出各种问题
* 哪些具体的功能在 `file://` 下不可用
* 一个简单的实验：对比两种打开方式的行为差异

## 一、两种打开方式，两个世界

假设你在桌面上有一个 `index.html` 文件。打开它有两种方式：

**方式 A**：直接双击文件（或者在文件资源管理器里双击）
* 浏览器地址栏显示：`file:///C:/Users/xxx/Desktop/index.html`

**方式 B**：通过 Live Server（或者任何 HTTP 服务器）打开
* 浏览器地址栏显示：`http://127.0.0.1:5500/index.html`

看起来只是地址栏开头的几个字母不一样，对吧？但实际上，**这两个打开方式背后是完全不同的两种协议**，而这种差异在很多场景下会导致"方式 A 出 bug，方式 B 正常"。

## 二、一个生活类比：自言自语 vs 在餐厅点餐

想象以下两个场景：

**场景 A：你在自己房间里自言自语（file://）**

你一个人在房间里，对自己说"给我拿一杯水"。这当然能实现——你自己去厨房倒水就行。你不需要和任何人交互，不需要遵守什么规则，你拥有控制一切的自由。

但是，如果你说"帮我查一下明天的天气"——这就做不到了。因为你自己不知道明天的天气，你需要向"外部世界"请求信息。

**场景 B：你在餐厅正常点餐（http://）**

你走进一家餐厅，对服务员说"一份宫保鸡丁"。服务员记下你的请求，传给厨房，厨房做好后由服务员端给你。整个过程中，你和服务员之间有明确的"协议"：你负责下单，服务员负责传菜，厨房负责做菜。

一些事情在场景 A 里做不了，在场景 B 里就能做：
* 你不能"自己给自己发一个网络请求"（没有服务器）
* 你不能在自己的房间里"模拟一个数据库查询"（没有数据库服务）

`file://` 就像场景 A——你直接读写本地文件，浏览器不需要网络通信。`http://` 就像场景 B——浏览器通过 HTTP 协议和服务器对话。

## 三、技术上的本质区别

### `file://` 协议

当浏览器加载 `file:///C:/xxx/index.html` 时：
* 浏览器**直接从硬盘读取文件**，不经过网络。
* 没有"服务器"这个概念——浏览器的角色更像是一个"文件查看器"。
* 没有域（Origin）。浏览器的安全策略里，`file://` 的 Origin 是 `null`。

### `http://` 协议

当浏览器加载 `http://127.0.0.1:5500/index.html` 时：
* 浏览器**向一个服务器发起 HTTP 请求**。
* 服务器监听 5500 端口，收到请求后返回文件内容。
* 有域（Origin）：`http://127.0.0.1:5500`。浏览器的安全策略会基于这个域做判断。

> **类比 C 语言**：`file://` 像是你用 `fopen("index.html", "r")` 直接读本地文件。`http://` 像是你用了某种 socket 编程，连接到远程主机，通过 send/recv 来收发数据。即使 `127.0.0.1` 指的是你自己的电脑，数据也是走网络协议栈的。

## 四、哪些功能在 `file://` 下会出问题

以下功能在 `file://` 协议下**不可用或不正常**：

### 1. Fetch API / XMLHttpRequest（发网络请求）

这是最常见的问题。在 `file://` 下，JavaScript 无法向其他地址发送 HTTP 请求。如果你尝试写：

```javascript
fetch('https://api.example.com/data.json')
    .then(response => response.json())
    .then(data => console.log(data));
```

在 `file://` 下，浏览器会报错（CORS 错误或 network error），因为浏览器不知道"从文件系统发出请求"这个操作的安全上下文的域（Origin）是什么。在 `http://` 下，Origin 是 `http://127.0.0.1:5500`，浏览器可以基于这个执行同源策略。

### 2. ES Module（ES 模块）

现代 JavaScript 支持用 `import` / `export` 语法拆分模块：

```html
<script type="module" src="main.js"></script>
```

在 `file://` 下，模块加载会因为 CORS 策略而被阻止。浏览器在加载模块时会检查 CORS 头，而 `file://` 协议下没有服务器来提供这些头。你会看到类似这样的错误：

```
Access to script at 'file:///C:/xxx/main.js' from origin 'null'
has been blocked by CORS policy
```

### 3. Cookie 和其他存储

Cookie 需要绑定到一个域（Domain），而 `file://` 没有域。不同浏览器对 `file://` 下的 Cookie 处理不一致，有的完全不支持，有的支持但行为诡异。类似地，LocalStorage 和 SessionStorage 的行为也可能因协议而异。

### 4. Service Worker

Service Worker（用于离线缓存、推送通知等功能）**必须在 HTTPS 或 localhost 下注册**。`file://` 协议下无法使用。

### 5. 相对路径和绝对路径的行为差异

在 `file://` 下，以 `/` 开头的绝对路径会指向**文件系统根目录**（比如 `C:/`），而不是服务器的根目录。这在项目中有子目录时会造成路径混淆。

### 6. 浏览器的同源策略沙箱

因为 `file://` 没有域（Origin 为 `null`），每个 `file://` 文件的 Origin 都不同。这意味着：如果你用 `file://` 打开了同一目录下的两个 HTML 文件，并试图让它们通过 iframe 或 window.open 通信——它们会被浏览器的同源策略隔离，无法访问彼此的内容。

## 五、一个对比实验

请你做以下实验，直观感受区别：

**第一步**：创建两个文件在同一个文件夹中。

文件 `page-a.html`：
```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
    <meta charset="UTF-8">
    <title>页面 A</title>
</head>
<body>
    <h1>这是页面 A</h1>
    <button id="fetchBtn">发一个 fetch 请求</button>
    <p id="result"></p>

    <script>
        document.getElementById('fetchBtn').addEventListener('click', function() {
            fetch('https://jsonplaceholder.typicode.com/todos/1')
                .then(function(response) { return response.json(); })
                .then(function(data) {
                    document.getElementById('result').textContent =
                        'fetch 成功: ' + JSON.stringify(data);
                })
                .catch(function(error) {
                    document.getElementById('result').textContent =
                        'fetch 失败: ' + error.message;
                });
        });
    </script>
</body>
</html>
```

文件 `page-b.html`（用了 ES Module）：
```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
    <meta charset="UTF-8">
    <title>页面 B</title>
</head>
<body>
    <h1>这是页面 B（ES Module 测试）</h1>
    <p id="msg">加载中...</p>

    <script type="module">
        // 这段代码在 file:// 下会报错
        // 在 http:// 下正常运行
        const msg = 'ES Module 加载成功! (当前协议: ' + location.protocol + ')';
        document.getElementById('msg').textContent = msg;
        console.log(msg);
    </script>
</body>
</html>
```

**第二步**：分别用两种方式打开这两个页面：
1. 先**直接双击** `page-a.html` 打开（地址栏是 `file:///...`），点击"发一个 fetch 请求"按钮，观察结果。
2. 然后用 VS Code 的 **Live Server** 打开同一个文件（地址栏是 `http://...`），点击按钮，观察结果。

你会发现：
* `file://` 下 fetch 请求**失败**，ES Module **可能**失败
* `http://` 下 fetch 请求**成功**，ES Module **正常工作**

同样的 HTML 文件，同样的代码，只是因为打开方式不同，结果天差地别。这就是为什么你需要一个 HTTP 服务器。

## 动手试试

1. 完成上面第五部分的对比实验，亲眼看到 `file://` 和 `http://` 的差异。
2. 打开 DevTools 的 Console 面板（F12），分别用两种方式打开 `page-a.html`，观察 Console 中的错误信息。
3. 在 `http://` 模式下，点击 Network 面板，刷新页面，看看你发起了一个怎样的 HTTP 请求。
4. 回答自己一个问题：为什么一个看起来很简单的 HTML 文件，不能直接双击打开就能用所有的功能？

## 本节小结

`file://` 像是在自己房间自言自语，很多需要"和外界对话"的功能无法使用；`http://` 像是在餐厅正常点餐，有服务器角色、有域、有完整的 HTTP 语义。现代前端开发中，你必须使用 HTTP 服务器来获得完整的功能支持。
