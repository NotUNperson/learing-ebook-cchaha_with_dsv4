# 0.3 浏览器怎样显示一个页面

## 本节你会学到什么

* 从服务器拿到 HTML 之后，浏览器内部做了哪些事
* 什么是 DOM 树，它长什么样
* CSS 是怎样被计算并应用到页面上的
* 布局（Layout）和绘制（Paint）分别指什么
* 如何用 Chrome DevTools 观察浏览器的内部工作

## 一、一个做饭的比喻

浏览器把 HTML 变成屏幕上可见的页面，这个过程很像**做一道菜**：

| 浏览器做的事 | 做饭的类比 | 发生了什么 |
|-------------|-----------|-----------|
| **解析 HTML，构建 DOM 树** | 读菜谱 | 把 HTML 标签一个一个读过去，搞清楚页面里有哪些"食材"（标题、段落、图片、按钮），以及它们之间的层级关系 |
| **解析 CSS，计算样式** | 判断"切丝还是切块"、"红烧还是清蒸" | 把 CSS 规则应用到对应的 DOM 节点上，算出每个元素最终的字体、颜色、大小 |
| **布局** | 装盘摆盘，估算每个菜能占多大地方 | 计算每个元素在页面上的确切位置和尺寸（x 坐标、y 坐标、宽、高） |
| **绘制** | 下锅出菜 | 把计算好的结果"画"到屏幕上，变成你实际看到的像素 |

下面一步一步拆开来看。

## 二、第一步：解析 HTML，构建 DOM 树

浏览器拿到下面这段 HTML：

```html
<!DOCTYPE html>
<html>
<head>
    <title>示例</title>
</head>
<body>
    <h1>我的博客</h1>
    <p>这是第一段。<strong>重点</strong>内容。</p>
</body>
</html>
```

浏览器会一字节一字节地读，识别出每个标签，然后构建一棵**树**：

```
document
 └── html
      ├── head
      │    └── title
      │         └── "示例"
      └── body
           ├── h1
           │    └── "我的博客"
           └── p
                ├── "这是第一段。"
                └── strong
                     └── "重点"
                └── "内容。"
```

这棵树叫 **DOM 树**（Document Object Model，文档对象模型）。每一个 HTML 标签（`<p>`、`<h1>`、`<strong>`）变成了树上的一个节点（Node），标签里的文字（文本内容）也变成了节点。

> **类比 C 语言的数据结构**：如果你学过二叉树或一般树，DOM 树就是一种多叉树。每个节点有 `parentNode`（父节点）、`childNodes`（子节点列表）、`nextSibling`（下一个兄弟节点）等指针。浏览器内部就是用这些指针来遍历整棵树的。

**DOM 树的特点**：
* 根节点永远是 `document`（代表整个文档）。
* HTML 标签形成节点之间的父子关系和兄弟关系。
* 你后面写的所有 JavaScript 代码，操作的就是这棵树上的节点。

## 三、第二步：解析 CSS，计算样式

CSS 可能是写在 `<style>` 标签里，也可能是通过 `<link>` 引入的外部文件。浏览器读了 CSS 后，需要回答一个问题：**DOM 树上每个节点，最终长什么样？**

CSS 的核心机制是**选择器匹配**。举个例子：

```css
p {
    color: red;
    font-size: 16px;
}

.card p {
    color: blue;
}
```

浏览器会遍历 DOM 树上的每一个节点，检查：有没有 CSS 规则匹配到这个节点？如果有，就把这些样式"挂"到节点上。如果多个规则冲突了（比如上面 `.card p` 和 `p` 都匹配到了同一个 `<p>` 标签），浏览器有一套精密的优先级计算规则（叫"层叠"，Cascading，CSS 的 C 就是这个意思），最终算出一个"最终样式"。

计算完成后，DOM 树上的每个节点都带上了它自己的"最终样式表"：
* 这个 `<p>` 的颜色是 `red`
* 那个 `<h1>` 的字体大小是 `2em`
* 那个 `<button>` 的背景色是 `#007acc`

这一步的结果叫 **CSSOM 树**（CSS Object Model），它和 DOM 树结合后形成一棵 **渲染树（Render Tree）**。渲染树只包含"需要显示"的节点——`<head>`、`<script>`、`display: none` 的元素不会出现在渲染树里。

## 四、第三步：布局（Layout）

现在浏览器知道了每个元素长什么样（颜色、字体、大小），但还不知道每个元素放在屏幕上的哪个位置。

布局阶段做的事情：
* 从渲染树的根节点开始，自顶向下地计算**每个可见元素的确切几何信息**。
* 几何信息包括：元素的**宽度、高度**，以及它在页面中的 **x 坐标和 y 坐标**。
* 这个计算过程需要考虑：屏幕宽度（视口大小）、盒模型（margin、border、padding、content）、文档流（块级元素独占一行，行内元素排在一行）、浮动、定位等等。

可以把布局想象成**拼图**：你有一堆不同大小的拼图块（各种 HTML 元素），你要把它们塞进一个固定宽度的画框（浏览器窗口），有些块必须占一整行，有些块可以排在同一行。布局引擎就是自动完成这个拼图的程序。

每次你改变窗口大小、修改 DOM、修改 CSS 时，浏览器可能都需要**重新布局**（这叫 reflow）。频繁的 reflow 是网页卡顿的常见原因——所以写 CSS 时要注意减少不必要的布局触发。

## 五、第四步：绘制（Paint）

布局完了，每个元素的"位置"和"大小"都确定了。最后一步是把这些东西画到屏幕上。

绘制阶段做的事情：
* 把渲染树上的每个节点，从后往前（或者说从背景到前景），一层一层画出来。
* 浏览器会把页面分成多个"图层"（Layer），有点像 Photoshop 里的图层。
* 先在每个图层里画好内容，然后把图层合成到一起（Composite），最终显示在屏幕上。

某些 CSS 属性（比如 `transform`、`opacity`）只会触发合成阶段，不会触发布局，所以这些动画性能比较好，不容易卡顿。

## 六、整个过程的关键渲染路径

总结成一个流水线：

```
HTML ──解析──> DOM 树 ──+
                         ├──> 渲染树 ──> 布局 ──> 绘制 ──> 显示在屏幕上
CSS  ──解析──> CSSOM ────+
```

每次修改 DOM 或 CSS，这个路径（或路径的一部分）会重新跑一次。浏览器做了大量优化，只重跑必要的部分。

## 七、用 Chrome DevTools 观察这些过程

你可以亲眼看到浏览器内部的"DOM 树"和"计算后的样式"：

1. 打开任意网页（比如 `https://www.example.com`）
2. 按 **F12** 打开 DevTools
3. 切换到 **Elements** 面板
4. 你会看到网页的 DOM 树——一个可折叠的树形结构。鼠标移上去，对应的页面区域会高亮。
5. 点击任意一个 DOM 节点，右侧的 **Styles** 面板会显示这个节点的"最终样式"——每个 CSS 属性的值，以及它来自哪个 CSS 规则。
6. 点击 **Computed** 子标签（在 Styles 旁边），会看到盒模型图和所有最终计算值。

你甚至可以在 Elements 面板中**直接修改 DOM**：双击一个文字，改掉它，页面会立刻更新。你也可以在 Styles 面板中**实时修改 CSS**：勾掉一个属性，或者改一个值，页面立刻反映变化。这就是在直接操作渲染树的"数据源"。

另外，打开 DevTools 的 **Performance** 面板，录制几秒钟的页面操作，你会看到浏览器的渲染流水线被分解成 Parse HTML、Recalculate Style、Layout、Paint、Composite 等阶段，一目了然。

## 动手试试

1. 打开 `https://example.com`（或任何网页），按 F12 打开 DevTools。
2. 切到 **Elements** 面板，把你看到的 DOM 树展开几层，观察 HTML 标签的嵌套关系。
3. 点击 `<h1>` 节点，在右侧 Styles 面板看看它的样式（字号、颜色、margin 等）。
4. 在 Styles 面板里，找到 `font-size` 属性，把它的值改成 `50px` 然后按回车。观察页面变化。
5. 在 Elements 面板里，双击 `<h1>` 标签里的文字（`Example Domain`），把它改成别的文字，观察页面变化。
6. 思考：你刚才做的操作，触发了渲染流水线的哪些阶段？

## 本节小结

浏览器把 HTML 变成屏幕上可见的页面，经历了"构建 DOM 树 -> 计算样式 -> 布局（算位置）-> 绘制（画出来）"四个关键步骤。Chrome DevTools 的 Elements 面板让你能直接看到并操控这棵 DOM 树。
