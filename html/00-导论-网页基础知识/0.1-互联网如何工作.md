# 0.1 互联网如何工作

## 本节你会学到什么

* 客户端和服务器分别是什么，它们怎么"对话"
* HTTP 协议是什么，为什么它是 web 世界的通用语言
* 一个 HTTP 请求和响应长什么样
* 状态码（200/404/500）代表什么含义
* 在浏览器地址栏输入一个网址后，背后发生了哪些事情

## 一、客户端-服务器模型：像叫外卖一样理解网络通信

想象你在家饿着肚子，拿起手机叫外卖。你会做三件事：

1. **下单**——告诉店家你要什么（一份宫保鸡丁 + 一碗米饭）
2. **等待**——店家在厨房里准备你的菜
3. **收货**——外卖小哥把菜送到你手里

互联网上的通信和这个过程几乎一模一样，只不过角色换了个名字：

* **你（客户端，Client）**：就是你电脑上的浏览器。它负责"下单"——向某个网站发起请求，说"我想看这个页面"。
* **店家（服务器，Server）**：就是存放网站的那台远程电脑。它收到请求后，在"厨房"里准备好数据（查询数据库、拼装 HTML），然后把结果打包发回来。
* **外卖小哥（网络传输）**：光纤、路由器、海底光缆等等。它们负责把数据从服务器送到你的浏览器，但你通常不需要关心它们怎么做到的。

> **类比 C 语言**：客户端调用一个"远程函数"（发起请求），服务器执行这个函数（处理请求），然后 return 一个结果（返回响应）。只不过不是函数调用，是网络通信。

**关键点**：你的电脑（客户端）没有存任何网站内容，它每次都是"现要现拿"。这和你在本地打开一个 Word 文档完全不同——网页是"远程的"，需要通过网络去获取。

## 二、HTTP：互联网世界的"通用语言"

HTTP 的全称是 **HyperText Transfer Protocol**（超文本传输协议）。名字听起来很唬，但其实就是一套"规则"——约定客户端和服务器之间怎么说话。

举个例子：中国人和美国人交流需要约定用哪种语言（英语）、哪种格式（书信？Email？）。HTTP 就是客户端和服务器之间的"英语"——双方都按这个格式说话，才能互相理解。

HTTP 的核心规则很简单：

* **客户端先说话**（发请求），服务器再回答（返回响应）。服务器不会主动给你发消息。
* 请求和响应都包含两部分：**头部（header）** 和 **体（body）**。头部像信封上的地址和寄件人信息，体像信封里的信纸内容。

## 三、一个 HTTP 请求和响应的真实样子

### 客户端发出的请求（Request）

当你打开 `https://www.example.com/hello` 时，浏览器发出的请求大致长这样：

```
GET /hello HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0 Chrome/120
Accept: text/html

（这里是 body，GET 请求的 body 通常为空）
```

逐行解释：

* `GET` —— 请求方法，意思是"我想要获取某个东西"。（后面会学到 POST，那是"我要提交数据给你"）
* `/hello` —— 我想要服务器上路径为 `/hello` 的那个资源
* `HTTP/1.1` —— 我用的是 HTTP 1.1 版本的规则
* `Host: www.example.com` —— 告诉服务器我访问的是哪个域名（因为一台服务器可能托管多个网站）
* `User-Agent` —— 告诉服务器我是什么浏览器（服务器可以根据这个返回不同版本）
* `Accept` —— 我想要 HTML 格式的内容

### 服务器返回的响应（Response）

服务器收到请求后，返回的东西长这样：

```
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
Content-Length: 1234

<!DOCTYPE html>
<html>
<head><title>你好</title></head>
<body><h1>Hello World</h1></body>
</html>
```

逐行解释：

* `HTTP/1.1 200 OK` —— 第一行永远包含**状态码**。200 表示"成功，你要的东西在这"。
* `Content-Type` —— 返回的内容是 HTML，编码是 UTF-8（支持中文）
* `Content-Length` —— 内容有 1234 个字节
* 空行之后就是 **body**，这里是真正的 HTML 代码

## 四、状态码：快递签收 vs 查无此人 vs 仓库爆炸

状态码用一个三位数字表示服务器端的处理结果。你不需要背下来，但常见的有这几类：

| 状态码 | 含义 | 通俗类比 |
|--------|------|----------|
| **200** OK | 成功 | 快递顺利签收，包裹完好 |
| **301** Moved Permanently | 永久搬家 | 这个地址已经换了，新的地址是XXX（浏览器会自动跳转） |
| **302** Found | 临时跳转 | 今天临时换个地方拿，明天可能还回原地址 |
| **403** Forbidden | 禁止访问 | 你站在门口，保安不让你进 |
| **404** Not Found | 找不到 | 快递员按了门铃，发现这栋楼根本不存在 |
| **500** Internal Server Error | 服务器内部错误 | 仓库爆炸了——不是你地址写错了，是店家自己出了问题 |
| **502** Bad Gateway | 网关错误 | 中间人（代理服务器）收到了上游服务器的无效响应 |
| **503** Service Unavailable | 服务不可用 | 店家临时关门，挂了个"暂停营业"的牌子 |

> **404 的故事**：HTTP 状态码 404 是 web 上最"出名"的代码。据说 404 这个数字来自 CERN（欧洲核子研究组织，也是 Web 的诞生地）的 404 号房间，那里原本应该有一个数据库，但一直没建好。

重点记住：**2xx 表示成功，3xx 表示重定向，4xx 表示你（客户端）搞错了，5xx 表示服务器自己挂了**。

## 五、在浏览器地址栏输入 URL 后发生了什么

这是一个经典的面试题，我们先做一个简化版的理解。你输入 `https://www.example.com/page` 然后按回车，背后发生的事情按时间顺序：

1. **浏览器检查本地缓存**：这个东西我最近访问过吗？如果有缓存且没过期，直接用。没有的话继续往下走。
2. **DNS 解析**：把 `www.example.com` 这个人类好记的域名，翻译成机器认识的 IP 地址（比如 `93.184.216.34`）。就像你手机通讯录里存了"老王"，拨号时手机会翻译成老王的手机号。
3. **建立 TCP 连接**：浏览器和服务器之间"握个手"（三次握手），确认双方都能正常收发数据。相当于打电话时先"喂？听到吗？"
4. **发起 HTTPS 握手**（因为用了 https）：双方协商加密方式，生成临时的"密钥"，保证后面的通信内容被加密，防止被人偷听。
5. **发送 HTTP 请求**：浏览器把请求（"GET /page"那一串东西）发过去。
6. **服务器处理请求**：服务器收到请求，可能需要查数据库、调用后端代码、拼装页面。
7. **服务器返回 HTTP 响应**：服务器把结果（HTML + 状态码 200）发回来。
8. **浏览器渲染页面**：收到 HTML 后，浏览器开始解析、计算样式、绘制页面。这一步的细节我们在 0.3 节展开。
9. **断开连接或保持连接**：HTTP/1.1 默认会保持连接一段时间，以便复用（省得每次都要重新"握手"）。

整个过程的真实耗时可以打开浏览器 DevTools 的 Network 面板看到（按 F12，切到 Network 标签，刷新页面）。你会惊讶地发现：**一个网页其实发了几十个甚至上百个 HTTP 请求**——HTML 本身只是一个，但里面的图片、CSS、JS 文件会各自产生新的请求。

## 动手试试

打开命令行（Windows 用户打开 PowerShell 或 CMD），输入以下命令并回车：

```
curl -v https://www.example.com
```

`curl` 是一个命令行工具，用来发送 HTTP 请求。`-v` 表示"verbose"，让它把请求和响应的详细信息都打印出来。你会看到：

* `> GET / HTTP/1.1` 开头的行是**你发出的请求**
* `< HTTP/1.1 200 OK` 开头的行是**服务器返回的响应**
* 中间会看到 DNS 解析、TCP 连接、TLS 握手的过程标记

如果没有 curl，可以在浏览器里打开 `https://www.example.com`，然后按 F12 -> Network 标签 -> 刷新页面，点击第一条请求，你就能看到完整的 request headers 和 response headers。

## 本节小结

互联网通信的本质就是**客户端发请求，服务器返回响应**，HTTP 是这套通信的标准格式。你在地址栏输入网址的那一刻，浏览器就在帮你完成一次完整的"点外卖"流程。
